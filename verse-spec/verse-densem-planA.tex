The semantic value domain has these components
\begin{itemize}
\item $\mathbb{Z}$, the integers
\item $\dtuple$ finite tuples of values.
\item $\semfcn$ (computable, provably terminating) functions from values to values.  The function can be partial,
  \ie, the function domain, $\dom$, can be a subset of $\Value$,
  and the range, $\ran$, can also be a subset of $\Value$.
  Note that a function always maps a single value to a single value.
  Functions are defined as a set of pairs, as in set theory.
  Since a function is a set of pairs of values, there is no notion of non-terminating functions.
\end{itemize}

The semantic valuation function $\semapp{E}{e}{\rho}$ maps an expression in the given environment to
a set of values.
It has to be a set since an expressions like \ensuremath{\exists\Varid{x}.\,\textbf{gt}(\langle\Varid{x},\mathrm{0}\rangle)} can be any non-negative integer.

The $\apply$ function does function application handling both tuples and functions.
If the argument is not in the domain of the function it returns an empty set,
otherwise a singleton with the result.

The denotation of \ensuremath{\textbf{gt}} and \ensuremath{\textbf{int}} are singleton functions,
whereas the denoation of \ensuremath{\textbf{add}} is the set of all function that behave like $+$ on (pairs of) integers,
but are otherwise unconstrained.

\vspace{2em}
Note that $\semapp{E}{e}{\rho}$ is the same as $\semappp{M}{e}{\Value}{\rho}$ for the non-function cases.
\lennart{
It is tantalizingly close to be (obviously) true for functions as well.
I think it's true for functions as well, but it's not clear to me.}
Experiments with the Haskell implementation all confirm that $\semapp{E}{e}{\rho} = \semappp{M}{e}{\Value}{\rho}$.

To make notation a little clearer in set comprehension we use $x \inb e$ to indicate a binding occurence of x,
and $x \in e$ for a membership test.
The \ensuremath{\mathbf{let}} is used in the same way as \ensuremath{\mathbf{let}} in Haskell list comprehensions.
The \ensuremath{\textbf{if}} construct is the logical connective, not the descriminator for Booleans.
The \textbf{T} stands for the always true proposition.

The equation for closed functions in words.
\begin{itemize}
\item The denotation is a set of functions drawn from all possible functions (partial and total),
  $\{ f | f \in \semfcn, \mydots \}$
\item For every possible argument, $x$, $\forall x \in \Value,$
\item Find all possible ways in which $e_1$ can match $x$.  The result is the original environment
  extended with the different ways to ``solve'' for the variables bound in $e_1$.
  $ \ensuremath{\mathbf{let}}\; \bar{\rho} = \semappp{B}{e_1}{x}{\rho} $
\item If $x$ does not match $e_1$ in any way, then $x$ is not in the domain of the function we are defining,
  so it must not be in the domain of $f$ either.
  $\ensuremath{\textbf{if}}\; \bar{\rho} = \emptyset\; \ensuremath{\textbf{then}}\; x \not\in \dom(f)$
\item If $x$ does match $e_1$ then $x$ must be in the domain of $f$.
  $ \ensuremath{\textbf{else}}\; x \in \dom(f) $
\item \textbf{And} the image of $x$ (\ie, $f(x)$) must be in the denotation of $e_2$
  for all the different ways we can match in $e_1$.
  $ \land \forall \rho' \in \bar{\rho}. f(x) \in \semapp{D}{e_2}{\rho'} $
\end{itemize}

The explanation for open functions is almost identical.  The only difference is that if $e_1$ does not match $x$,
\ie, the $x$ is not in the domain of the function we are defining, then we accept $f$ without further ado.

The denotation of \ensuremath{\textbf{if}} is not obvious.  What should it be?
Consider the following program\\
\ensuremath{\textbf{if}(\Varid{x}\mathbin{:}\textbf{int};\,\Varid{x}\mathbin{*}\Varid{x}\mathrel{=}\mathrm{1})\{\Varid{x}\}\textbf{else}\{\mathrm{999}\}} \\
There are two solutions for \ensuremath{\Varid{x}}, namely $-1$ and $1$.
They are not ordered so it is not clear what the value of \ensuremath{\Varid{x}} should have then the \ensuremath{\textbf{then}} branch.
One alternative would be to say that it is \ensuremath{\Conid{Wrong}}.
The semantic equation takes another approach and says that we should union all possible \ensuremath{\textbf{then}} branches,
and so the result is $\{-1,1\}$.

\textbf{Note:} The given semantics does not treat tuples as functions, except for function application.
This can probably be easily fixed.
