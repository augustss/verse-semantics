\section{A short introduction to Verse}

This section gives an informal introduction to Verse, leaving a formal description
for subsequent sections.

Verse is a statically typed functional-logic language: the main
syntactic construct is the \emph{expression}.  Some expressions are
part of \emph{Core Verse}, defined in \Cref{fig:core}.  Others are
mere syntactic sugar, which can be described by translation into
simpler constructs (see \Cref{fig:sugar}). This Figure covers two
forms of syntactic sugar: sugar that can be translated with a local rewrite;
and sugar that requires some care with scoping when translating.  The
latter is dealt with by \emph{scope extrusion}, which we discuss in
\Cref{sec:extrusion}.

\subsection{Expressions yield multiple values}

Verse's most salient and unusual feature is that \emph{an expression yields a (possibly-empty) sequence of values}.
\lennart{Sequences is a semantic concept; not something at the language level.}
For example:
\begin{itemize}
\item The expression \vtt{2} yields (a sequences consisting of) exactly one value, namely $2$.
\item The expression \vtt{(2 | 7 | 2)} yields a sequence of three values, namely $2$, $7$, and $2$.  We write this sequence $[2,7,2]$.
In general, the expression \vtt{($e_1$ | $e_2$)} yields all the values of $e_1$ followed by all the values of $e_2$.
\item The expression \vtt{:false} yields no values at all, the empty sequence $[]$.  We will see in \Cref{sec:booleans} why \vtt{:false} is the canonical name for the expression yielding the empty sequence.
\end{itemize}

An expression that yields no values is said to \emph{fail}; it if returns one or more, it is said to \emph{succeed}.

\subsection{Function application is lifted over the argment sequences}
\label{sec:lifting}

All functions are lifted to work over sequences, by taking the Cartesian product of their argument sequences.
For example, the expression \vtt{($e_1$ + $e_2$)} returns the sequence of all values $v_1 + v_2$ where
$v_1$ is drawn from the sequence yielded by $e_1$, and
$v_2$ is drawn from the sequence yielded by $e_2$.
Thus \vtt{(2 | 7) + (4 | 11)} evaluates to the sequence $[6, 13, 11, 18]$.

From this lifting semantics, you can see that failure propagates
upward; for example, \vtt{(:false + $e$)} fails for any $e$, since there
are no values $v_1$ returned by \vtt{:false}.

The semicolon operator is an ordinary binary operator that simply returns its right-hand argument. Because of the lifting semantics, however, it still still iterates over its left-hand argument.  For example, \vtt{(:false ; $e$)} fails, while \vtt{((1 | 2) ; (3 | 4))} yields the sequence $[3,4,3,4]$.


\subsection{Comparisons and conditionals} \label{sec:bool}

In Verse, there are no Boolean values \vtt{true} and \vtt{false}.  Instead, failure plays the role of falsity \cite{Icon}.  In a
conditional expression \vtt{(if $e_1$ then $e_2$ else $e_3$)},
\begin{itemize}
\item if $e_1$ fails (yields no values), the conditional returns $e_3$;
  \item if $e_1$ returns one or more values, the conditional returns $e_2$.
\end{itemize}

The comparison operators such as $(>)$, $(<)$, $(=)$ and so on, use the same idiom: \vtt{($e_1$ < $e_2$)} returns $e_1$ if the condition
holds, and fails if it does not.  That makes \vtt{(if (3 > 4) then 10 else 20)} return $20$ as you would expect.

Amusingly, this semantics allows us to write
\vtt{(0 < x < 10)}, an expression that succeeds (returning $10$) if and only if \vtt{x} is between 0 and 10.
This only works because \vtt{(<)} is right associative, so the expression means \vtt{0 < (x<10)}. (Exercise for the reader:
what does \vtt{((0<x) < 10)} do?)

What if $e_1$ and/or $e_2$ return a \emph{sequence} of values?  Remembering the lifting semantics of \Cref{sec:lifting},
we take all pairs of values $(v_1,v_2)$ yielded by $e_1$ and $e_2$.  For each such pair, if $v_1 < v_2$ return $v_1$,
otherwise return nothing.   So \vtt{(1 | 8) < (2 | 10)} yields the sequence $[1,1,8]$.

\subsection{Variables}
\label{sec:any} % for def x in x
\label{sec:variables}

In functional programming, a variable is bound to a specified immutable value as soon as it comes into scope.
Not so in Verse.  Instead, a variable can be brought into scope \emph{without a specified value}.  Here is an example:
\begin{verbatim}
  def x in x + (x = 7; 5)
\end{verbatim}
The construct \vtt{def x in $e$} brings \vtt{x} into scope in $e$.\footnote{
In actual Verse there is no \vtt{def} construct; it is an explanatory device in this paper.
In actual Verse you would write \vtt{x:any} to introduce an unbound variable.
}  Its semantics is unusual:
\begin{itemize}
\item The expression \vtt{(def x in $e$)} binds $x$ successively to
  \emph{all possible values}, and returns all the values yielded
  by $e$ under those bindings.
\item Although an expression yields multiple values, a variable is always bound to
  a \emph{single} value.
\end{itemize}
For which value(s) of \vtt{x} will this particular expression succeed?  Answer: because of the
\vtt{x=7} in the second argument of the \vtt{+}, all execuctions will fail except the one that binds \vtt{x}
to $7$.  So the expression yields just one value, namely 12.

Of course, no implementation could possibly work by binding \vtt{x} to every possible value and
evaluating $e$ -- there are infinitely many such values.  But that is the semantics.
Notice that there is nothing special about \vtt{(=)}; it is just the ordinary equality operator
described in \Cref{sec:bool}.

It is easy to write an expression that yields an infinite sequence of values,
by omitting all equalities.
For example \vtt{def x in x} returns a sequence of all possible values.

\paragraph{Scoping.} Verse allows you to omit the \vtt{def} entirely, using a ``\vtt{:=}'' to introduce the variable instead. We may write our running example like this, omitting the \vtt{def}:
\vtt{(x + (x:=7; 5)}.
The binding \vtt{x:=7} is not required to precede uses of \vtt{x}.
Verse has simple, static rules that explain exactly where to place any missing \vtt{def}s,
as we describe in \Cref{sec:scope}.
In pracical Verse code, however, a programmer will typically write an equality before any uses;
so \vtt{(x:=7; x+5)} would be a more typical way to write our running example.

occurrences are binding and which are just usage.
In the core language the lambda expressions are written \vtt{\clam{$x$}{$e$}} where there is just a single
variable being bound.

\subsection{Let-bindings}

Unusually, in Verse a \vtt{let} binding takes an \emph{expression} not a \emph{binding}.  Thus we have \vtt{(let $e_1$ in $e_2$)}.  For example, \vtt{let x := p*q in x+x}. But this \vtt{let}-expression is simply syntactic sugar for
\begin{center}
  \vtt{def x in x = p*q ; x+x}
\end{center}
That is, we bring \vtt{x} into scope with the \vtt{def}; then constrain it to be equal to \vtt{p+q}; and return \vtt{x+x}.

A \vtt{let} may bind many variables.  For example
$$
\begin{array}{l}
  \vtt{let (x:=7+y; y:=10) in x+y} \\
  \equiv \vtt{def \{x,y\} in (x=7+y; y=10) ; x+y}
\end{array}
$$

In principle, the \vtt{def} binds \vtt{x} and \vtt{y} to (all combinations of) all possible values;
but the \vtt{y=10} makes execution fail unless \vtt{y} is bound to 10; and the \vtt{x=7+y} makes
  execution fail unless \vtt{x} is bound to 17; so the whole expression returns (the singeton) 27
  as you would expect.

  \simon{I can use comma or semicolon in a 'let', can't I?  It makes no difference.}
  \lennart{Yes, since the actual expression is discarded , and ; do the same thing.}

A \vtt{let} is not required to bind \emph{any} variables, in which case it behaves just like ``\vtt{;}''.  For example \vtt{(let 7 in 9)} behaves like \vtt{(7 ; 9)}, which in turn is equivalent to \vtt{9}.

In the most general form \vtt{let $e_1$ in $e_2$}, we must know which variables are \emph{bound} in $e_1$, because they are brought into scope in $e_2$.   These bindings may in principle be nested; for example, here is another way of writing the previous example:
\begin{center}
  \vtt{let x:=7 + (y:=10) in x+y}
\end{center}  
Here the binding for \vtt{y} is buried inside the argument to ``\vtt{+}''.  We will formalise the scoping rules in \Cref{sec:extrusion}.

\subsection{Arrays}

Verse's only data structure is a one-dimensional \emph{array}.  When the array is of small fixed
size, it is often called a \emph{tuple}.
\begin{itemize}
\item A literal array is built with comma notation. For example \vtt{(2,7,7)} is an tuple of three elements.  As with other operators, tuple construction yields all combinations of values yielded by the sub-expressions.  For example, \vtt{( (3 | 1), (7 | 22) )} yields the sequence $[(3,7), (3,22), (1,7), (1,22)]$.
\item An array is treated by Verse as a function, so that indexing is just function application.  For example, in the expression \vtt{(x := (2,7,3); x[1])} we apply the function (= array, = tuple) \vtt{x} to 1, yielding a singleton value 7.  Indexing is zero-based.
  \item Indexing can fail, if the index is out of bounds, yielding an empty sequence of values. This is why Verse uses square brackets (which programmers may think of as array indexing) for a ``function application that can fail'' (\Cref{sec:application}).
\end{itemize}

\subsubsection{The \vtt{for} construct}

The \vtt{for} construct builds an array.  Specifically \vtt{for $e_1$ in $e_2$} takes each value returned by $e_1$, ignores it, and evaluates $e_2$ in the presence of the bindings from $e_1$.
For example:
\begin{verbatim}
for (i := (1 | 2 | 3)) (i*i)
\end{verbatim}
So \vtt{for} reifies the sequence of values yielded by an expression into a value (the array)
containing all those values.
A \vtt{for} always yields a single value.

\section{Types}

Verse is designed to be statically type-checked.
As in any typed language, a type describes a set of values.  In
Verse, unusually, \emph{a ``type'' is just a particular sort of Verse expression}.
Specifically a type is a partial identity function;
it fails on values that do not belong to the type, and is the identity function (returning a singleton sequence) for the values that do belong.  A mathematician would call it the
\emph{characteristic function} of the set of values; a denotational semantics person might call it a \emph{retraction}.

So, for example:
\begin{itemize}
\item The function \vtt{((x:int) => x>0)} is the type of positive integers.  Since \vtt{x>0} succeeds, returning {x} for integers greater than zero, if we lambda-abstract over \vtt{x} we get a partial identity function.
\item The function \vtt{(x:int,y:int) => (x>y); (x,y)} is the type of pairs in which the second component is less than the first.
\end{itemize}
So in Verse, it is easy to carve out arbitrarily complex sets of
values; refinement types are built in.  Of course, that puts a complete type checker well
out of reach, but an incomplete typechecker can still give strong static guarantees about some
Verse programs.

\subsection{Built in types}

Verse has some built-in types:
\begin{itemize}
\item \vtt{any} is defined as \vtt{any := x \clamsym x}, the function that always succeeds.  So type type \vtt{any} is inhabited by all values.
\item \vtt{int} is a function that, given an argument value $v$, yields the singleton sequence $[v]$ if $v$ is an integer; and fails otherwise.
\item \vtt{false} is the function that always fails; \vtt{:false} denotes the empty set of values.
  \simon{I still don't see the logic of calling this \vtt{false}. E.g. \vtt{if false then..else..} does not do what I expect.}

\end{itemize}

\subsection{Function types}

A function type \vtt{($t_1$ -> $t_2$)} is, like any Verse type, an expression.
  It is partial identity that, given a function, either leaves it alone, or makes it fail.
  Here is the definition of \vtt{op`->`}:
\begin{verbatim}
   op`->` := arg_ty \clamsym res_ty \clamsym f \clamsym x \clamsym res_ty[ f[ arg_ty[x] ] ]
\end{verbatim}
That is, given an argument and result type (each of which is a partial identity function),
and the function \vtt{f} itself, it returns a function that
checks that the argument \vtt{x} satisfies
\vtt{arg\_ty} and the result satisfies \vtt{res\_ty}.  This construction uses the now-standard
method used by Findler and Felleisen for \emph{contracts} in Scheme; indeed Verse's types
are very like Findler/Felleisen contracts.

Notice that \vtt{($t_1$ -> $t_2$)} does not \emph{immediately} fail on functions
that do not satisfy the type; it fails lazily, when the function is applied, which
is the best we can hope for.
\lennart{We could give a non-executable semantics for the type that fails at the right time.}

\subsection{Typedef}

If we have an expression $e$ that yields a sequence of
values $[v_1,\ldots,v_n]$, can we define a type (that is, a Verse function) that denotes that
set of values?  Yes, of course: it is just \vtt{(x \clamsym x=$e$)}; that is, a function that,
given a value $v$, will succeed (returning $v$) iff $v$ is among the values yielded by $e$.
This idiom is sufficiently useful that Verse defines \vtt{typedef\{$e$\}} as syntactic sugar
for that lambda (see \Cref{fig:sugar}).

So, for example \vtt{typedef\{3 | 7 | 21\}} is the type consisting of 3, 7, and 21.
The type of natural numbers can be defined by \vtt{nat := typedef\{x:int where x>=0\}}.

Figure \ref{extrusion-rules} has rules for collecting variable definitions to the beginning of blocks.
Notation: \defunion{}{} represents multiset union, $\#D$ is a predicate for multiset uniqueness,
\defempty is the empty multiset.
The relation \defarrow{$e$}{$e'$}{$D$} denotes that the topmost scope of $e$ contains the
\vtt{def}s for the variables in $D$, and $e'$ is the expression $e$ with all \vtt{def}s removed.
The relation \defdefin{$e$}{$e'$} denotes that $e'$ is equivalent to $e$,
but with all definitions replaced by \vtt{def-in}.

  2+5   # evaluates to 7

  if(3<2) then 10 else 20  # evaluates to 20
  if(1|2) then 10 else 20  # evaluates to 10

  xs[3]  # fails

  :xs  # evaluates to 2; 3; 10

  if(y:=xs[0]) then y else 20  # evaluates to 2
  if(y:=xs[3]) then y else 20  # evaluates to 20
  if(y:= :xs; y>8) then y else 20  # evaluates to 10
  if(y:xs; y>8) then y else 20  # evaluates to 10
