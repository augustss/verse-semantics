\section{A short introduction to Verse}

This section gives an informal introduction to Verse, leaving a formal description
for subsequent sections.

Verse is a statically typed functional-logic language: the main
syntactic construct is the \emph{expression}.  Some expressions are
part of \emph{Core Verse}, defined in \Cref{fig:core}.  Others are
mere syntactic sugar, which can be described by translation into
simpler constructs (see \Cref{fig:sugar}). This Figure covers two
forms of syntactic sugar: sugar that can be translated with a local rewrite;
and sugar that requires some care with scoping when translating.  The
latter is dealt with by \emph{scope extrusion}, which we discuss in
\Cref{sec:extrusion}.

\subsection{Expressions yield multiple values} \label{sec:multi-value}

Verse's most salient and unusual feature is that \emph{an expression yields a (possibly-empty) sequence of values}.
This is not a new idea --- Verse inherits it from Ontic \cite{ontic} and $\lambda_\aleph$ \cite{lambda-aleph}
--- but it makes Verse look and feel very different to a vanilla functional language.
\lennart{Sequences is a semantic concept; not something at the language level.}

Here are some examples:
\begin{itemize}
\item The expression \vtt{2} yields (a sequences consisting of) exactly one value, namely $2$.
\item The expression \vtt{(2 | 7 | 2)} yields a sequence of three values, namely $2$, $7$, and $2$.  We write this sequence $[2,7,2]$.
In general, the expression \vtt{($e_1$ | $e_2$)} yields all the values of $e_1$ followed by all the values of $e_2$.
\item The expression \vtt{:false} yields no values at all, the empty sequence $[]$.  We will see in \Cref{sec:booleans} why \vtt{:false} is the canonical name for the expression yielding the empty sequence.
\end{itemize}
\noindent
An expression is said to \emph{fail} if it returns no values at all; and to \emph{succeed} if it returns one or more values.

A value can be:
\begin{itemize}
\item An integer
\item A floating point number
\item A character
\item A function value
\item \vtt{WRONG}
\end{itemize}
where \vtt{WRONG} is the value used when something ``impossible'' happens, such as adding an integer to a character.
Note that we distinguish strongly between \emph{failing} which is perfectly normal, and \emph{going wrong} (i.e. returning \vtt{WRONG})
which should never happen.
Verse is statically typed, and its verifier guarantees that no \vtt{WRONG} values will ever occur (\Cref{sec:types}).


\subsection{Cross-product semantics}
\label{sec:lifting}

All functions are lifted to work over sequences, by taking the Cartesian product of their argument sequences.
For example, the expression \vtt{($e_1$ + $e_2$)} returns the sequence of all values $v_1 + v_2$ where
$v_1$ is drawn from the sequence yielded by $e_1$, and
$v_2$ is drawn from the sequence yielded by $e_2$.
Thus \vtt{(2 | 7) + (4 | 11)} evaluates to the sequence $[6, 13, 11, 18]$.
We call this the \emph{cross-product semantics}, because the result sequence is obtained a cross product of the argument sequences.

From the cross-product semantics, you can see that failure propagates
upward; for example, \vtt{(:false + $e$)} fails for any $e$, since there
are no values $v_1$ returned by \vtt{:false}.

The semicolon operator is an ordinary binary operator that simply returns its right-hand argument. Because of the cross-product semantics, however, it still still iterates over its left-hand argument.  For example, \vtt{(:false ; $e$)} fails, while \vtt{((1 | 2) ; (3 | 4))} yields the sequence $[3,4,3,4]$.

\subsection{Built in values}

Verse has built-in integers (\vtt{1}, \vtt{21}, \vtt{-2} etc), and characters (\vtt{'x'}, \vtt{';'} etc).
The integers come equipped with standard arithmetic operations like addition \vtt{$e_1$+$e_2$}, etc.   Infix operations can be
written prefix using the C++-derived \vtt{operator} notation, thus: \vtt{operator'+'($e_1$,$e_2$)}.

Verse's only data structure is a one-dimensional \emph{array}.  When
the array is of small fixed size, it is often called a \emph{tuple}. A
literal array is built with comma notation. For example \vtt{(2,7,7)}
is an tuple of three elements.  As with other operators, tuple
construction yields all combinations of values yielded by the
sub-expressions.  For example, \vtt{( (3 | 1), (7 | 22) )} yields the
sequence $[(3,7), (3,22), (1,7), (1,22)]$.

Arrays of variable size, array indexing, and for-loops, are discussed in \Cref{sec:arrays}.

\subsection{Comparisons and conditionals} \label{sec:bool} \label{sec:conditionals}

In Verse, there are no Boolean values \vtt{true} and \vtt{false}.  Instead, failure plays the role of falsity \cite{Icon}.  In a
conditional expression \vtt{(if $e_1$ then $e_2$ else $e_3$)},
\begin{itemize}
\item if $e_1$ fails (yields no values), the conditional returns $e_3$;
  \item if $e_1$ returns one or more values, the conditional returns $e_2$.
\end{itemize}

Conjunction (and) and disjunction (or) are readily expressed:
\begin{itemize}
\item \emph{Conjunction.} The term \vtt{($e_1$ ; $e_2$)} succeeds iff both $e_1$ succeeds and $e_2$ succeeds.
\item \emph{Disjunction.} The term \vtt{($e_1$ | $e_2$)} succeeds if either $e_1$ succeeds or $e_2$ succeeds.
\item \emph{Negation.} Verse provides a negation operator \vtt{!$e$}, which is syntactic sugar
  for \vtt{(if x then :false else ())}.  That is, if $e$ fails, \vtt{!$e$} succeeds, returning the value \vtt{()};
  but if $e$ succeeds, \vtt{!$e$} fails.  (Remember, \vtt{:false} is the expression that fails (\Cref{sec:multi-value}).
\end{itemize}

The comparison operators such as $(>)$, $(<)$, $(=)$ and so on, do not return a Boolean value; instead
they fail if the comparison is false, or succeed otherwise.  More specifically, when a comparision succeeds
it returns its left argument.
That makes \vtt{(if (3 > 4) then 10 else 20)} return $20$ as you would expect.

Amusingly, this semantics allows us to write
\vtt{(0 < x < 10)}, an expression that succeeds (returning $10$) if and only if \vtt{x} is between 0 and 10.
This works because \vtt{(<)} is right associative, so the expression means \vtt{0 < (x<10)}. (Exercise for the reader:
what does \vtt{((0<x) < 10)} do?)

What if $e_1$ and/or $e_2$ return a \emph{sequence} of values?  Remembering the cross-product semantics of \Cref{sec:lifting},
we take all pairs of values $(v_1,v_2)$ yielded by $e_1$ and $e_2$.  For each such pair, if $v_1 < v_2$ return $v_1$,
otherwise return nothing.   So \vtt{(1 | 8) < (2 | 10)} yields the sequence $[1,1,8]$.

\subsection{Variables}
\label{sec:any} % for def x in x
\label{sec:variables}

In functional programming, a variable is bound to a specified immutable value as soon as it comes into scope.
Not so in Verse.  Instead, a variable can be brought into scope \emph{without a specified value}.  Here is an example:
\begin{verbatim}
  def x in x + (x = 7; 5)
\end{verbatim}
The construct \vtt{(def $v$ in $e$)} brings $v$ into scope in $e$.\footnote{
In actual Verse there is no \vtt{def} construct; it is an explanatory device in this paper.
In actual Verse you would write \vtt{v:any} to introduce an unbound variable.
}  Its semantics is unusual:
\begin{itemize}
\item The expression \vtt{(def $v$ in $e$)} binds $v$ successively to
  \emph{all possible values}, and returns all the values yielded
  by $e$ under those bindings.
\item Although an expression yields multiple values, a variable is always bound to
  a \emph{single} value.
\end{itemize}
For which value(s) of \vtt{x} will this particular expression succeed?  Answer: because of the
\vtt{x=7} in the second argument of the \vtt{+}, all execuctions will fail except the one that binds \vtt{x}
to $7$.  So the expression yields just one value, namely 12.

Of course, no implementation could possibly work by binding \vtt{x} to every possible value and
evaluating $e$ -- there are infinitely many such values.  But that is the semantics.
Notice that there is nothing special about \vtt{(=)}; it is just the ordinary equality operator
described in \Cref{sec:bool}.

It is easy to write an expression that yields an infinite sequence of values,
by omitting all equalities.
For example \vtt{def x in x} returns a sequence of all possible values.

\subsection{Scoping and \vtt{:=}}

Verse allows you to omit the ``\vtt{def}'' entirely, using a ``\vtt{:=}'' to introduce the variable instead.
Intead of writing
\begin{verbatim}
  def x in x + (x = 7; 5)
\end{verbatim}
we may write this, omitting the ``\vtt{def}'':
\begin{verbatim}
  x + (x:=7; 5)
\end{verbatim}
Note that the binding \vtt{x:=7} is not required to precede uses of \vtt{x}.
In pracical Verse code, however, a programmer will typically write an equality before any uses;
so \vtt{(x:=7; x+5)} would be a more typical way to write our running example.

Verse has simple, static rules that explain exactly where to place any missing \vtt{def}s.
Specifically, every variable bound by \vtt{:=} has a scope that begins at the innermost
enclosing \emph{scope-limiting construct}.  The scope-limiting constructs are:
\begin{itemize}
\item The arguments of ``\vtt{|}''.  E.g. in \vtt{((x:=7; x+1) | (y>v; y:=9; 7))}, the bindings of \vtt{x} and \vtt{y} are limited to the first and second arms of the alternative ``\vtt{|}'' respectively, so the expression could also be written
\begin{verbatim}
  (def x in (x=7; x+1)) | (def y in (y>v; y=9; 7))
\end{verbatim}
\item Body of lambda.
\item All three arguments of \vtt{if}.
\item \vtt{do} block.  Limiting scope is the only purpose of \vtt{do}.  For example, \vtt{(x + do\{ x:=7, x+1 \})} means
\begin{verbatim}
  x + (def x in (x=7; x+1))
\end{verbatim}
where the first \vtt{x} is presumably bound in some outer scope.
\end{itemize}
These rules are formalised in \Cref{sec:scope}.

\subsection{Let-bindings}

Unusually, in Verse a \vtt{let} binding takes an \emph{expression} not a \emph{binding}.  Thus we have \vtt{(let $e_1$ in $e_2$)}.  For example, \vtt{let x := p*q in x+x}. But this \vtt{let}-expression is simply syntactic sugar for
\begin{center}
  \vtt{def x in x = p*q ; x+x}
\end{center}
That is, we bring \vtt{x} into scope with the \vtt{def}; then constrain it to be equal to \vtt{p+q}; and return \vtt{x+x}.

Some specifics:
\begin{itemize}
\item \emph{Binding multiple variables.}  A \vtt{let} may bind many variables.  For example
$$
\begin{array}{l}
  \vtt{let (x:=7+y; y:=10) in x+y} \\
  \equiv \vtt{def \{x,y\} in (x=7+y; y=10) ; x+y}
\end{array}

In principle, the \vtt{def} binds \vtt{x} and \vtt{y} to (all combinations of) all possible values;
but the \vtt{y=10} makes execution fail unless \vtt{y} is bound to 10; and the \vtt{x=7+y} makes
  execution fail unless \vtt{x} is bound to 17; so the whole expression returns (the singeton) 27
  as you would expect.

It does not matter whether you use a comma or a semicolon in a

\item \emph{Binding no variables.} A \vtt{let} is not required to bind \emph{any} variables, in which case it behaves just like ``\vtt{;}''.  For example \vtt{(let 7 in 9)} behaves like \vtt{(7 ; 9)}, which in turn is equivalent to \vtt{9}.

\section{Types and type checking} \label{sec:types}

Verse is designed to be statically type-checked.  The type checker is called the \emph{verifier}.
Paraphrasing Milner: \emph{verified programs don't go wrong}.

\subsection{Going wrong}

\item The body of a function fails (returns zero values), but the function's multiplicity effects

The job of the verifier is to ensure that \emph{verified programs don't go wrong}.
Like any type system, the verifier may reject programs that do not, in fact, go wrong.
But it must never accept a program that does go wrong.

Failure is fully expected as part of normal execution -- for example, failure is used
instead of falsehood in a conditional (\Cref{sec:conditionals}).  So verified programs
(or, rather, sub-expressions thereof) may certain fail, but they cannot go wrong.

\subsection{What is a ``type''?} \label{sec:what-is-a-type}

As in any typed language, a type describes a set of values.  In
Verse, unusually, \emph{a ``type'' is just a particular sort of Verse expression}.
Specifically a \emph{type} is a function that
\begin{itemize}
\item Returns zero or one results
\item Is idempotent
\end{itemize}
The \emph{inhabitants of a type} $t$ are the values $v$ such that $t(v) = [v]$; that is, the values
on which $t$ succeeds, and returns its input.
A denotational semantics person might describe a type as a \emph{retraction}, and
indeed Cardelli used retractions as types in \cite{cardelli:type-in-type}.

So, for example:
\begin{itemize}
\item The function \vtt{((x:int) => x>0)} is the type of positive integers.  Since \vtt{x>0} succeeds, returning {x} for integers greater than zero, if we lambda-abstract over \vtt{x} we get a partial identity function.
\item The function \vtt{(x:int,y:int) => (x>y); (x,y)} is the type of pairs in which the second component is less than the first.
\end{itemize}
So in Verse, it is easy to carve out arbitrarily complex sets of
values; refinement types are built in.  Of course, that puts a complete type checker well
out of reach, but an incomplete typechecker can still give strong static guarantees about some
Verse programs.

Moreover, a Verse type can perform conversions.  For example the built-in function \vtt{abs} take the absolute value of an integer, and fails on a non-integer argument.  So \vtt{abs} is a type: it is idempotent, but also converts a negative integer to a positive one.  We will see how these value-converting types are useful in \Cref{sec:definitions}.

\item \vtt{int} is a function that, given an argument value $v$, yields the singleton sequence $[v]$ if $v$ is an integer; and fails otherwise.
\item \vtt{false} is the function that always fails; \vtt{:false} denotes the empty set of values.

\subsection{Definitions} \label{sec:definitions}

So far we have seen bindings of form \vtt{x := $e$}, but the general form is \vtt{$d$ := $e$} where $d$ is a \emph{definition} (\Cref{fig:syntax}).  As well as \vtt{x:=$e$} there are three other forms of definition $d$:
\begin{itemize}
\item A \emph{function binding} \vtt{$d(p)$ := $e$}, which means the same as binding $d$ with a lambda on the right hand side, thus $\vtt{$d$ := $p$=>$e$}$.  This applies recursively, so \vtt{f(x:int)(y:int) := $e$} means the curried binding
  \vtt{f := (x:int) => (y:int) => $e$}.
\item A \emph{tuple binding} \vtt{($d_0$,$\cdots$,$d_n$):= $e$}, which means the same as
  \begin{center}
  \vtt{x := $e$;} \vtt{$d_0$ := x[0];} $\cdots$\vtt{;} \vtt{$d_n$ := x[n]}
  \end{center}
where \vtt{x} is fresh. That is, we bind \vtt{x} to the right hand side, and take the components apart
\item A \emph{typed binding} \vtt{($d$:$t$) := $e$}, which means the same as \vtt{$d$ := $t$[$e$]}.
\end{itemize}
These desugarings are given in \Cref{fig:pat-sugar}.  The first two are unsurprising, but the third is unusual.
For a typed binding we take the type and \emph{apply it to the right-hand side}.  Remember from \Cref{sec:what-is-a-type} that a type is just an idempotent function.  By applying it to the right hand side $e$, the type $t$ will either return an inhabitant of the type (possibly applying a conversion), or it will fail (if the input lies outside the domain of the type).

For example, \vtt{(x:int := $e$)} means \vtt{(x := int[$e$]}.  Suppose $e$ yields the sequence $[3, 'x', 4.5, 20]$.
The type (function) \vtt{int} fails (i.e. returns no values) when applied to a character or float. Like any
function call, \vtt{int[$e$]} concatenates the results of \vtt{int[$3$]}, \vtt{int[$'x'$]}, \vtt{int[$4.5$]},
and \vtt{int[$20$]}.  The middle two calls fail, returning no values; so \vtt{int[$e$]} returns $[3,20]$, and these
are the values to which \vtt{x} is successively bound.

Now consider \vtt{(x:abs := $e$)} when $e$ yields the sequence $[3, 4.3, -5]$.

\subsection{Divergence and sequencing}

In general, the evaluation of an expression may:
\begin{itemize}
\item \emph{Diverge}.
\item \emph{Terminate} with no further values.
\item \emph{Yield a value}, and then behave in one of these three ways.
\end{itemize}
Another way to say this is that the semantics of an expression is a sequence of values,
where the sequence may be infinite, may terminate, or may at some point diverge.

Notice that we say \emph{sequence} of values rather than \emph{set} or \emph{bag} of values.
The order is semantically significant in two places:
\begin{itemize}
\item Consider \vtt{(if (loop | 3) then $e_1$ else $e_2$)}. To evaluate \vtt{(loop | 3)} Verse
  will try to return all the values of \vtt{loop}.  So if \vtt{loop} diverges, the \vtt{if} will diverge too.
  In contrast \vtt{(if (3 | loop) then $e_1$ else $e_2$)} will immediately return $e_1$.  In terms of
  semantics, \vtt{loop | 3} evaluates to $\bot$, whereas \vtt{3 | loop} evaluates to $3 : \bot$.

\item The expression \vtt{for($e_1$) in $e_2$} iterates over the values returned by $e_1$, forming an
  array of the values of $e_2$ for each iteration.  So $for(i := (1|7)) in (i+1)$ returns the array $(2,8)$,
  not the array $(8,2)$; the order of the results yielded by $(1|7)$ is reified into the array
  returned by the \vtt{for}.
\end{itemize}
