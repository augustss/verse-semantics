\begin{abstract}
  Functional logic languages have a rich literature, but it is tricky
  to give them a satisfying semantics.  In this paper we describe the
  Verse calculus, \versecalc{}, a new core calculus for functional
  logical programming. Our main contribution is to equip \versecalc{} with a
  small-step rewrite semantics, so that we can reason
  about a \versecalc{} program in the same way as one does with lambda
  calculus; that is, by applying successive rewrites to it.
  This semantics elegantly fills the gap between very high level
  ``magical'' semantics, and very low-level operational semantics.

  \emph{Note: please don't circulate this draft, because the work is incomplete and we
    are likely to change details.  But we'd be happy to share it with others, on request.}
\end{abstract}

\section{Introduction}

Functional logic programming languages add expressiveness to functional programming
by introducing logical variables, equality constraints among those variables,
and choice to allow multiple alternatives to be explored.  Here is a tiny example:
$$
\ensuremath{\exists\Varid{x}\hspace{0.3mm}\Varid{y}\hspace{0.3mm}\Varid{z}.\,\Varid{x}\mathrel{=}\langle{}\Varid{y},\mathrm{3}\rangle;\,\Varid{x}\mathrel{=}\langle{}\mathrm{2},\Varid{z}\rangle;\,\Varid{y}}
$$
This expression introduces three logical (or existential) variables \ensuremath{\Varid{x},\Varid{y},\Varid{z}},
constrains them with two equalities \ensuremath{(\Varid{x}\mathrel{=}\langle{}\Varid{y},\mathrm{3}\rangle} and \ensuremath{(\Varid{x}\mathrel{=}\langle{}\mathrm{2},\Varid{z}\rangle)}, and finally returns \ensuremath{\Varid{y}}.
The only solution to the two equalities is \ensuremath{\Varid{y}\mathrel{=}\mathrm{2}}, \ensuremath{\Varid{z}\mathrel{=}\mathrm{3}}, and \ensuremath{\Varid{x}\mathrel{=}\langle{}\mathrm{2},\mathrm{3}\rangle}; so the result of the
whole expression is \ensuremath{\mathrm{2}}.

Functional logic programming has a long history and a rich literature.
But it is somewhat tricky for programmers to \emph{reason} about
functional logic programs: they must think about logical variables,
narrowing, backtracking, Horn clauses, resolution, and the like.  This
contrasts with functional programming, where one can say ``just apply
rewrite rules, such as beta reduction, let-inlining, and
case-of-known-constructor.''  Moreover, these rewrite rules apply even
in the presence of free variables; {\eg}, under lambdas.

We therefore seek a precise expression of functional logic programming as
a term-rewriting system, to give us both a formal semantics (via small-step reductions),
and a powerful set of equivalences that programmers can use to reason about their programs,
and compilers can use to optimize them.

We make the following contributions in this paper.
First, we describe a new core calculus, the Verse calculus or \versecalc{} for short,
for functional logic programming (\cref{sec:informal} and \ref{sec:verse}).
Like any functional logic language,
\versecalc{} supports logical variables, equalities, and choice, but it is distinctive in several ways:
\begin{itemize}
\item \versecalc{} natively supports \emph{higher-order programming}, just like the lambda calculus.
  Indeed, \emph{every lambda calculus program is a \versecalc{} program}.
  In contrast, most of the functional-logic literature is rooted in a first-order world, and
  addresses higher-order features via an encoding called defunctionalisation~\cite[3.3]{hanus:survey}.
\item All functional logic languages have some notion of ``flexible'' {\vs} ``rigid'' variables,
  or ``suspending'' {\vs} ``narrowing'' operations.  \versecalc{} offers a new
  way to address these notions, namely the operators \ensuremath{\textbf{one}} (\cref{sec:one})
  and \ensuremath{\textbf{all}} (\cref{sec:all}).  This enables an elegant economy of concepts: for example,
  there is just one equality (other languages have a suspending equality and a narrowing equality),
  and conditional expressions are driven by failure rather than booleans (\cref{sec:if}).
\item \versecalc{} uses \emph{constructive choice}, meaning that the choice operator behaves
  a bit like a data constructor: it appears in normal forms (\cref{sec:choice-rules}).
  This makes \versecalc{} \emph{deterministic}, unlike most
  functional logic languages which are non-deterministic (\cref{sec:deterministic}).
  In \versecalc{} choices are laid out in \emph{space},
  in the syntax of the term, rather than being explored by non-deterministic selection.
\end{itemize}
As always with a calculus, the idea is that \versecalc{} distills the essence of functional logic programming.
Each construct does just one thing, and \versecalc{} cannot be made smaller without losing key features.
We believe that it is possible to use \versecalc{} as the compilation target for a variety of
functional logic languages such as Curry~\cite{hanus:curry-report} (although see \cref{sec:laziness}).
We are ourselves working on Verse, a new general purpose programming language, built directly on \versecalc{}; indeed,
our motivation for developing \versecalc{} is practical rather than theoretical.
No single aspect of \versecalc{} is unique, but we believe that their combination
is particularly harmonious and orthogonal.
We discuss the rich related work in \Cref{sec:related}, and design alternatives in \cref{sec:variations}.

Our second contribution is to equip \versecalc{} with a
\emph{small-step term-rewriting semantics} (\cref{sec:rewrite-rules}).
We said that the lambda calculus is a subset of \versecalc{}, so it is natural to give its
semantics using rewrite rules, just like the lambda calculus.  That seems problematical, however, because
logical variables and unification involve sharing and non-local communication that seems hard
to express in a rewrite system.

Exactly the same difficulty arises with call-by-need.  For a long time, the only semantics of
call-by-need that was faithful to its sharing semantics (in which thunks are evaluated at most once)
was an operational semantics that sequentially threads a global heap through execution~\cite{launchbury:natural}.
But then Ariola {\etal}, in a seminal paper, showed how to reify the heap into the term itself,
and thereby build a rewrite system that is completely faithful to lazy evaluation~\cite{ariola:call-by-need}.
Inspired by their idea, we present a new rewrite system for functional logic program, that reifies
logical variables and unification into the term itself,
and exploits our notion of constructive choice to replace non-deterministic
search with a (deterministic) tuple of successful results.

gives programmers a set of equivalences with which

We begin by presenting the Verse calculus, \versecalc, informally.  We will give its
rewrite rules precisely in \cref{sec:rewrite-rules}.
\versecalc{} is modeled on the lambda calculus, and indeed the lambda calculus is
a sub-language of \versecalc{}.  \emph{Every lambda calculus term is a \versecalc{} term},
and has the same semantics. Just like the lambda calculus, \versecalc{} is untyped;
adding a type system is an excellent goal, but is the subject of another paper.

The syntax of \versecalc{} is given in \cref{fig:syntax}.
It has a very conventional sub-language that is just the lambda calculus with
some built-in operations and tuples as data constructors:
\begin{itemize}
\item \emph{Head-normal forms}.
  A head-normal form, \ensuremath{\mathit{hnf}}, is a conventional value: a built-in constant \ensuremath{\Varid{k}} or operator \ensuremath{\Varid{op}}, a tuple
  \ensuremath{\langle{}\Varid{v}_{\mathrm{1}},\cdots,\Varid{v}_{\Varid{n}}\rangle}, or a lambda term \ensuremath{\lambda\Varid{x}.\,\Varid{e}}.
For the built-in constants and operators we use only integer constants, and two representative
functions \ensuremath{\textbf{gt}} and \ensuremath{\textbf{add}}.
\item \emph{Values.}
In a functional logic language an expression may evaluate to an as-yet-unknown logical variable,
so a \emph{value} \ensuremath{\Varid{v}} is either a head-normal form, or a variable \ensuremath{\Varid{x}}.
\item \emph{Expressions} \ensuremath{\Varid{e}} includes values \ensuremath{\Varid{v}}, and applications \ensuremath{\Varid{v}_{\mathrm{1}}\,\Varid{v}_{\mathrm{2}}}.\footnote{
For clarity we usually write \ensuremath{\Varid{v}_{\mathrm{1}}(\Varid{v}_{\mathrm{2}})} rather than \ensuremath{\Varid{v}_{\mathrm{1}}\,\Varid{v}_{\mathrm{2}}} when \ensuremath{\Varid{v}_{\mathrm{2}}} is not a tuple. }
Remembering that a value can be a lambda, this gives us the full lambda calculus.
\item \emph{A program}, \ensuremath{\Varid{p}}, is a closed expression from which we extract one result using \ensuremath{\textbf{one}} (see \cref{sec:one}).
\end{itemize}
Expressions also include two other key collections of constructs:
logical variables and unification (\cref{sec:unify}), and choice (\cref{sec:choice}).
The details of choice and unification, and especially their
interaction, are rather tricky, so this section will do a lot of
arm-waving.  But fear not: \cref{sec:rewrite-rules} will make all this
precise.

\subsection{Logical variables and unification} \label{sec:unify}

The Verse calculus includes first class logical variables and unification:
you can bring a fresh logical variable into scope with \ensuremath{\exists}; equate two expressions with \ensuremath{\Varid{e}_{\mathrm{1}}\mathrel{=}\Varid{e}_{\mathrm{2}}}; and
sequence two expressions with \ensuremath{\Varid{e}_{\mathrm{1}};\,\Varid{e}_{\mathrm{2}}} (see \cref{fig:syntax}).
As an example, what might be written \ensuremath{\textbf{let}\;\Varid{x}=\Varid{e}_{\mathrm{1}}\;\textbf{in}\,\{\Varid{e}_{\mathrm{2}}\}} in a conventional functional language can be written
\ensuremath{\exists\Varid{x}.\,\Varid{x}\mathrel{=}\Varid{e}_{\mathrm{1}};\,\Varid{e}_{\mathrm{2}}} in \versecalc{}.

\emph{A program executes by solving its equations} to get values for some of the logical variables.
For example,
$$
\ensuremath{\exists\Varid{x}\hspace{0.3mm}\Varid{y}\hspace{0.3mm}\Varid{z}.\,\Varid{x}\mathrel{=}\langle{}\Varid{y},\mathrm{3}\rangle;\,\Varid{x}\mathrel{=}\langle{}\mathrm{2},\Varid{z}\rangle;\,\Varid{y}}
$$
is solved by unifying \ensuremath{\Varid{x}} with \ensuremath{\langle{}\Varid{y},\mathrm{3}\rangle} and with \ensuremath{\langle{}\mathrm{2},\Varid{z}\rangle}; that in turn unifies \ensuremath{\langle{}\Varid{y},\mathrm{3}\rangle} with \ensuremath{\langle{}\mathrm{2},\Varid{z}\rangle}, which
unifies \ensuremath{\Varid{y}} with \ensuremath{\mathrm{2}} and \ensuremath{\Varid{z}} with \ensuremath{\mathrm{3}}.  Finally \ensuremath{\mathrm{2}} is returned as the result.
Note carefully that, like any declarative language,
\emph{logical variables are not mutable}.
We use ``\ensuremath{\exists}'' to bring a fresh logical variable into scope, because we really
mean ``there exists an \ensuremath{\Varid{x}} such that ...''.  Logical variables \emph{are} existential variables.

High-level functional languages usually provide some kind of
pattern matching; in such a language, we might define \ensuremath{\Varid{first}} by
\ensuremath{\Varid{first}\langle\Varid{a},\Varid{b}\rangle\mathrel{=}\Varid{a}}. Such pattern matching is typically desugared to
more primitive \ensuremath{\textbf{case}} expressions.  But in \versecalc{} we do not need \ensuremath{\textbf{case}} expressions:
unification does the job.  For example we can define \ensuremath{\Varid{first}} like this:
$$
\ensuremath{\Varid{first}\mathrel{=}\lambda\Varid{pr}.\,\exists\Varid{a}\hspace{0.3mm}\Varid{b}.\,\Varid{pr}\mathrel{=}\langle{}\Varid{a},\Varid{b}\rangle;\,\Varid{a}}
$$
The lambda unifies its argument (presumably a pair)
with \ensuremath{\langle{}\Varid{a},\Varid{b}\rangle}, where \ensuremath{\Varid{a},\Varid{b}} are fresh logical
variables in the body of the lambda; and returns the first of the two, namely \ensuremath{\Varid{a}}.
For convenience, in this presentation we allow ourselves to write a term like
\ensuremath{\Varid{first}\langle\mathrm{2},\mathrm{5}\rangle}, where we define \ensuremath{\Varid{first}} separately. Formally, you can imagine
each example \ensuremath{\Varid{e}} being wrapped with a binding for \ensuremath{\Varid{first}}, thus \ensuremath{\exists\Varid{first}.\,\Varid{first}\mathrel{=}\mathbin{...};\,\Varid{e}};
and similarly for all other library functions.

This way of desugaring pattern matching means that the input
to \ensuremath{\Varid{first}} is not required to be fully determined when the function is called. For example:
$$
\ensuremath{\exists\Varid{x}\hspace{0.3mm}\Varid{y}.\,\Varid{x}\mathrel{=}\langle{}\Varid{y},\mathrm{5}\rangle;\,\Varid{first}(\Varid{x})\mathrel{=}\mathrm{2};\,\Varid{y}}
$$
Here \ensuremath{\Varid{first}(\Varid{x})} evaluates to \ensuremath{\Varid{y}}, which we then unify with \ensuremath{\mathrm{2}}.
Another way to say this is that, as usual in logic programming, we may
constrain the \emph{output} of a function (here \ensuremath{\Varid{first}(\Varid{x})\mathrel{=}\mathrm{2}}), and thereby
affect its \emph{input} (here \ensuremath{\langle{}\Varid{y},\mathrm{5}\rangle}).

Although ``\ensuremath{;\,}\!'' is called ``sequencing'', the order of that sequence
is immaterial for equations.
For example
% foothand not needed, the figure introduces this shorthand
%\footnote{We informally write |3+y| rather than |applyN add (3,y)|, and similarly with other operators.},
consider \ensuremath{\exists\Varid{x}\hspace{0.3mm}\Varid{y}.\,\Varid{x}\mathrel{=}\mathrm{3}\mathbin{+}\Varid{y};\,\Varid{y}\mathrel{=}\mathrm{7};\,\Varid{x}}.  In \versecalc{} we can only unify \ensuremath{\Varid{x}} with a \emph{value};
we will see why in \cref{sec:call-time-choice}. So the equation \ensuremath{\Varid{x}\mathrel{=}\mathrm{3}\mathbin{+}\Varid{y}} is stuck.
No matter!  We simply leave it and try some other equation. In this case, we can make progress with \ensuremath{\Varid{y}\mathrel{=}\mathrm{7}};
and that in turn unlocks \ensuremath{\Varid{x}\mathrel{=}\mathrm{3}\mathbin{+}\Varid{y}} because now we know that \ensuremath{\Varid{y}} is \ensuremath{\mathrm{7}}, so we can evaluate \ensuremath{\mathrm{3}\mathbin{+}\mathrm{7}} to \ensuremath{\mathrm{10}}
and unify \ensuremath{\Varid{x}} with that.  The idea of leaving stuck expressions aside, and executing other parts of the
program is called
\emph{residuation}~\cite{hanus:survey}\footnote{Hanus did not invent the terms ``residuation'' and``narrowing'', but
his survey is an excellent introduction and bibliography.}, and is at the heart of our mantra ``just solve the equations.''

\subsection{Choice} \label{sec:choice} \label{sec:call-time-choice}

In conventional functional programming, an expression evaluates to a single value.
In contrast, a \versecalc{} expression evaluates to a choice of zero, one, or many values (or it can get stuck, which is different from producing zero values).
The expression \ensuremath{\textbf{fail}} yields no values; a head-normal form \ensuremath{\mathit{hnf}} or a logical variable \ensuremath{\Varid{x}} yields one value;
and the choice \ensuremath{\Varid{e}_{\mathrm{1}}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\Varid{e}_{\mathrm{2}}} yields all the values
yielded by \ensuremath{\Varid{e}_{\mathrm{1}}} and all the values yielded by \ensuremath{\Varid{e}_{\mathrm{2}}}.
Duplicates are not eliminated and, as we shall see in \cref{sec:for}, order is maintained; in short,
an expression yields a sequence of values, not a bag, and certainly not a set.

The equations we saw in \cref{sec:unify} can fail, if the arguments
are not equal, yielding no results.  Thus \ensuremath{\mathrm{3}\mathrel{=}\mathrm{3}} succeeds, returns a single result, namely \ensuremath{\mathrm{3}},
while \ensuremath{\mathrm{3}\mathrel{=}\mathrm{4}} fails, returning no results.  In general, we use ``fail''
and ``returns no results'' synonymously.

What if the choice was not at the top level of an expression?  For example,
what does \ensuremath{\langle{}\mathrm{3},(\mathrm{7}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\mathrm{5})\rangle} mean?
In \versecalc{} it does \emph{not} mean a pair with some kind of multi-value in its second component.
Indeed, as you can see from \cref{fig:syntax}, this expression is syntactically ill-formed.
We must instead give a name to that choice, and then we can put it in the pair, thus: \ensuremath{\exists\Varid{x}.\,\Varid{x}\mathrel{=}(\mathrm{7}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\mathrm{5});\,\langle{}\mathrm{3},\Varid{x}\rangle}.
This is syntactically legal, but what does it mean?  In \versecalc{} a variable is never bound to a multi-value.
Instead, \ensuremath{\Varid{x}} is successively bound to \ensuremath{\mathrm{7}}, and then to \ensuremath{\mathrm{5}}, like this:
$$
\ensuremath{\exists\Varid{x}.\,\Varid{x}\mathrel{=}(\mathrm{7}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\mathrm{5});\,\langle{}\mathrm{3},\Varid{x}\rangle} \quad \movesto \quad \ensuremath{(\exists\Varid{x}.\,\Varid{x}\mathrel{=}\mathrm{7};\,\langle{}\mathrm{3},\Varid{x}\rangle)\;\rule[-0.3mm]{0.5mm}{2.5mm}\;(\exists\Varid{x}.\,\Varid{x}\mathrel{=}\mathrm{5};\,\langle{}\mathrm{3},\Varid{x}\rangle))}
$$
We duplicate the context surrounding the choice, and ``float the choice outwards.''

\subsection{Mixing choice and unification} \label{sec:mixing}

We saw in \cref{sec:unify} that \emph{equations} are insensitive to sequencing---but \emph{choice} is not.
Consider
\ensuremath{\exists\Varid{x}\hspace{0.3mm}\Varid{y}.\,\Varid{x}\mathrel{=}(\mathrm{3}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\mathrm{4});\,\Varid{y}\mathrel{=}(\mathrm{20}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\mathrm{30});\,\langle{}\Varid{x},\Varid{y}\rangle}.  The choices are made \emph{left-to-right}, so that the result is
\ensuremath{(\langle{}\mathrm{3},\mathrm{20}\rangle\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\langle{}\mathrm{3},\mathrm{30}\rangle\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\langle{}\mathrm{4},\mathrm{20}\rangle\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\langle{}\mathrm{4},\mathrm{30}\rangle)}.

So much for choice under unification.  What if we have
unification under choice?  For example:
$$
\ensuremath{\exists\Varid{x}.\,(\Varid{x}\mathrel{=}\mathrm{3};\,\Varid{x}\mathbin{+}\mathrm{1})\;\rule[-0.3mm]{0.5mm}{2.5mm}\;(\Varid{x}\mathrel{=}\mathrm{4};\,\Varid{x}\mathbin{*}\mathrm{2})}
$$
Intuitively, either unify \ensuremath{\Varid{x}} with \ensuremath{\mathrm{3}} and return \ensuremath{\Varid{x}\mathbin{+}\mathrm{1}}, or unify \ensuremath{\Varid{x}} with \ensuremath{\mathrm{4}} and return \ensuremath{\Varid{x}\mathbin{*}\mathrm{2}}.
But so far we have said only ``a program executes by solving its equations'' (\cref{sec:unify}).
Well, we can see two equations here, \ensuremath{(\Varid{x}\mathrel{=}\mathrm{3})} and \ensuremath{(\Varid{x}\mathrel{=}\mathrm{4})}, which are mutually contradictory,
so clearly we need to refine our notion of ``solving.''
The answer is pretty clear: in a branch of a choice, solve the equations in that branch
to get the value for some logical variables, \emph{and propagate those values to occurrences in
  that branch (only)}. Occurrences of that variable outside the choice are unaffected.  We call
this \emph{local propagation}.
This local-propagation rule would allow us to reason thus:
$$
\ensuremath{\exists\Varid{x}.\,(\Varid{x}\mathrel{=}\mathrm{3};\,\Varid{x}\mathbin{+}\mathrm{1})\;\rule[-0.3mm]{0.5mm}{2.5mm}\;(\Varid{x}\mathrel{=}\mathrm{4};\,\Varid{x}\mathbin{*}\mathrm{2})} \movesto  \ensuremath{\exists\Varid{x}.\,(\Varid{x}\mathrel{=}\mathrm{3};\,\mathrm{4})\;\rule[-0.3mm]{0.5mm}{2.5mm}\;(\Varid{x}\mathrel{=}\mathrm{4};\,\mathrm{8})}
$$
Are we stuck now?  No, we can float the choice out as
before\footnote{Indeed we could have done so first, had we wished.},
$$
\ensuremath{\exists\Varid{x}.\,(\Varid{x}\mathrel{=}\mathrm{3};\,\mathrm{4})\;\rule[-0.3mm]{0.5mm}{2.5mm}\;(\Varid{x}\mathrel{=}\mathrm{4};\,\mathrm{8})} \movesto \ensuremath{(\exists\Varid{x}.\,\Varid{x}\mathrel{=}\mathrm{3};\,\mathrm{4})\;\rule[-0.3mm]{0.5mm}{2.5mm}\;(\exists\Varid{x}.\,\Varid{x}\mathrel{=}\mathrm{4};\,\mathrm{8})}
$$
and now it is apparent that the sole occurrence of \ensuremath{\Varid{x}} in each \ensuremath{\exists} is the equation
\ensuremath{(\Varid{x}\mathrel{=}\mathrm{3})}, or \ensuremath{(\Varid{x}\mathrel{=}\mathrm{4})} respectively; so we can drop the \ensuremath{\exists} and the equation, yielding \ensuremath{(\mathrm{4}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\mathrm{8})}.

\end{hscode}\resethooks
If pattern matching on the first equation fails, we want to fall through to the second.
Fortunately, choice allows us to express this idea directly\footnote{
We use the empty tuple \ensuremath{\langle\rangle} to represent the empty list and pairs to represent cons cells;
and we allow ourselves to write \ensuremath{\lambda\langle{}\Varid{x},\Varid{y}\rangle.\,\;\Varid{body}} rather than \ensuremath{\lambda\Varid{p}.\,\exists\Varid{x}\hspace{0.3mm}\Varid{y}.\,\Varid{p}\mathrel{=}\langle{}\Varid{x},\Varid{y}\rangle;\,\Varid{body}}
  }:
$$
\ensuremath{\Varid{append}\mathrel{=}\lambda\langle{} xs , ys \rangle.\,( xs \mathrel{=}\langle\rangle;\, ys )\;\hspace{1mm}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\hspace{1mm}\;(\exists\Varid{x}\hspace{0.3mm}\Varid{xrest}.\, xs \mathrel{=}\langle{}\Varid{x},\Varid{xrest}\rangle;\,\langle{}\Varid{x},\Varid{append}\langle\Varid{xrest}, ys \rangle\rangle)}
$$
If \ensuremath{ xs } is \ensuremath{\langle\rangle}, the left-hand choice succeeds, returning \ensuremath{ ys }; and the right-hand choice
fails (by attempting to unify \ensuremath{\langle\rangle} with \ensuremath{\langle{}\Varid{x},\Varid{xrest}\rangle}).
If \ensuremath{ xs } is of the form \ensuremath{\langle{}\Varid{x},\Varid{xrest}\rangle},
the right-hand choice succeeds, and we make a recursive call to \ensuremath{\Varid{append}}.
Finally if \ensuremath{ xs } is built with head-normal forms other than the empty tuple and pairs, both choices fail, and \ensuremath{\Varid{append}} returns no
results at all.

This approach to pattern matching is akin to \emph{narrowing}~\cite{hanus:survey}.
Suppose \ensuremath{\Varid{single}\mathrel{=}\langle{}\mathrm{1},\langle\rangle\rangle}, a singleton list whose only element is 1.
Consider this call:
$$
\ensuremath{\exists\Varid{zs}.\,\Varid{append}\langle\Varid{zs},\Varid{single}\rangle\mathrel{=}\Varid{single};\,\Varid{zs}}
$$
The call to \ensuremath{\Varid{append}} expands into a choice
$$
  \ensuremath{(\Varid{zs}\mathrel{=}\langle\rangle;\,\Varid{single})} \quad \ensuremath{\rule[-0.3mm]{0.5mm}{2.5mm}} \quad
  \ensuremath{(\exists\Varid{x}\hspace{0.3mm}\Varid{xrest}.\,\Varid{zs}\mathrel{=}\langle{}\Varid{x},\Varid{xrest}\rangle;\,\langle{}\Varid{x},\Varid{append}\langle\Varid{xrest},\Varid{single}\rangle\rangle)}
$$
which amounts to exploring the possibility that \ensuremath{\Varid{zs}} is headed by \ensuremath{\langle\rangle} or a pair---the
essence of narrowing.  It should not take long to reassure yourself that the program
evaluates to \ensuremath{\langle\rangle}, effectively running \ensuremath{\Varid{append}} backwards in the classic logic-programming manner.

This example also illustrates that \versecalc{} allows an equality (for \ensuremath{\Varid{append}})
that is recursive. As in any functional language with recursive bindings, you can
go into an infinite loop if you keep fruitlessly inlining the function in its own right-hand side.
It is the business of an \emph{evaluation strategy} to do only rewrites that make
progress towards a solution (\cref{sec:eval-strategy}).

\subsection{Conditionals and \ensuremath{\textbf{one}}} \label{sec:rigid} \label{sec:one} \label{sec:if}

Every source language will provide a conditional, such as \ensuremath{\textbf{if}\;(\Varid{x}\mathrel{=}\mathrm{0})\;\textbf{then}\;\Varid{e}_{\mathrm{2}}\;\textbf{else}\;\Varid{e}_{\mathrm{3}}}.  But what is the equality operator in \ensuremath{(\Varid{x}\mathrel{=}\mathrm{0})}?  One
possibility, adopted by Curry, is this: there is one ``='' for
equations (as in \cref{sec:unify}), and another, say ``=='', for
testing equality (returning a boolean with constructors \ensuremath{\Conid{True}} and \ensuremath{\Conid{False}}).  \versecalc{} takes a
different, more minimalist position. In \versecalc{} \emph{there is just one equality operator}, written ``='' just as in
\cref{sec:unify}. The expression \ensuremath{\textbf{if}\;(\Varid{x}\mathrel{=}\mathrm{0})\;\textbf{then}\;\Varid{e}_{\mathrm{2}}\;\textbf{else}\;\Varid{e}_{\mathrm{3}}} tries to unify \ensuremath{\Varid{x}} with \ensuremath{\mathrm{0}}.  If that succeeds (returns one or more values)
the \ensuremath{\textbf{if}} returns \ensuremath{\Varid{e}_{\mathrm{2}}}; otherwise it returns \ensuremath{\Varid{e}_{\mathrm{3}}}.  There are no data constructors \ensuremath{\Conid{True}} and \ensuremath{\Conid{False}}; instead
failure plays the role of falsity.

But something is terribly wrong here.  Consider \ensuremath{\exists\Varid{x}\hspace{0.3mm}\Varid{y}.\,\Varid{y}\mathrel{=}(\textbf{if}\;(\Varid{x}\mathrel{=}\mathrm{0})\;\textbf{then}\;\mathrm{3}\;\textbf{else}\;\mathrm{4});\,\Varid{x}\mathrel{=}\mathrm{7}}.  Presumably this is meant
to set \ensuremath{\Varid{x}} to \ensuremath{\mathrm{7}}, test if it is equal to \ensuremath{\mathrm{0}} (it is not), and unify \ensuremath{\Varid{y}} with 4.
But what is to stop us instead unifying \ensuremath{\Varid{x}} with \ensuremath{\mathrm{0}} (via \ensuremath{(\Varid{x}\mathrel{=}\mathrm{0})}), unifying \ensuremath{\Varid{y}} with \ensuremath{\mathrm{3}}, and then
failing when we try to unify \ensuremath{\Varid{x}} with \ensuremath{\mathrm{7}}?  Not only is that not what we intended, but it also looks very
non-deterministic: the result is affected by the order in which we did unifications!

To address this, we give \ensuremath{\textbf{if}} a special property: in the expression \ensuremath{\textbf{if}\;\Varid{e}_{\mathrm{1}}\;\textbf{then}\;\Varid{e}_{\mathrm{2}}\;\textbf{else}\;\Varid{e}_{\mathrm{3}}}, unifications inside \ensuremath{\Varid{e}_{\mathrm{1}}} (the condition of the \ensuremath{\textbf{if}})
can only unify variables bound inside \ensuremath{\Varid{e}_{\mathrm{1}}}; variables bound outside \ensuremath{\Varid{e}_{\mathrm{1}}} are called ``rigid.''  So in our example, the \ensuremath{\Varid{x}} in \ensuremath{(\Varid{x}\mathrel{=}\mathrm{0})} is
rigid and cannot be unified.  Instead, the \ensuremath{\textbf{if}} is stuck, and we move on to unify \ensuremath{\Varid{x}\mathrel{=}\mathrm{7}}.  That unblocks
the \ensuremath{\textbf{if}} and all is well.  This special property is precisely the local propagation rule that we sketched for
choice (\cref{sec:mixing}).

In fact, \versecalc{} distills the three-part \ensuremath{\textbf{if}} into something simpler, the unary construct \ensuremath{\textbf{one}\{\Varid{e}\}}.
Its specification is this:
if \ensuremath{\Varid{e}} fails, \ensuremath{\textbf{one}\{\Varid{e}\}} fails; otherwise \ensuremath{\textbf{one}\{\Varid{e}\}} returns the first of the values yielded by \ensuremath{\Varid{e}}.
Now, \ensuremath{\textbf{if}\;\Varid{e}_{\mathrm{1}}\;\textbf{then}\;\Varid{e}_{\mathrm{2}}\;\textbf{else}\;\Varid{e}_{\mathrm{3}}} can (nearly) be re-expressed like this:
$$
\ensuremath{\textbf{one}\{(\Varid{e}_{\mathrm{1}};\,\Varid{e}_{\mathrm{2}})\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\Varid{e}_{\mathrm{3}}\}}
$$
If \ensuremath{\Varid{e}_{\mathrm{1}}} fails, the first branch of the choice fails, so we get \ensuremath{\Varid{e}_{\mathrm{3}}}; if \ensuremath{\Varid{e}_{\mathrm{1}}} succeeds, we
get \ensuremath{\Varid{e}_{\mathrm{2}}}, and the outer \ensuremath{\textbf{one}} will select it from the choice.  But this isn't right:
what if \ensuremath{\Varid{e}_{\mathrm{2}}} or \ensuremath{\Varid{e}_{\mathrm{3}}} themselves fail or return multiple results?  Here is a better translation,
which wraps the then and else branches in a thunk:
$$
\ensuremath{(\textbf{one}\{(\Varid{e}_{\mathrm{1}};\,(\lambda x.\,\Varid{e}_{\mathrm{2}}))\;\rule[-0.3mm]{0.5mm}{2.5mm}\;(\lambda x.\,\Varid{e}_{\mathrm{3}})\})\langle\rangle}
$$
The argument of \ensuremath{\textbf{one}} evaluates to either \ensuremath{((\lambda x.\,\Varid{e}_{\mathrm{2}})\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\cdots)} or \ensuremath{(\lambda x.\,\Varid{e}_{\mathrm{3}})}
depending on whether \ensuremath{\Varid{e}_{\mathrm{1}}} succeeds or fails, respectively, and \ensuremath{\textbf{one}} then picks that lambda and applies it to \ensuremath{\langle\rangle}.
As a bonus, provided we do no evaluation under a lambda, then \ensuremath{\Varid{e}_{\mathrm{2}}} and \ensuremath{\Varid{e}_{\mathrm{3}}} will
remain un-evaluated until the choice is made, just as we expect.

We use the same local-propagation rule for \ensuremath{\textbf{one}} that we do for choice (\Cref{sec:mixing});
together with the desugaring for \ensuremath{\textbf{if}} into \ensuremath{\textbf{one}},
we get the ``special property'' of \ensuremath{\textbf{if}} described above.

\subsection{Tuples and \ensuremath{\textbf{all}}} \label{sec:tuple}
The main data structure in \versecalc{} is the tuple.
A tuple is a finite sequence of values, \ensuremath{\langle{}\Varid{v}_{\mathrm{1}},\cdots,\Varid{v}_{\Varid{n}}\rangle}.
It can be used like a function: indexing is simply function application
with the argument being integers from 0 and up.  Indexing out of range is \ensuremath{\textbf{fail}}.
For example, \ensuremath{\exists\Varid{t}.\,\Varid{t}\mathrel{=}\langle{}\mathrm{10},\mathrm{27},\mathrm{32}\rangle;\,\Varid{t}(\mathrm{1})} reduces to \ensuremath{\mathrm{27}} and \ensuremath{\Varid{t}(\mathrm{3})} reduces to \ensuremath{\textbf{fail}}.
The reduction rule for indexing in tuples admits multi-valued index expressions.
For instance, \ensuremath{\exists\Varid{t}.\,\Varid{t}\mathrel{=}\langle{}\mathrm{10},\mathrm{27},\mathrm{32}\rangle;\,\Varid{t}(\mathrm{1}\hspace{1mm}\mathop{\rule[-0.3mm]{0.5mm}{2.5mm}}\hspace{1mm}\mathrm{0}\hspace{1mm}\mathop{\rule[-0.3mm]{0.5mm}{2.5mm}}\hspace{1mm}\mathrm{1})} reduces to \ensuremath{(\mathrm{27}\hspace{1mm}\mathop{\rule[-0.3mm]{0.5mm}{2.5mm}}\hspace{1mm}\mathrm{10}\hspace{1mm}\mathop{\rule[-0.3mm]{0.5mm}{2.5mm}}\hspace{1mm}\mathrm{27})}.

Tuples can be constructed by collecting all the results from a multi-valued expression, using the \ensuremath{\textbf{all}} construct:
if \ensuremath{\Varid{e}} reduces to \ensuremath{(\Varid{v}_{\mathrm{1}}\hspace{1mm}\mathop{\rule[-0.3mm]{0.5mm}{2.5mm}}\hspace{1mm}\cdots\hspace{1mm}\mathop{\rule[-0.3mm]{0.5mm}{2.5mm}}\hspace{1mm}\Varid{v}_{\Varid{n}})} then \ensuremath{\textbf{all}\{\Varid{e}\}} reduces to
the tuple \ensuremath{\langle{}\Varid{v}_{\mathrm{1}},\cdots,\Varid{v}_{\Varid{n}}\rangle};
as a consequence, if $e$ fails, the \ensuremath{\textbf{all}} simply produces the empty sequence.
Note that \ensuremath{\rule[-0.3mm]{0.5mm}{2.5mm}} is associative, which means that we (and the compiler)
can think of a sequence or tree of binary choices as really being a single
$n$-way choice.
\label{sec:all}

You might think that tuple indexing would be stuck until we know the index,
but \versecalc{} uses narrowing to make progress.
The expression \ensuremath{\exists\Varid{t}.\,\Varid{t}\mathrel{=}\langle{}\mathrm{10},\mathrm{27},\mathrm{32}\rangle;\,\exists\Varid{i}.\,\Varid{t}(\Varid{i})}
looks stuck because we have no value for \ensuremath{\Varid{i}}, but in fact it rewrites to
$$
\ensuremath{\exists\Varid{i}.\,(\Varid{i}\mathrel{=}\mathrm{0};\,\mathrm{10})\;\rule[-0.3mm]{0.5mm}{2.5mm}\;(\Varid{i}\mathrel{=}\mathrm{1};\,\mathrm{27})\;\rule[-0.3mm]{0.5mm}{2.5mm}\;(\Varid{i}\mathrel{=}\mathrm{2};\,\mathrm{32})}
$$
which (as we will see in \Cref{sec:rewrite-rules}) simplifies
to just \ensuremath{(\mathrm{10}\hspace{1mm}\mathop{\rule[-0.3mm]{0.5mm}{2.5mm}}\hspace{1mm}\mathrm{27}\hspace{1mm}\mathop{\rule[-0.3mm]{0.5mm}{2.5mm}}\hspace{1mm}\mathrm{32})}.
So \ensuremath{\textbf{all}} allows a choice to be reified into a tuple; and \ensuremath{(\exists\Varid{i}.\,\Varid{t}(\Varid{i}))} allows a
tuple to be turned back into a choice.

Do we even need \ensuremath{\textbf{one}} as a primitive construct, given that we have
\ensuremath{\textbf{all}}? Can we not use \ensuremath{(\textbf{all}\{\Varid{e}\})(\mathrm{0})} instead of \ensuremath{\textbf{one}\{\Varid{e}\}}? Indeed
they behave the same if \ensuremath{\Varid{e}} fully reduces to finitely many choices of
values. But \ensuremath{\textbf{all}} really requires the evaluation of \emph{all}
choices before proceeding, while \ensuremath{\textbf{one}} only needs to evaluate the
first choice. So, supposing that \ensuremath{\Varid{loop}} is a non-terminating function,
\ensuremath{\textbf{one}\{\mathrm{1}\hspace{1mm}\mathop{\rule[-0.3mm]{0.5mm}{2.5mm}}\hspace{1mm}\Varid{loop}\langle\rangle\}} reduces to 1, while
\ensuremath{(\textbf{all}\{\mathrm{1}\hspace{1mm}\mathop{\rule[-0.3mm]{0.5mm}{2.5mm}}\hspace{1mm}\Varid{loop}\langle\rangle\})(\mathrm{0})} loops.

\subsection{List functions}

We can define the typical list functions one would expect from functional programming
by using the duality between tuples and choices, as seen in \cref{fig:list-functions}.
A tuple can be turned into choices by indexing with a logical variable \ensuremath{\Varid{i}}.
Conversely, choices can be turned into a tuple using \ensuremath{\textbf{all}}.
The choice operator, \ensuremath{\rule[-0.3mm]{0.5mm}{2.5mm}}, serves as both \ensuremath{\Varid{cons}} and \ensuremath{\Varid{append}} for choices.
%and |oneKW| is like |head| for choices.
With that in mind, here is how to understand
the definitions in \cref{fig:list-functions}:
\begin{itemize}
\item \ensuremath{\Varid{head}}.
To obtain the first element of a tuple, we can simply use array indexing. This fails if the tuple is empty.

\item \ensuremath{\Varid{tail}}.
To get the tail of a tuple, we apply the tuple to a fresh variable, and restrict that variable with \ensuremath{\Varid{i}\mathbin{>}\mathrm{0}}.
This way, the expression fails at the first value and only the other values are collected by \ensuremath{\textbf{all}}.
It returns the empty tuple if \ensuremath{ xs } is empty or not even a tuple.

\item \ensuremath{\Varid{cons}} and \ensuremath{\Varid{append}}.
We can implement \ensuremath{\Varid{cons}\langle\Varid{x}, xs \rangle} by collecting with \ensuremath{\textbf{all}} the choices we get from choosing \ensuremath{\Varid{x}}
or all the elements of \ensuremath{ xs }, and similarly we can define \ensuremath{\Varid{append}}.

\item \ensuremath{\Varid{flatMap}}.
For the function \ensuremath{\Varid{map}}, which applies a function to each element of a tuple and collects the results,
two possible implementation come to mind, and they differ when the function passed to \ensuremath{\Varid{map}} fails or produces more than a single value.
\ensuremath{\Varid{flatMap}} flattens the choices from iterating through \ensuremath{ xs } with
the choices taken by \ensuremath{\Varid{f}}, so \ensuremath{\Varid{flatMap}\langle(\lambda\Varid{x}.\,(\Varid{x}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\Varid{x}\mathbin{+}\mathrm{1})),\langle{}\mathrm{10},\mathrm{20}\rangle\rangle)} $\equiv$ \ensuremath{\langle{}\mathrm{10},\mathrm{11},\mathrm{20},\mathrm{21}\rangle}.

\item \ensuremath{\Varid{map}}.
But often this is not what we want: we may want the effects of \ensuremath{\Varid{f}}, such as failure or returning multiple values,
to affect the whole expression and obtain the cartesian product of these choices.
The expression \ensuremath{\Varid{map}\langle(\lambda\Varid{x}.\,(\Varid{x}\hspace{1mm}\mathop{\rule[-0.3mm]{0.5mm}{2.5mm}}\hspace{1mm}\Varid{x}\mathbin{+}\mathrm{1})),\langle{}\mathrm{10},\mathrm{20}\rangle\rangle} should reduce to \ensuremath{\langle{}\mathrm{10},\mathrm{20}\rangle\hspace{1mm}\mathop{\rule[-0.3mm]{0.5mm}{2.5mm}}\hspace{1mm}\langle{}\mathrm{10},\mathrm{21}\rangle\hspace{1mm}\mathop{\rule[-0.3mm]{0.5mm}{2.5mm}}\hspace{1mm}\langle{}\mathrm{11},\mathrm{20}\rangle\hspace{1mm}\mathop{\rule[-0.3mm]{0.5mm}{2.5mm}}\hspace{1mm}\langle{}\mathrm{11},\mathrm{21}\rangle}.
We get that behaviour if we simply define \ensuremath{\Varid{map}} much like in any other functional programming language:
if taking the head of the tuple fails, we return an empty tuple.
Else we apply \ensuremath{\Varid{f}} to the head, recur on the \ensuremath{\Varid{tail}} and assemble the result with \ensuremath{\Varid{cons}}.\label{sec:map}

\item \ensuremath{\Varid{filter}}.
How can we filter a tuple with a predicate \ensuremath{\Varid{p}}?
We turn the tuple into the sequence of choices, and let those fail where \ensuremath{\Varid{p}} fails before collecting the other values into a tuple.

\item \ensuremath{\Varid{find}}.
If we use \ensuremath{\textbf{one}} instead of \ensuremath{\textbf{all}}, we get the function that checks if \ensuremath{\Varid{p}} holds for some element of the tuple,
and returns the first matching element, or fails if \ensuremath{\Varid{p}} does not hold for any element.

\item \ensuremath{\Varid{every}}.
To test if a predicate \ensuremath{\Varid{p}} holds for every element of a tuple \ensuremath{ xs }, we can simply use \ensuremath{\Varid{map}\langle\Varid{p}, xs \rangle}!
If \ensuremath{\Varid{p}} fails for any value, the whole call to \ensuremath{\Varid{map}\langle\Varid{p}, xs \rangle} fails.
\end{itemize}

\subsection{Choice comprehensions} \label{sec:for}

Source languages can use this to build more featureful constructs like \ensuremath{\textbf{for}},
mimicking the list comprehension in languages like Haskell or Python:
The expression \ensuremath{\textbf{for} (\Varid{e}_{\mathrm{1}})\;\textbf{do}\;\Varid{e}_{\mathrm{2}}} will evaluate \ensuremath{\Varid{e}_{\mathrm{2}}} for each of the choices in \ensuremath{\Varid{e}_{\mathrm{1}}}.
The \ensuremath{\Varid{e}_{\mathrm{1}}} is called the \emph{domain} of the loop.

The scoping is peculiar\footnote{But similar to C++.}
in that variables bound in \ensuremath{\Varid{e}_{\mathrm{1}}} also scope over \ensuremath{\Varid{e}_{\mathrm{2}}}.
So, {\eg}, \ensuremath{\textbf{for} (\exists\Varid{x}.\,\Varid{x}\mathrel{=}\mathrm{2}\hspace{1mm}\mathop{\rule[-0.3mm]{0.5mm}{2.5mm}}\hspace{1mm}\mathrm{3}\hspace{1mm}\mathop{\rule[-0.3mm]{0.5mm}{2.5mm}}\hspace{1mm}\mathrm{5})\;\textbf{do}\;(\Varid{x}\mathbin{+}\mathrm{1})} will reduce to the tuple \ensuremath{\langle{}\mathrm{3},\mathrm{4},\mathrm{6}\rangle}.
This example can, with the \ensuremath{\mathrel{\coloneqq}} shorthand, be written \ensuremath{\textbf{for} (\Varid{x}\mathrel{\coloneqq}\mathrm{2}\hspace{1mm}\mathop{\rule[-0.3mm]{0.5mm}{2.5mm}}\hspace{1mm}\mathrm{3}\hspace{1mm}\mathop{\rule[-0.3mm]{0.5mm}{2.5mm}}\hspace{1mm}\mathrm{5})\;\textbf{do}\;(\Varid{x}\mathbin{+}\mathrm{1})}.

Like list comprehension, it supports filtering, but in Verse this falls out naturally by just using
a possibly failing expression in the domain.
So, \ensuremath{\textbf{for} (\Varid{x}\mathrel{\coloneqq}\mathrm{2}\hspace{1mm}\mathop{\rule[-0.3mm]{0.5mm}{2.5mm}}\hspace{1mm}\mathrm{3}\hspace{1mm}\mathop{\rule[-0.3mm]{0.5mm}{2.5mm}}\hspace{1mm}\mathrm{5};\,\Varid{x}\mathbin{>}\mathrm{2})\;\textbf{do}\;(\Varid{x}\mathbin{+}\mathrm{1})} reduces to the tuple \ensuremath{\langle{}\mathrm{4},\mathrm{6}\rangle}.

Nested iteration in a \ensuremath{\textbf{for}} works as expected, and requires nothing special.
So, \ensuremath{\textbf{for} (\Varid{x}\mathrel{\coloneqq}\mathrm{10}\hspace{1mm}\mathop{\rule[-0.3mm]{0.5mm}{2.5mm}}\hspace{1mm}\mathrm{20};\,\Varid{y}\mathrel{\coloneqq}\mathrm{1}\hspace{1mm}\mathop{\rule[-0.3mm]{0.5mm}{2.5mm}}\hspace{1mm}\mathrm{2}\hspace{1mm}\mathop{\rule[-0.3mm]{0.5mm}{2.5mm}}\hspace{1mm}\mathrm{3})\;\textbf{do}\;(\Varid{x}\mathbin{+}\Varid{y})} reduces to \ensuremath{\langle{}\mathrm{11},\mathrm{12},\mathrm{13},\mathrm{21},\mathrm{22},\mathrm{23}\rangle}

Just like \ensuremath{\textbf{if}} is defined in terms of the primitive \ensuremath{\textbf{one}}, we can define \ensuremath{\textbf{for}}
in terms of the primitive \ensuremath{\textbf{all}\{\Varid{e}\}}.
Again, we have to be careful when \ensuremath{\Varid{e}_{\mathrm{2}}} itself fails or produces multiple results;
simply writing \ensuremath{\textbf{all}\{\Varid{e1};\,\Varid{e2}\}} would give the wrong semantics. So we put \ensuremath{\Varid{e}_{\mathrm{2}}} under a lambda, and
apply each element of the tuple to \ensuremath{\langle\rangle} afterwards, using the \ensuremath{\Varid{map}} function defined above.
The full desugaring is thus
\[
\ensuremath{\textbf{for} (\Varid{e}_{\mathrm{1}})\;\textbf{do}\;\Varid{e}_{\mathrm{2}}} \equiv \ensuremath{\Varid{v}\mathrel{\coloneqq}\textbf{all}\{\Varid{e}_{\mathrm{1}};\,\lambda x.\,\Varid{e}_{\mathrm{2}}\};\,\Varid{map}\langle\lambda\Varid{z}.\,\Varid{z}\langle\rangle,\Varid{v}\rangle}
\]
for a fresh variable \ensuremath{\Varid{v}}.

Note how this achieves that peculiar scoping rule: variables defined in \ensuremath{\Varid{e}_{\mathrm{1}}} are in scope in \ensuremath{\Varid{e}_{\mathrm{2}}}.
Any effects (like being multivalued) in \ensuremath{\Varid{e}_{\mathrm{2}}} will not affect the choices defined by \ensuremath{\Varid{e}_{\mathrm{1}}} since they are in a thunk.
So, {\eg}, \ensuremath{\textbf{for} (\Varid{x}\mathrel{\coloneqq}\mathrm{10}\hspace{1mm}\mathop{\rule[-0.3mm]{0.5mm}{2.5mm}}\hspace{1mm}\mathrm{20})\;\textbf{do}\;\{\mskip1.5mu \Varid{x}\hspace{1mm}\mathop{\rule[-0.3mm]{0.5mm}{2.5mm}}\hspace{1mm}\Varid{x}\mathbin{+}\mathrm{1}\mskip1.5mu\}} will reduce to
\ensuremath{\langle{}\mathrm{10},\mathrm{20}\rangle\hspace{1mm}\mathop{\rule[-0.3mm]{0.5mm}{2.5mm}}\hspace{1mm}\langle{}\mathrm{10},\mathrm{21}\rangle\hspace{1mm}\mathop{\rule[-0.3mm]{0.5mm}{2.5mm}}\hspace{1mm}\langle{}\mathrm{11},\mathrm{20}\rangle\hspace{1mm}\mathop{\rule[-0.3mm]{0.5mm}{2.5mm}}\hspace{1mm}\langle{}\mathrm{11},\mathrm{21}\rangle}.
At this point it is crucial to use \ensuremath{\Varid{map}}, not \ensuremath{\Varid{flatMap}}.

Given that tuple indexing expands into choices, we can iterate over tuple indices and elements using \ensuremath{\textbf{for}}.
For example \ensuremath{\textbf{for} (\exists\Varid{i}.\,\Varid{x}\mathrel{\coloneqq}\Varid{t}(\Varid{i}))\;\textbf{do}\;(\Varid{x}\mathbin{+}\Varid{i})} produces a tuple with the elements of \ensuremath{\Varid{t}}, increased by their index in \ensuremath{\Varid{t}}.

\subsection{First class patterns}
Pattern matching for function definitions is simply done by unification of ordinary expressions.
This means that we can use ordinary abstraction mechanisms for patterns.
For example, here is a function that should be called like \ensuremath{\Varid{fcn}\langle\mathrm{88},\mathrm{1},\mathrm{99},\mathrm{2}\rangle}.
$$
\begin{array}{l}
\ensuremath{\Varid{fcn}(\Varid{t})\mathrel{\coloneqq}\exists\Varid{x}\hspace{0.3mm}\Varid{y}.\,\Varid{t}\mathrel{=}\langle{}\Varid{x},\mathrm{1},\Varid{y},\mathrm{2}\rangle;\,\Varid{x}\mathbin{+}\Varid{y}}
\end{array}
$$
If we want to give a name to the pattern, it is simple to do so:
$$
\begin{array}{l}
\ensuremath{\Varid{pat}\langle\Varid{v},\Varid{w}\rangle\mathrel{\coloneqq}\langle{}\Varid{v},\mathrm{1},\Varid{w},\mathrm{2}\rangle;\,} \\
\ensuremath{\Varid{fcn}(\Varid{t})\mathrel{\coloneqq}\exists\Varid{x}\hspace{0.3mm}\Varid{y}.\,\Varid{t}\mathrel{=}\Varid{pat}\langle\Varid{x},\Varid{y}\rangle;\,\Varid{x}\mathbin{+}\Varid{y}}
\end{array}
$$

\section{Rewrite rules} \label{sec:rewrite-rules}

How can we give a precise semantics to a non-strict functional
programming language?  Here are some possibilities:
\begin{itemize}
\item \emph{A denotational semantics} is the classical approach,
  but it is hard to see how to give a (perspicuous) denotational semantics to a functional logic language,
  because of the logical variables; see \cite{christiansen:denotational} for one approach.
  As a way to explain a programmer what their program means, it has its challenges.
\item \emph{A big-step operational semantics} typically involves explaining how a (heap, expression) starting point
  evaluates to a (heap, value) pair; Launchbury's natural semantics for lazy evaluation~\cite{launchbury:natural} is the classic paper.  The heap, threaded through the semantics, accounts for updating thunks as they are evaluated.
\item \emph{A small-step operational semantics}. Despite its ``operational semantics'' title, the big-step approach
  does not convey accurate operational intuition, because it goes all the way to a value
  in one step. So-called ``small-step'' operational semantics are therefore widely used;
  they typically describe how a (heap, expression, stack) configuration evolves, one small
  step at a time ({\eg}, ~\cite{pj:stg-machine}).
  The difficulty is that the description is now so low level that it is again hard to explain to programmers.
\item \emph{A rewrite semantics} steers between these two extremes.
  For example, Ariola {\etal}'s ``A call by need lambda calculus''~\cite{ariola:call-by-need} shows how
  to give the semantics of a call-by-need language as a set of rewrite rules. The great
  advantage of this approach is that it is readily explicable to programmers.
  Indeed teachers almost always explain the execution of Haskell or ML programs as a succession of rewrites
 of the program ({\eg}, inline this call, simplify this \ensuremath{\textbf{case}} expression, \etc).
\end{itemize}

Up to this point there has been no satisfying rewrite semantics for functional logic languages
(see \cref{sec:related} for previous work).
Our main technical contribution is to fill this gap with a rewrite semantics for \versecalc{}, one that
has the following properties:
\begin{itemize}
\item The semantics is expressed as a set of rewrite rules.
\item Any rule can be applied, in either direction,
  anywhere in the program term (including under lambdas) to obtain an equivalent program.
\item The rules are oriented, with the intent that using them left-to-right makes progress.
\item Even given this orientation, the rules do not say which rule should be
  applied where; that is the task of a separate \emph{evaluation strategy}.
\item The rules can be applied by programmers, to reason about what their program does;
  and by compilers, to transform (and hopefully optimise) the program.
\item There is no ``magical rewriting'' (\cref{sec:magical}): all the variables on the
  right-hand side of a rule are bound on the left.
\end{itemize}

The rewrite rules are given in Figures~\ref{fig:rules1} and \ref{fig:rules2},
and are described next.

\subsection{Primitive rewrites}

Looking at \cref{fig:general-axioms}, rules \rulename{p-add} and \rulename{p-gt}
should be familiar: they simply rewrite applications of \ensuremath{\textbf{add}} and \ensuremath{\textbf{gt}} to integer
constants.  For example $\ensuremath{\textbf{add}\langle\mathrm{3},\mathrm{4}\rangle} \movesto \ensuremath{\mathrm{7}}$.  The only unusual point is
that \ensuremath{\textbf{gt}} fails if $k_1 \leq k_2$ (rather than returning \ensuremath{\Conid{False}} as is more conventional),
and returns \ensuremath{\Varid{k}_{\mathrm{1}}} otherwise.

% Rule \rulename{p-map} implements a map operation over tuples: It applies the function |f| to each value
% in turn, and collects the results into a tuple. If the function applications produce a choice
% of values, this will, as it should, produce a choice of tuples:
% \[
% |applyN map ((lam x (x choice  x+1)), (10,20))| \movesto^\ast |(10,20) choice (10,21) choice (11,20) choice (11,21)|

Sequencing (rule \rulename{seq}) is very simple: \ensuremath{(\Varid{v};\,\Varid{e})} discards \ensuremath{\Varid{v}}, leaving just \ensuremath{\Varid{e}}.
The rule only fires with a \emph{value} on the left; we must rewrite the expression on
the left for effects.

\subsection{Function application} \label{sec:beta}

Beta-reduction is performed quite conventionally by \rulename{app-beta}; the
only unusual feature is that we use a \ensuremath{\exists} to bind \ensuremath{\Varid{x}} on the right-hand side,
and an equality \ensuremath{(\Varid{x}\mathrel{=}\Varid{v})} to bind \ensuremath{\Varid{x}} to the argument.
The rule may appear to use call-by-value, because the argument is a value \ensuremath{\Varid{v}},
but remember that values include variables, and that variable may be bound to an
as-yet-unevaluated expression.  For example:
$$
\ensuremath{\exists\Varid{y}.\,\Varid{y}\mathrel{=}\mathrm{3}\mathbin{+}\mathrm{4};\,(\lambda\Varid{x}.\,\Varid{x}\mathbin{+}\mathrm{1})(\Varid{y})}
\; \movesto \;
\ensuremath{\exists\Varid{y}.\,\Varid{y}\mathrel{=}\mathrm{3}\mathbin{+}\mathrm{4};\,\exists\Varid{x}.\,\Varid{x}\mathrel{=}\Varid{y};\,\Varid{x}\mathbin{+}\mathrm{1}}
$$
Finally, the side condition $x \not\in \freevars{v}$ in \rulename{app-beta}
ensures that the \ensuremath{\exists\;\Varid{x}}
does not capture any variables free in \ensuremath{\Varid{v}}.  If \ensuremath{\Varid{x}} appears free in \ensuremath{\Varid{v}}, just use
\textalpha-conversion to rename \ensuremath{\Varid{x}} to $x' \not\in \freevars{v}$.

In \versecalc{}, tuples behave like (finite) functions, in which application is
indexing.  Rule \rulename{app-tup} describes how tuple application works.
Notice that \rulename{app-tup} does not require the argument to be an integer $k$; instead
the rule works by narrowing.  So the expression \ensuremath{\exists\Varid{x}.\,\langle{}\mathrm{2},\mathrm{3},\mathrm{2},\mathrm{7},\mathrm{9}\rangle(\Varid{x})\mathrel{=}\mathrm{2};\,\Varid{x}}
does not suspend awaiting a value for \ensuremath{\Varid{x}}; instead it explores all the
alternatives, returning \ensuremath{(\mathrm{0}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\mathrm{2})}.  This is a free design decision:
a suspending semantics would be equally easy to express.

\subsection{Unification: the easy bits}

Next we study unification, again in \cref{fig:general-axioms}.
Rules \rulename{ulit}, \rulename{utup}, \rulename{ux1} and \rulename{ux2}
are the standard rules for unification, going back at least 40 years \cite{MartelliMontanari82}.
The only special thing
to note is that \ensuremath{(\Varid{k}_{\mathrm{1}}\mathrel{=}\Varid{k}_{\mathrm{2}})} either fails if the values differ,
or yields common value if they are the same; and similarly for
tuples.%
\footnote{An alternative choice would
for unification to yield \ensuremath{\langle\rangle} on success; but it does not make much difference
either way.}.
\rulename{ux1} and \rulename{ux2} make unification fail if
you attempt to unify an integer with a tuple.
Slightly less obviously,
\rulename{ux3} through \rulename{ux6} make unification fail if you attempt to unify a
lambda or a built-in operation with any other value.  We discuss this design choice further in \cref{sec:unifying-lambda}.

Failure, written \ensuremath{\textbf{fail}}, propagates upwards, as \rulename{fail-def} and \rulename{fail} show. 
The latter rule makes use of an \emph{execution context}, \ensuremath{X},
whose syntax is given in \cref{fig:contexts}.
In general, a ``context'' is a syntactic construct (in this case an expression)
containing a single hole, written \ensuremath{\hole} \cite{felleisen:contexts1,felleisen:contexts2}.
The notation \ensuremath{X [\mskip1.5mu \Varid{e}\mskip1.5mu]} is the expression obtained by filling the hole
in \ensuremath{X} with \ensuremath{\Varid{e}}.
For example \ensuremath{(\Varid{y}\mathrel{=}\langle{}\hole,\mathrm{4}\rangle)} is a possible execution context, \ensuremath{X}; filling its hole with 99
gives the expression $\ensuremath{X [\mskip1.5mu \mathrm{99}\mskip1.5mu]} \equiv \ensuremath{(\Varid{y}\mathrel{=}\langle{}\mathrm{99},\mathrm{4}\rangle)}$.
So \rulename{fail} says that if \ensuremath{\textbf{fail}} occurs in an \ensuremath{X}-context, thus \ensuremath{X [\mskip1.5mu \textbf{fail}\mskip1.5mu]}, we can rewrite
it to \ensuremath{\textbf{fail}}, discarding \ensuremath{X} altogether. For example \ensuremath{(\Varid{x}\mathrel{=}\Varid{y};\,\textbf{fail};\,\mathrm{3})} rewrites to \ensuremath{\textbf{fail}}.

\subsection{Unification: the \rulename{subst} rule}

The key innovation in \versecalc{} is the way bindings (that is, just ordinary equalities)
of logical variables are propagated.
The key rule is \rulename{subst}:
$$
\rulename{subst} \quad \ensuremath{X [\mskip1.5mu \Varid{x}\mathrel{=}\Varid{v}\mskip1.5mu]}  \movesto  \ensuremath{(\subst{X}{\Varid{x}}{\Varid{v}}) [\mskip1.5mu \Varid{x}\mathrel{=}\Varid{v}\mskip1.5mu]}
        \quad \text{if $\ensuremath{\Varid{x}} \in \freevars{X}$, $\ensuremath{\Varid{x}} \notin \freevars{v}$} \\
$$
The notation \ensuremath{\subst{X}{\Varid{x}}{\Varid{v}}} is the context obtained by replacing all
free occurrences of \ensuremath{\Varid{x}} in \ensuremath{X} by \ensuremath{\Varid{v}}.

So \rulename{subst} says that if we have an equality \ensuremath{(\Varid{x}\mathrel{=}\Varid{v})} in a term \ensuremath{X [\mskip1.5mu \Varid{x}\mathrel{=}\Varid{v}\mskip1.5mu]}
we can replace occurrences of \ensuremath{\Varid{x}} in the
\ensuremath{X}-context surrounding an equality \ensuremath{(\Varid{x}\mathrel{=}\Varid{v})}, thereby propagating \ensuremath{\Varid{x}}'s value to those occurrence sites.
There are several things to notice:
\begin{itemize}
\item \rulename{subst} fires only when the right-hand side of the unification is a \emph{value} \ensuremath{\Varid{v}},
so that the substitution does not risk duplicating either work or choices. This
restriction is precisely the same as the \rulename{let-v} rule of \cite{ariola:call-by-need},
and (by not duplicating choices) it neatly implements so-called \emph{call-time choice}~\cite{hanus:survey}.
We do not need a heap, or thunks, or updates; the equalities of the program elegantly
suffice to express the necessary sharing.
The fact that we substitute only values for variables also ensures that substitution is syntactically
valid; if we tried to substitute an expression \ensuremath{\Varid{e}} for a variable in \ensuremath{\langle{}\Varid{x},\mathrm{2}\rangle} we would get the
ill-formed expression \ensuremath{\langle{}\Varid{e},\mathrm{2}\rangle}.
\item \rulename{subst}
  replaces the occurrences of \ensuremath{\Varid{x}} in \ensuremath{X}, \emph{but leaves the original \ensuremath{(\Varid{x}\mathrel{=}\Varid{v})} undisturbed}.
  For example, we can rewrite \ensuremath{((\Varid{y}\mathrel{=}\Varid{x}\mathbin{+}\mathrm{1};\,\Varid{x}\mathrel{=}\mathrm{3});\,\Varid{z}\mathrel{=}\Varid{x}\mathbin{+}\mathrm{3})} to \ensuremath{((\Varid{y}\mathrel{=}\mathrm{3}\mathbin{+}\mathrm{1};\,\Varid{x}\mathrel{=}\mathrm{3});\,\Varid{z}\mathrel{=}\Varid{x}\mathbin{+}\mathrm{3})}, using
  \ensuremath{X\mathrel{=}(\Varid{y}\mathrel{=}\Varid{x}\mathbin{+}\mathrm{1};\,\hole)}.  We must not drop the \ensuremath{(\Varid{x}\mathrel{=}\Varid{v})} because there may be other occurrences of \ensuremath{\Varid{x}}
  ``further out'', such as \ensuremath{(\Varid{z}\mathrel{=}\Varid{x}\mathbin{+}\mathrm{3})} in this example.
\item \rulename{subst} does not specify how big the execution context \ensuremath{X} is:
  it can be anything from a local expression immediately surrounding the \ensuremath{(\Varid{x}\mathrel{=}\Varid{v})} to an
  arbitrarily large expression.
But note carefully  that \emph{execution contexts \ensuremath{X} do not include \ensuremath{\textbf{one}}, \ensuremath{\textbf{all}}, and choice} (\cref{fig:contexts}).
That restriction is the key to localizing unification (\cref{sec:mixing}), and
the flexible/rigid distinction of
\cref{sec:rigid}. For example, consider the expression
\ensuremath{(\Varid{y}\mathrel{=}((\Varid{x}\mathrel{=}\mathrm{3};\,\Varid{x}\mathbin{*}\mathrm{2})\;\rule[-0.3mm]{0.5mm}{2.5mm}\;(\Varid{x}\mathrel{=}\mathrm{4}));\,\langle{}\Varid{x}\mathbin{+}\mathrm{1},\Varid{y}\rangle)}.
We must not propagate the binding \ensuremath{(\Varid{x}\mathrel{=}\mathrm{3})} to the expression \ensuremath{(\Varid{x}\mathbin{+}\mathrm{1})},
because the latter is outside the choice, and a different branch of the choice binds \ensuremath{\Varid{x}} to \ensuremath{\mathrm{4}}.
But rule \rulename{subst}
\emph{can} propagate it locally
within the first arm of the choice, thus\footnote{
You may wonder if this local propagation is useful, a point we return
to in \cref{sec:local-subst}.}:
$$
\ensuremath{\Varid{y}\mathrel{=}((\Varid{x}\mathrel{=}\mathrm{3};\,\Varid{x}\mathbin{*}\mathrm{2})\;\rule[-0.3mm]{0.5mm}{2.5mm}\;(\Varid{x}\mathrel{=}\mathrm{4}));\,\langle{}\Varid{x}\mathbin{+}\mathrm{1},\Varid{y}\rangle}
\movesto
\ensuremath{\Varid{y}\mathrel{=}((\Varid{x}\mathrel{=}\mathrm{3};\,\mathrm{3}\mathbin{*}\mathrm{2})\;\rule[-0.3mm]{0.5mm}{2.5mm}\;(\Varid{x}\mathrel{=}\mathrm{4}));\,\langle{}\Varid{x}\mathbin{+}\mathrm{1},\Varid{y}\rangle}
$$
To make further progress, we need a rule for choice, see \cref{sec:choice-rules}.
\item The $x \in \freevars{X}$ in \rulename{subst} prevents the rule from
  firing infinitely often, doing nothing each time.  The other guard $x \not\in \freevars{v}$
  also prevents \rulename{subst} from applying infinitely often, this time unrolling a recursive loop; but
  see \cref{sec:recursion}.
\end{itemize}


\subsection{Unification: recursion and the occurs check} \label{sec:occurs} \label{sec:recursion}

The classic ``occurs check'', rule \rulename{ux-occurs}, makes an equality like \ensuremath{(\Varid{x}\mathrel{=}\langle{}\Varid{x},\mathrm{3}\rangle)} fail, because the only successful
unification would yield an infinitely-nested pair.
$$
    \rulename{ux-occurs} \quad \ensuremath{\Varid{x}\mathrel{=}\Conid{V} [\mskip1.5mu \Varid{x}\mskip1.5mu]} \; \movesto \; \ensuremath{\textbf{fail}} \quad \text{if }\ensuremath{\Conid{V}} \ne \ensuremath{\hole}
$$
In this rule, \ensuremath{\Conid{V}} is a \emph{value context}, defined in \cref{fig:syntax}.
So \rulename{ux-occurs} says that the equality fails if there is a \ensuremath{\Conid{V}} such the equality looks like \ensuremath{(\Varid{x}\mathrel{=}\Conid{V} [\mskip1.5mu \Varid{x}\mskip1.5mu])};
that is, if \ensuremath{\Varid{x}} occurs under a ``shell'' of tuples.  The guard $\ensuremath{\Conid{V}} \ne \ensuremath{\hole}$ is to avoid failure for \ensuremath{\Varid{x}\mathrel{=}\Varid{x}}.

What if \ensuremath{\Varid{x}} occurs inside a lambda?  That is very useful, and we do not want to fail; for example
$$
\ensuremath{\Varid{factorial}\mathrel{=}\lambda\Varid{n}.\,\;\textbf{if}\;(\Varid{n}\mathrel{=}\mathrm{0})\;\textbf{then}\;\mathrm{1}\;\textbf{else}\;\Varid{n}\mathbin{*}\Varid{factorial}(\Varid{n}\mathbin{-}\mathrm{1})}
$$
Our solution, \rulename{subst-rec}, is standard: if the occurrence of \ensuremath{\Varid{x}} is inside a lambda \ensuremath{(\lambda\Varid{y}.\,\Varid{e})},
we create a new local binding for \ensuremath{\Varid{x}} under the lambda.  Now the original equality is
non-recursive, and \rulename{subst} can apply.

If the equality \ensuremath{(\Varid{x}\mathrel{=}\Varid{v})} turns out to be the sole remaining occurrence of \ensuremath{\Varid{x}} in
the body of an \ensuremath{\exists}, then rule \rulename{def-eliml} can drop both the \ensuremath{\exists}

A number of rules shuffle expressions about in order to make
the main unification rules apply.
The rule \rulename{swap} swaps the arguments to unification to put the variable on the left.

Rule \rulename{def-float} moves an \ensuremath{\exists} ``out of the way.'' For example,
the expression \ensuremath{(\Varid{y}\mathrel{=}(\exists\Varid{x}.\,\langle{}\Varid{x},\mathrm{2}\rangle);\,\cdots)} does not bind \ensuremath{\Varid{y}} to a value, but if we float the \ensuremath{\exists} out
we get \ensuremath{\exists\Varid{x}.\,\Varid{y}\mathrel{=}\langle{}\Varid{x},\mathrm{2}\rangle;\,\cdots}, and now we may be able to make progress by substituting for \ensuremath{\Varid{y}}.

Similarly, rules \rulename{unify-seql} and \rulename{unify-seqr} move sequencing out of unification,
again to make it more likely to get the form \ensuremath{(\Varid{x}\mathrel{=}\Varid{v})}.  Notice that \ensuremath{\Varid{e}_{\mathrm{1}}}, \ensuremath{\Varid{e}_{\mathrm{2}}} and \ensuremath{\Varid{e}_{\mathrm{3}}} stay in the
same left-to-right order, there is no risk of re-ordering choices (c.f. \cref{sec:mixing}).

Rules \rulename{unify-unifyl} and \rulename{unify-unifyr} move a unification between values out of another unification.
Without these rules \ensuremath{(\Varid{x}\mathrel{=}\Varid{y})\mathrel{=}\mathrm{1};\,\Varid{y}\mathbin{>}\mathrm{0}} would be stuck; but \rulename{unify-unifyl}
rewrites it to \ensuremath{((\Varid{x}\mathrel{=}\Varid{y});\,(\Varid{y}\mathrel{=}\mathrm{1});\,\Varid{y}\mathbin{>}\mathrm{0}}, thus exposing \ensuremath{\Varid{y}\mathrel{=}\mathrm{1}} for further use by \rulename{subst}. Furthermore, in order
to prove confluence, we need to be able to freely move the equation \ensuremath{(\Varid{v}_{\mathrm{1}}\mathrel{=}\Varid{v}_{\mathrm{2}})},
as discussed in \cref{sec:conf-move-eqn}.

\subsection{Local substitution} \label{sec:local-subst}

Consider this (extremely) tricky term:
$$
\ensuremath{\exists\Varid{x}.\,\Varid{x}\mathrel{=}\textbf{if}\;(\Varid{x}\mathrel{=}\mathrm{0};\,\Varid{x}\mathbin{>}\mathrm{1})\;\textbf{then}\;\mathrm{33}\;\textbf{else}\;\mathrm{55}}
$$
What should this do? At first you might think it was stuck; how can we simplify the \ensuremath{\textbf{if}}
when its condition mentions \ensuremath{\Varid{x}} which is not yet defined?  But in fact, rule \rulename{subst} allows
us to substitute \emph{locally} in any X-context surrounding the equality \ensuremath{(\Varid{x}\mathrel{=}\mathrm{0})} thus:
$$
\begin{array}{l@{\;}l}
                                & \ensuremath{\exists\Varid{x}.\,\Varid{x}\mathrel{=}\textbf{if}\;(\Varid{x}\mathrel{=}\mathrm{0};\,\Varid{x}\mathbin{>}\mathrm{1})\;\textbf{then}\;\mathrm{33}\;\textbf{else}\;\mathrm{55}} \\
\movestoR{subst}                & \ensuremath{\exists\Varid{x}.\,\Varid{x}\mathrel{=}\textbf{if}\;(\Varid{x}\mathrel{=}\mathrm{0};\,\mathrm{0}\mathbin{>}\mathrm{1})\;\textbf{then}\;\mathrm{33}\;\textbf{else}\;\mathrm{55}} \\
\movestoR{fail}                 & \ensuremath{\exists\Varid{x}.\,\Varid{x}\mathrel{=}\textbf{if}\;\textbf{fail}\;\textbf{then}\;\mathrm{33}\;\textbf{else}\;\mathrm{55}}\\
\movestoC{\text{simplify \ensuremath{\textbf{if}}}} & \ensuremath{\exists\Varid{x}.\,\Varid{x}\mathrel{=}\mathrm{55}} \\
\movestoR{def-eliml}            & \ensuremath{\mathrm{55}}
\end{array}
$$
Minor variants of the same example get stuck instead of reducing. For example, if you replace
the \ensuremath{(\Varid{x}\mathrel{=}\mathrm{0})} with \ensuremath{(\Varid{x}\mathrel{=}\mathrm{100})} then rewriting gets stuck, as the reader may verify; and yet there
is a solution to the equations, namely $x=55$.
But if you replace \ensuremath{(\Varid{x}\mathrel{=}\mathrm{0})} with \ensuremath{(\Varid{x}\mathrel{=}\mathrm{55})} then rewriting again
gets stuck, and reasonably so, since in this case there are \emph{no} valid solutions to the equations.

It would be nicer if all three variants got stuck. Perhaps we can simplify the system, by only allowing
substitution over the entire scope of the \ensuremath{\exists}, and combining it with \rulename{def-eliml} thus:
$$
\rulename{global-subst} \quad \ensuremath{\exists\Varid{x}.\,X [\mskip1.5mu \Varid{x}=\Varid{v}\mskip1.5mu]}  \movesto  \ensuremath{(\subst{X}{\Varid{x}}{\Varid{v}}) [\mskip1.5mu \Varid{v}\mskip1.5mu]}
        \quad \text{if $\ensuremath{\Varid{x}} \in \freevars{X}$, $\ensuremath{\Varid{x}} \notin \freevars{v}$} \\
$$
Now all three variants get stuck.  But alas, we lose confluence.  Consider this term:
$$
\ensuremath{\exists\Varid{x}.\,\Varid{x}\mathrel{=}\textbf{if}\;(\exists\Varid{y}.\,\Varid{y}\mathrel{=}\Varid{x};\,\Varid{y}\mathrel{=}\mathrm{0};\,\Varid{y}\mathbin{>}\mathrm{1})\;\textbf{then}\;\mathrm{33}\;\textbf{else}\;\mathrm{55}}
$$
In the condition of the \ensuremath{\textbf{if}} we have introduced a local variable \ensuremath{\Varid{y}}, and set it equal to \ensuremath{\Varid{x}}.
Suppose we use \rulename{global-subst} instead of \rulename{subst}.  Then we can rewrite this term in two
different ways:
\begin{itemize}
\item Use \rulename{global-subst} on \ensuremath{(\Varid{y}\mathrel{=}\Varid{x})}; doing so recovers the original tricky term, which is
stuck, because \rulename{global-subst} cannot rewrite it.
\item Use \rulename{global-subst} on \ensuremath{(\Varid{y}\mathrel{=}\mathrm{0})}; then we can simplify the \ensuremath{\textbf{if}}, and reduce the entire thing to \ensuremath{\mathrm{55}}.
\end{itemize}
We have lost confluence: two different rewrites starting from the same term lead to non-joinable terms.
This is what drives us to adopt the tricky local-substitution rule \rulename{subst} rather than
the simpler \rulename{global-subst}: to establish confluence.

\subsection{Rules for choice} \label{sec:choice-rules}

The rules for choice are given in \cref{fig:choice}.
Rules \rulename{one-value}, \rulename{one-choice}
and \rulename{one-fail} describe the semantics for \ensuremath{\textbf{one}}, just as in \cref{sec:one}.
Similarly \rulename{all-fail}, \rulename{all-choice} and \rulename{all-value} describe
the semantics of \ensuremath{\textbf{all}} (\cref{sec:all}).

The most interesting rule is \rulename{choose} which, just as described in \cref{sec:choice},
``floats the choice outwards'', duplicating the surrounding context.
But what ``surrounding context'' precisely? We use two new contexts, \ensuremath{\Conid{SX}} and \ensuremath{CX}, both
defined in \cref{fig:syntax}. A \emph{choice context} \ensuremath{CX} is like an
execution context \ensuremath{X}, \emph{but with no possible
  choices to the left of the hole}:
$$
CX ::= \hole \vb \ensuremath{CX=\Varid{e}} \vb \ensuremath{\Varid{ce}=CX} \vb \ensuremath{CX;\,\Varid{e}} \vb \ensuremath{\Varid{ce};\,CX}
\vb \ensuremath{\exists\Varid{x}.\,CX}
$$
Here, \ensuremath{\Varid{ce}} is guaranteed-choice-free expression (syntax in \cref{fig:syntax}).
This syntactic condition is necessarily conservative; for example, a call \ensuremath{\Varid{f}(\Varid{x})}
is considered not guaranteed-choice-free, because it depends on what function \ensuremath{\Varid{f}} does.
We must guarantee not to have choices to the left so that we preserve order---see \cref{sec:mixing}.

The context \ensuremath{\Conid{SX}} (\cref{fig:syntax}) is a \emph{scope context}; it
ensures that \ensuremath{CX} is as large as possible.  This is a subtle point:
without this restriction we lose confluence.  To see this, consider\footnote{Remember, \ensuremath{\textbf{if}} is syntactic sugar for
  a use of \ensuremath{\textbf{one}}, see \cref{sec:one}, but using \ensuremath{\textbf{if}} makes the example easier to understand.}:
$$
\begin{array}{l@{\;}l}
                                 & \ensuremath{\exists\Varid{x}.\,(\textbf{if}\;(\Varid{x}\mathbin{>}\mathrm{0})\;\textbf{then}\;\mathrm{55}\;\textbf{else}\;\mathrm{44});\,(\Varid{x}\mathrel{=}\mathrm{1});\,(\mathrm{77}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\mathrm{99})}\\
 \movestoR{subst}                & \ensuremath{\exists\Varid{x}.\,(\textbf{if}\;(\mathrm{1}\mathbin{>}\mathrm{0})\;\textbf{then}\;\mathrm{55}\;\textbf{else}\;\mathrm{44});\,(\Varid{x}\mathrel{=}\mathrm{1});\,(\mathrm{77}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\mathrm{99})}\\
 \movestoC{\text{simplify \ensuremath{\textbf{if}}}} & \ensuremath{\exists\Varid{x}.\,\mathrm{55};\,(\Varid{x}\mathrel{=}\mathrm{1});\,(\mathrm{77}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\mathrm{99})}\\
 \movestoR{seq}                  & \ensuremath{\exists\Varid{x}.\,(\Varid{x}\mathrel{=}\mathrm{1});\,(\mathrm{77}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\mathrm{99})}\\
% \movestoR{def-eliml}           & |1; (77 choice 99)|\\
 \movestoRR{def-eliml}{seq}      & \ensuremath{\mathrm{77}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\mathrm{99}}\\
\end{array}
$$
But suppose instead we floated the choice out, \emph{part-way}, like this:
$$
\begin{array}{ll}
\multicolumn{2}{l}{\ensuremath{\exists\Varid{x}.\,(\textbf{if}\;(\Varid{x}\mathbin{>}\mathrm{0})\;\textbf{then}\;\mathrm{55}\;\textbf{else}\;\mathrm{44});\,(\Varid{x}\mathrel{=}\mathrm{1});\,(\mathrm{77}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\mathrm{99})}}\\
\quad \movestoC{\text{Bogus \rulename{choose}}} & \ensuremath{\exists\Varid{x}.\,(\textbf{if}\;(\Varid{x}\mathbin{>}\mathrm{0})\;\textbf{then}\;\mathrm{55}\;\textbf{else}\;\mathrm{44});\,((\Varid{x}\mathrel{=}\mathrm{1});\,\mathrm{77})\;\rule[-0.3mm]{0.5mm}{2.5mm}\;((\Varid{x}\mathrel{=}\mathrm{1});\,\mathrm{99})}\\
\end{array}
$$
Now the \ensuremath{(\Varid{x}\mathrel{=}\mathrm{1})} is inside the choice branches, so we cannot use \rulename{subst} to rewrite the
condition of the \ensuremath{\textbf{if}}.  Nor can we use \rulename{choose} again to float the choice further out,
because the \ensuremath{\textbf{if}} is not guaranteed choice-free (for example, the branches might contains choices).
So, alas, we are stuck!
Our not-entirely-satisfying solution is to force \rulename{choose} to float the choice all the way
to the innermost enclosing scope construct; hence the \ensuremath{\Conid{SX}} in the rule.

Rule \rulename{choose} moves choices around; only \rulename{one-choice} and \rulename{all-choice}
\emph{decompose} choices.
So choice behaves a bit like a data constructor, or normal form, of the language.
For this reason we call
this approach \emph{constructive choice}, in contrast to approaches that eliminate choice by non-deterministically
picking one branch or the other, which immediately gives up confluence.

The rules for \ensuremath{\textbf{one}} and \ensuremath{\textbf{all}} expect multiple choices to be
normalized into a right-associative list of non-failing values, and
the administrative rules \rulename{assoc-choice}, \rulename{fail-l}
and \rulename{fail-r} bring nested choices into that form. But why do these rules
need a \ensuremath{\Conid{SX}} context? Again, they are needed to guarantee confluence. Suppose we had a rule \rulename{fail-l-no-sx}
that unconditionally rewrites \ensuremath{(\textbf{fail}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\Varid{e})} to \ensuremath{\Varid{e}}. Now consider these
two reduction sequences, starting from the same expression:
$$
\begin{array}{rll}
\ensuremath{\Varid{f}\langle\rangle;\,(\textbf{fail}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;(\mathrm{3}\mathrel{=}(\mathrm{1}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\mathrm{3})))} & \movestoR{fail-l-no-sx} & \ensuremath{\Varid{f}\langle\rangle;\,\mathrm{3}\mathrel{=}(\mathrm{1}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\mathrm{3})} \\
\ensuremath{\Varid{f}\langle\rangle;\,(\textbf{fail}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;(\mathrm{3}\mathrel{=}(\mathrm{1}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\mathrm{3})))} & \movestoR{choose} & \ensuremath{\Varid{f}\langle\rangle;\,(\textbf{fail}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;((\mathrm{3}\mathrel{=}\mathrm{1})\;\rule[-0.3mm]{0.5mm}{2.5mm}\;(\mathrm{3}\mathrel{=}\mathrm{3})))} \\
 & \movestoRn{ulit}{2}  & \ensuremath{\Varid{f}\langle\rangle;\,(\textbf{fail}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;(\textbf{fail}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\mathrm{3}))} \\
 & \movestoRn{fail-l-no-sx}{2} & \ensuremath{\Varid{f}\langle\rangle;\,\mathrm{3}}
\end{array}
$$
The first sequence gets stuck after one step\footnote{The strange \ensuremath{\Varid{f}\langle\rangle} prevents
  us using \rulename{choose} to float the \ensuremath{(\mathrm{1}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\mathrm{3})} upwards.}, while the second makes more progress;
and the two results are not joinable.

\subsection{\versecalc{} is lenient} \label{sec:lenient}

\versecalc{} is \emph{lenient} \cite{lenience},
not lazy (call-by-need), nor strict (call-by-value).
Under lenient
evaluation, everything is eventually evaluated, but functions can run
before their arguments have a value.
Consider a function
call \ensuremath{\Varid{f}(\Varid{e})}.  In \versecalc{} applications are in administrative normal form (ANF),
so we must actually write \ensuremath{(\Varid{x}\mathrel{=}\Varid{e};\,\Varid{f}(\Varid{x}))}. This expression will not return a value until
\ensuremath{\Varid{e}} reduces to a value: that is, everything is eventually evaluated.  But even so \ensuremath{\Varid{f}(\Varid{x})} can proceed
to beta reduce (\cref{sec:beta}), assuming we know the definition of \ensuremath{\Varid{f}}.

Lenience supports \emph{abstraction}.  For example, we can replace an expression \ensuremath{(\Varid{x}\mathrel{=}\langle{}\Varid{y},\mathrm{3}\rangle;\,\Varid{y}\mathbin{>}\mathrm{7})} by
$$
\ensuremath{\exists\Varid{f}.\,\Varid{f}\mathrel{=}(\lambda\langle{}\Varid{p},\Varid{q}\rangle.\,\Varid{p}\mathrel{=}\langle{}\Varid{q},\mathrm{3}\rangle;\,\Varid{q}\mathbin{>}\mathrm{7});\,\Varid{f}\langle\Varid{x},\Varid{y}\rangle}
$$
Here, we abstract over the free variables of the expression, and define a named function \ensuremath{\Varid{f}}.
Calling the function is just the same as writing the original expression.
This transformation would not be valid under call-by-value.

This is not just a way to get \emph{parallelism}, which was the original motivation for introducing
lenience in the data-flow language Id~\cite{lenience}; it affects
\emph{semantics}. Consider
$$
\ensuremath{\exists\Varid{f}\hspace{0.3mm}\Varid{x}\hspace{0.3mm}\Varid{y}.\,\Varid{f}\mathrel{=}(\lambda\Varid{p}.\,\Varid{x}\mathrel{=}\mathrm{7};\,\Varid{p});\,\Varid{y}\mathrel{=}(\textbf{if}\;(\Varid{x}\mathbin{>}\mathrm{0})\;\textbf{then}\;\mathrm{7}\;\textbf{else}\;\mathrm{8});\,\Varid{f}(\Varid{y})}
$$
Here, \ensuremath{\Varid{y}} does not get a value until \ensuremath{\Varid{x}} is known; but \ensuremath{\Varid{x}} does not get its
value (in this case \ensuremath{\mathrm{7}}) until \ensuremath{\Varid{f}} is called.  Without lenience this program would be stuck.
Laziness would be another possible design choice, one that is even more expressive, as we discuss in
\Cref{sec:laziness}.

\subsection{Evaluation strategy} \label{sec:eval-strategy}

The rewrite rules say which rewrites are \emph{allowed}, but they do not say which
rewrite rule to apply next if you want to execute the program. That is
the role of an \emph{evaluation strategy} that, given a closed term, says which redex
to rewrite first.  For example, in the pure lambda calculus, \emph{normal-order reduction},
sometimes called \emph{leftmost outermost reduction}, is an evaluation strategy that
guarantees to terminate if any strategy does so.

Any decent evaluation strategy should (a) guarantee to terminate if there
is \emph{any} terminating sequence of reductions; and (b) be amenable
to compilation into efficient code.
It would be even better if the strategy could (c) guarantee to find the
result in the minimal number of rewrite steps---so called ``optimal reduction'' \cite{levy1978reductions,lamping1989algorithm,asperti1998optimal}---but optimal reduction is typically very hard, even in theory, and
invariably involves reducing under lambdas, so for practical purposes it is well out of reach.

Formalising an evaluation strategy for \versecalc{} is beyond the scope of this paper, but
we can make some informal comments.  First, in service of (b) we envisage compiling
lambdas to code, and thus we never rewrite under a lambda \cite{pj-book}.
Second, it never makes sense to evaluate in the right-hand argument of
a choice\footnote{Except perhaps in parallel, of course.}, because
\versecalc{}'s strong-ordering semantics mean that we must first find out what the left-hand argument
is (especially, whether it fails) before the right-hand one can be used.
So the basic plan is: rewrite the leftmost-outermost redex, subject to these two constraints.

The trouble is that it is hard to say what the ``leftmost'' redex is.  For example in \ensuremath{(\Varid{e};\,\langle{}\Varid{x},\mathrm{3}\rangle\mathrel{=}\langle{}\mathrm{2},\Varid{y}\rangle)},
the equality may or may not be the leftmost redex, depending on whether \ensuremath{\Varid{e}} is stuck ({\ie}, contains no
redexes); and whether or not \ensuremath{\Varid{e}} is stuck is a not syntactic property, and
(worse) may depend not only on \ensuremath{\Varid{e}} itself, but on its context. Even worse, \ensuremath{\Varid{e}} may subsequently
become un-stuck when we rewrite the equality.
Any calculus in which a redex to the ``right'' may unblock one to the
``left''---that is, residuation---must grapple with this problem,
so we leave evaluation strategy and compilation for future work.

\section{Confluence and equational theory}
\label{sec:confluence}

The rules of our rewrite semantics can be applied anywhere, in any order, and they give meaning to programs without committing to a particular evaluation strategy. But then it had better be the case that no matter how the rules are applied, one always obtains the same result!

If the rewrite rules were terminating, then we would expect that all possible execution paths lead to the same normal form. But since {\versecalc} has recursion, we can aim for \emph{confluence} instead: If $e_2 \movesfrom^\ast e_1 \movesto^\ast e_3$, can we find $e_4$ so that $e_2 \movesto^\ast e_4 \movesfrom^\ast e_3$?

Confluence would give us the assurance that we will not get different results when choosing different rules, or get stuck with some rules and not with others.

\subsection{subst {\vs} subst, a critical pair}

The main source for possible non-confluence is rule \rulename{subst}: If we have multiple equations for a variable $x$ available, we can use any of them to rewrite. A typical example would be
\[ \ensuremath{\exists\Varid{x}.\,\Varid{x}\mathrel{=}\Varid{v}_{\mathrm{1}};\,\Varid{x}\mathrel{=}\Varid{v}_{\mathrm{2}};\,\Varid{e}} \]
which we can rewrite by \rulename{subst} (and administrative \rulename{def-eliml} and \rulename{seq}) to either
\[ \ensuremath{\Varid{v}_{\mathrm{1}}\mathrel{=}\Varid{v}_{\mathrm{2}};\,\subst{\Varid{e}}{\Varid{x}}{\Varid{v}_{\mathrm{1}}}} \quad \text{or} \quad \ensuremath{\Varid{v}_{\mathrm{2}}\mathrel{=}\Varid{v}_{\mathrm{1}};\,\subst{\Varid{e}}{\Varid{x}}{\Varid{v}_{\mathrm{2}}}}. \]
Can we still join these two expressions? It seems that the unification rules decompose an equality \ensuremath{\Varid{v}_{\mathrm{1}}\mathrel{=}\Varid{v}_{\mathrm{2}}} completely: Either it reduces to \ensuremath{\textbf{fail}}, which will swallow the whole expression, or to a sequence of equalities with a variable on (at least) one side. Using \rulename{subst} on these equations, we can join the occurrences of $v_1$ resp.\ $v_2$ in \ensuremath{\subst{\Varid{e}}{\Varid{x}}{\Varid{v}_{\mathrm{1}}}} resp.\ \ensuremath{\subst{\Varid{e}}{\Varid{x}}{\Varid{v}_{\mathrm{2}}}}.

Confluence seems to be within reach? Unfortunately, not quite, as we discuss next.

\subsection{Order of quantifiers, position of equations and variable names}
\label{sec:confluence-extra}


There are some sources of non-confluence that we consider insignificant. For example
\ensuremath{(\exists\Varid{x}.\,\Varid{x}\mathrel{=}\Varid{e}_{\mathrm{1}});\,(\exists\Varid{y}.\,\Varid{y}\mathrel{=}\Varid{e}_{\mathrm{2}})}
can be reduced to
\ensuremath{(\exists\Varid{x}\hspace{0.3mm}\Varid{y}.\,\Varid{x}\mathrel{=}\Varid{e}_{\mathrm{1}};\,\Varid{y}\mathrel{=}\Varid{e}_{\mathrm{2}})}
or
\ensuremath{(\exists\Varid{y}\hspace{0.3mm}\Varid{x}.\,\Varid{x}\mathrel{=}\Varid{e}_{\mathrm{1}};\,\Varid{y}\mathrel{=}\Varid{e}_{\mathrm{2}})}. These are different expressions, but they only differ in the order the variables are listed in \ensuremath{\exists}, and that is not a difference we care about. So we want to consider these expression equivalent. To that end we introduce separate rewrite system $\movestoX$, with rules in \cref{fig:rules-confluence}:

\begin{itemize}
\item \rulename{def-swap} makes the order of \ensuremath{\exists} irrelevant.
\item As we do not care about where, within the same $X$-context, an equality is asserted, we allow ourselves to move any such equation to the front of its $X$-context, using \rulename{unify-move}. This rule leaves $v_2$ in place, just like \rulename{def-eliml} does.
\item In the presence of an equation between two variables, we do not care which of the two variables is mentioned, nor the orientation of that equality, and hence \rulename{var-swap}; else we’d get non-confluence from \ensuremath{\exists\Varid{x}.\,\Varid{x}\mathrel{=}\Varid{y};\,\Varid{x}\mathrel{=}\Varid{z}}, which can reduce to \ensuremath{\Varid{y}\mathrel{=}\Varid{z};\,\Varid{z}} or to \ensuremath{\Varid{z}\mathrel{=}\Varid{y};\,\Varid{y}}.
\item We include rule \rulename{seq} in $\movestoX$, so that we can clean up after \rulename{unify-move}.
\end{itemize}
We can now consider two expressions the same up to these administrative differences if they are convertible via \ensuremath{\movestoX}.

\subsection{Recursion}%
\label{sec:confluence-and-recursion}

There remains still another source of non-confluence, due to recursion. Depending on whether we apply \rulename{subst} first to \ensuremath{\Varid{x}} or \ensuremath{\Varid{y}} we get these two reductions:
\[ \ensuremath{\langle{}\mathrm{1},\Varid{v}_{\Varid{y}}\rangle\movesfrom^{\ast}\exists\Varid{x}\hspace{0.3mm}\Varid{y}.\,\Varid{x}\mathrel{=}\langle{}\mathrm{1},\Varid{y}\rangle;\,\Varid{y}\mathrel{=}(\lambda\Varid{z}.\,\Varid{x});\,\Varid{x}\movesto^{\ast}\Varid{v}_{\Varid{x}}}\]
where $\ensuremath{\Varid{v}_{\Varid{y}}} \equiv \ensuremath{(\lambda\Varid{z}.\,\exists\Varid{y}.\,\Varid{y}\mathrel{=}(\lambda\Varid{z}.\,\langle{}\mathrm{1},\Varid{y}\rangle);\,\langle{}\mathrm{1},\Varid{y}\rangle)}$ and $\ensuremath{\Varid{v}_{\Varid{x}}} \equiv \ensuremath{\langle{}\mathrm{1},\lambda\Varid{z}.\,\exists\Varid{x}.\,\Varid{x}\mathrel{=}\langle{}\mathrm{1},(\lambda\Varid{z}.\,\Varid{x})\rangle;\,\Varid{x}\rangle}$.
%
%we can apply \rulename{subst} to $x$ first and obtain the following reduction:
%\[
%|-> def y (y=(lam z (tup (1,y))); tup (1,y)) -> def y (y=v_y; tup (1,y)) -> tup (1,v_y)|
%\]
%where $|v_y| \equiv |(lam z (def y (y = (lam z (tup (1,y))); tup (1,y))))|$.
%Alternatively, we can apply \rulename{subst} to |y = (lam z x)| first, and obtain
%\[
%| -> def x (x=tup (1,lam z x); x) -> def x (x=v_x; x) -> v_x|
%\]
%where $|v_x| \equiv |tup (1,lam z (def x (x = tup (1,(lam z x)); x)))|$.

Can we join \ensuremath{\langle{}\mathrm{1},\Varid{v}_{\Varid{y}}\rangle} and \ensuremath{\Varid{v}_{\Varid{x}}}? Note that $\ensuremath{\Varid{v}_{\Varid{x}}} \movesto^\ast \ensuremath{\langle{}\mathrm{1},\lambda\Varid{z}.\,\Varid{v}_{\Varid{x}}\rangle}$ and $\ensuremath{\Varid{v}_{\Varid{y}}} \movesto^\ast \ensuremath{\lambda\Varid{z}.\,\langle{}\mathrm{1},\Varid{v}_{\Varid{y}}\rangle}$, so we get
\[
\begin{array}{c@{\,}c@{\,}c@{\,}c@{\,}c@{\,}c@{\,}c@{\,}c}
& \ensuremath{\langle{}\mathrm{1},\Varid{v}_{\Varid{y}}\rangle} & \movesto^\ast & \ensuremath{\langle{}\mathrm{1},\lambda\Varid{z}.\,\langle{}\mathrm{1},\Varid{v}_{\Varid{y}}\rangle\rangle} & \movesto^\ast & \ensuremath{\langle{}\mathrm{1},\lambda\Varid{z}.\,\langle{}\mathrm{1},(\lambda\Varid{z}.\,\langle{}\mathrm{1},\Varid{v}_{\Varid{y}}\rangle)\rangle\rangle} & \movesto^\ast & \cdots \\
\ensuremath{\Varid{v}_{\Varid{x}}}     & \movesto^\ast & \ensuremath{\langle{}\mathrm{1},\lambda\Varid{z}.\,\Varid{v}_{\Varid{x}}\rangle} & \movesto^\ast & \ensuremath{\langle{}\mathrm{1},\lambda\Varid{z}.\,\langle{}\mathrm{1},(\lambda\Varid{z}.\,\Varid{v}_{\Varid{x}})\rangle\rangle} & \movesto^\ast && \cdots
\end{array}
\]
and we find that we \emph{cannot} join them; they keep “missing each other”. The relation $\movesto$ is \emph{not} confluent.
%
But notice that as we keep evaluating these fixed points, they produce the same “infinite expression” \ensuremath{\langle{}\mathrm{1},\lambda\Varid{z}.\,\langle{}\mathrm{1},(\lambda\Varid{z}.\,\cdots)\rangle\rangle}. Put differently, we can reduce the two expressions be equal up to an arbitrary depth, which is good enough.

\begin{definition}[Depth cutting]
To make this formal, we define a function on terms and values that removes subexpressions at depth $n$ (measured by counting nested lambdas), defined by
$\ensuremath{\operatorname{cut}_{\mathrm{0}}(\Varid{e})} \coloneqq \ensuremath{\textbf{fail}}$, $\ensuremath{\operatorname{cut}_{\Varid{n}\mathbin{+}\mathrm{1}}(\lambda\Varid{x}.\,\Varid{e})} \coloneqq \ensuremath{(\lambda\Varid{x}.\,\operatorname{cut}_{\Varid{n}}(\Varid{e}))}$ and compositionally otherwise.
%When we need to write the function as a relation, we use |cutr n|.
The relation \ensuremath{\equiv_{\Varid{n}}} is the equivalence relation defined by \ensuremath{\Varid{e}_{\mathrm{1}}\equiv_{\Varid{n}}\Varid{e}_{\mathrm{2}}} iff \ensuremath{\operatorname{cut}_{\Varid{n}}(\Varid{e}_{\mathrm{1}})\mathrel{\equiv}\operatorname{cut}_{\Varid{n}}(\Varid{e}_{\mathrm{2}})}.
\end{definition}

Clearly for every $n \in \mathbb N$ we can find \ensuremath{\Varid{e}_{\mathrm{1}}} and \ensuremath{\Varid{e}_{\mathrm{2}}} so that \ensuremath{\langle{}\mathrm{1},\Varid{v}_{\Varid{y}}\rangle\movesto^{\ast}\Varid{e}_{\mathrm{1}}\equiv_{n}\Varid{e}_{\mathrm{2}}\movesfrom^{\ast}\Varid{v}_{\Varid{x}}} and this is the notion of confluence we aim for.

\subsection{Confluence}

So although \ensuremath{\to } is not confluent on its own, we can say that it is confluent up to \ensuremath{\movestoX} and \ensuremath{\equiv_{\Varid{n}}}:

\begin{definition}
We consider \ensuremath{\Varid{e}_{\mathrm{1}}} and \ensuremath{\Varid{e}_{\mathrm{2}}} to be \emph{joinable}, written \ensuremath{\Varid{e}_{\mathrm{1}}\mathrel{\sim}\Varid{e}_{\mathrm{2}}}, if for every $n \in \mathbb N$ there exist \ensuremath{\Varid{e}_{\mathrm{3}}}, \ensuremath{\Varid{e}_{\mathrm{4}}}, \ensuremath{\Varid{e}_{\mathrm{5}}} and \ensuremath{\Varid{e}_{\mathrm{6}}} so that
\[
\ensuremath{\Varid{e}_{\mathrm{1}}\movesto^{\ast}\Varid{e}_{\mathrm{3}}\;\movestoX^{\ast}\;\Varid{e}_{\mathrm{4}}\equiv_{n}\Varid{e}_{\mathrm{5}}\;\movesfromX^{\ast}\;\Varid{e}_{\mathrm{6}}\movesfrom^{\ast}\Varid{e}_{\mathrm{2}}}.
\]
\end{definition}

\begin{proposition}[Confluence]
\label{thm:confluence}%
For all \ensuremath{\Varid{e}_{\mathrm{2}}\movesfrom^{\ast}\Varid{e}_{\mathrm{1}}\movesto^{\ast}\Varid{e}_{\mathrm{3}}} we have \ensuremath{\Varid{e}_{\mathrm{2}}\mathrel{\sim}\Varid{e}_{\mathrm{3}}}.
\end{proposition}

In general, confluence up to an equivalence relation is not necessarily the same as Church-Rosser up-to-an equivalence relation. For \versecalc, we believe both hold:

\begin{proposition}[Church-Rosser]
\label{thm:church-rosser}%
\ensuremath{\Varid{e}_{\mathrm{1}}\mathrel{\convertible}\Varid{e}_{\mathrm{2}}} implies \ensuremath{\Varid{e}_{\mathrm{1}}\mathrel{\sim}\Varid{e}_{\mathrm{2}}}.
\end{proposition}

Because {\versecalc} is non-terminating, confluence does not simply follow from local confluence ({\ie}, \ensuremath{\Varid{e}_{\mathrm{2}}\leftarrow \Varid{e}_{\mathrm{1}}\to \Varid{e}_{\mathrm{3}}} $\implies$ \ensuremath{\Varid{e}_{\mathrm{2}}\mathrel{\sim}\Varid{e}_{\mathrm{3}}}), and requires a more involved proof, which is still work in progress.
%(see \cref{sec:confluence-proof-plan} for a high-level proof plan).
%
We have, however, reasons to be optimistic: our incomplete proof
effort has already helped us root out various forms of unwanted
non-confluence. Moreover, we have an implementation of the complete
reduction semantics in Haskell and we are using property-based testing
with QuickCheck \citep{quickcheck} to search for non-confluence.

\subsection{Equational theory}

The relation \ensuremath{\mathrel{\sim}} gives rise to an equational theory for {\versecalc}. Assuming our confluence conjecture holds, it is a congruent equivalence relation and and contains all reductions. Furthermore, we expect that this equivalence is respected by substitution,
\[ \ensuremath{\Varid{e}_{\mathrm{1}}\mathrel{\sim}\Varid{e}_{\mathrm{2}}} \text{ implies }  \ensuremath{\subst{\Varid{e}_{\mathrm{1}}}{\Varid{x}}{\Varid{v}}\mathrel{\sim}\subst{\Varid{e}_{\mathrm{2}}}{\Varid{x}}{\Varid{v}}}, \]
allows equational reasoning with local equations,
\[ \ensuremath{\Varid{v}_{\mathrm{1}}\mathrel{=}\Varid{v}_{\mathrm{2}};\,(\subst{\Varid{e}}{\Varid{x}}{\Varid{v}_{\mathrm{1}}})\mathrel{\sim}\Varid{v}_{\mathrm{1}}\mathrel{=}\Varid{v}_{\mathrm{2}};\,(\subst{\Varid{e}}{\Varid{x}}{\Varid{v}_{\mathrm{2}}})}, \]
and even allows rewriting with a local equation within that equation's right-hand side,
\[ \ensuremath{\Varid{v}_{\mathrm{1}}\mathrel{=}(\subst{\Varid{v}_{\mathrm{2}}}{\Varid{x}}{\Varid{v}_{\mathrm{1}}})\mathrel{\sim}\Varid{v}_{\mathrm{1}}\mathrel{=}(\subst{\Varid{v}_{\mathrm{2}}}{\Varid{x}}{\subst{\Varid{v}_{\mathrm{2}}}{\Varid{x}}{\Varid{v}_{\mathrm{1}}}})}. \]

The joinability relation \ensuremath{\mathrel{\sim}} is still weaker than \emph{contextual equivalence}, which relates expressions that are observably indistinguishable, even if they are not directly joinable. One possible definition of contextual equivalence is that $e_1$ and $e_2$ are contextually equivalent, written \ensuremath{\Varid{e}_{\mathrm{1}}\mathrel{\simeq}\Varid{e}_{\mathrm{3}}}, if for all closing contexts $C$, if $C[e_1] \movesto^\ast \ensuremath{\textbf{fail}}$ then $C[e_2] \movesto^\ast \ensuremath{\textbf{fail}}$. We expect that \ensuremath{\mathrel{\simeq}} not only contains \ensuremath{\mathrel{\sim}}, but admits additional useful equations, such as the commutativity of sequencing
(\ensuremath{(\Varid{e}_{\mathrm{1}};\,\Varid{e}_{\mathrm{2}});\,\Varid{e}_{\mathrm{3}}\mathrel{\simeq}\Varid{e}_{\mathrm{1}};\,(\Varid{e}_{\mathrm{2}};\,\Varid{e}_{\mathrm{3}})}),
naming of subexpressions
(\ensuremath{\Varid{e}\mathrel{\simeq}(\exists\Varid{x}.\,\Varid{x}\mathrel{=}\Varid{e};\,\Varid{x})} if $x \notin\freevars{e}$),
and the removal of repeated choice-free expressions
(\ensuremath{(\Varid{ce};\,\Varid{ce})\mathrel{\simeq}\Varid{ce}}).  The development of this theory is future work.

\subsection{Fixed points}

To conclude this section on metatheory, let use have a closer look at \rulename{subst-rec}, and consider an equation \ensuremath{\Varid{x}\mathrel{=}\Varid{v}} that is not failed by the occurs check, {\ie}, \ensuremath{\Varid{x}} may occur in \ensuremath{\Varid{v}}, but only under lambda. Here, \rulename{subst-rec} changes of \emph{some} \ensuremath{(\lambda\Varid{y}.\,\Varid{e})} in \ensuremath{\Varid{v}}, not nested under another lambda and with $x\in\freevars{e}$, to \ensuremath{(\lambda\Varid{y}.\,\exists\Varid{x}.\,\Varid{x}\mathrel{=}\Varid{v};\,\Varid{e})}. Let \ensuremath{(\mu\Varid{x}.\,\Varid{v})} denote the expression where we do this to \emph{every} such lambda simultaneously. This is not quite the same expression that we get by applying \rulename{subst-rec} exhaustively, as that may also affect lambdas nested more deeply, but they are joinable, {\ie}, \ensuremath{\Varid{x}\mathrel{=}\Varid{v}\mathrel{\sim}\Varid{x}\mathrel{=}(\mu\Varid{x}.\,\Varid{v})}.
%
This expression behaves much as one would expect from a fixed-point operator:
\begin{itemize}
\item $\freevars{\ensuremath{\mu\Varid{x}.\,\Varid{v}}} = \freevars v \setminus \{x\}$
\item If $x \notin \freevars{v}$ then $\ensuremath{(\mu\Varid{x}.\,\Varid{v})\mathrel{\equiv}\Varid{v}}$.
\item The fixpoint can be unrolled: \ensuremath{\mu\Varid{x}.\,\Varid{v}\mathrel{\sim}\subst{\Varid{v}}{\Varid{x}}{(\mu\Varid{x}.\,\Varid{v})}}.
\end{itemize}

In the example from \cref{sec:confluence-and-recursion}, we find \ensuremath{\Varid{v}_{\Varid{y}}\mathrel{\equiv}(\mu\Varid{y}.\,\lambda\Varid{x}.\,\langle{}\mathrm{1},\Varid{y}\rangle)} and \ensuremath{\Varid{v}_{\Varid{x}}\mathrel{\equiv}(\mu\Varid{x}.\,\langle{}\mathrm{1},(\lambda\Varid{y}.\,\Varid{x})\rangle)} and we can join \ensuremath{\langle{}\mathrm{1},\Varid{v}_{\Varid{y}}\rangle\mathrel{\sim}\Varid{v}_{\Varid{x}}} using a general lemma about partially unrolling a fixed point:
\[
\ensuremath{\mu\Varid{x}.\,\subst{\Varid{v}_{\mathrm{1}}}{\Varid{y}}{\Varid{v}_{\mathrm{2}}}\mathrel{\sim}\subst{\Varid{v}_{\mathrm{1}}}{\Varid{y}}{(\mu\Varid{y}.\,\subst{\Varid{v}_{\mathrm{2}}}{\Varid{x}}{\Varid{v}_{\mathrm{1}}})}} \quad \text{if } x \notin \freevars{v_1}, y \notin\freevars{v_2}.
\]

It is satisfying that in {\versecalc} a form of fixed-point operator emerges, without us having to add it explicitly to the syntax of values. In particular, because \ensuremath{(\mu\Varid{x}.\,\Varid{v})} and \ensuremath{\Varid{v}} look the same until you peek under a lambda, we do not need rules dedicated to unrolling \ensuremath{(\mu\Varid{x}.\,\Varid{v})} in equations or applications. The design choice to allow recursion through lambdas, but not recursive (co-inductive) data structures, plays a role here.


\section{\versecalc{} in context: reflections and related work} \label{sec:related}

Functional logic programming has a rich literature; an excellent
starting point is Hanus's survey~\cite{hanus:survey}.  Now that we
know what \versecalc{} is, we can identify its distinctive features,
and compare them to other approaches.

\subsection{Choice and non-determinism} \label{sec:deterministic}

A significant difference between our presentation and earlier works is our
treatment of choice. Consider an expression like \ensuremath{(\mathrm{3}\mathbin{+}(\mathrm{20}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\mathrm{30}))}.
This is typically handled by a pair of non-deterministic rewrite rules:
$$
\ensuremath{\Varid{e}_{\mathrm{1}}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\Varid{e}_{\mathrm{2}}} \movesto \ensuremath{\Varid{e}_{\mathrm{1}}} \hspace{1cm} \ensuremath{\Varid{e}_{\mathrm{1}}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\Varid{e}_{\mathrm{2}}} \movesto \ensuremath{\Varid{e}_{\mathrm{2}}}
$$
So our expression rewrites (non-deterministically) to either \ensuremath{(\mathrm{3}\mathbin{+}\mathrm{20})} or \ensuremath{(\mathrm{3}\mathbin{+}\mathrm{30})};
and that in turn allows the addition to make progress.
Of course, including non-deterministic choice means the rules are non-confluent by
construction.  Instead, one must generalize to say that a reduction does not change the
\emph{set} of results; in the context of lambda calculi see
for example \cite{schmidt-schauss:non-det1,schmidt-schauss:non-det2}.

In contrast, our rules never pick one side or the other of a choice.
And yet \ensuremath{(\mathrm{3}\mathbin{+}(\mathrm{20}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\mathrm{30}))} can still make progress, by floating out the
choice (rule \rulename{choose} in \cref{fig:general-axioms}), thus
\ensuremath{(\mathrm{3}\mathbin{+}\mathrm{20})\;\rule[-0.3mm]{0.5mm}{2.5mm}\;(\mathrm{3}\mathbin{+}\mathrm{30})}.  In effect, \emph{choices are laid out in space} (in
the syntax of the term), rather than being explored by
non-deterministic selection.  Rule \rulename{choose} is not a new idea:
it is common in calculi with choice, see {\eg}, ~\citetext{\citealp[Section 6.1]{liguoro:non-det}, \citealp[Section 3]{lago:probabilistic}}
and, more recently, has been used to describe functional logic languages where it is variously
called \emph{bubbling} \cite{antoy:bubbling} or \emph{pull-tabbing} \cite{antoy:pull-tabbing}.
However, our formulation appears simpler, because we avoid the need for attaching
an identifier to each choice with its attendant complications.

\subsection{One and all}

Logical variables, choice, and equalities are present in many functional logic
languages.  However \ensuremath{\textbf{one}} and \ensuremath{\textbf{all}} are distinctive features of \versecalc{},
with the notable exception of Smolka {\etal}'s language Fresh.
Introduced in a technical report nearly 40 years ago~\cite{smolka:fresh},
Fresh has \emph{confinement} (equivalent to \ensuremath{\textbf{one}}) and \emph{collection} (equivalent to \ensuremath{\textbf{all}}).
It is a very interesting design, but one does not appear to have been implemented,
and its treatment of equality and thus logical variables is rather different to ours.

Several aspects of \ensuremath{\textbf{all}} and \ensuremath{\textbf{one}} are worth noting.
First, \ensuremath{\textbf{all}} \emph{reifies} choice (a control operator) into a tuple (a data structure);
for example, \ensuremath{\textbf{all}\{\mathrm{1}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\mathrm{7}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\mathrm{2}\}} returns the tuple \ensuremath{\langle{}\mathrm{1},\mathrm{7},\mathrm{2}\rangle}.
In the other direction, indexing turns a tuple into choice ({\eg}, \ensuremath{\exists\Varid{i}.\,\langle{}\mathrm{1},\mathrm{7},\mathrm{2}\rangle(\Varid{i})} yields \ensuremath{(\mathrm{1}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\mathrm{7}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\mathrm{2})}).
Other languages can reify choices into a (non-deterministic)
list, via an operator called \text{\ttfamily bagof}, or a mechanism called
\emph{set-functions} in an extension of Curry \cite[Section 4.2.7]{curry:tutorial},
implemented in the Kiel Curry System interpreter \cite{brassel:encapsulated,kics,hanus:set-functions}.
But this is regarded as a somewhat sophisticated feature,
whereas it is part of the foundational fabric of \versecalc{}.
Curry's set-functions need careful explanation about sharing across non-deterministic choices,
or what is ``inside'' and ``outside'' the set function, something that appears as a straightforward
consequence of \versecalc{}'s single rule \rulename{choose}.

Second, even under the reification of \ensuremath{\textbf{all}},
\versecalc{} is \emph{deterministic}.  Choice is not
non-deterministic: \versecalc{} takes pains to maintain order, so that
when reifying choice into a tuple, the order of elements in that
tuple is completely determined.  Anything else would leak
non-determinism into the language.  This determinism has a price: as we saw in \cref{sec:mixing}
and \cref{sec:choice-rules}, we have to take care to maintain the
left-to-right order of choices.
However, maintaining that order has other payoffs. For example, it is relatively easy to add
effects other than choice, including mutable variables and input/output, to \versecalc{}.
%% To substantiate this claim, \cref{sec:store} gives the additional syntax and rewrite rules
%% for mutable variables.

Thirdly, \ensuremath{\textbf{one}} allows us to reify failure; to try something and take different actions
depending on whether or not it succeeds. Prolog's ``cut'' operator has a similar flavour,
and Curry's set-functions allow one to do the same thing.

Finally, \ensuremath{\textbf{one}} and \ensuremath{\textbf{all}} neatly encapsulate the idea of ``flexible'' {\vs} ``rigid'' logical variables.
As we saw in \cref{sec:rigid}, logical variables bound outside \ensuremath{\textbf{one}}/\ensuremath{\textbf{all}} cannot be unified inside it;
they are ``rigid.''  This notion is nicely captured by the execution context \ensuremath{X}, which limits the
scope of the \rulename{subst} rule, and specifically \emph{excludes} \ensuremath{\textbf{one}} and \ensuremath{\textbf{all}}.

\subsection{The semantics of logical variables} \label{sec:magical}

Our logical variables, introduced by \ensuremath{\exists}, are often called
\emph{extra variables} in the literature, because they
are typically introduced as variables that appear on the right-hand side of a function
definition, but are not bound on the left.  For example, in Curry we can write
\begin{tabbing}\ttfamily
~~~~~~first~x~\char124{}~x~\char61{}\char58{}\char61{}~\char40{}a\char44{}b\char41{}~\char61{}~a~where~a\char44{}b~free
\end{tabbing}
Here \text{\ttfamily a} and \text{\ttfamily b} are logical variables, not bound on the left; they
get their values through unification (written ``\text{\ttfamily \char61{}\char58{}\char61{}}'').   In Curry they are explicitly introduced
by the ``\text{\ttfamily where~a\char44{}b~free}'' clause, while in many other papers their
introduction is implicit in the top-level rules, simply by not being bound
on the left.  Logical variables are
at the heart of the ``logic'' part of functional logic programming.

Constructor-based ReWrite Logic (CRWL) \cite{crwl} is the brand leader for
high-level semantics for non-strict, non-deterministic functional logic languages.
%
CRLW is a ``big-step'' rewrite semantics that rewrites a term to a value in a single
step.
%
\Citet{fraguas:rewriting} make a powerful case for instead giving the semantics
of a functional logic language using ``small-step'' rewrite rules, more like those
of the lambda calculus, that successively rewrite the term, one step at
a time, until it reaches a normal form.
Their paper does exactly this, and proves equivalence to the
CRWL framework.  Their key insight (like us, inspired by
\citet{ariola:call-by-need}'s formalisation of the call-by-need lambda calculus)
is to use \ensuremath{\mathbf{let}} to make sharing explicit.

However both CRWL and Fraguas {\etal} suffer from a major problem:
they require something we call \emph{magical rewriting}. A key rewrite rule is this:
$$
\begin{array}{l}
f( \theta(e_1),\ldots, \theta(e_n) ) \movesto \theta( rhs ) \\
\text{if}
\begin{array}[t]{l}
  (e_1, \ldots, e_n) \movesto rhs
   \; \text{is a top-level function binding, and} \\
\text{$\theta$ is a substitution mapping variables to closed values, s.t
$dom(\theta) = \mathit{fvs}(e_1, \ldots, e_n, rhs)$}
\end{array}
\end{array}
$$
The substitution for the free variables of the left-hand side can
readily be chosen by matching the left-hand side against the call.
But the substitution for the extra variables must be chosen ``magically''
\cite[Section 7]{fraguas:rewriting} or clairvoyantly, so as to make the future execution
work out. This is admirably high level, because it hides everything
about unification, but it is not much help to a programmer trying to
understand a program, nor is it directly executable.
In a subsequent journal paper they refine CRWL to avoid magical rewriting using
``let-narrowing'' \cite[Section 6]{fraguas:let-narrowing}; this system
looks rather different to ours, especially in its treatment of choice,
but is rather close in spirit.

To explain actual execution, the state of the art is described by
\citet{hanus:semantics}.  They give both a big-step operational
semantics (in the style of \citep{launchbury:natural}), and
a small-step operational semantics.  These two approaches both thread a
\emph{heap} through the execution, which holds the unification variables and their unification
state; the small-step semantics also has a \emph{stack}, to specify
the focus of execution.
%
The trouble is that heaps and stacks are difficult to explain to a programmer,
and do not make it easy to reason about program equivalence.
%
In addition to this machinery, the model is further complicated with
concurrency to account for residuation.

In contrast, our rewrite rules give a complete, executable ({\ie}, no ``magic'')
account of logical variables and choice, directly as small-step
rewrites on the original program, rather than as the evolution of a
(heap, control, stack) configuration. Moreover, we have no problem with
residuation.

\subsection{Flat {\vs} higher order}

When giving the semantics
of functional logic languages, a first-order presentation is almost universal.
User-defined functions can be defined at top level only;
and function symbols (the names of such functions) are syntactically
distinguished from ordinary variables. As Hanus describes, it is possible
to translate a higher-order program into a first-order
form\footnote{
  Hanus does not mention this, but for a language with arbitrarily nested lambdas
  one would need to do lambda-lifting as well, but that is a small point.}
using defunctionalisation~\cite[Section 3.3]{hanus:survey}, and a built-in
\ensuremath{\textbf{apply}} function.
Sadly, this encoding is hardly a natural rendition of the lambda calculus: 
it obstructs the goal of using rewrite rules to explain to programmers how
their program might execute. In contrast, a strength of our \versecalc{} 
presentation is that it deals natively with the full lambda calculus.

\subsection{Intermediate language}

Hanus's \emph{Flat Language}~\cite[Fig 1]{hanus:semantics}, FLC, plays the same role as \versecalc{}:
it is a small core language
into which a larger surface language can be desugared.
There are some common features: variables, literals, constructor applications,
and sequencing (written \text{\ttfamily hnf} in FLC).
However, it seems that \versecalc{} has a greater economy of concepts.
In particular, FLC
has two forms of equality \text{\ttfamily \char40{}\char61{}\char61{}\char41{}} and \text{\ttfamily \char40{}\char61{}\char58{}\char61{}\char41{}}, and two forms
of case-expression, \text{\ttfamily case} and \text{\ttfamily fcase}.  In each pair, the former suspends
if it encounters a logical variable; the latter unifies or narrows
respectively. In contrast, \versecalc{}
has a single equality \ensuremath{(\mathrel{=})}, and the orthogonal \ensuremath{\textbf{one}} construct, to deal with all four concepts.

FLC has let-expressions \text{\ttfamily \char40{}let~x\char61{}e~in~b\char41{}}, where \versecalc{} uses
\ensuremath{\exists} and (again) unification.
FLC also uses the same construct for a different purpose, to
bring a logical variable into scope, using the strange binding \text{\ttfamily x\char61{}x}, thus \text{\ttfamily \char40{}let~x\char61{}x~in~e\char41{}}.
In contrast, \ensuremath{\exists\Varid{x}.\,\Varid{e}} seems more direct.

\section{Variations and choices} \label{sec:variations}

%% \lennart{Mention binary {\vs} n-ary for choice and tuples?}

In a calculus like \versecalc{} there is room for many design variations.
We discuss some of them here.

\subsection{Unifying lambdas} \label{sec:unifying-lambda}

In \versecalc{} we say that any attempt to unify two lambdas (or primitive operations) fails---even if the
lambdas are semantically identical.  The problem, obviously, is that it is hard
to tell if they \emph{are} semantically identical.

We cannot say that the attempt to unify gets stuck, because that leads to non-confluence.
Here is an expression that rewrites in two different ways, depending on which equality we \rulename{subst} first:
\[ \ensuremath{(\lambda\Varid{p}.\,\mathrm{1})\mathrel{=}(\lambda\Varid{q}.\,\mathrm{2});\,\mathrm{1}\movesfrom^{\ast}\exists\Varid{x}.\,\Varid{x}\mathrel{=}(\lambda\Varid{p}.\,\mathrm{1});\,\Varid{x}\mathrel{=}(\lambda\Varid{q}.\,\mathrm{2});\,\Varid{x}\;\langle\rangle\movesto^{\ast}(\lambda\Varid{q}.\,\mathrm{2})\mathrel{=}(\lambda\Varid{p}.\,\mathrm{1});\,\mathrm{2}} \]
% $$
% \begin{array}{lll}
%  \multicolumn{3}{l}{|def x (x = (lam p 1); x = (lam q 2); x emptytup)|} \\
% &    \movesto \{\textit{use \rulename{SUBST} on first equality}\} & |(lam p 1) = (lam q 2); 1| \\
% OR & \movesto \{\textit{use \rulename{SUBST} on second equality}\} & |(lam q 2) = (lam p 1); 2|
% \end{array}
% $$
These two outcomes cannot be joined.  But making unification on lambdas fail, both outcomes
lead to \ensuremath{\textbf{fail}}, and confluence is restored.  A perhaps-unexpected consequence is that we cannot add a rule
$\rulename{urefl}\; \ensuremath{(\Varid{x}\mathrel{=}\Varid{x})} \movesto \ensuremath{\Varid{x}}$, because that would lead to non-confluence if \ensuremath{\Varid{x}} turns out to be a lambda.

\subsection{Administrative normal form}

We have chosen to restrict tuple formation and function application to
\emph{values} and not \emph{expressions}; thus we allow \ensuremath{\Varid{v}_{\mathrm{1}}(\Varid{v}_{\mathrm{2}})}, but not \ensuremath{\Varid{e}_{\mathrm{1}}(\Varid{e}_{\mathrm{2}})} unless
\ensuremath{\Varid{e}_{\mathrm{1}}} and \ensuremath{\Varid{e}_{\mathrm{2}}} are values; and similarly \ensuremath{\langle{}\Varid{v}_{\mathrm{1}},\Varid{v}_{\mathrm{2}}\rangle} but not \ensuremath{\langle{}\Varid{e}_{\mathrm{1}},\Varid{e}_{\mathrm{2}}\rangle}.
It would be entirely straightforward to allow expressions rather than values, but doing so
would require many more rules, to describe how to distribute, {\eg}, application over sequencing.
Restricting to values involves no loss of expressive power:
it is simply a form of administrative normal form (ANF) \cite{felleisen:anf}.

We could take the ANF idea a bit further, by restricting the syntax of equalities so
that the left-hand-side is always a value.
Thus $e ::= \ldots \vb \ensuremath{\Varid{v}\mathrel{=}\Varid{e}} \vb \ldots$. It is easy to get the program into this form,
by desugaring \ensuremath{(\Varid{e}_{\mathrm{1}}\mathrel{=}\Varid{e}_{\mathrm{2}})} to \ensuremath{(\exists\Varid{x}.\,\Varid{x}\mathrel{=}\Varid{e}_{\mathrm{1}};\,\Varid{x}\mathrel{=}\Varid{e}_{\mathrm{2}})}.
The payoff is that we can then drop \rulename{unify-seql} and \rulename{unify-unifyl}.
We have chosen not to do this, mainly because it makes \ensuremath{(\mathrel{=})} asymmetrical and adds a lot of clutter;
a debatable decision, but one of no great consequence.

\subsection{Dead existentials}

Consider the term \ensuremath{(\exists\Varid{x}.\,\mathrm{99})}.  We have no rule to rewrite this to \ensuremath{\mathrm{99}}; \rulename{def-eliml} comes closest,
but it insists that every existential is given a value. It would be easy to add a rule to discard an existential
whose bound variable is not used in its body, and so far as we know, doing so would not lead to
new difficulties for (say) confluence.  It is mostly a philosophical issue.  For example, the term
\ensuremath{(\exists\Varid{x}.\,\Varid{x}\mathrel{=}(\mathrm{1}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\mathrm{2});\,\mathrm{99})} rewrites to \ensuremath{(\mathrm{99}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\mathrm{99})}, producing two results, \emph{one for each
  solution for \ensuremath{\Varid{x}}}.  So, if \ensuremath{\Varid{x}} is entirely unconstrained, maybe we should return an infinite number of results?
For now, we make a conservative choice, and insist that every existential gets a value:
only \rulename{def-eliml} and \rulename{def-elimr} eliminate an existential.

\subsection{Ordering and choices}

As we discussed in \cref{sec:choice-rules}, rule \rulename{choose} is less than satisfying, for two reasons.
First, the \ensuremath{CX} context uses a conservative, syntactic analysis for choice-free expressions; and second,
the \ensuremath{\Conid{SX}} context is needed to force \ensuremath{CX} to be maximal.  A rule like this would be more satisfying:
$$
\rulename{simpler-choose} \quad   \ensuremath{X [\mskip1.5mu \Varid{e}_{\mathrm{1}}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\Varid{e}_{\mathrm{2}}\mskip1.5mu]} \movesto \ensuremath{X [\mskip1.5mu \Varid{e}_{\mathrm{1}}\mskip1.5mu]\;\rule[-0.3mm]{0.5mm}{2.5mm}\;X [\mskip1.5mu \Varid{e}_{\mathrm{2}}\mskip1.5mu]}
$$
The trouble with that is that it may change the order of the results (\cref{sec:mixing}).
Another possibility would be to accept that results may come out in the ``wrong'' order, but have
some kind of sorting mechanism to put them back into the ``right'' order.  Something like this:
$$
\rulename{labeled-choose} \quad   \ensuremath{X [\mskip1.5mu \Varid{e}_{\mathrm{1}}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\Varid{e}_{\mathrm{2}}\mskip1.5mu]} \movesto \ensuremath{X [\mskip1.5mu \Conid{L};\,\Varid{e}_{\mathrm{1}}\mskip1.5mu]\;\rule[-0.3mm]{0.5mm}{2.5mm}\;X [\mskip1.5mu \Conid{R};\,\Varid{e}_{\mathrm{2}}\mskip1.5mu]}
$$
Here the two branches are labeled with \ensuremath{\Conid{L}} and \ensuremath{\Conid{R}}.  We can add new rules to reorder such
labelled expressions, something in the spirit of
$$
\rulename{sort} \quad   \ensuremath{(\Conid{R};\,\Varid{e}_{\mathrm{1}})\;\rule[-0.3mm]{0.5mm}{2.5mm}\;(\Conid{L};\,\Varid{e}_{\mathrm{2}})} \movesto \ensuremath{(\Conid{L};\,\Varid{e}_{\mathrm{2}})\;\rule[-0.3mm]{0.5mm}{2.5mm}\;(\Conid{R};\,\Varid{e}_{\mathrm{1}})}
$$
We believe this can be made to work, and it would allow more programs to evaluate, but it adds unwelcome clutter to program terms, and the cure
may be worse than the disease.

\subsection{Generalizing \ensuremath{\textbf{one}} and \ensuremath{\textbf{all}}}

In \versecalc{}, we introduced \ensuremath{\textbf{one}} and \ensuremath{\textbf{all}} as the primitive choice-consuming operators, and neither is more general than the other, as discussed in \cref{sec:all}. We could have introduced a more general operator \ensuremath{\textbf{split}} as $e ::=  \ensuremath{\cdots} \vb \ensuremath{\textbf{split}\{\Varid{e},\Varid{v}_{\mathrm{1}},\Varid{v}_{\mathrm{2}}\}}$ and rules
\[\begin{array}{lrcll}
    \rulename{split-fail} & \ensuremath{\textbf{split}\{\textbf{fail},\Varid{f},\Varid{g}\}} & \movesto & \ensuremath{\Varid{f}\langle\rangle} \\
    \rulename{split-choice} & \ensuremath{\textbf{split}\{\Varid{v}\hspace{1mm}\mathop{\rule[-0.3mm]{0.5mm}{2.5mm}}\hspace{1mm}\Varid{e},\Varid{f},\Varid{g}\}} & \movesto & \ensuremath{\Varid{g}\langle\Varid{v},\lambda x.\,\Varid{e}\rangle} & \text{\ensuremath{\Varid{x}} fresh}  \\
    \rulename{split-value} & \ensuremath{\textbf{split}\{\Varid{v},\Varid{f},\Varid{g}\}} & \movesto & \ensuremath{\Varid{g}\langle\Varid{v},\lambda x.\,\textbf{fail}\rangle} \\
\end{array}\]
The intuition behind \ensuremath{\textbf{split}} is that it distinguishes a failing computation from one that returns at least one value. If \ensuremath{\Varid{e}} fails, it calls \ensuremath{\Varid{f}}, and if \ensuremath{\Varid{e}} returns at least one value, passes that to \ensuremath{\Varid{g}} together with the remaining computation, safely tucked away behind a lambda.

Indeed, this is more general, as we can implement \ensuremath{\textbf{one}} and \ensuremath{\textbf{all}} with \ensuremath{\textbf{split}}:
\[\begin{array}{r@{\;}l@{\;}l@{\;}l}
    \ensuremath{\textbf{one}\{\Varid{e}\}} & \equiv \ensuremath{\Varid{f}(\Varid{x})\mathrel{\coloneqq}\textbf{fail};\,} & \ensuremath{\Varid{g}\langle\Varid{x},\Varid{y}\rangle\mathrel{\coloneqq}\Varid{x};\,} & \ensuremath{\textbf{split}\{\Varid{e},\Varid{f},\Varid{g}\}} \\
    \ensuremath{\textbf{all}\{\Varid{e}\}} & \equiv \ensuremath{\Varid{f}(\Varid{x})\mathrel{\coloneqq}\langle\rangle;\,}    & \ensuremath{\Varid{g}\langle\Varid{x},\Varid{y}\rangle\mathrel{\coloneqq}\Varid{cons}\langle\Varid{x},\textbf{split}\{\Varid{y}\langle\rangle,\Varid{f},\Varid{g}\}\rangle;\,} & \ensuremath{\textbf{split}\{\Varid{e},\Varid{f},\Varid{g}\}}
\end{array}\]
For this paper we stuck to the arguably simpler \ensuremath{\textbf{one}} and \ensuremath{\textbf{all}},
to avoid confusing the presentation with these higher-order encodings, but we do not expect any complications using \ensuremath{\textbf{split}} instead.

\subsection{Laziness} \label{sec:laziness}

As \Cref{sec:lenient} discussed, \versecalc{} is lenient. Unlike Curry however, \versecalc{} is not \emph{lazy}.
For example this
program does not yield a value:
$$
\ensuremath{\exists\Varid{ones}\hspace{0.3mm}\Varid{y}\hspace{0.3mm}\Varid{z}.\,\Varid{ones}\mathrel{=}\langle{}\mathrm{1},\Varid{ones}\rangle;\,\langle{}\Varid{y},\Varid{z}\rangle\mathrel{=}\Varid{ones};\,\Varid{y}}
$$
Here, \ensuremath{\Varid{ones}} denotes an infinite list of \ensuremath{\mathrm{1}}'s, and we might hope that the expression returns
\ensuremath{\mathrm{1}}; but in \versecalc{} it won't because rule \rulename{subst} declines to substitute for the equality
\ensuremath{\Varid{ones}\mathrel{=}\langle{}\mathrm{1},\Varid{ones}\rangle}, which has an ``occurs check.''
Remembering the duality between values and choices,
one might also want \ensuremath{\textbf{all}} to return a lazy stream
of results, one by one, rather than waiting for them all to terminate.  For example, one might hope that
this program would converge:
$$
\ensuremath{\exists\Varid{y}\hspace{0.3mm}\Varid{z}.\,\langle{}\Varid{y},\Varid{z}\rangle\mathrel{=}\textbf{all}\{\exists\Varid{onec}.\,\Varid{onec}\mathrel{=}(\mathrm{1}\;\rule[-0.3mm]{0.5mm}{2.5mm}\;\Varid{onec})\};\,\Varid{y}}
$$
Here we suppose that \ensuremath{\textbf{all}} returns a lazy stream of values (represented as nested
pairs), from which we may pick the first and discard the rest.

A lazy variant of \versecalc{} looks entirely feasible, but we have not worked out the details.

\section{Looking back, looking forward}

The semantics of \versecalc{} is designed at a level intended to capture the
\emph{computational model} of the language; not all formal semantics do so.
Defining a language by giving its low-level semantics is \emph{precise} but not necessarily \emph{illuminating}.
For example, giving a reference compiler that compiles
the program to x86 instructions is precise, but is not helpful
to a human who is trying to understand exactly what the original program
meant. Likewise, a high-level semantics simply provides the eventual answer
produced by the program, without insight into the \emph{computational steps}
that got us from program start to program completion.

The moral here is that \emph{formal specifications can be obfuscatory---or
illuminating.} The latter kind shed light because they are defined \emph{in
terms of the intended mechanisms of the language}. \versecalc{} does this; it
respects the conceptual structures of the Verse language.

Note that when we say ``illuminating'' we mean that in multiple ways.
A semantics can be illuminating for humans who are trying to understand
what a particular program does, or how a proposed change to the language
will affect the language. It can drive analyses that help the compiler
optimize programs. It underlies the use of formal methods and verification
to provide machine-derived and -checkable proofs of correctness. All of
these applications depend on the semantics being defined at the appropriate
level: the level of the computational model that underlies the language.
This has been our goal in this work.

We have much left to do.  The full Verse language has
statically checked types.  In the dynamic semantics, the types can be
represented by partial identity functions---identity of the values of
the type and \ensuremath{\textbf{fail}} otherwise.  This gives a distinctive new
perspective on type systems, one that we intend to develop in future
work.  The full Verse language also has a statically-checked effect system,
including both mutable references and input/output.  All these effects
must be \emph{transactional}, {\eg}, when the condition of an \ensuremath{\textbf{if}}
fails, any store effects in the condition must be rolled back.
We have preliminary reduction rules for updateable references,
but they are not included here.

The \ensuremath{\textbf{store}\;\Varid{s}\;\textbf{in}\,\{\Varid{e}\}} indicates that \ensuremath{\Varid{e}} should be reduced using the store \ensuremath{\Varid{s}}.
A store is simply a mapping from references to values (one mapping being \ensuremath{\Varid{r}\mapsto\!\Varid{v}}).
A reference is some opaque type that supports equality (unification) and creation of a new reference.

There are three new operations, \ensuremath{\textbf{alloc}}, \ensuremath{\textbf{read}}, and \ensuremath{\textbf{write}}.
Their interaction with the store can be seen from the axioms.
The \ensuremath{\textbf{alloc}(\Varid{v})} operation creates a new reference and adds a binding with \ensuremath{\Varid{v}} to the store.
The \ensuremath{\textbf{read}(\Varid{r})} operation retrieves the value for reference \ensuremath{\Varid{r}} from the store,
and \ensuremath{\textbf{write}\langle\Varid{r},\Varid{v}\rangle} updates the reference \ensuremath{\Varid{r}} with \ensuremath{\Varid{v}} in the store.
Note that these reductions only happen when the store is exactly at the operation.

Most of the store axioms just move the store around from left-to-right, ``visiting'' every subexpression,
and moving on when the subexpression has reduced to a value.
This way every subexpression (in a non-looping program) will get a ``visit'' and all reference operations
can reduce.

The interesting rules are those that cover \ensuremath{\textbf{split}}.
When reducing the \ensuremath{\Varid{e}} in \ensuremath{\textbf{split}\{\Varid{e},\Varid{f},\Varid{g}\}} the reduction of \ensuremath{\Varid{e}} is treated as a transaction, {\ie},
if the reduction fails any changes to the store are rolled back.
This is handled by a combination of \rulename{st-split-copy}, \rulename{st-fail}, and \rulename{st-store}.
First, when a \ensuremath{\textbf{store}} is at a \ensuremath{\textbf{split}\{\Varid{e},\Varid{f},\Varid{g}\}} then the store is copied to reduce the \ensuremath{\Varid{e}}.
If the evaluation of \ensuremath{\Varid{e}} fails, then rule \rulename{st-fail} will simply abandon the copy of the store.
If the evaluation of \ensuremath{\Varid{e}} is successful then the \rulename{st-value} or \rulename{st-choice} will move the store out of the \ensuremath{\textbf{split}}.
In that position the \rulename{st-store} rule can ``commit'' and throw away the old store, replacing it with the new.
Note that the \rulename{st-split-copy} has a side condition to stop it from firing when it has just fired,
and also when the \ensuremath{\Varid{e}} is a result.

To support limited store operations ({\eg}, \ensuremath{\textbf{read}}, but not \ensuremath{\textbf{write}}) we can equip the store with a set of currently allowed operations.
We also need some extra expressions that modify this set as the store ``moves'' around.

\section{Confluence proof plan}\label{sec:confluence-proof-plan}

In this section we outline our strategy for a rigorous proof of \cref{thm:confluence}, which is still in the making. The main steps are:
\begin{itemize}
\item Split the depth-preserving unification rules from the depth-collapsing evaluation rules (\ref{sec:conf-proof-split}).
\item The unification fragment is terminating, assuming we cut at a given depth (\ref{sec:conf-proof-term}).
\item Furthermore, it is confluent if we add the additional rules (\ref{sec:conf-proof-u}).
\item The additional rules and the depths-cutting can be moved to the end of any sequence (\ref{sec:conf-proof-split}).
\end{itemize}

\subsection{Notation}

We briefly recapitulate some standard notation about abstract rewriting. If $\rightarrow$ is a relation on a set $A$, then $\leftarrow$ is its inverse, $\rightarrow^\ast$ its reflexive transitive closure and $\rightarrow^=$ its reflexive closure. If $\rightsquigarrow$ is also a relation on $A$, then $\leftarrow;\rightsquigarrow$ is its composition.

Relations $\rightarrow$ and $\rightsquigarrow$ commute if $\leftarrow^\ast;\rightsquigarrow^\ast\subseteq \rightsquigarrow^\ast;\leftarrow^\ast$. A self-commuting relation is confluent. A relation $\rightarrow$ has the diamond property if $\leftarrow;\rightarrow\subseteq \rightarrow;\leftarrow$.

\subsection{Separating the evaluation rules}
\label{sec:conf-proof-split}

The first problem we need to deal with is that, as discussed in \cref{sec:confluence-and-recursion}, we can only hope for confluence up to a certain depth $n$. But simply fixing a single depth for the whole proof is unlikely to work, as rule \rulename{app-beta} collapses one layer of lambdas. The more beta-reductions are still to happen, the deeper we have to unroll fixpoints.

We can address this problem with the following general lemma (which we have proven using Isabelle)
\begin{lemma}\label{lem:u-e}
Let \ensuremath{\underset{A}\movesto}, \ensuremath{\underset{B}\movesto} and \ensuremath{\equiv_{\Varid{n}}}, $n\in \mathbb{N}$, be relations on a common set. If
\begin{enumerate}
\item \ensuremath{\equiv_{\Varid{n}}} is an equivalence relation
\item \ensuremath{\equiv_{\Varid{n}};\,\underset{A}\movesto\mathrel{\subseteq}\underset{A}\movesto^{=};\,\equiv_{\Varid{n}}}
\item \ensuremath{\equiv_{\Varid{n}\mathbin{+}\mathrm{1}};\,\underset{B}\movesto\mathrel{\subseteq}\underset{B}\movesto;\,\equiv_{\Varid{n}}}
\item \ensuremath{\underset{A}\movesfrom^{\ast};\,\underset{A}\movesto^{\ast}\mathrel{\subseteq}\underset{A}\movesto^{\ast};\,\equiv_{\Varid{n}};\,\underset{A}\movesfrom^{\ast}}
\item \ensuremath{\underset{B}\movesfrom;\,\underset{B}\movesto\mathrel{\subseteq}\underset{B}\movesto;\,\underset{B}\movesfrom}
\item \ensuremath{\underset{A}\movesfrom;\,\underset{B}\movesto\mathrel{\subseteq}\underset{B}\movesto;\,\underset{A}\movesfrom^{\ast}}
\end{enumerate}
then \ensuremath{\underset{AB}\movesto\mathrel{\coloneqq}(\underset{A}\movesto\mathop{\cup}\underset{B}\movesto)} is confluent up to \ensuremath{\equiv_{\Varid{n}}}; {\ie},
\ensuremath{\underset{AB}\movesfrom^{\ast};\,\underset{AB}\movesto^{\ast}\mathrel{\subseteq}\underset{AB}\movesto^{\ast};\,\equiv_{\Varid{n}};\,\underset{AB}\movesfrom^{\ast}}.
\end{lemma}

We divide our relation $\movesto$ into two sets:
\begin{enumerate}
\item The unification and choice fragment, consisting of the rules from sections Unification,  Unification variables, Sequencing, Choice and fail propagation, as well as the extra rules mentioned in \cref{fig:rules-confluence}. This becomes the \ensuremath{\underset{A}\movesto} of \cref{lem:u-e}.

This fragment respects $\equiv_n$, as it does not look under lambdas.

\item The remaining rules, {\ie}, resolving applications and \ensuremath{\textbf{one}} and \ensuremath{\textbf{all}}, form the evaluation fragment.

These rules look under at most one level of lambdas, so take terms related by \ensuremath{\underset{\Varid{n}\mathbin{+}\mathrm{1}}\movesto} to terms related by \ensuremath{\underset{\Varid{n}}\movesto}.

Furthermore, the rules are \emph{orthogonal} (they do not overlap) and left-linear, so not only is this fragment on its own confluent, but its parallel application has the diamond property and still reduces the depth of expressions at most one level.

The \emph{parallel} application becomes the \ensuremath{\underset{B}\movesto} of \cref{lem:u-e}.
\end{enumerate}

The two relations  commute, and crucially, commuting them does not increase the number of (parallel) evaluation steps needed. By \cref{lem:u-e} it remains to show that \ensuremath{\underset{A}\movesto} is confluent up to \ensuremath{\equiv_{\Varid{n}}}.

Because \ensuremath{\underset{A}\movesto} respects $\equiv_n$, we can equivalently prove that the relation \ensuremath{\underset{U_n}\movesto\mathrel{\coloneqq}\underset{A}\movesto;\,\underset{\Varid{n}}\movesto} is confluent. This brings us to the world of “conventional” confluence.

\label{sec:conf-proof-u}

Unfortunately, the relation is not locally confluent, as we have seen in \cref{sec:confluence-extra}, and we need the extra rules in \cref{fig:rules-confluence} to equate them. \subsection{Moving equivalences to the end}
\label{sec:conf-proof-moving}

Finally, to get to the precise statement of \cref{thm:confluence}, we have to show that the extra rules and \ensuremath{\underset{\Varid{n}}\movesto} can be moved to the end of a sequence, and commute them past \ensuremath{\underset{A}\movesto} and \ensuremath{\underset{B}\movesto}. There are three new primitive operations, \ensuremath{\textbf{alloc}}, \ensuremath{\textbf{read}}, and \ensuremath{\textbf{write}}.
Their interaction with the store can be seen from the axioms.
The \ensuremath{\textbf{alloc}(\Varid{v})} operation creates a new reference and adds a binding with \ensuremath{\Varid{v}} to the store.
The \ensuremath{\textbf{read}(\Varid{r})} operation retrieves the value for reference \ensuremath{\Varid{r}} from the store,
and \ensuremath{\textbf{write}\langle\Varid{r},\Varid{v}\rangle} updates the reference \ensuremath{\Varid{r}} with \ensuremath{\Varid{v}} in the store.
All of these operations use the context \ensuremath{\Conid{S}} which ensures that there are no store operations
to the left of the hole, i.e., a store operation in the hole is the next one that should execute.

The interesting rules involve choice and \ensuremath{\textbf{split}} because store operations are transactional in the sense
that when an expressions fails, none of its store operations will happen.

When reducing \ensuremath{\textbf{split}\{\Varid{e},\Varid{f},\Varid{g}\}} in an \ensuremath{\Conid{S}} hole, rule \rulename{st-split-dup}, the store is duplicated.
Any store operations inside the \ensuremath{\textbf{split}} will happen in this local copy of the store.
Note the two occurrences of \ensuremath{\Varid{h}} in the right hand side of \rulename{st-split-dup}.
If the reduction of \ensuremath{\Varid{e}} results in \ensuremath{\textbf{fail}} then rule \rulename{fail} is used,
and the store from the failing computation is simply thrown away.
If the reduction of \ensuremath{\Varid{e}} results in a value (with or without more alternatives) then rule
\rulename{st-split} is used.  This rule replaces the outer store with the inner store,
since we know the inner computation has succeeded.

Similarely, the reduction of \ensuremath{\Varid{e}_{\mathrm{1}}\hspace{1mm}\mathop{\rule[-0.3mm]{0.5mm}{2.5mm}}\hspace{1mm}\Varid{e}_{\mathrm{2}}} will duplicate the store into the first branch, \rulename{st-choice-dup}.
Here \ensuremath{\Varid{e}_{\mathrm{1}}} must not contain any store operation nor any choices.
And again, similarely, \rulename{st-choice} commits the new store and throws away the old.

The use of \ensuremath{\Varid{oe}} in the rules is to ensure that the rules cannot get stuck in a loop.
Using \ensuremath{\Varid{e}} instead of \ensuremath{\Varid{oe}} would mean that failing or committing would make the expression match the duplication rule again.
It also prevents the duplication rule from repeatedly duplicating the \ensuremath{\textbf{store}}.

Note that \ensuremath{\textbf{store}} is part of the \ensuremath{X} context, which means that the \ensuremath{\textbf{store}} can float inside
existentials.

a variable that is not in scope.

but not \ensuremath{\textbf{write}}) we can equip the store with a set of currently allowed operations.
