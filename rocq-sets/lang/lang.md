# `lang/`: typed λ-calculi with set-theoretic semantics

This directory collects several typed lambda calculi, each given a
*denotational* semantics in the quotient set theory `ZFSet` and proved
**sound**: a well-typed term denotes an element of (the denotation of) its
type. They are independent developments that share one recipe.

## The common recipe

Every module follows the same shape:

- **`evalTy : Ty → ZFSet`** — a type denotes a set. The base type of naturals
  denotes `Omega` (`ω`); a function type `A → B` denotes the set-theoretic
  function space `Pi A (fun _ => B)` (total functional graphs, i.e. subsets of
  `A × B`).
- **`evalTm`** — a term denotes either an *element* of `ZFSet` (the functional
  models) or a *set of values* (the relational PCF models). A `λ` denotes its
  function **graph** `iUnion A (fun a => {⟨a, ⟦body⟧(a·ρ)⟩})`; application is
  `applyFun` (relational `image`).
- **`models Γ ρ`** — the environment `ρ` assigns each variable a value in its
  type's denotation.
- **`soundness`** — `has_type Γ e T → models Γ ρ → ⟦e⟧ρ ∈ evalTy T` (or `⊆`
  for the relational models).

The shared `ZFSet` toolkit: `Omega`/`ω` (the naturals), `Pi` (`Π[ _ ∈ A ] B`,
the function space), `applyFun` (`f [ v ]`), `iUnion`-of-couples (function
graphs), `Big` (a Grothendieck universe of "small" sets), and
`natZ`/`natZAdd`.

## `STLC.v` — simply-typed λ-calculus

Types `TNat`/`TArr`; terms are de Bruijn `tvar`/`tlam`/`tapp`/`tcon`/`tadd`.
The model is **functional**: `evalTm e ρ` is the value itself (a `λ` is its
graph, `tapp` is `applyFun`, `tcon n` is `natZ n`, `tadd` is `natZAdd`).

- **`soundness`** — `has_type Γ e T → models Γ ρ → evalTm e ρ ∈ evalTy T`.
- **Call-by-name reduction** `step` (`-->`): `S_beta` (β on the unevaluated
  argument), `S_app1`, and the strict `+` rules `S_add1`/`S_add2`/`S_add`;
  plus `multistep` (`-->*`).
- **`step_sound`** — every reduction rule preserves the denotation on
  well-typed terms: `e --> e' → has_type Γ e T → models Γ ρ → evalTm e ρ =
  evalTm e' ρ`. The interesting case is β, via the substitution lemma
  `evalTm_subst` and `applyFun_graph` (applying a λ-graph at an in-domain
  point yields the body's value); supported by `evalTm_lift` and the
  environment-insertion operator `env_ins`.
- **β-equivalence** `beta_equiv` (`<-->`) — the equivalence closure of `-->`
  (`beta_equiv_Equivalence`); `step_beta_equiv`/`multistep_beta_equiv` include
  reduction.
- **βη-equivalence** `beta_eta_equiv` (`<==>`) — the *congruence* generated by
  β, η, and the `+` δ-rule (`beta_eta_equiv_Equivalence`). It extends β-equiv
  (`beta_equiv_beta_eta`), and the η witness `λf. f <==> λf. λx. f x`
  (`eta_example`) — which β-equivalence cannot prove — holds.
- **`eta_sound`** — the model validates η: an η-expansion `λx. (lift e) x` of a
  well-typed `e : T1→T2` has the same denotation as `e` (graph reconstruction
  via `pi_value_edge`/`pi_edge_value`). With `step_sound` this means `evalTm`
  validates every rule of `<==>`.
- **`preservation`** (*axiom*) — subject reduction, taken as an axiom (the
  standard, provable STLC property). With it, **`multistep_sound`** lifts
  `step_sound` to `-->*`. Full β-equivalence soundness needs confluence (a
  `<-->` chain can pass through ill-typed terms, since subject expansion
  fails), noted but not done.
- **Completeness for βη** — by **Friedman's theorem** the model is *complete*
  (equal denotation ⟹ βη-equal), since `evalTy ι` is infinite and arrows are
  the *full* function space. A detailed proof sketch (normalisation + a
  logical relation gluing the model to the syntax via an injection of normal
  forms into the infinite base) is given in the file; it is not formalized.

## `Intersection.v` — Coppo–Dezani intersection types

Types are mutually inductive: a `Ty` is a base type or an arrow whose
*domain* is an intersection `DomTy`, and a `DomTy` is a nonempty intersection
of `Ty`s (so intersections sit only in argument position, matching
`Syntax.IntersectionTypes`). A domain type denotes a **set intersection**
(`evalDomTy`, using `∩`); membership `InDom T D` records that `T` is a
component of `D`.

- **`InDom_Inc`** — a domain's denotation is included in each component's.
- **`soundness_and`** — the mutual soundness statement for `has_type` /
  `has_dom_type`, proved by mutual induction (`has_type_has_dom_type_mut`).
- **`soundness`** / **`soundness_dom`** — its two projections: a well-typed
  term is in its type's denotation; a term of domain type `D` is in
  `evalDomTy D` (the intersection of all components).

## `PolyF.v` — predicative System F

Monotypes `Mono` (base, type variables, arrows) versus polytypes `Poly`
(monotypes, arrows, `∀`). Set-theoretically a monotype denotes a **small** set
(a member of the universe `Big`), type variables range over `Big`, and a
quantifier `∀.σ` denotes the **predicative** product `Π[ X ∈ Big ] ⟦σ⟧(X·δ)`.
Type application instantiates a quantifier with a *monotype only*
(predicativity), which keeps the argument in `Big`. Semantics is relative to a
type environment `δ : tenv` (variables ↦ small sets).

- **`Arr_small`** — `Big` is closed under function space (so monotype
  denotations stay small); **`mono_small`** — every monotype denotes a member
  of `Big` under an OK type environment.
- **`shiftMono_eval`/`shiftPoly_eval`** and **`substMono_eval`/`substPoly_eval`**
  — the type-level shift/substitution lemmas: type substitution is denotation
  insertion (`insert`), shifting is cancelled by insertion.
- **`soundness`** — `has_type Γ e s → tenv_ok δ → models Γ ρ δ → evalTm e ρ δ
  ∈ evalPoly s δ`. The `Λ` case builds the type-function graph over `Big`; the
  type-application case instantiates at the small monotype via
  `applyFun_mem_of_pi` (`substPoly_eval` + `insert0`).

## `PCF/` — Plotkin's PCF, four models

A self-contained sub-suite with its own design note, [`PCF/PCF.md`](PCF/PCF.md).
PCF adds a fixpoint operator, so terms can diverge; the models differ in how
they represent divergence, and the through-line is **when β-reduction is
sound**. Unlike the calculi above, a term denotes a *set of values* (`∅` =
divergence), so soundness is `⊆`.

- **`PCFSyntax.v`** — the shared syntax, typing, and CBN small-step reduction
  (`lift`/`subst`/`value`/`step`/`multistep`); no semantics.
- **`PCFStrict.v`** — the original *strict* relational model (divergence =
  `∅`); `soundness`, the big-step semantics (`⇓`) and contextual equivalence,
  and `S_beta_unsound`: **β is unsound** (strict application + total `Pi`).
- **`PCFLifted.v`** — a bottom `⊥` at the *ground* types and a unique-fixed-point
  `tfix`; `soundness`, `evalTm_subsingleton` (every term is single-valued or
  `∅`), and `S_beta_sound` (β sound for single-valued arguments). The
  `PCFStrict` β counterexample flips from `∅` to `{0}` (`tot_app_lifted`).
- **`PCFPointed.v`** — a *type-indexed* bottom `botT` at **every** type and a
  type-directed evaluator `eval`; **`eval_total`**: every well-typed term
  denotes *exactly one* value (never `∅`), subsuming soundness
  (`eval_sound`) and determinism (`eval_single_valued`, `eval_nonempty`).

See [`PCF/PCF.md`](PCF/PCF.md) for the full story.

---

*All modules compile against the `ZFSet` library and rely only on the
development's standard axioms (the quotient axioms, propositional and
functional extensionality, and Hilbert's `epsilon`; `PCFPointed.v` additionally
uses classical excluded middle, and `STLC.v` assumes `preservation` — subject
reduction — as an axiom).*
