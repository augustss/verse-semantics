========================================================================================================================
Terse Library Specification: Terse.h for Verse interoperability with C++.
========================================================================================================================

Terse is a systems-level C++ library for memory-safe, thread-safe garbage-collected containers, transactional
concurrency, and asynchronous functional logic evaluation. It serves as both a foundation for writing C++ programs
and for interoperating with Verse.

Terse data types are shared between C++ and the Verse runtime and transpiler, with names and conventions that are
consistent across languages. These data structures include values (any, integer), futures (future<t>), transactional
pointers (ptr<t>), containers (map<k,v>, bag<t>, array<t>, option<t>), and functions (function<r(ps...)>).

Terse exposes sufficient functionality that a C++ programmer can write any kind of functional logic code that a
Verse programmer can write, and can write a Verse compiler and interpretter that interoperates live with native
code and data implemented in C++.

Terse is designed for three Verse usage scenarios:

1) Dynamically typed Verse runtime: Here, the runtime works without any compile-time type information, passing
   around values as future<>, and coercing to general types like integer, function<>, and array<> as needed,
   with the expectation that the coercions can't fail when running a successfully verified program.

2) Simply typed Verse runtime: Here, the Verse compiler would generate simple runtime types coinciding with
   types defined in Terse, and we'd run them, minimizing the need for dynamic memory allocation.

3) Transpilation of Verse code to C++ using simple types generated by the compiler. Here we'd run C++ code
   coinciding with Verse code, with no interpretive overhead.

Note that this specification makes reference to the separate Verse Language Specification.

Decomposition Of Types & Values
-------------------------------

The most general type is future<>, equivalent to future<any>. A value of any type t can be converted
to future<> via future::future(const t&) as follows:
	Implicitly by default.
	Explicitly only if you implement void ExposeExplicit(const t&).
	Not at all if you implement void ExposeValue(const t&).

Every Terse value belongs to one of 6 base types: rational, path, nominal, primitive, function<>, ptr<>.
Given "any A", you can use "Cast<t>(A)" to determine which one of these six types contains it, and then
recursively decompose it further.

Any type that you expose nominally using the VERSE_* macros or Expose* functions will belong to "nominal".
Any type that is exposed automatically without such guidance will belong to "primitive".
For example, though the verse "char" type coincides with the Terse "char8" type and carries no runtime
overhead, Cast<nominal>(char C) succeeds and makes it look like characters are values belonging to the
Verse type "char=struct {CodeUnit:nat8}".

Further, values built out of types rational, path, nominal, and map<> are well-founded and can be
recursively decomposed without any possibility of looping. Scripting languages often call this capability
"reflection", but in Terse you can accomplish this just through casting, without rummaging through any
sort of type-description metadata.

True Subtyping
--------------

To provide rigor despite the non-value-preserving conversion rules of C++, we provide IsSubtype<t0,t1> which
expresses a true subtyping relationship: IsSubtype<t0,t1> means that all values of t0 are also values
of t1, and that converting a value from t0 to t1 and back is lossless.

Subtyping Managed Types: Covariance and Contravariance
------------------------------------------------------

Managed types support rich subtyping with covariance and contravariance, free of any cost that's proportional
to the size of the types (e.g. without copying or conversion). Besides the subtyping rules that the hierarchy
above expresses (e.g. integer <= rational <= any <= future<any>, we also have:

	function<r0(ps0...)> <= function<r1(ps1...)> if r0<=r1 and ps0 has same number of parameters as ps1 
	                                             and for each valid index i, ps1[i] <= ps0[i]
	map<k0,v0>            <= map<k1,v1>          if k0<=k1 and v0<=v1
	array<t0>             <= array<t1>           if t0<=t1
	bag<t0>               <= bag<t1>             if t0<=t1
	box<t0>               <= box<t1>             if t0 (possibly multiple-) inherits from t1 in C++

Managed Types: Confluence, Curry, and Church
--------------------------------------------

Our managed types are Curry types, meaning they purely characterize whether a given value (of type any) 
belongs to the type or not. Our values thus have a universal meaning that is unchanged by the managed type
that it's currently held in. We expect (though C++ cannot enforce) that any templated function
on managed types produces the same result when invoked with the same values, even when those values are
represented though different types, e.g. Given integers a and b, rational(a+b) == rational(a)+rational(b).
This property is known as confluence.

Comparison: Equality and Ordering
------------------------------------------------------------------------------------------------------------------------

Equality is a rigid concept in the library: Equality implies "observable indistinguishability": if a==b, no
high-level library operation should be able to distinguish between a and b, though you kernel operations may
distinguish them.

Equality of managed values must be confluent: it must produce the same result independent of the choice of types
for each variable, provided those types actually allow for comparison. More precisely, if we have
a of type t and b of type u, and we have types t1,u1 where IsSubtype<t,t1> and IsSubtype<u,u1>,
then a==b and t1(a)==u1(b) must be identical.

Values of type float64, float32, and rational which "notionally represent the same value",
like 1.5, 1.5f, and 1.5r, are not equal in the library's view. To decree them equal would break
confluence, as 0.3 and 0.3f are distinguishable by calling an overloaded function.

The prevent confusion, operator== and operator!= on library-provided types are defined to disallow
comparison across unrelated families of values. Use CompareDynamic to compare values across families.
And be careful of these operations on built-in C++ types, which the library can't override.

Numeric equality and order
--------------------------

Verse demands equality be reflexive (x=x) and universal (x=y implies f(x)=f(y) for pure functions f).
The IEEE 754 floating point specification takes a contradictory position, which we minimally reformulate as follows:

IEEE 754 Position    Verse Position   How we expose in Terse
-----------------    --------------   ----------------------
NaN <> NaN           NaN = NaN        float64 Divide (float64 a,float64 b) {return a/(+0+b);}
1.0/-0.0 = -Inf      1.0/-0.0 = +Inf  bool operator<=(float64 a,float64 b) {return a==a||b==b? a<=b: true;}

Terse agrees with IEEE 754 on -0==+0 and all arithmetic and comparison except for NaN<>NaN and division by -0.

Paths and Signatures
------------------------------------------------------------------------------------------------------------------------

To provide rigor for open-world programming, with nominal types declared across many nested modules in which
name clashes might occur, we provide the "path" type with values such as "/Verse" and "/UnrealEngine/Actor"
to describe the location of a resource, including the origin of a nominal type.

For any value, call Signature(v) to get the path where the value's nominal type was defined.
For native types like integer, the system provides a pre-defined signature.
We use the term signature because it's a security property: the only way to construct an instance of a type
with a given signature is by through the interface (public or private) of the module the type resides in.

Values with unequal signatures are guaranteed to be unequal.

Arithmetic
------------------------------------------------------------------------------------------------------------------------

The library exposes managed exact numeric types with subtype relations real <= rational <= integer <= natural:

	real:     A base type with no current implementation, reserved for the computable real numbers.
	rational: The mathematical rational numbers, with fully support for exact math.
	integer:  The mathematical integers, without overflow.
	natural:  The mathematical natural numbers (integers >= 0).

Integers that fit into 48 bits are stored directly with no memory management overhead; larger integers
are garbage collected. Integer operations never overflow.

The library also exposes aliases for standard C++ types. Their math does overflow.
	int8, int16, int32, int64
	nat8, nat16, nat32, nat64

IsSubtype reflects subset inclusion on these types, e.g. nat8<=int16.

C++ and Verse behaviour differ as follows:
	C++ type aliases like int8 suffer overflow. In Verse, they produce exact integer results.
	C++ a/b on integer divides a by b and rounds down. In Verse, a/b is a rational number.

Division and modulus are Euclidean: a/b=floor(a divided by b), a%b is between 0 and b-1, b*(a/b)+a%b==a.

Mutability
------------------------------------------------------------------------------------------------------------------------

There are two primary mutability models in current languages. 

1. Ad-hoc mutability in C++, C#, and Java dictates that variables like "int x" be mutable by default
   and subject to a complicated set of copying rules specifying what is passed by value versus reference
   and lifetime rules defining when things remain valid. Such languages start simple, and grow
   complex, e.g. C# adding structs (passed by value) and "ref" for passing parameters by reference, and 
   each adding complicated "lambda capture" rules.

2. Explicit mutability, layering explicit mutable variable types on top of an otherwise pure functional
   language, like Haskell (IORef t), ML (t ref), and Verse (^t). These languages distinguish a mutable
   variable (like x:^int) from the value it contains, which must be explicitly read (like x^, yielding an int).
   The benefit is everything is passed by value, captured by value, and variable lifetimes are always safe.
   The drawback is that mutables must be explicitly declared (x^:int) and explicitly read (x^).

We implement option #2, using ptr<t> to represent a life-time-safe mutable variable of type t.
When passing a mutable variable around, each instance refers to the same underlying mutable variable.

Containers like array<t> and map<k,v> are immutable.

To manipulate containers with structural mutation similar to C++ vector<t> or Unreal TArray<t> or TMap<k,v>,
use ptr<array<t>> and ptr<map<k,v>> (NOTE: general implementation pending; use ptr<map<k,v>> for now).

Container Model
------------------------------------------------------------------------------------------------------------------------

To recap Verse: failure is a first-class control-flow notion; the domain of a function is determined by the values
for which the function succeeds; tables are a special kind of function (which is precomputed and accessible without
effects); and arrays are a special kind of map (whose keys are consecutive natural numbers 0..length-1).

Containers provided by the library like map<k,v> and array<t> are notionally immutable. You can't change one,
but can produce a new one efficiently from an old one and an operation like adding, removing, or updating a value.

You can wrap a container into a mutable variable like ptr<array<integer>> as, then read it with *as and write it
with as+=array{3,5}.

Exposing User C++ Functions
------------------------------------------------------------------------------------------------------------------------

The managed type function<r(ps...)> represents a callable function which respects subtype relationships.
You may construct such a function from a lambda, function pointer, or function object, and call it
from C++ just like calling any other function or function object with the same parameter types and return type.

Multiple Parameters
-------------------

C++ functions take multiple parameters, whereas Verse functions take a single parameter, using a tuple to
achieve the effect if more than one value are to be passed.

To accomodate this, the translation maps 1-parameter C++ functions like int(int) to a Verse function taking
exactly that parameter, and maps n-parameter C++ functions (where n>1) like int(int,int) taking an n-tuple.

Effects
-------

Effects are described by enum class fx and are as described in the Verse Language Specification.
Before C++ code executes an operation that must be sequenced relative to lenient Verse evaluation,
it must use WhenFxStep.

Asynchronous Execution Model
----------------------------

Immediate library operations:
	Most library operations execute immediately and simply return their results.

Suspending library operations:
	Operations with names of the form *Suspend take callback functions as parameters and may suspend while time
	passes. The callbacks may be called immediately, or at some later. When called later, they execute in
	the same context, possibly in a different thread, however no visible operations in the same context
	will run in parallel on different threads.

Stepping library operations:
	Operations with names of the form *Step take a parameter "step Step0" representing the next code to run
	after the current operation, and return a step - that is usually either the passed-in Step0 or a
	newly-created step that lambda-captures and eventually executes the original step.
	This is known as "continuation passing style", and it naturally supports the Verse semantics of:
		- Verse failure as control flow
		- Verse exceptions causing non-local control flow without relying on C++ or operating system exceptions
		- Verse iteration and forking
		- Verse suspending asynchronous execution

		Define nat ExposeHash(const t&) and dynamic_ordering ExposeCompare(const t&,const t&) to support
		comparison for equality on some values of t but not necessarily all of them. This is useful for
		polymorphic or dynamically typed containers where some instances are comparable and others aren't.
		Optionally define bool ExposeIsComparable(const t&) to say whether a given value of type t is
		comparable or not.
		The library provides only HashDynamic, CompareDynamic.

	Structural equality
		Define equality_ordering ExposeCompare(t,t), which must provide a sound equality relationship.
		Define nat ExposeHash(const t&).
		The library provides Hash, HashDynamic, Compare, CompareDynamic, operator==, and operator!=.

	Structural partial ordering
		Define partial_ordering ExposeCompare(const t&,const t&) which must provide a sound partial ordering.
		Define nat ExposeHash(const t&).
		The library provides Hash, HashDynamic, Compare, CompareDynamic, operator==, and operator!=,
		operator<, operator<=, operator>, operator>=.

	Structural total ordering (structural ordering)
		Define total_ordering ExposeCompare(const t&,const t&) which must provide a sound total ordering.
		Define nat ExposeHash(const t&).
		The library provides Hash, HashDynamic, Compare, CompareDynamic, operator==, and operator!=,
		operator<, operator<=, operator>, operator>=.

	By defining ExposeCompare(const t&,const t&), you are promising that values of type t are
	not equal to values in any type other than t. Hence it's not possible to expose ad hoc equality
	relationships among different types.

Exposing Signatures for User Types
----------------------------------

For a user type t, there are two ways of defining t's signature.
If all values of type t have the same signature, specialize expose<t> and implement ExposeTypeSignature:

	template<> struct expose<t>: default_expose<t> {
		static path ExposeTypeSignature() {return "/MyOrganization.com/MyModule/t"_VP;}
	};

If different values of type t may have different signatures, overload ExposeSignature:

	path ExposeSignature(const t&) {return "/MyOrganization.com/MyModule/t"_VP;}

Exposing ToString for User Types
--------------------------------

Overload ExposeToString on type. Then, ToString will automatically work on that type, and
will even work when your type is up-converted to comparable, any, or another polymorphic type.

	string ExposeToString(const t&);

Mutation of user box<t> in C++
------------------------------

	Verse expects all visible mutation to occur through ptr<t>. If you expose a member variable to Verse
	through box<t>, and use ExposeUnique to make it mutable through operator->() in C++,
	the behavior is undefined.

Using box<t> in C++
-------------------

[const] t* box<t>::operator->()
	If t is a struct or class, then box<t>::operator-> accesses its fields.
	The t* is constant unless you declare ExposeByMutableReference(t).
	Use B->FieldName only; B::operator->() produces a t* that may dangle if B is modified or goes out of scope.

t box<t>::operator*()
	Extracts t, if t has an accessible copy constructor.

box<t>::operator()
box<t>::operator[]
	If t is callable, then box<t> is callable with the same parameters.

box<t>::operator bool()
	If t is a container, this returns true if it's non-empty.

box<t>::box(const t&)
	Given a t&, you can use the box<t> "copy constructor" box it or rebox it.
	If the t& already is in managed memory as part of a class inheriting from t, then this references the
	existing object without creating a copy! If t is elsewhere, it creates a new managed copy.

Memory Management Of User box<t>
--------------------------------

Memory that box<t>(..) allocates directly for t is fully managed by the garbage collector.
However, if t's constructor uses C++ heap functions to allocate other memory, any managed objects inside
that other memory will be treated as garbage collector roots, leading to possible cycles and memory leaks.

When the garbage collector determines the managed box<t> is no longer reachable, it
calls t's destructor, then frees the memory. The destructor is called at a random time,
from a random garbage collector thread. When using box<t>, don't use t's destructor for objects
which require more complicated lifetime management than this supports.

Gotchas
-------

	Given a managed box<t>, there are various ways to obtain a t& or t*. You must not hold this raw pointer
	or reference after the box<t> has gone out of scope or mutated. The library goes to efforts to make this
	difficult, but you must watch out for the following:

	1) Retained "this":
	   class C {C* Tricky() {return this;}};
	   C* Unsafe = box<C>().Tricky(); // Retains C* after box<C>'s lifetime.
	   SOLUTION: Use box(*this) to wrap "this" back in its box (a cheap no-copy operation) while it's live.

	2) Lambda captured "this":
	   class C {int x=3; auto Tricky() {return [=]{return x;};}};
	   auto Lambda = box<C>(3).Tricky();
	   auto Unsafe = Lambda(); // Accesses C* after box<C>'s lifetime.
	   SOLUTION: If a lambda must access this, capture and name it like [=,Self=box(*this)](..){..}.

	3) Misusing operator->():
	   class C {int x=3;};
	   C* Unsafe = box<C>(3)->operator->(); // Retains C* after box<C>'s lifetime.
	   If you need to capture a t, use box<t>::operator*, which returns a copy.

	4) Multithreaded races on pin<t>:
	   Pinning is for local variables on the stack which are not visible to other threads.
	   Exposing a mutable pin<t> to multiple threads for reading/writing can create multithreaded
	   garbage collector race conditions leading to unsound automatic memory management.
	   NOTE: Lambda-captured pin<t> is safe because it's immutable.

The Terse Kernel
========================================================================================================================

The Terse kernel implements low-level operations for memory management, garbage collection, forking, and transactions.
We aim to minimize the size of the kernel by writing as much code as possibly in high-level parts of the library.

Managed Values
--------------

The type future<any> and its subtypes are lifetime-safe, thread-safe C++ wrappers on top of a 64-bit machine
word, called its "Payload", which characterizes the value. The value is one of the following:

1. An encoded immediate constant, like a small integer, which fits in the 64-bit word and requires no memory management.

2. An unencoded non-null pointer t* into the contents of a live object managed by the garbage collector.
   This can either be a pointer t* into the top of a managed allocation of type box<t>, or it can be an interior
   pointer inside a instance of class u where u (possibly multiple-) inherits from t.

3. An interior pointer to a future<t> or ptr<t> which resides anywhere inside a live managed object.

4. An encoded non-copyable interior value which resides anywhere inside a live managed object, meaning that
   the address of the future<any>& is significant. It represents either:
   A. The state of an unresolved future - storing its context or first waiting suspension.
   B. The state of a ptr<t> - storing an encoded form of its current value.

Memory Management
-----------------

We provide a page-based allocator with two categories of allocations:
* Small allocations (2KB or less) are binned into power-of-2 sizes and allocated within a larger 64KB set of
  pages which service only allocations in the same size bin.
* Large allocations (over 2KB) are allocated using operating system page management functions with mid-sized
  allocations (under 64KB) coalesced into contiguous pages.

To support garbage collection with interior pointers, Terse provides a very efficient AllocationBase to determine
the starting address coinciding with a pointer into the interior of an allocation.

To track allocations, we have two large sparse virtually-allocated global arrays:
* PageTags: For every 4KB of managed allocation space, we have 8B tracking the page status.
* MemTags:  For every 8B of managed allocation space, we have 1B tracking what it contains.

Garbage Collection
------------------

The library provides garbage collection which "just works" for the provided managed types.
It's currently a concurrent non-blocking single-generation collector based on Dijkstra's algorithm.

Organizing Source Code in Verse and C++

A top-level module m (e.g. verse, unreal) should go in a top-level directory of the same name and capitalization.
File extensions .ext stands for .h, .cpp, and .verse

Source code comprising the module m resides entirely in that directory.
Source code for a direct submodule of m named n can either go:
	1) in a single file named m/n.ext
	2) in one or more files in a subdirectory m/n
Source code that contains a single module-level type named t (e.g. class) and associated Verse module-level
	functions pertaining directly to that type (similar to C++ class static functions) should be named m/t.ext.
Source code that contains a single module-level value named named V should be named m/V.ext.
Source code in the module that contains various code organised by purpose or feature (e.g. Printing) should
	be named Printing.ext with an UpperCamelCase name.
