========================================================================================================================
Verse Language Specification 0.1.1
========================================================================================================================

We define Verse by saying how to parse, verify, and run a program:
- Verse Source Code is ASCII, UTF8, or UTF32 text accepted by the Verse Grammar's File production below.
- Verse syntax s accepted by the Verse Grammar is verified and compiled into a Operations op by using
  sequent calculus inference rules specified in the Syntax Reduction Rules, Verifier Inference Rules,
  and Verifier Data Flow Inference Rules starting with the goal:
  empty |- explore(top_allows,fxv) {scope c {exists i x; syntax(opv,i,x){s}}} where fxv,c,i,x,opv fresh.
- Verified Operations are run according to the runtime subset (TBD) of the Verifier Inference Rules,
  with the same goal.

------------------------------------------------------------------------------------------------------------------------
Grammar Conventions
------------------------------------------------------------------------------------------------------------------------

We provide an unambiguous EBNF grammar describing input text in either UTF-8, UTF-32, or ASCII format.
Our grammar is a core context-free LL(k) grammar, extended with context for significant indentation.
When we're parsing with context, we can push and pop the context, and set context variables.
We provide two equivalent Unicode definitions of Printable, one for UTF-8, and one for UTF-32.

Our EBNF grammar is defined using the following symbols:

    A:=b      - Define parsing rule named A as parsing operation b.
    a|b       - Parse either a or b.
    [a]       - Optionally parse a.
    {a}       - Parse zero or more a.
    (a)       - Parse a. Parenthesis only for grouping.
    'c'       - Parse a literal character c.
    "abc"     - Parse a sequence of literal characters a, b, c.
    0oH       - Parse ASCII or Unicode UTF-8 Code Unit octet, where H are hex digits.
    0uH       - Parse Unicode UTF-32 Code Point; note that 0o0..0o7F and 0u0..0u7F are equivalent.
    &s        - Looks ahead for presence of s without consuming it.
    !s        - Looks ahead for absence of s without consuming anything.
    end       - Recognizes end of file.
    push, pop - Save and restore parser context.
    set a=b   - Set parser context variable a to value b.
    parse a=b - Parse b and set parser context variable a to the text parsed by b.
    ^a        - Parse exactly the text in the context variable a.
    a..b      - Parse a character in the range specified by literals characters a and b.
    if(e) then s else t - Based on context-dependent condition e, parse either s or t.

Alpha     := 'A'..'Z' | 'a'..'z' | '_'
Alnum     := 'A'..'Z' | 'a'..'z' | '_' | '0'..'9'
Hex       := 'A'..'F' | 'a'..'f' |       '0'..'9'
Digits    := '0'..'9' {'0'..'9'}

U8        := 0o80..0oBF
UTF8      :=                                      0o00..0o7F
          |                                       0oC2..0oDF U8
          |  !(0oE0 0o80..0o9F | 0oED 0oA0..0oBF) 0oE0..0oEF U8 U8
          |  !(0oF0 0o80..0o8F | 0oF4 0o90..0oBF) 0oF0..0oF4 U8 U8 U8
UTF32     := !(0uD800..0uDFFF) 0u0..0u10FFFF
Printable := 0o09 | !("<#" | "#>" | 0o0..0o1F | 0o7F | 0oC2 0o80..0o9F | 0oE2 0o80 0oA8..0oA9 ) UTF8
          |  0u09 | !("<#" | "#>" | 0u0..0u1F | 0o7F..0o9F             | 0u2028 | 0u2029      ) UTF32

Space     := {0o09 | 0o20 | Comment}
NewLine   := 0o0D [0o0A] | 0o0A
Ending    := &(NewLine | end)
Ind       := Ending push; set Nest=false; set BlockInd=LineInd; set LinePrefix=""
Ded       := Ending pop
Line      := NewLine; parse i:={0o09|0o20}; (Ending | !(0o09|0o20) Space
             if     (i>BlockInd | Nest and i=BlockInd) then set LineInd=i
             else if(not i<=BlockInd                 ) then error)
Scan      := Space {Line}
ScanKey   := Space (&NewLine Scan ^LinePrefix Space | !NewLine)

LineCmt   :=  "#" !'>' {Text        } Ending
BlockCmt  := "<#" !'>' {Text|NewLine} !'<' "#>"
IndCmt    := "<#>"     {Text        } Ind {Text|Line} Ded
Comment   := LineCmt   | BlockCmt | IndCmt
Text      := Printable | BlockCmt | "<#>"

Exp       := [('e'|'E') ['+'|'-'] Digits] !(('e'|'E') ('+'|'-'|Digit))
Units     := [Alpha {Alpha}] !Alpha
Num       := ("0x" Hex {Hex} | !(("0b"|"0o"|"0u"|"0x") Hex) Digits ['.' Digits] Exp Units) !'.' !Alnum
    TODO: allow dot alnum for extension methods
    TODO: separate units out

Special   := '\'|'{'|'}'|'#'|'<'|'>'|'&'|'~'
CharEsc   := '\' ('r'|'n'|'t'|'''|'"'|Special)
CharLit   := ''' Printable ''' !''' | ''' CharEsc '''
Char8     := "0o" (       Hex) [Hex]                   !Alnum
Char32    := "0u" ("10" | Hex) [Hex] [Hex] [Hex] [Hex] !Alnum
Char      := CharLit | Char8 | Char32

Ident     := Alpha {Alnum} !Alnum ["'" {!("<#"|"#>"|'\'|'{'|'}'|'"'|''') 0o20..0o7E} "'"]
Path      := '/' Label ('@' Label | !'@') {'/' ['(' Path ":)"] Ident} !'/'
Label     := Alnum {Alnum|'-'|'.'} !(Alnum|'-'|'.')

Interp    := '{' List '}'
Ampersand := push; parse LinePrefix:='&'; Space Def (';'|Ending); pop
String    := '"' {Interp | CharEsc |                                       !('\'|'{'|'}'|'"') Text} '"'
Content   :=     {Interp | CharEsc | Markup | Ampersand | Comment | Line | !Special           Text}
Contents  := Content | '~' Content {'~' Content}
Markup    := '<' Scan Tags Scan ":>" Space Ind Contents Ded 
          |  '<' Scan Tags Scan ';'  Scan      Contents '>'
          |  '<' Scan Tags Scan '>'  Scan      Contents "</" Ident Space {'/' Ident Space} '>'

Key       := !Alnum Space !":="
Return    := ("return"|"yield"|"break"|"continue") Key
Reserved  := ("catch"|"do"|"else"|"if"|"in"|"is"|"not"|"then"|"until"|"where"|"with"
             |"alias"|"const"|"live"|"mutable"|"ref"|"set"|"var") Key | Return

DotSpace  := '.' &(0o09 | 0o20 | Ending)
Brace     := Scan '{' List '}' Space
Block     := Brace | DotSpace Space Def Space | ':' Space Ind List Ded
BraceInd  := Brace | Ind List Ded
KeyBlock  := Block
Defs      := Def {Space ',' Scan Def}
Paren     :=  '(' List  ')' Space
QualIdent := ['(' List ":)" Space] Ident
Specs     := [ScanKey "with" Key] '<' Scan Choose Space '>' Space (Specs | !Specs)
Tags      := Space (!'/' Call ScanKey '.' | !Reserved) QualIdent Space {Invoke} [',' Scan Tags]

Do        := ScanKey "do"    Key (KeyBlock | Def)
Until     := ScanKey "until" Key (KeyBlock | Def) | ScanKey "catch" Key Invoke
Then      := ScanKey "then"  Key (KeyBlock | Def)
Else      := ScanKey "else"  Key (ScanKey If | !(ScanKey If) (KeyBlock | Def))

Invoke    :=          [Specs] (Paren [Specs] (Block | Do  ) | Block [[Specs] Do  ]) (Until | !Until)
If        := "if" Key [Specs] (Paren         (Block | Then) | Block [        Then]) (Else  | !Else )
In        := ("in" Key | ':') Space (In | &Choose NotEq [Space Specs])
Var       := ("var"|"set"|"ref"|"alias") Key Space Choose

Base      := '(' List ')' | Num | Char | Path | String | Markup | If | !Reserved QualIdent
Call      := Base    {Space Postfix}
Postfix   := Invoke  | !Invoke (Paren | Specs) | ("at"|"of") Key (KeyBlock | Fun)
                     | ('^' | '?' | '[' List ']' | ScanKey '.' QualIdent)
Prefix    := Call    | ('^' | '?' | '[' List ']' | '+' | '-' | '*') Space (Brace | Prefix)
Mul       := Prefix  { Space ('*' | '/' | '&'       ) Scan  Prefix  }
Add       := Mul     { Space ('+' | '-'             ) Scan  Mul     }
To        := Add     [ Space ("to" Key | ".." | "->") Scan  To      ]
Choose    := To      [ Space ('|'                   ) Scan  Choose  ]
Greater   := Choose  [ Space ('>'  | ">="           ) Scan  Greater ]
Less      := Greater [ Space ('<'  | "<="           ) Scan  &(Choose Space !'>' !">=") Less]
NotEq     := Less    { Space ("<>"                  ) Scan  Choose  }
Eq        := NotEq   { Space ('='                   ) Scan  NotEq   }
Not       := Eq      |       ("not" Key             ) Space Not
And       := Not     { Space ("and" Key             ) Scan  And     }
Or        := And     { Space ("or"  Key             ) Scan  Or      }
Def       := (Or | (In | Var) Space (('='|":="|"+="|"*="|"/=") Space (BraceInd | Def) | !'=' !":="))
             { &In Def
             | Space   ":="    Space (BraceInd | Def )
             | Space   "where" Key   (KeyBlock | Defs)
             | ScanKey "is"    Key   (KeyBlock | Def ) } StopDef
          |  ('&'|"..") Space Def | Return [Block | Def] StopDef
Fun       := Def { Space ("over" | "when" | "while") Key (KeyBlock | Defs)
                 | Space ("=>" Space | "next" Key) (BraceInd | Fun) } StopFun
Expr      := Fun {Space '@' Space Call} StopExpr
          |  '@' Space Call Scan &('@' | QualIdent) Expr

StopExpr  := &(Space (":)" | ')' | ']' | '}' | ';' | ',' | Ending !(ScanKey "is" Key)))
StopFun   := &(Space ("@")) | StopExpr
StopDef   := &(Space ("=>" | ("over"|"when"|"while"|"next") Key) | StopFun)

Commas    := Expr {',' Scan Expr}
Separator := (';' | Ending) Scan
List      := push; set LinePrefix=""; Scan [Commas {Separator Commas} (Separator | !Separator)]; pop
File      := [0oEF 0oBB 0oBF] set Nest=true; set BlockInd=""; set LineInd=""; List end

Verse Grammar Extension to Directories
------------------------------------------------------------------------------------------------------------------------

Here we specify how to translate the files in a directory or zip file D into text interpretable by the
Verse Grammar. We form a Verse module "module {\n...\n}". We recursively gather a list of all filenames and
directory names FS residing in D and (recursive) subdirectories whose names matches *.more[.*]. For each name
F in FS, sorted lexocographically (by case-sensitive UCS-32 hence UTF-8 sort order), we handle F as follows:

1) If F is a *.zip file, we treat it as a directory with the .zip extension omitted.

2) Otherwise we ignore F if it:
   - matches .*
   - contains *.ignore[.*]
   - is a file containing no '.'
   - is a directory containing *.more[.*]

3) Otherwise if F is a directory, we break it into a Name (up to the first '.') and Specifiers (following each '.').
  We produce an error if the Name or any Specifiers don't match Verse Grannar Ident syntax.
  We add a field:
      Name<Specifiers_0>...<Specifiers_n>:=module{...};
  and recursively translate the contents of directory F into a module as above.

4) Otherwise if F is a file named Self.verse, we append its contents with leading and trailing \n to the module text.

5) Otherwise if F is a file, we break it into a Name (up to the first '.'), Specifiers (between each '.' pair),
  and Extension (following last '.'). We produce an error if Name, Specifiers, or Extension don't match Verse Grammar
  Ident syntax.

  We append to the module declaration according to Extension as follows:
      .verse: <precise file contents>\n
      .value: Name<Specifiers_0>...<Specifiers_n>:=(<precise file contents>)\n
      .txt:   Name<Specifiers_0>...<Specifiers_n>:string="<file contents as String literal with proper escaping>"\n
      .bin:   Name<Specifiers_0>...<Specifiers_n>:[]nat8=array{<file bytes as comma-separated Num 0-255>}\n

  If Specifiers includes 'closed', then we omit <closed> from the declaration above, and we omit any ":type" there.
  Other file formats (.png, .gltf, etc) will be added over time with similar mappings.

Verse Core Grammar
------------------------------------------------------------------------------------------------------------------------

The full Verse grammar can be simplified to the following core grammar subset, which is canonical if Num is canonical:

    Expr   := Num
           |  ['(' Expr ':)'] Ident
           |  Expr ['(' Expr ')' {'<' Expr '>'}] '{' List '}' ['until' '{' List '}']
           |  'if' ['(' Expr ')' {'<' Expr '>'}] '{' List '}' ['else'  '{' List '}']
           |  Expr '(' List ')'
           |  Expr '[' List ']'
           |  Expr '<' Expr '>'
           |  '&' Expr
    Commas := Expr   {',' Expr  }
    List   := Commas {';' Commas}

Syntax Equivalences between full grammar (left) and core grammar (right):

    Num Units     ---> units'Units'[Num]
    s0(s1). op2   ---> s0(s1){op2} <--> s0(s1) do op2

    s0:s1=s2      ---> s0 := (:s1 := s2)
    s0:s1<fx>=s2  ---> s0 := (:s1<fx> := s2)
    s0=>s1        ---> function(s0){s1}
    s0^           ---> operator'^'[s0]     # Same for postfix ^ ?
    s0&s1         ---> operator'&'{s0,s1}  # Same for infix & -> to | <> = and or += -= *= /=
    s0+s1         ---> operator'+'[s0,s1]  # Same for infix * / + - >= > <= < .. 
    :s0           ---> prefix'-'{s0}       # Same for prefix : not
    -s0           ---> prefix'-'[s0]       # Same for prefix ^ ? + - * []
    -{s0}         ---> :-type{s0}          # Same for prefix ^ ? + - * []

Inference Rule Conventions
------------------------------------------------------------------------------------------------------------------------

Sequent Calculus:

    We use the notation of Sequent Calculus (https://en.wikipedia.org/wiki/Sequent_calculus)
    to specify how we verify programs. This is a brief overview of what it means.

    Facts are statements like succeeds<=decides that are true or false.

    We use the turnstile written |- to say the following:

        Under these Assumptions |- this Proposition is proven

    We write our inference rules bottom-up as follows:

        (3) ...is to make these assumptions... |- (4) ...and prove this proposition.
        -------------------------------------------------------------------------------------
        (1) Under these assumptions...         |- (2) ...one way to prove this proposition...

    Given a program with syntax s, we start inference by writing at the bottom of a sheet of paper,
    the turnstile:

        empty |- explore(top_allows,fxv) {scope c {exists i x; syntax(opv,i,x){s}}} # fxv,i,x,opv fresh

    Then we choose an inference rule to apply to the top of the stack based by matching our program
    on the right hand side of the |- turnstile to one of our inference rules. We then apply the inference
    rule and write a new turnstile |- above the line, using the inference rule to transform our assumptions
    on the left hand side of the top turnstile and our program on the right hand side.

    When looking for an inference rule we can apply to make further progress, we say a given
    inference rule G... |- Program is available if all of the assumption patterns expressed in the syntax
    of G... are matched by assumptions contained in G, and Program matches the goal we're trying to prove.

    We're finished when the (ProgramElim) inference rule is applicable. It has no content above its
    horizontal line, signifying that inference has completed successfully.

    If we've applied all of the inference rules that are available, and still aren't able to apply
    (program-elim), then our attempted verification of the program is rejected.

Our Use of Sequent Calculus:

    Within Sequence Calculus, we achieve bidirectional functional logic inference and effects sequencing by
    providing inference rules of the form:
        G |- Program,         meaning: under the assumptions in G, our program is valid;
        G |- ProgramOp[c,op], meaning: under the assumptions in G, our program containing
                                       a flexibility context c and operation op is valid.

    EBNF patterns like ProgramOp[c,op] defined as paramterized EBNF grammars below and enable us to match
    constructs nested arbitrarily deeply within Verse programs, while binding symbols (like c and op in this
    example) which are available to the inference rules:

        Below the horizontal line, c and op in ProgramOp[c,op] are wildcards: if the program
        matches this EBNF grammar, then c and op are bound to particular program operations.

        Above the horizontal line, those EBNF patterns and wildcards have the meaning bound to them
        below the line. In this way, we can easily express parameteric transformations of program syntax,
        and parametric analysis of program contents.

        Above the horizontal line, new EBNF terms like 'x' that aren't defined below the line are
        assumed to be "fresh", meaning: when the inference rule is applied, we replace it with a newly-named
        EBNF term of the same type which hasn't been seen elsewhere in inference.

    All of our rules are structured to contain the whole program on the right-hand side of the |-, so that each
    available inference adds more assumptions while preserving our ability to make further inference about
    the whole program.

Sequent Calculus Extensions and Conventions

    We use several special forms of inference:

    Condition or Definition
    G... |- Program1 #
    ---------------- # Convention: Conditions and definitions above the top turnstile must for the inference rule to apply.
    G... |- Program2 #

    G... |- Program1 #
    G... |- Program2 # 
    ---------------- # Convention: A stack of turnstiles above the horizontal line means we take their conjunction.
    G... |- Program3 #
    
    G... |- Program1 #
    ---------------- # Convention: A stack of turnstiles below the horizontal line means disjunction: we apply the 
    G... |- Program2 # inference if either applies.
    G... |- Program3 #

    Rule names preceded by '*' have not yet been tested through implementation.

Inference Rule Definitions
------------------------------------------------------------------------------------------------------------------------

Definitions (as EBNF grammars) used by our inference rules:

    EBNF Expressions         E := Anything defined here
    Natural Numbers        m n := 0 | 1 | ...
    Integers                 Z := n | -n where n<>0
    Rational Numbers         Q := Z | Z0/Z1 where Z1<>0 and Z1<>1 and GCD(Z0,Z1)=1
    Pointers                   := P
    Path                       := Path production from grammar above, with Label part canonized as lower case
    Variables                  := f | g | h | i | j | k | t | w | x | y | z
    Contexts                   := c | d | e | ec | ic | vc | fc | sc
    Operation Variables        := opv
    Effects Variables          := fxv
    Syntax and Metasyntax    s := List | Expr | Num | Ident | Unquote(u)
    Effects                 fx := cardinalities | ambiguates | succeeds | decides | resolves | abstracts | fails | contradicts
                               |  diverges | unifies | demands | varies | allocates
                               |  iterates         | reads         | writes         | interacts         | throws         | suspends
                               |  iterates_pending | reads_pending | writes_pending | interacts_pending | throws_pending | suspends_pending
                               |  imperatives | pendings | transacts | top_allows | lambda_allows | rejects | effects
                               |  fx0\/fx1 (canonized from lattice)
                               |  ~fx      (canonized from lattice)
    Open World Specifier    oc := open|closed
    Equality Strength       st := succeeds\/ambiguates | decides\/ambiguates | abstracts
    Atom Values           atom := Q | P | Path | () | Print | operator'^' | prefix'set' | fx | oc
    Implied fx             ifx := none | fx
    Lambdas                  L := lambda(oc,fx,u) c0 i w {op0} range c1 j z {op1}
    Heads                 head := atom                                             # atom value
                               |  L                                                # lambda_vertex
                               |  macro L                                          # macro_vertex
                               |  tuple(p) x                                       # array_vertex
                               |  nom(Path) x                                      # nominal data type wrapper
    Vertices         p,q,r,u,v := x                                                # variable with top shape
                               |  head                                             # variable with specific shape
                               |  u(v)                                             # call_vertex
    Programs           Program := explore(top_allows,fxv) {...; scope c {op}; ...}
    Operations              op := u=v                                              # op::atom, op::unify, op::span, op::lambda, op::call
                               |  u=fail                                           # op::fail
                               |  op0; op1                                         # op::sequence
                               |  op0|op1                                          # op::choice
                               |  exists x_0 ... x_n                               # op::exists
                               |  scope c {op}                                     # op::scope
                               |  beta L(p) c0 {op0} range c1 {op1}                # op::call logic to track beta conversion effects in c
                               |  in(fx,fxv|none) c i x {op}                       # op::in
                               |  stage(fx,fxv) c0 i x {op0} value c1 y {op1}      # op::stage
                               |  verify(fx,err) c     {op}                        # op::verify
                               |  assume(fx)     c     {op}                        # op::assume
                               |  iterate         {op_0; ...; op_n}                # op::iterate
                               |  fold(x0) c0 {op0} then c1 x1 {op1} else c2 {op2} # op::iterate
                               |  explore(fx,fxv) {op_0; ...; op_n}                # part of op::lambda
                               |  cast(u)         {HeadPattern_0 => {op_0}; ...}   # WhenCastStep (dynamic, paraconsistent)
                               |  cast(u,opv,err) {HeadPattern_0 => {op_0}; ...}   # WhenCastStep (static, consistent)
                               |  err                                              # op::stuck
                               |  syntax(opv,u,v){s}                               # op::reduce (NewRedex)
                               |  resolved(opv){op}
                               |  define Ident {definable}
    Head Pattern   HeadPattern := head | prose unambiguously describing a nested head normal form

                               EBNF patterns matching operations that start a new context and its top level operation
    ScopeStartOp[SC,OP]        := scope           SC     {OP }
                               |  beta L(p)       SC     {OP } range c1 {op1}
                               |  beta L(p)       c0     {op0} range SC {OP }
    FlexibleStartOp[FC,OP]     := in(fx,fxv|none) FC i x {OP }
                               |  stage(fx,fxv)   FC i x {OP } value c1 y {op1}
                               |  stage(fx,fxv)   c0 i x {op0} value FC y {OP }
                               |  verify(fx,err)  FC     {OP }
                               |  assume(fx)      FC     {OP }
                               |  fold(x0)        FC     {OP } then d1 x1 {op1} else d2 {op2}
    ExploreStartOp[EC,OP]      := explore(fx,fxv) {...; ScopeStartOp    [EC,OP]; ...}
    IterateStartOp[EC,OP]      := iterate         {...; FlexibleStartOp [EC,OP]; ...}

                               BNF pattern that matches splits in program verification.
    SplitOp[C_0,...,C_N]       := explore(fx,fxv) {ScopeStartOp   [C_0,op_0]; ...; ScopeStartOp   [C_n,op_n]}
                               |  iterate         {FlexibleStartOp[C_0,op_0]; ...; FlexibleStartOp[C_n,op_n]}

                               EBNF patterns matching operations within the span of a context
    ScopeSpan[OP]              := OP
                               |  ScopeSpan[OP]; op
                               |  op; ScopeSpan[OP]
                               |  resolved(opv){OP}
    FlexibleSpan[OP]           := ScopeSpan   [OP | ScopeStartOp    [c,FlexibleSpan[OP]]]
    IterateSpan[OP]            := FlexibleSpan[OP | FlexibleStartOp [c,IterateSpan [OP]]]
    ExploreSpan[OP]            := IterateSpan [OP | IterateStartOp  [c,ExploreSpan [OP]]]
    ProgramSpan[OP]            := ExploreSpan [OP | ExploreStartOp  [c,ProgramSpan [OP]]]

                               EBNF patterns matching a context-starting operation and an operation in its span:
    ScopeOp[SC,OP]             := (ExploreStartOp|IterateStartOp|FlexibleStartOp|ScopeStartOp)[SC,ScopeSpan   [OP]]
    FlexibleOp[FC,OP]          := (ExploreStartOp|IterateStartOp|FlexibleStartOp             )[FC,FlexibleSpan[OP]]

                               EBNF patterns matching a program span exposing an operation OP in a flexibility context FC.
    ProgramOp[FC,OP]           := ProgramSpan[FlexibleOp[FC,OP]]

Effects Lattice:

    Effects form a lattice where fx0\/fx1 is lattice join, and fx0/\fx1 is lattice meet.
    Cardinality effects are information-ordered by the following lattice:

                                  cardinalities
                                   |         |
                               abstracts     |
                              /    |         |      UNDECIDABLE
                             /  resolves     | 
            ================/======|=========|================
            decides\/ambiguates    |    iterates
                         |     \   |   /     |       DECIDABLE
                         |      decides      |
           ==============|=========|===\=====|================
           succeeds\/ambiguates    |    \    |
                              \    |     \   |
                               succeeds   fails        DECIDED
                                   \        / 
                                   contradicts

    For all effects fx, we have contradicts<=fx<=effects.
    We have varies<=allocates and varies<=reads.
    We define compound effects:
        cardinalities   = abstracts\/iterates
        transacts       = reads\/writes\/allocates
        pendings        = iterates_pending\/reads_pending\/writes_pending\/interacts_pending\/throws_pending\/suspends_pending
        imperatives     = unifies\/diverges\/transacts\/suspends\/interacts\/throws\/pendings
        top_allows      = succeeds      \/ unifies\/diverges\/transacts\/interacts
        lambda_allows   = resolves      \/ imperatives
        lambda_defaults = succeeds      \/ unifies\/diverges\/transacts,
        effects         = cardinalities \/ imperatives \/ rejects

    The effects sequencing operation fx0*fx1 captures the cumulative effects of running op0 with effects fx0
    then running op1 with efffects fx1. It's associative, transitive, and commutative.

        fx0*fx1 = (fx0\/fx1)/\
            if      fx0/\fx1 <= fails   then fails\/imperatives\/rejects
            else if fx0/\fx1 <= demands then effects
            else                             effects/\~demands

        Note that fails*contradicts=fails, because an op having effects <contradicts><converges> means the context
        containing it cannot commit with <succeeds>, but it doesn't preclude other ops in the context failing first.

    Pending Effects

        The operation PendingAfter[fx] says what pending effects should be held subsequent to this operation having fx.
        During inference, each operation receives *_pending effects from its sequentially prior operation,
        calculates its actual suspended effects fx, and propagates PendingAfter[fx] to the sequentially next operation.

        PendingAfter[fx] :=
            if(iterates <=fx) {iterates_pending } else {none} \/
            if(reads    <=fx) {reads_pending    } else {none} \/
            if(writes   <=fx) {writes_pending   } else {none} \/
            if(throws   <=fx) {throws_pending   } else {none} \/ # throws&interacts can't run where fails allowed,
            if(interacts<=fx) {interacts_pending} else {none} \/ # so we needn't relatively sequence them.
            fx \/ ~pendings

        PendingDown[fx] says how a compound operation's pending effects propagate to its sequentially first sub-operation.
        PendingDown[fx] := fx\/~pendings

    Effects Compliance

        The operation CompliesFx(ActualFx,SpecifiedFx) describes to what extent ActualFx is a subset of SpecifiedFx.
        It's used in analyzing actual versus specified effects in 'lambda' and 'stage.
        When its cardinalities aren't a subset, we capture that as pootential failure cardinalities.
        When its imperatives aren't a subset, we capture those as excess effects.

        CompliesFx(ActualFx,SpecifiedFx) = ActualFx /\ ~SpecifiedFx /\ ~cardinalities \/
            if      cardinalities/\ActualFx              <= SpecifiedFx then succeeds
            else if cardinalities/\ActualFx/\SpecifiedFx <= contradicts then fails
            else                                                             cardinalities/\ActualFx

        The operation GateFx(SpecifiedFx,CompliedFx) widenes SpecifiedFx by CompliedFx, where CompiledFx is expected
        to be derived from prior CompliesFx calculations.

        GateFx(SpecifiedFx,CompliedFx) =
            if      cardinalities/\CompliedFx <= contradicts then contradicts
            else if cardinalities/\CompliedFx <= succeeds    then SpecifiedFx \/ CompliedFx/\imperatives
            else                                                  SpecifiedFx \/ CompliedFx

Bound Variables:

    Inferences like beta reduction ensure they are instantiating fresh variables by identifying the
    relevant bound variables BVS[op] bound in their Operations op, and renaming them when applied.
    We define BVS as the smallest set of Variables and Contexts such that:

    BVS[ProgramScope   [c,op]] includes {c}
    BVS[ProgramFlexible[c,x ]] includes {x}

Inference Rule Conventions

    (*Syntax)  general    syntax desugaring
    (*Define)  definition syntax desugaring
    (*Macro)   macro      syntax desugaring
    (*Set)     mutation   syntax desugaring

    (*Equal)   Equality inference.
    (*Intro)   Inference rules add initial assumptions for op, propagate information up and down from the op to
               subops, and possibly add subsequent ops.
    (*Up)      Inference rules propagate information up from subops to ops and subcontexts to contexts
    (*Down)    Inference rules propagate information down from ops to subops and contexts to subcontexts
    (*Elim)    Inference rules that reduce an op to a different op.
    (*Reach)   Data flow reachability rules

Verifier Syntax Reduction Rules
------------------------------------------------------------------------------------------------------------------------

Overview

    When we verify Verse, we run inference rules that simultaneously reduce syntax and perform all other aspects
    of program verification.

    We translate Syntax s defined by the Verse Core Grammar into Operations op using these Syntax Reduction Rules,
    which can be full inference rules as defined earlier, or can be written in shorthand as follows:
  
                         Optional Conditions
             (RuleName)  syntax(opv,u,v){s} ---> op

    The shorthand above translates precisely to the inference rule below:

                         Optional Conditions
                         G |- ProgramOp[c,resolved(opv){op} ]
             (RuleName)  ------------------------------------
                         G |- ProgramOp[c,syntax(opv,u,v){s}]

    As with inference rules, all variables to the right side of the ---> that aren't defined on the left side are fresh.
    The resolved(opv){op} op is plumbing involved in ensuring syntax resolves consistently everywhere.

Abstraction Syntax:

    Verse's x:t and x:t=v constructs are very powerful and expose three general abstraction mechanisms:
    - Type abstraction (x:nat says x belongs to the set of natural numbers)
    - Effects abstraction (when implying, x:nat stands for a natural-number-producing computation with any allowed effects)
    - Type constructors as functors (we can generalize pair(t:type):=type{x:t,y:t} to pair(f:function):=type{x:f,y:f})

    A simple intuition is that syntax(opv,u,v){:f} translates to something like v=f[u],
    but it's wrapped in an 'in' op handling abstraction of types and effects.

    This syntax is designed to nest so you can write x:xs, i->x:xs, i->j->x::xs, etc.
    To handle this, we rely on an Verse Grammar EBNF pattern to match syntax containing zero or more ':':

        UnwrapColons[s] := :s
                        |  s (where s is any syntax not of the form :s)

    On the left hand side of a definition, :UnwrapColons[s0] stands for greedy :s0 or ::s0 or :::s0, etc.
    Right-hand UnwrapColons[s1] stands for s1 prefixed by the number of ':' the left-hand matched.

             (InSyntax)  syntax(opv,u,v){:UnwrapColons[s0]} ---> syntax(opv,u,v){_->_:UnwrapColons[s0]}

         (InSpecSyntax)  TODO, and expand UnwrapColons to capture specifiers

Overview of flow graphs and domination:

    We use flow graph theory in Verse verification to conservatively determine which unifications are guaranteed to
    succeed or to be decidable at runtime under all possible orders of head normal forms arriving from various sources.

    In graph theory, a directed graph consists of Vertices u,v,w,x,... and directed edges of the form v->u.
    We say v->*u (vertex v reaches vertex u), if v=u or v->w and w->*u. When v->*u, the set of paths
    of v->*u includes every sequence v->...->u of vertices that go from v to u.

    A control flow graph is a graph containing an entry node x reaching all vertices in the graph.
    Control flow graph dominators are defined in https://en.wikipedia.org/wiki/Dominator_(graph_theory),
    which says that v>>u (vertex v dominates vertex u) if every path from the entry node x to u passes through v.

    For a general directed graph (not necessarily having a unique entry node), we can define dominators more generally
    as v>>u if v->*u and for all y->*u, we have either v->*w (w is between v and u) or every path w->*u passes
    through v (w is behind v).

    In practice, we determine the dominators of a graph by calculating the fixed point, starting with the set of
    dominators candidates where v>>u if v->*u, keeping v>>u if for all w where w->u, we have v=w or v>>w.

    When our graph is a data flow graph in which vertices are variables and edges are data flows into flexible 
    unificands as assumed by our inference rules, v>>u implies v=u. However, this gives us only weak equality 
    inference in Verse, whereas we want a stronger data flow inference that effectively considers equality
    assumptions we've made through other rules such as modus ponens.

Strength-cut dominators:

    For Equality Strengths st and contexts c, we define the strength st-cut subgraph of our data flow graph as
    the subgraph consisting of edges v->u where our fully saturated assumption set G doesn't contain an
    assumption st{v=u}@c.

    Now we define the strength-cut domination relationship st{v>>w}@c, in a context c lacking pending <unifies>, as:
    st{v>>u}@c means that v>>u in the strength st-cut subgraph.

    Why do we do this? Because succeeds{v>>w} means that there is guaranteed to be no runtime dispute about whether
    v=w will succeed at runtime, hence in the verifier we can conclude (and add a new assumption) succeeds{v=w}@c.
    And when decides{v>>w}, we can assume decides{v=w}.

About head normal forms and head shapes:

    We define data flow Vertices as including atoms like 7 and other head-normal forms such as tuple(u) t.
    Such head normal forms impede some forms of data flow, and to reason about this, we define:

    HeadShape[v]@c produces values in a lattice where:
        HeadShape[u]          <= any    (for all Vertices u)
        HeadShape[atom]@c     <= atom   (where atom comes from Atom Values in the spec)
        HeadShape[tuple(x) t] <= []any  (for all tuples)
        HeadShape[tuple(x) t] <= [n]any (when our assumptions G satisfy G[abstracts{x=n}@c] for n in the Natural Numbers)

About data flow among tuples and their elements:

    When a program contains unifications among tuples and among elements of those tuples, our data flow
    analysis must consider not only the data flow among unified tuples and among unified tuple elements,
    but also the implicit data flow among tuple elements arising from data flow among the tuples containing them.

    One way we could do that is to notice that an equation contains a tuple, and for every vertex v in that
    tuple equation, create 'virtual elements' v[0] etc; and for every edge v->u in the tuple equation,
    create a 'virtual edges' v[0]->u[0] etc. This works, but increases the complexity the verifier's analysis.

    Fortunately, there's a better way: We add data flow rules relating flow among tuples to flow among elements,
    and relating element dominators to tuple dominators, which precisely capture the necessary flow relationships.
    This approach relies on the idempotence of data flow inference: given an inference turnstile,
    running this inference rule once guarantees an immediately subsequent (dominator) inferences
    will infer nothing now.

Data Flow Analysis Process

    We define a meta inference rule (DataFlow) that is available at all times.
    When it's run, it copies our assumptions G to local flow assumptions FG, then runs a four-stage process.
    Stages 1-3 each run to saturation in sequence, manipulating the local flow assumptions FG.
    Stage 4 takes the final flow assumptions FG and adds certain conclusions back to G, discarding the rest.

    Data Flow Analysis Stage 1: We run the following inference rules to saturation. At the end, we have:
    - The Strength-cut data flow graph, defined by direct edges u->v and product-navigible direct reachability u->*v.
    - Calculation of vertex strength-equated shape inclusions v<=u.

                         FG, v->u |- ProgramUnify[c,op,u,v]
        (UnifyFlexible)  ---------------------------------------------------------
                         FG       |- ProgramUnify[c,op,u,v] & ProgramFlexible[c,u]

                         (Intersection over all x where FG[st{x=v}] of HeadShape[x]) <= HeadShape[u]
                         FG, st{v<=u} |- Program
          (VertexShape)  -----------------------
                         FG           |- Program

                         Not FG[st{v=u}@c]
                         FG, st{v->*u}@c |- ProgramFlexible[c,u]
          (DirectReach)  ---------------------------------------
                         FG, v->u        |- ProgramFlexible[c,u]

                         Not FG[st{v=u}@c]
                         FG, st{w->*u}@c        |- Program
      (TransitiveReach)  ---------------------------------
                         FG[st{w->*v}@c, v->u]  |- Program

                         F is tuple(p) t | L
                         FG, succeeds{us->*us at p}@c              |- Program
         (ProductReach)  ----------------------------------------------------
                         FG[abstracts{us=F}@c, abstracts{us(p)}@c] |- Program

                         FG, st{vs->*us at q}                   |- Program
    (EqualProductReach)  -------------------------------------------------
                         FG[st{vs->*us at p}, abstracts{p=q}@c] |- Program

                         For all xs&q where FG[abstracts{xs(q)}@c], we do not have FG[st{xs=ws}@c, abstracts{q=p}@c].
                         FG, st{ws->*us at p}@c               |- Program
(TransitiveProductReach) -----------------------------------------------
                         FG[st{ws->vs}@c, st{vs->*us at p}@c] |- Program

                         Not FG[st{ws=vs}@c]
                         FG, st{ws(q)->*us(p)}@c                                            |- Program
         (ElementReach)  -----------------------------------------------------------------------------
                         FG[ws->vs, abstracts{ws(q)}@c, abstracts{q=p}, st{vs->*us at p}@c] |- Program

                         For all xs&ys&qs, we do not have FG[xs->*ws at q, st{xs=ys}@c, abstracts{ys(q)}, abstracts{q=p}]
                         FG, st{z->*us(p)}@c    |- Program (z fresh)
    (BlockElementReach)  ---------------------------------
                         FG[st{ws->*us at p}@c] |- Program

                         Not FG[abstracts{xs(r)}@c, abstracts{r=p}@c]
                         FG, st{z->*us(p)}@c, st{z=ys(q)}                                                |- Program (z fresh)
    (EqualElementReach)  ------------------------------------------------------------------------------------------
                         FG[xs->ws, abstracts{ys(q)}@c, abstracts{q=p}, st{ws->*us at p}@c, st{xs=ys}@c] |- Program

    Data Flow Analysis Stage 2: We run the following rules to saturation to treat strength-reachable vertices as strength-dominator
    candidates. The output is a set of dominators candidates st{v>>u}@c which the next stage will narrow down by enforcing dominator
    consistency rules.

                         FG, st{u>>u}@c |- ProgramFlexible[c,u]
        (SelfDominator)  --------------------------------------
                         FG             |- ProgramFlexible[c,u]

                         FG, st{w>>u}@c            |- Program
       (EqualDominator)  ------------------------------------
                         FG[st{w=v}@c, st{v>>u}@c] |- Program

                         Not FG[st{u=w}@c]
                         FG, st{v>>u}@c                       |- ProgramFlexible[c,u]
        (FlowDominator)  ----------------------------------------------------------------------------------------------------------
                         FG[st{v>>w}@c, w->u, ~unifies{op}@c] |- ProgramFlexible[c,u]
                                                              &  ProgramSpan[(ExploreStartOp|IterateStartOp|FlexibleStartOp)[c,op]]

                         Candidate dominators of w flow across non-strength-cut edges w->u to become candidate dominators of u.
                         We only do this once u's context is free of <unifies> effects, meaning no further unifications (and thus
                         data flow arrows) may be introduced.

                         FG, st{vs(q)>>us(p)}@c                                                 |- Program
        (CallDominator)  ---------------------------------------------------------------------------------
                         FG[abstracts{vs(q)}@c, st{vs>>us}@c, abstracts{us(p)}@c and st{p=q}@c] |- Program

    Data Flow Analysis Stage 3: We run the following rules to saturation to remove inconsistent candidate dominators fx{v>>u}@c.
    Upon completion, the remaining candidate dominators are guaranteed to be actual dominators.

                         Not:
                             for all all w where FG[w->u] and not FG[st{w=u}]:
                                 (FG[st{v=w}@c] or FG[st{v>>w}@c]) and FG[st{v<=u}@c]
                         FG with st{v>>u}@c removed |- Program
      (FlowConsistency)  -------------------------------------
                         FG[st{v>>u}@c]             |- Program

                         Not:
                             FG[st{us(p)>>us(p)}@c] or
                             exists ws.   FG[st{xs(p)>>us(p)}@c, st{xs=vs}@c] or
                             exists xs q. FG[st{xs(q)>>us(p)}@c, st{xs=vs}@c, st{q=p}@c,(st\/computes){us(p)}@c, (st\/computes){ws(q)}@c]
                         FG with st{vs>>us} removed         |- Program
  (FunctionConsistency)  ---------------------------------------------
                         FG[st{vs>>us}, abstracts{us(p)}@c] |- Program

                         Not:
                             (exists ws cs q. FG[st{v=ws(q)}@c, abstracts{q=p}, st{cs>>us}@c, st{cs>>ws}@c)] or
                             (for all w where FG[succeeds{w->*u}]. (FG[st{v=w}@c] or FG[st{v>>w}@c]) and FG[st{v<=u}@c])
                         FG with st{v>>us(p)}@c removed |- Program
      (CallConsistency)  -----------------------------------------
                         FG[st{v>>us(p)}@c]             |- Program

    Data Flow Analysis Stage 4: We apply our local dominator assumptions FG to global equality assumptions G.
        
                         G[st{v=u}@c]   |- Program
        (DominatorElim)  -------------------------
                         FG[st{v>>u}@c] |- Program

                         G[st{as(p)=bs(q)}@c]                                                       |- Program
 (ElementDominatorElim)  -------------------------------------------------------------------------------------
                         FG[st{cs>>as}@c, st{cs>>bs}@c, abstracts{p=q}@c, abstracts{as(p)=bs(q)}@c] |- Program

Verse Verification Aftermath
------------------------------------------------------------------------------------------------------------------------

Verification of a program with syntax s starts by introducing fresh c,u,v,opv and making the initial inference:
    empty |- explore(top_allows,fxv) {scope c {exists u v; syntax(opv,u,v){s}}}

We then apply available inference rules beginning with (ProgramIntro) until one of the following occurs:

    1) The (ProgramElim) inference rule is discharged, so a verifier should accept the program.
       Our fully verified desugared program is then syntax(opv,u,v){s} with the following substitutions
       named (Desugar) applied where G are the Assumptions available to (ProgramElim) and c is the
       top-level scope from above:

       ProgramOp[d,syntax(opv1,u1,v1){s}]    ---> op1 where G[opv1:=op1@c]
       ProgramOp[d,scope e {op}]             ---> ProgramOp[c,op[d/e]]
       ProgramOp[d,define Ident {definable}] ---> ProgramOp[c,exists]

       The resulting program has the following properties:
       - It's object code: It contains no syntax to be desugared or identifiers to be resolved.
       - It's proof-carrying: The verified program can be verified again and it will be accepted.

    2) The verifier runs forever or appears to, due to an infinite expansion sequence like f():=f().
       The verifier should provide an optional dead-mans switch or interrupt mechanism so it can report
       the program locations involved in the looping.

    3) The verifier runs to saturation (meaning: no inference rules can introduce new information into the program)
       without the (ProgramElim) inference rule becoming available, so a verifier should reject the program.
       We then produce an error message according to the terminal Assumptions G according to
       a specification TBD that considers:
       - Program sequential order.
       - Dependencies among the program which may assist in identifying the ultimate cause of the error.
       - Errors err embedded in various Operations.
       - Operations whose inferred effects are disallowed (we have fx1{op}@c and fx2:_:_@c without fx1<=fx2).

Verse Runtime
------------------------------------------------------------------------------------------------------------------------

The runtime is defined on simpler Operations translated with the rule named (RuntimeSimplify):

    ProgramOp[c,lambda(oc,fx,u) d0 i w {op0}; range d1 j z {op1}] ---> ProgramOp[c,icfp_lambda(u) i z {exists j w; op0; op1[c/d0][c/d1]}]
    ProgramOp[c,in(fx,fxv) d i x {op}]                            ---> ProgramOp[c,exists i x; op[c/d]]
    ProgramOp[c,stage(fx,fxv) d0 i x {op0} value y {op1}]         ---> ProgramOp[c,exists i x y; i=y; u=x; op0[c/d0]; op1[c/d0]]
    ProgramOp[c,verify(fx,err) d {op}]                            ---> ProgramOp[c,op[c/d]]
    ProgramOp[c,assume(fx)      d {op}]                           ---> ProgramOp[c,op[c/d]]

We specify the contents of the native environment module /verse.org/Verse which is available to all
programs without requiring import{...} or using{...} statements, as follows:

1) Every field of the module /verse.org/Verse is defined in the file Verse.verse.
   Where the field defined in Verse.verse succeeds, it's complete. This includes types like:
   char8, char16, char32, float32, float64, string, path.

2) Where the field defined in Verse.verse is abstract, we have elaborated on it as appropriate
   in the verifier and/or runtime inference rules above.

Expected Command Line Tooling
------------------------------------------------------------------------------------------------------------------------

We'll provide command line tools for Verse programming supporting both transient (traditional)
and persistent (metaverse style) compilation, deployment, and running of Verse code.
It will look something like:

verse [VerifierArgs] [Filename[.verse]] [ShellArgs] [do VerseExpr]
    If [do VerseExpr] verifies: let(Args:=(comma separated ShellArgs); Envs:=...) {file_text; VerseExpr}
    Else              verifies: let(Args:=(comma separated ShellArgs); Envs:=...) {file_text}
    If success, runs it.

verse [VerifierArgs] Directory[.zip] [ShellArgs] [do VerseExpr]
    Verifies a module from files in Directory as specified in "Verse Grammar Extension to Directories".
    Verifies the module then, if [do VerseExpr], verifies: using{module_contents}; VerseExpr;
    If success and [do VerseExpr], runs it.

In both cases:

- If compiling a top-level directory containing a Self.verse, or a Filename[.verse], and the .verse
  file contains #!bin/verse [ShebangArgs], then adds [ShebangArgs] to [CompilerArgs].

- If these files contain any imports or usings of the funny format import{"relative_path"} or
  using{"relative_path"} where relative_path is relative to the file containing it and uses '/' to
  delimit directories, then we recursively verify those modules and make them visible as imports to
  the contents of the file.

- If we're compiling in a directory containing a '.metaverse' file, we expect our directory to be
  the root (virtual file system) directory of a revision controlled branch of the metaverse,
  containing subdirectories coinciding with domain names, when then recursively contain files and directories.

  This branch must inherit from a verified parent branch as specified in the .metaverse file.
  The verified parent branch can be the live master branch of the metaverse, or can inherit from it.

  In this case, we verify this branch revision-control aware:
  * We demand the parent branch previously verified successfully and is unchanged.
  * Module compilation further requires that what we're compiling be Verse backwards-compatible with
    the verified parent branch, by starting top-level verifier inference of the new version
    with an input existential bound to the prior version.
  * If module verification succeeds, then this branch is marked verified.
  * So long as this branch is marked verified, it's unchanged, and the verifier parent branch is unchanged,
    it can be pushed to the verified parent branch, representing a backwards-compatible evolution.
