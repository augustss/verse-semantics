========================================================================================================================
Verse Runtime Sketch 3
========================================================================================================================

This is a core Verse runtime intended to be compatible with ICFP Verse.
We define runtime Operations, runtime Assumptions, and runtime inference rules.
We don't cover desugaring and verification, which are documented in VerseSpecification.txt.

Given a program "program c x {op}", we begin with the inference described in (ProgramIntro):

    empty, unblocked{op}@c |- program c x {op}

We then apply available inference rules.
The inference rules match specific running parts of the program through the lens of EBNF wildcards like ProgramOp.
The inference rules keep the entire program on the right hand side of the |- turnstile.
The inference rules add to the assumptions on the left of the turnstile; this is monotonic except for mutable writes.
The EBNF wildcards only match running parts of the program; parts that aren't yet available to run are not matched.

We have completed successfully when (ProgramElim) becomes available.

Runtime Definitions
------------------------------------------------------------------------------------------------------------------------

Definitions (as EBNF grammars) used by our inference rules:

    Natural Numbers        m n := 0 | 1 | ...
    Atom Values           atom := n | Print | operator'^' | prefix'set'

    Variables                  := f | g | h | i | j | k | t | w | x | y | z
    Contexts                   := c | d | e

    Effects Sublattices:
        CardinalityFx          := abstracts   | succeeds  | fails
        TransactFx             := transacts   | varies    | allocates | reads    | writes  | no_transacts
        ImperativeFx           := imperatives | interacts | throws    | suspends | no_imperatives
        UnblockFx              := blocked     | unblocked_iterates | unblocked_reads | unblocked_writes | unblocked_imperatives | unblocked
    Effects                    := CardinalityFx & TransactFx & ImperativeFx & UnblockFx

    ICFP Lambdas        lambda := lambda i x {op}
    Heads                 head := atom | lambda | tuple{u_0,...,u_n-1}
    Known (Ground) Value    kv := atom | lambda | tuple{kv_0,...,kv_m-1}
    Vertices         p,q,r,u,v := x | head

    Operations              op := u=v
                               |  u=v(p)
                               |  op0; op1
                               |  op0|op1
                               |  exists u_0 ... u_n
                               |  iterate(u0) x0 c {op0} then x1 {op1} else {op2}
                               |  case(u) {head_0 => {op_0}; ...}
    Programs           Program := program c x {op}

                               Matches any running operation that starts a new context C and its top level operation op.
    ContextStartOp[c,op]       := program c x {op}
                               |  iterate(u0) x0 c {op} then x1 {op1} else {op2}

                               Matches any running operation within the current flexible span.
    ContextSpan[op]            := op
                               |  ContextSpan[op]; op
                               |  op; ContextSpan[op]

                               Matches a program containing a running context c and an operation op in its span.
    ProgramOp[c,op]            := ContextStartOp[c,ContextSpan[op]]
                               |  ContextStartOp[d,ContextSpan[ProgamOp[c,op]]]

                               Matches a unification operation op that unifies U and V.
    Unify[u,v]                 := u=v | v=u

                               Matches a program exposing a context c and a flexible vertex within it.
    ProgramFlexible[c,u]       := ProgramOp[c,exists ... u ...]
                               |  ProgramOp[c,exists ... tuple{...,u,...} ...]

    Assumptions              A := fx{op}@c | P^:=kv@c
    Assumption Sets          G := empty | G, A

    E0 & E1                    := Concrete syntax matches both EBNF E0 and E1, with consistent bindings in both
    E0 | E1                    := Concrete syntax matches either BNF E0 or E1, exposing a binding only if it's bound in both branches
    ~E                         := Concrete syntax doesn't match EBNF E
    G[A0, ..., A1]             := assumption set G containing assumptions A0, A1, ... in any order.
    E[E1/E0]                   := EBNF E with EBNF E1 replacing EBNF E0.
    op[v_0/u_0,...,*/*]        := op[if p is u_i for some i then v_i/p else p_i/p | p:BVS[op]] where p_i fresh
    BVS[op]                    := Bound variables; the least set of Contexts and Variables such that:
                                      BVS[ProgramOp      [c,op]] includes {c}
                                      BVS[ProgramFlexible[c,x ]] includes {x}

Effects:

    Effects is a lattice (https://en.wikipedia.org/wiki/Lattice_(order)) describing the effects of a computation.
    The EBNF definition of Effects above describes its canonical form; unequal Effects EBNF means unequal effects.

    Here we also write effects computations non-canonically as shorthand for their canonical form using:

        fx0 & fx1          := the lattice meet of effects fx0 and fx1
        fx0 + fx1          := the lattice join of effects fx0 and fx1

        effects            := abstracts   & transacts    & imperatives  & blocked                    # top of the whole lattice
        no_effects         := contradicts & no_transacts & no_transacts & no_imperatives & unblocked # bottom of the whole lattice

        only_cardinalities :=               no_transacts & no_imperatives & unblocked
        only_succeeds      := succeeds    & no_transacts & no_imperatives & unblocked
        only_transacts     := contradicts &                no_imperatives & unblocked
        only_imperatives   := contradicts & no_transacts &                  unblocked
        only_blocked       := contradicts & no_transacts & no_imperatives

        We can write effects from any of the Effects Sublattices as shorthand for meeting that
        sublattice effect with the top effect in each of the other sublattices. The top sublattice effects
        are cardinalities, transacts, imperatives, and blocked.

    The Effects lattice is the product of the following sublattices:

    - CardinalityFx describes the potential multiplicity of values:

                                   abstracts
                                   /       \
                               succeeds   fails
                                   \       /
                                  contradicts

    - TransactFx is a complete lattice on fx in the set {varies,allocates,reads,writes}
      where no_transacts<=fx<=transacts.

    - ImperativeFx is a complete lattice on fx in the set {interacts,throws,suspends}
      where no_imperatives<=fx<=imperatives.

    - UnblockFx is a complete lattice on the set fx in {unblocked_iterates,unblocked_reads,unblocked_writes,unblocked_imperatives}
      where unblocked<=fx<=blocked. This describes which operations fx{op} are blocked from running now, to ensure proper
      sequencing of effects amidst lenient evaluation.

    Effects Inclusion

        The inequality fx0 <= fx1 (the lattice partial order inclusion) says fx0 has no more effects than fx1.
        The Effects lattice is information-ordered. That is, if fx0{op} holds (meaning operation op has at most the effects fx0)
        and fx0<=fx1, then fx1{op} also holds. As we learn more information about a program's effects, the information
        we have about it strictly increases.

        Given a computation with effects fx, we can ask if it definitely fails via fails<=contradicts+fx or
        only_cardinalities&fails<=fx.

    Sequencing Effects

        The effects sequencing operation SequenceFx(fx0,fx1) captures the cumulative effects of running
        op0 with effects fx0 then running op1 with effects fx1. It's associative, transitive, and commutative.
        We have fx = only_succeeds*fx = fx*only_succeeds

            SequenceFx(fx0,fx1) := (fx0 + fx1) &
                if(fx0&fx1 <= fails) then fails
                else                      effects

	    An op that contradicts neither fails nor allows the current context to commit successfully,
	    when sequenced with an op that fails, the context may fail, so we have fails*contradicts = fails.

    Blocking Effects
    
        In a language with lenient evaluation and effects, we must ensure that operations with unsatisfied
        effects dependencies, such as read operations dependent on unfinished prior write operations,
        are executed in-order while enabling all other operations to execute out-of-order.

        The operation AfterFx(fx) says what effects should be unblocked subsequent to this operation having effects fx.
        During inference, each operation receives unblocked_* effects from its sequentially prior operation,
        calculates its actual suspended effects fx, and propagates AfterFx(fx) to the sequentially next operation.

            AfterFx(fx) := (unblocked + fx) &
	            (if(abstracts <= contradicts    + fx) then blocked else unblocked_iterates              ) & # While iterates    pending, blocked subsequent iterates.
	            (if(reads     <= no_transacts   + fx) then blocked else unblocked_writes                ) & # While reads       pending, blocked subsequent writes.
	            (if(writes    <= no_transacts   + fx) then blocked else unblocked_reads&unblocked_writes) & # While writes      pending, blocked subsequent reads and writes.
	            (if(throws    <= no_imperatives + fx) then blocked else unblocked_imperatives           ) & # While imperatives pending, blocked subsequent imperatives.
	            (if(suspends  <= no_imperatives + fx) then blocked else unblocked_imperatives           ) &
	            (if(interacts <= no_imperatives + fx) then blocked else unblocked_imperatives           )

            DownFx(fx) := unblocked + fx
                This says how a compound operation's pending effects propagate to its sequentially first sub-operation.

    Explanation of iterate:
        
        Here we explain the meaning of: iterate(u0) x0 c {op0} then x1 {op1} else {op2}

        We're going to iterate over one or more of the values produced by op0, executing op1 on success,
        and op2 on failure. While doing this, we're folding a value (initially u0) into the variable x0,
        and giving op1 the opportunity to ask for the next iteration by unifying x1 with tuple{u1},
        or to stop by unifying x1 with tuple{}.

        - u0 is the value we're folding through the current iteration.
        - x0 is a variable that will be unified with u0(0) upon success or failure, before running op1 or op2.
          We have to keep x0 separate from u0 until success, so that we can rename it in (ChoiceElim).
          If we tried to combine u0 and x0 into a single vertex, substitution would make it unrenamable.
        - We are running op0 in context c, until (IterateSucceedsElim) or (IterateFailsElim) kick in.
        - Our EBNF patterns only see op0 running; op1 and op2 aren't visible until we reduce the 'iterate'
          into something simpler in (IterateSucceedsElim) or (IterateFailsElim).
        - On success, (IterateSucceedsElim) does housekeeping and runs op1.
          This op1 has the opportunity to read the current fold value (stored in the 1-tuple x0).
          This op1 is must unify the next fold value (stored in x1) with either tuple{u1} to continue iterating,
          or with tuple{} to stop. When op1 decides to continue iterating, (ChoiceElim)'s case runs op2.
        - Making all of this useful is the job of (ChoiceElim), which says essentially:
          When we saw this choice a|b, we were iterating in op0 with then & else prongs op1 and op2.
          So now we eliminate the choice by continuing to iterate op in op0 (but with a|b rewritten to a),
          keep op1 as our then-prong, but replace our else-prong op2 with a new 'iterate' that's a copy of
          this one (but with a|b rewritten to b), with a copy of our original then-prong op1, and
          we move op2 from our original iterate else-prong to the new iterate's else-prong.
          Effectively, we're inserting a new op1 prong into a linked list.
          This approach ensures choices are properly sequenced.

    Here are translations loosely using desugar{vertex=term}:

        desugar{u=one{e0}} --->
            iterate(tuple{tuple{}}) x0 c {exists z; desugar{z=e0}}
                then x1                  {u=z; x1=tuple{}}
                else                     {u=tuple{}()}

        desugar{u=if(e0) then e1 else e2} --->
            iterate(tuple{tuple{}}) x0 c {exists z; desugar{z=e0}}
                then                x1   {desugar{u=e1}; x1=tuple{}}
                else                     {desugar{u=e2}}

        desugar{u=all{e0}} --->
            iterate(tuple{0})       x0 c {exists z; desugar(z=e0}}
                then                x1   {exists y; y=x0(0); z=u(y); case(y) {n => {x1=tuple{n+1}}}
                else                     {x0=length(u)}

        desugar{u=for(e0) do e1} --->
            iterate(tuple{0})       x0 c {desugar{e0}}
                then                x1   {exists y; y=x0(0); desugar{u(y)=e1}; x1=tuple{y+1}}
                else                     {x0=length(u)}
