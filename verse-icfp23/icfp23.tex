\begin{abstract}
  Functional logic languages have a rich literature, but it is tricky
  to give them a satisfying semantics.  In this paper we describe the
  Verse calculus, \versecalc{}, a new core calculus for deterministic functional
  logic programming. Our main contribution is to equip \versecalc{} with a
  small-step rewrite semantics, so that we can reason
  about a \versecalc{} program in the same way as one does with lambda
  calculus; that is, by applying successive rewrites to it.
  We also show that the rewrite system is confluent.
%   This semantics elegantly fills the gap between very high level
%  ``magical'' semantics, and very low-level operational semantics.
  \par
  \vspace{1em}
\end{abstract}

\section{Introduction}

Functional logic programming languages add expressiveness to functional programming
by introducing logical variables, equality constraints among those variables,
and choice to allow multiple alternatives to be explored.  Here is a tiny example:
$$
\ensuremath{\exists\Varid{x}\hspace{0.1em}\Varid{y}\hspace{0.1em}\Varid{z}.\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\langle{}\Varid{y},\mathrm{3}\rangle;\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\langle{}\mathrm{2},\Varid{z}\rangle;\,\Varid{y}}
$$
This expression introduces three logical (or existential) variables \ensuremath{\Varid{x},\Varid{y},\Varid{z}},
constrains them with two equalities (\ensuremath{\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\langle{}\Varid{y},\mathrm{3}\rangle} and \ensuremath{\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\langle{}\mathrm{2},\Varid{z}\rangle}), and finally returns \ensuremath{\Varid{y}}.
The only solution to the two equalities is \ensuremath{\Varid{y}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{2}}, \ensuremath{\Varid{z}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{3}}, and \ensuremath{\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\langle{}\mathrm{2},\mathrm{3}\rangle}; so the result of the
whole expression is \ensuremath{\mathrm{2}}.

Functional logic programming has a long history and a rich literature \cite{antoy-hanus:survey}.
But it is somewhat tricky for programmers to \emph{reason} about
functional logic programs: they must think about logical variables,
needed narrowing, unification, and the like.
This contrasts with functional programming, where one can say ``just apply
rewrite rules, such as \textbeta-reduction, let-inlining, and
case-of-known-constructor.''
% Moreover, these rewrite rules apply even
% in the presence of free variables; {\eg}, under lambdas.
We therefore seek \emph{a precise expression of functional logic programming as
a term-rewriting system}, to give us both a formal semantics (via small-step reductions),
and a powerful set of equivalences that programmers can use to reason about their programs,
and that compilers can use to optimize them.

We make two main contributions.
{\bf Our first contribution} is a new core calculus for functional logic programming,
the Verse calculus or \versecalc{} for short
(\cref{sec:informal}).
Like any functional logic language,
\versecalc{} supports logical variables, equalities, and choice, but it is distinctive in several ways:
\begin{itemize}
\item \emph{Natively higher order.} \versecalc{} directly supports higher-order functions, just like the lambda calculus.
  Indeed, \emph{every lambda calculus program is a \versecalc{} program}.
  In contrast, most of the functional logic literature is rooted in a first-order world, and
  addresses higher-order features via an encoding called defunctionalization~\cite[3.3]{hanus:survey}.
\item \emph{Deterministic, with native encapsulation}.  \versecalc{} is \emph{deterministic}, in the sense that when an expression yields more than one value (as is often the case in functional logic programs), those values are returned in a well-specified order.  This makes it easy to solve the notoriously tricky issue \cite{bernd-hanus:encapsulating} of
  how to \emph{encapsulate} the result of a search as a data structure, using the \ensuremath{\textbf{all}} operator (\cref{sec:all}).
  It opens up a new approach to dealing with so-called ``flexible'' {\vs} ``rigid'' variables
  (\cref{sec:one}).  It supports an elegant economy of concepts: for example,
  there is just one equality (other languages may have a suspending equality and a narrowing equality),
  and conditional expressions are driven by failure rather than booleans (\cref{sec:if}).
  On the other hand, it pretty much rules out \emph{laziness} (\cref{sec:laziness})
  and parallel first-come first-returned search strategies.
  %
  Most other functional-logic languages (Curry is the brand leader~\cite{hanus:curry-report})
  are non-deterministic by design; \versecalc{}
  explores a different (and less well-examined) part of the design space.
\end{itemize}
{\bf Our second contribution} is to equip \versecalc{} with a
\emph{small-step term-rewriting semantics} (\cref{sec:rewrite-rules}).
We said that the lambda calculus is a subset of \versecalc{}, so it is natural to give its
semantics using rewrite rules, just as for the lambda calculus.  That seems challenging, however, because
logical variables and unification involve sharing and non-local communication. How can that
be expressed in a rewrite system?

Happily, we can build on prior work: exactly the same difficulty arises with call-by-need in the lambda calculus.
For a long time, the only semantics of
call-by-need that was faithful to its sharing semantics (in which thunks are evaluated at most once)
was an operational semantics that sequentially threads a global heap through execution~\cite{launchbury:natural}.
But then Ariola {\etal}, in a seminal paper, showed how to \emph{reify the heap into the term itself},
and thereby build a rewrite system that is completely faithful to lazy evaluation~\cite{ariola:call-by-need}.
Inspired by their idea, we present a new rewrite system for functional logic programs that reifies
logical variables, unification, and choice into the term itself,
and replaces non-deterministic search with a (deterministic) tree of successful results.
In \versecalc{} the choices are ``laid out in space'' (in the syntax of the term)
rather than, as is more typical, ``laid out in time'' (via non-deterministic rewrites and backtracking).

Rules may be applied anywhere they match, again just like the lambda calculus.

so we show that \versecalc{} is \emph{confluent}, in \cref{sec:metatheory}.

As always with a calculus, the idea is that \versecalc{} distills the essence of

Each construct does just one thing, and \versecalc{} cannot be made smaller without losing key features.

We are working on Verse, a new general purpose programming language, built directly on \versecalc{}; indeed,

No single aspect of \versecalc{} is unique, but we believe that their combination

\section{The Verse calculus, informally} \label{sec:informal}

\figureSyntax

We begin by presenting the Verse calculus, \versecalc, informally.  We will describe its
rewrite rules precisely in \cref{sec:rewrite-rules}.
The (abstract) syntax of \versecalc{} is given in \cref{fig:syntax}.
It has a very conventional sub-language that is just the lambda calculus with
some built-in operations and tuples as data constructors:
\begin{itemize}
\item \emph{Values.}  A value \ensuremath{\Varid{v}} is either a variable \ensuremath{\Varid{x}} or a head-normal form \ensuremath{\mathit{hnf}}.
  In \versecalc{}, a variable counts as a value because in a functional logic language an expression
  may evaluate to an as-yet-unknown logical variable.
  A head-normal form is a conventional value: a built-in constant \ensuremath{\Varid{k}}, an operator \ensuremath{\Varid{op}}, a tuple, or
  a lambda.  Our tiny calculus offers only integer constants \ensuremath{\Varid{k}} and two illustrative integer operators \ensuremath{\Varid{op}}, namely \ensuremath{\textbf{gt}} and \ensuremath{\textbf{add}}.
\item \emph{Expressions} \ensuremath{\Varid{e}} include values \ensuremath{\Varid{v}}, and applications \ensuremath{\Varid{v}_{\mathrm{1}}\,\Varid{v}_{\mathrm{2}}}; we will
  introduce the other constructs as we go.
For clarity, we often write \ensuremath{\Varid{v}_{\mathrm{1}}(\Varid{v}_{\mathrm{2}})} rather than \ensuremath{\Varid{v}_{\mathrm{1}}\,\Varid{v}_{\mathrm{2}}} when \ensuremath{\Varid{v}_{\mathrm{2}}} is not a tuple.
\item A term \ensuremath{eq} is either an ordinary expression \ensuremath{\Varid{e}}, or
an \emph{equation} \ensuremath{\Varid{v}\hspace{-0.14em}=\hspace{-0.14em}\Varid{e}}; this syntax ensures that equations can only occur to the left of a ``\ensuremath{;\,}'' (\cref{sec:unify}).
\item \emph{A program}, \ensuremath{\Varid{p}}, contains a closed expression from which we extract one result
  using \ensuremath{\textbf{one}} (see \cref{sec:one})---unless the expression fails, in which case the program fails (\cref{sec:choice}).
%%  \lennart{It's not really supposed top be |OneKW|, but |succeeds|.  But we don't have |succeeds|.}
\end{itemize}
The formal syntax for \ensuremath{\Varid{e}} allows only applications of \emph{values}, \ensuremath{(\Varid{v}_{\mathrm{1}}\;\Varid{v}_{\mathrm{2}})}, but the desugaring
rules in \cref{fig:syntax} show how to desugar more applications \ensuremath{(\Varid{e}_{\mathrm{1}}\;\Varid{e}_{\mathrm{2}})}. This restriction
not fundamental; it simply reduces the number of rewrite rules we need\footnote{
This is a common pattern, often called
  ``administrative normal form'', or ANF \cite{felleisen:anf}}.
The desugaring rules are more suggestive than precise; we aim to be precise about \versecalc{} but
less so about the source language.

Modulo this desugaring, every lambda calculus term is a \versecalc{} term,
and has the same semantics. Just like the lambda calculus, \versecalc{} is untyped;
adding a type system is an excellent goal, but is the subject of another paper.

Expressions also include two other key collections of constructs:
logical variables and the use of equations to perform unification (\cref{sec:unify}), and choice (\cref{sec:choice}).
The details of choice and unification, and especially their
interaction, are subtle, so this section does a lot of
arm-waving.  But fear not: \cref{sec:rewrite-rules} spells out the precise details.
We only have space to describe one incarnation of \versecalc{}; \cref{sec:variations}
explores some possible alternative design choices.

\subsection{Logical variables and equations} \label{sec:unify}

The Verse calculus includes first class \emph{logical variables} and \emph{equations} that constrain their values.
You can bring a fresh logical variable into scope with \ensuremath{\exists{}}, constrain a value to be equal to
an expression with an equation
\ensuremath{\Varid{v}\hspace{-0.14em}=\hspace{-0.14em}\Varid{e}}, and compose expressions in sequence with \ensuremath{eq;\,\Varid{e}} (see \cref{fig:syntax}).
As an example, what might be written \ensuremath{\textbf{let}\;\Varid{x}=\Varid{e}_{\mathrm{1}}\;\textbf{in}\,\Varid{e}_{\mathrm{2}}} in a conventional functional language can be written
\ensuremath{\exists\Varid{x}.\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\Varid{e}_{\mathrm{1}};\,\Varid{e}_{\mathrm{2}}} in \versecalc{}.  The syntax carefully constrains both the form of
equations and where they can appear: an equation \ensuremath{(\Varid{v}\hspace{-0.14em}=\hspace{-0.14em}\Varid{e})} always equates a \emph{value} $v$ to an expression $e$;
and an equation can only appear to the left of a ``\ensuremath{;\,}'' (see \ensuremath{eq} in \cref{fig:syntax}).
The desugaring rules in \cref{fig:syntax} rewrite a general equation \ensuremath{\Varid{e}_{\mathrm{1}}\hspace{-0.14em}=\hspace{-0.14em}\Varid{e}_{\mathrm{2}}} into this simpler form.

\emph{A program executes by solving its equations}, using the process of unification.
For example,
$$
\ensuremath{\exists\Varid{x}\hspace{0.1em}\Varid{y}\hspace{0.1em}\Varid{z}.\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\langle{}\Varid{y},\mathrm{3}\rangle;\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\langle{}\mathrm{2},\Varid{z}\rangle;\,\Varid{y}}
$$
is solved by unifying \ensuremath{\Varid{x}} with \ensuremath{\langle{}\Varid{y},\mathrm{3}\rangle} and with \ensuremath{\langle{}\mathrm{2},\Varid{z}\rangle}; that in turn unifies \ensuremath{\langle{}\Varid{y},\mathrm{3}\rangle} with \ensuremath{\langle{}\mathrm{2},\Varid{z}\rangle}, which
unifies \ensuremath{\Varid{y}} with \ensuremath{\mathrm{2}} and \ensuremath{\Varid{z}} with \ensuremath{\mathrm{3}}.  Finally, \ensuremath{\mathrm{2}} is returned as the result.
Note carefully that, as in any declarative language,
\emph{logical variables are not mutable}; a logical variable stands for a single, immutable value.
We use ``\ensuremath{\exists{}}'' to bring a fresh logical variable into scope, because we really
mean ``there exists an \ensuremath{\Varid{x}} such that\ldots''.
%% Logical variables \emph{are} existential variables.

High-level functional languages usually provide some kind of
pattern matching; in such a language, we might define \ensuremath{\Varid{first}} by
\ensuremath{\Varid{first}\langle\Varid{a},\Varid{b}\rangle\hspace{-0.14em}=\hspace{-0.14em}\Varid{a}}. Such pattern matching is typically desugared to
more primitive \ensuremath{\textbf{case}} expressions, but in \versecalc{} we do not need \ensuremath{\textbf{case}} expressions:
unification does the job.  For example we can define \ensuremath{\Varid{first}} like this:
$$
\ensuremath{\Varid{first}\hspace{-0.14em}:=\hspace{-0.14em}\lambda\Varid{p}.\,\exists\Varid{a}\hspace{0.1em}\Varid{b}.\,\Varid{p}\hspace{-0.14em}=\hspace{-0.14em}\langle{}\Varid{a},\Varid{b}\rangle;\,\Varid{a}}
$$
% The lambda unifies its argument (presumably a pair)
%with |tup(a,b)|, where |a,b| are fresh logical
% variables in the body of the lambda; and returns the first of the two, namely |a|.
For convenience, we allow ourselves to write a term like
\ensuremath{\Varid{first}\langle\mathrm{2},\mathrm{5}\rangle}, where we define the library function \ensuremath{\Varid{first}} separately with ``\ensuremath{\hspace{-0.14em}:=\hspace{-0.14em}}''; formally, you can imagine
each example \ensuremath{\Varid{e}} being wrapped with a binding for \ensuremath{\Varid{first}}, thus \ensuremath{\exists\Varid{first}.\,\Varid{first}\hspace{-0.14em}=\hspace{-0.14em}\mathbin{...};\,\Varid{e}},
and similarly for other library functions.

This way of desugaring pattern matching means that the input
to \ensuremath{\Varid{first}} is not required to be fully determined when the function is called. For example:
$$
\ensuremath{\exists\Varid{x}\hspace{0.1em}\Varid{y}.\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\langle{}\Varid{y},\mathrm{5}\rangle;\,\mathrm{2}\hspace{-0.14em}=\hspace{-0.14em}\Varid{first}(\Varid{x});\,\Varid{y}}
$$
Here \ensuremath{\Varid{first}(\Varid{x})} evaluates to \ensuremath{\Varid{y}}, which we then unify with \ensuremath{\mathrm{2}}.
Another way to say this is that, as usual in logic programming, we may
constrain the \emph{output} of a function (here \ensuremath{\mathrm{2}\hspace{-0.14em}=\hspace{-0.14em}\Varid{first}(\Varid{x})}), and thereby
affect its \emph{input} (here \ensuremath{\langle{}\Varid{y},\mathrm{5}\rangle}).

Although ``\ensuremath{;\,}\!'' is called ``sequencing,'' the order of that sequence
is immaterial for equations that do not contain choices (see \Cref{sec:choice} for the latter caveat).
For example,
% foothand not needed, the figure introduces this shorthand
%\footnote{We informally write |3+y| rather than |applyN add (3,y)|, and similarly with other operators.},
consider \ensuremath{(\exists\Varid{x}\hspace{0.1em}\Varid{y}.\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{3}\mathbin{+}\Varid{y};\,\Varid{y}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{7};\,\Varid{x})}.  The sub-expression \ensuremath{\mathrm{3}\mathbin{+}\Varid{y}} is stuck until \ensuremath{\Varid{y}} gets a value.
In \versecalc{}, we can unify \ensuremath{\Varid{x}} only with a \emph{value}---we will see why in \cref{sec:call-time-choice}---and
hence the equation \ensuremath{\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{3}\mathbin{+}\Varid{y}} is also stuck.
No matter!  We simply leave it and try some other equation. In this case, we can make progress with \ensuremath{\Varid{y}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{7}},
and that in turn unlocks \ensuremath{\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{3}\mathbin{+}\Varid{y}} because now we know that \ensuremath{\Varid{y}} is \ensuremath{\mathrm{7}}, so we can evaluate \ensuremath{\mathrm{3}\mathbin{+}\mathrm{7}} to \ensuremath{\mathrm{10}}
and unify \ensuremath{\Varid{x}} with that.  The idea of leaving stuck expressions aside and executing other parts of the
program is called
\emph{residuation}~\cite{hanus:survey}\footnote{Hanus did not invent the terms ``residuation'' and``narrowing,'' but
his survey is an excellent introduction and bibliography.}, and is at the heart of our mantra ``just solve the equations.''

\subsection{Choice} \label{sec:choice} \label{sec:call-time-choice}

In conventional functional programming, an expression evaluates to a single value.
In contrast, a \versecalc{} expression evaluates to zero, one, or many values;
or it can get stuck, which is different from producing zero values.
The expression \ensuremath{\textbf{fail}} yields no values; a value \ensuremath{\Varid{v}} yields one value;
and the choice \ensuremath{\Varid{e}_{\mathrm{1}}\;\rule[-0.1em]{0.15em}{0.75em}\;\Varid{e}_{\mathrm{2}}} yields all the values
yielded by \ensuremath{\Varid{e}_{\mathrm{1}}} followed by all the values yielded by \ensuremath{\Varid{e}_{\mathrm{2}}}.
Order is maintained and duplicates are not eliminated; we shall see why in \cref{sec:for}.
In short, an expression yields a \emph{sequence} of values, not a bag, and certainly not a set.

The equations we saw in \cref{sec:unify} can fail, if the arguments
are not equal, yielding no results.  Thus \ensuremath{\mathrm{3}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{3}} succeeds,
while \ensuremath{\mathrm{3}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{4}} fails, returning no results.  In general, we use ``fail''
and ``returns no results'' synonymously.

What if the choice was not at the top level of an expression?  For example,
what does \ensuremath{\langle{}\mathrm{3},(\mathrm{7}\;\rule[-0.1em]{0.15em}{0.75em}\;\mathrm{5})\rangle} mean?
In \versecalc{}, it does \emph{not} mean a pair with some kind of multi-value in its second component.
Indeed, as you can see from \cref{fig:syntax}, this expression is syntactically ill-formed.
We must use the desugaring rules of \cref{fig:syntax}, which give a name to that choice, thus: \ensuremath{\exists\Varid{x}.\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}(\mathrm{7}\;\rule[-0.1em]{0.15em}{0.75em}\;\mathrm{5});\,\langle{}\mathrm{3},\Varid{x}\rangle}.
Now the expression is syntactically legal, but what does it mean?  In \versecalc{}, a variable is never bound to a multi-value.
Instead, \ensuremath{\Varid{x}} is successively bound to \ensuremath{\mathrm{7}}, and then to \ensuremath{\mathrm{5}}, like this:
$$
\ensuremath{\exists\Varid{x}.\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}(\mathrm{7}\;\rule[-0.1em]{0.15em}{0.75em}\;\mathrm{5});\,\langle{}\mathrm{3},\Varid{x}\rangle} \quad \movesto \quad \ensuremath{(\exists\Varid{x}.\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{7};\,\langle{}\mathrm{3},\Varid{x}\rangle)\;\rule[-0.1em]{0.15em}{0.75em}\;(\exists\Varid{x}.\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{5};\,\langle{}\mathrm{3},\Varid{x}\rangle)}
$$
We duplicate the context surrounding the choice, and ``float the choice outwards''.
The same thing happens when there are multiple choices.  For example:
$$
\ensuremath{\exists\Varid{x}\hspace{0.1em}\Varid{y}.\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}(\mathrm{7}\;\rule[-0.1em]{0.15em}{0.75em}\;\mathrm{22});\,\Varid{y}\hspace{-0.14em}=\hspace{-0.14em}(\mathrm{31}\;\rule[-0.1em]{0.15em}{0.75em}\;\mathrm{5});\,\langle{}\Varid{x},\Varid{y}\rangle}
\quad
\text{yields the sequence \ensuremath{\langle{}\mathrm{7},\mathrm{31}\rangle,\langle{}\mathrm{7},\mathrm{5}\rangle,\langle{}\mathrm{22},\mathrm{31}\rangle,\langle{}\mathrm{22},\mathrm{5}\rangle}}
$$
Notice that the order of the two equations now \emph{is} significant:
$$
\ensuremath{\exists\Varid{x}\hspace{0.1em}\Varid{y}.\,\Varid{y}\hspace{-0.14em}=\hspace{-0.14em}(\mathrm{31}\;\rule[-0.1em]{0.15em}{0.75em}\;\mathrm{5});\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}(\mathrm{7}\;\rule[-0.1em]{0.15em}{0.75em}\;\mathrm{22});\,\langle{}\Varid{x},\Varid{y}\rangle}
\quad
\text{yields the sequence \ensuremath{\langle{}\mathrm{7},\mathrm{31}\rangle,\langle{}\mathrm{22},\mathrm{31}\rangle,\langle{}\mathrm{7},\mathrm{5}\rangle,\langle{}\mathrm{22},\mathrm{5}\rangle}}
$$
Readers familiar with list comprehensions in Haskell and other
languages will recognize this nested-loop pattern, but here it
emerges naturally from choice as a deeply built-in primitive,
rather than being a special construct for lists.

Just as we never bind a variable to a multi-value, we never bind it to \ensuremath{\textbf{fail}} either; rather
we iterate over zero values, and that iteration of course returns zero
values.  So:
$$
\ensuremath{\exists\Varid{x}.\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\textbf{fail};\,\mathrm{33}} \quad \movesto \quad \ensuremath{\textbf{fail}}
$$

\subsection{Mixing choice and equations} \label{sec:mixing}

In the last section, we discussed what happens if there is a choice in the right-hand side (RHS) of
an equation. What if we have equations under choice?  For example:
$$
\ensuremath{\exists\Varid{x}.\,(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{3};\,\Varid{x}\mathbin{+}\mathrm{1})\;\rule[-0.1em]{0.15em}{0.75em}\;(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{4};\,\Varid{x}\mathbin{+}\mathrm{4})}
$$
Intuitively, ``either unify \ensuremath{\Varid{x}} with \ensuremath{\mathrm{3}} and yield \ensuremath{\Varid{x}\mathbin{+}\mathrm{1}}, or unify \ensuremath{\Varid{x}} with \ensuremath{\mathrm{4}} and yield \ensuremath{\Varid{x}\mathbin{+}\mathrm{4}}''.
But there is a problem: so far we have said only ``a program executes by solving its equations'' (\cref{sec:unify}).
Here, we can see two equations, \ensuremath{(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{3})} and \ensuremath{(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{4})}, which are mutually contradictory,
so clearly we need to refine our notion of ``solving.''
The answer is pretty clear: in a branch of a choice, solve the equations in that branch
to get the value for some logical variables, \emph{and propagate those values to occurrences in
  that branch (only)}. Occurrences of that variable outside the choice are unaffected.  We call
this \emph{local propagation}.
This local-propagation rule would allow us to reason thus:
$$
\ensuremath{\exists\Varid{x}.\,(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{3};\,\Varid{x}\mathbin{+}\mathrm{1})\;\rule[-0.1em]{0.15em}{0.75em}\;(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{4};\,\Varid{x}\mathbin{+}\mathrm{4})} \movesto  \ensuremath{\exists\Varid{x}.\,(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{3};\,\mathrm{4})\;\rule[-0.1em]{0.15em}{0.75em}\;(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{4};\,\mathrm{8})}
$$
Are we stuck now?  No, we can float the choice out as
before\footnote{Indeed, we could have done so first, had we wished.},
$$
\ensuremath{\exists\Varid{x}.\,(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{3};\,\mathrm{4})\;\rule[-0.1em]{0.15em}{0.75em}\;(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{4};\,\mathrm{8})} \movesto \ensuremath{(\exists\Varid{x}.\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{3};\,\mathrm{4})\;\rule[-0.1em]{0.15em}{0.75em}\;(\exists\Varid{x}.\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{4};\,\mathrm{8})}
$$
and now it is apparent that the sole occurrence of \ensuremath{\Varid{x}} in each \ensuremath{\exists{}} is the equation
\ensuremath{(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{3})}, or \ensuremath{(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{4})} respectively; so we can drop the \ensuremath{\exists{}} and the equation, yielding \ensuremath{(\mathrm{4}\;\rule[-0.1em]{0.15em}{0.75em}\;\mathrm{8})}.

\subsection{Pattern matching and narrowing}

We remarked in \cref{sec:unify} that we can desugar the pattern matching of a high-level
language into equations. But what about multi-equation pattern matching, such as this definition in Haskell:
\begin{hscode}\SaveRestoreHook
\column{B}{@{}>{\hspre}l<{\hspost}@{}}%
\column{3}{@{}>{\hspre}l<{\hspost}@{}}%
\column{19}{@{}>{\hspre}l<{\hspost}@{}}%
\column{E}{@{}>{\hspre}l<{\hspost}@{}}%
\>[3]{}\Varid{append}\;[\mskip1.5mu \mskip1.5mu]\;{}\<[19]%
\>[19]{} ys \hspace{-0.14em}=\hspace{-0.14em} ys {}\<[E]%
\\
\>[3]{}\Varid{append}\;(\Varid{x}\mathbin{:} xs )\;{}\<[19]%
\>[19]{} ys \hspace{-0.14em}=\hspace{-0.14em}\Varid{x}\mathbin{:}\Varid{append}\; xs \; ys {}\<[E]%
\ColumnHook
\end{hscode}\resethooks
If pattern matching on the first equation fails, we want to fall through to the second.
Fortunately, choice allows us to express this idea directly, where
we use the empty tuple \ensuremath{\langle\rangle} to represent the empty list and pairs to represent cons cells
(see \cref{fig:syntax} to desugar the pattern-matching lambda):
$$
\ensuremath{\Varid{append}\hspace{-0.14em}:=\hspace{-0.14em}\lambda\langle{} xs , ys \rangle.\,(( xs \hspace{-0.14em}=\hspace{-0.14em}\langle\rangle;\, ys )\;\rule[-0.1em]{0.15em}{0.75em}\;(\exists\Varid{x}\hspace{0.1em}\Varid{xr}.\, xs \hspace{-0.14em}=\hspace{-0.14em}\langle{}\Varid{x},\Varid{xr}\rangle;\,\langle{}\Varid{x},\Varid{append}\langle\Varid{xr}, ys \rangle\rangle))}
$$
If \ensuremath{ xs } is \ensuremath{\langle\rangle}, the left-hand choice succeeds, returning \ensuremath{ ys }; and the right-hand choice
fails (by attempting to unify \ensuremath{\langle\rangle} with \ensuremath{\langle{}\Varid{x},\Varid{xr}\rangle}).
If \ensuremath{ xs } is of the form \ensuremath{\langle{}\Varid{x},\Varid{xr}\rangle},
the right-hand choice succeeds, and we make a recursive call to \ensuremath{\Varid{append}}.
Finally, if \ensuremath{ xs } is built with head-normal forms other than the empty tuple and pairs, both choices fail, and \ensuremath{\Varid{append}} returns no
results at all.

This approach to pattern matching is akin to \emph{narrowing}~\cite{hanus:survey}.
Suppose \ensuremath{\Varid{single}\hspace{-0.14em}=\hspace{-0.14em}\langle{}\mathrm{1},\langle\rangle\rangle}, a singleton list whose only element is 1.
Consider the call \ensuremath{\exists zs .\,\Varid{append}\langle zs ,\Varid{single}\rangle\hspace{-0.14em}=\hspace{-0.14em}\Varid{single};\, zs }.
The call to \ensuremath{\Varid{append}} expands into a choice:
$$
  \ensuremath{( zs \hspace{-0.14em}=\hspace{-0.14em}\langle\rangle;\,\Varid{single})} \quad \ensuremath{\rule[-0.1em]{0.15em}{0.75em}} \quad
  \ensuremath{(\exists\Varid{x}\hspace{0.1em}\Varid{xr}.\, zs \hspace{-0.14em}=\hspace{-0.14em}\langle{}\Varid{x},\Varid{xr}\rangle;\,\langle{}\Varid{x},\Varid{append}\langle\Varid{xr},\Varid{single}\rangle\rangle)}
$$
which amounts to exploring the possibility that \ensuremath{ zs } is headed by \ensuremath{\langle\rangle} or a pair---the
essence of narrowing.  It should not take long to reassure yourself that the program
evaluates to \ensuremath{\langle\rangle}, effectively running \ensuremath{\Varid{append}} backwards in the classic logic-programming manner.

This example also illustrates that \versecalc{} allows an equation (for \ensuremath{\Varid{append}})
that is recursive. As in any functional language with recursive bindings, you can
go into an infinite loop if you keep fruitlessly inlining the function in its own right-hand side.
It is the business of an \emph{evaluation strategy} to do only rewrites that make
progress toward a solution (\cref{sec:eval-strategy}).

\subsection{Conditionals and \texorpdfstring{\ensuremath{\textbf{one}}}{``one''}} \label{sec:rigid} \label{sec:one} \label{sec:if}

Every source language will provide a conditional, such as \ensuremath{\textbf{if}\;(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{0})\;\textbf{then}\;\Varid{e}_{\mathrm{2}}\;\textbf{else}\;\Varid{e}_{\mathrm{3}}}.  But what is the equality operator in \ensuremath{(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{0})}?  One
possibility, adopted by Curry \cite[\S 3.4]{curry:tutorial}, is this: there is one ``='' for
equations (as in \cref{sec:unify}), and another, say ``=='', for
testing equality (returning a boolean with constructors \ensuremath{\Conid{True}} and \ensuremath{\Conid{False}}).  \versecalc{} takes a
different, more minimalist position, following Icon's lead, see \cref{sec:icon}.
In \versecalc{}, \emph{there is just one equality operator}, written ``='' just as in
\cref{sec:unify}. The expression \ensuremath{\textbf{if}\;(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{0})\;\textbf{then}\;\Varid{e}_{\mathrm{2}}\;\textbf{else}\;\Varid{e}_{\mathrm{3}}} tries to unify \ensuremath{\Varid{x}} with \ensuremath{\mathrm{0}}.  If that succeeds (yields one or more values),
the \ensuremath{\textbf{if}} returns \ensuremath{\Varid{e}_{\mathrm{2}}}; otherwise it returns \ensuremath{\Varid{e}_{\mathrm{3}}}.  There are no data constructors \ensuremath{\Conid{True}} and \ensuremath{\Conid{False}}; instead
failure (returning zero values) plays the role of falsity.

But something is terribly wrong here.  Consider \ensuremath{\exists\Varid{x}\hspace{0.1em}\Varid{y}.\,\Varid{y}\hspace{-0.14em}=\hspace{-0.14em}(\textbf{if}\;(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{0})\;\textbf{then}\;\mathrm{3}\;\textbf{else}\;\mathrm{4});\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{7};\,\Varid{y}}.
Presumably this is meant
to set \ensuremath{\Varid{x}} to \ensuremath{\mathrm{7}}, test whether it is equal to \ensuremath{\mathrm{0}} (it is not), and unify \ensuremath{\Varid{y}} with 4.
But what is to stop us instead unifying \ensuremath{\Varid{x}} with \ensuremath{\mathrm{0}} (via \ensuremath{(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{0})}), unifying \ensuremath{\Varid{y}} with \ensuremath{\mathrm{3}}, and then
failing when we try to unify \ensuremath{\Varid{x}} with \ensuremath{\mathrm{7}}?  Not only is that not what we intended, but it also looks very
non-deterministic: the result is affected by the order in which we did unifications.

To address this, we give \ensuremath{\textbf{if}} a special property: in the expression \ensuremath{\textbf{if}\;\Varid{e}_{\mathrm{1}}\;\textbf{then}\;\Varid{e}_{\mathrm{2}}\;\textbf{else}\;\Varid{e}_{\mathrm{3}}}, equations inside \ensuremath{\Varid{e}_{\mathrm{1}}} (the condition of the \ensuremath{\textbf{if}})
can only unify variables bound inside \ensuremath{\Varid{e}_{\mathrm{1}}}; variables bound outside \ensuremath{\Varid{e}_{\mathrm{1}}} are called ``rigid.''  So in our example, the \ensuremath{\Varid{x}} in \ensuremath{(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{0})} is
rigid and cannot be unified.  Instead, the \ensuremath{\textbf{if}} is stuck, and we move on to unify \ensuremath{\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{7}}.  That unblocks
the \ensuremath{\textbf{if}} and all is well.
% This special property is precisely the local propagation rule that we sketched for
% choice (\cref{sec:mixing}).

In fact, \versecalc{} desugars the three-part \ensuremath{\textbf{if}} into something simpler, the unary construct \ensuremath{\textbf{one}\{\Varid{e}\}}.
Its specification is this:
if \ensuremath{\Varid{e}} fails, \ensuremath{\textbf{one}\{\Varid{e}\}} fails; otherwise \ensuremath{\textbf{one}\{\Varid{e}\}} returns the first of the values yielded by \ensuremath{\Varid{e}}.
Now, \ensuremath{\textbf{if}\;\Varid{e}_{\mathrm{1}}\;\textbf{then}\;\Varid{e}_{\mathrm{2}}\;\textbf{else}\;\Varid{e}_{\mathrm{3}}} can (nearly) be re-expressed like this:
$$
\ensuremath{\textbf{one}\{(\Varid{e}_{\mathrm{1}};\,\Varid{e}_{\mathrm{2}})\;\rule[-0.1em]{0.15em}{0.75em}\;\Varid{e}_{\mathrm{3}}\}}
$$
This isn't right yet, but the idea is this: if \ensuremath{\Varid{e}_{\mathrm{1}}} fails, the first branch of the choice fails, so we get \ensuremath{\Varid{e}_{\mathrm{3}}}; if \ensuremath{\Varid{e}_{\mathrm{1}}} succeeds, we
get \ensuremath{\Varid{e}_{\mathrm{2}}}, and the outer \ensuremath{\textbf{one}} will select it from the choice.
But what if \ensuremath{\Varid{e}_{\mathrm{2}}} or \ensuremath{\Varid{e}_{\mathrm{3}}} \emph{themselves} fail or return multiple results?  Here is a better translation,
the one given in \cref{fig:syntax}\footnote{The translation in the figure also allow variables bound in the condition to scope over the \ensuremath{\textbf{then}} branch.}, which wraps the \ensuremath{\textbf{then}} and \ensuremath{\textbf{else}} branches
in a thunk\footnote{Using thunks for the branches of a conditional is another very old idea;
for example, see \cite[p.\ 54]{steele:rabbit}.}:
$$
\ensuremath{(\textbf{one}\{(\Varid{e}_{\mathrm{1}};\,(\lambda \langle\rangle.\,\Varid{e}_{\mathrm{2}}))\;\rule[-0.1em]{0.15em}{0.75em}\;(\lambda \langle\rangle.\,\Varid{e}_{\mathrm{3}})\})\langle\rangle}
$$
The argument of \ensuremath{\textbf{one}} reduces to either \ensuremath{(\lambda \langle\rangle.\,\Varid{e}_{\mathrm{2}})\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}(\lambda \langle\rangle.\,\Varid{e}_{\mathrm{3}})} or \ensuremath{(\lambda \langle\rangle.\,\Varid{e}_{\mathrm{3}})} depending on whether \ensuremath{\Varid{e}_{\mathrm{1}}} succeeds or fails,
respectively; \ensuremath{\textbf{one}} then picks the first value, that is \ensuremath{\lambda \langle\rangle.\,\Varid{e}_{\mathrm{2}}} if \ensuremath{\Varid{e}_{\mathrm{1}}} succeeded, or \ensuremath{\lambda \langle\rangle.\,\Varid{e}_{\mathrm{3}}} if \ensuremath{\Varid{e}_{\mathrm{1}}} failed, and
applies it to \ensuremath{\langle{}\rangle}.
As a bonus, provided we do no evaluation under a lambda, then \ensuremath{\Varid{e}_{\mathrm{2}}} and \ensuremath{\Varid{e}_{\mathrm{3}}} will
remain unevaluated until the choice is made, just as we expect from a conditional.

We use the same local-propagation rule for \ensuremath{\textbf{one}} that we do for choice (\cref{sec:mixing}).
This, together with the desugaring for \ensuremath{\textbf{if}} into \ensuremath{\textbf{one}}, gives the ``special property'' of \ensuremath{\textbf{if}} described above.

\subsection{Tuples and \texorpdfstring{\ensuremath{\textbf{all}}}{``all''}} \label{sec:tuple}

The main data structure in \versecalc{} is the \emph{tuple}.
A tuple is a finite sequence of values, \ensuremath{\langle{}\Varid{v}_{\mathrm{1}},\mydots,\Varid{v}_{\Varid{n}}\rangle}, where $n\geq 0$.
A tuple can be used like a function: indexing is simply function application
with the argument being integers from 0 and up.  Indexing out of range is \ensuremath{\textbf{fail}}, as is indexing with
a non-integer value.
For example, \ensuremath{\Varid{t}\hspace{-0.14em}:=\hspace{-0.14em}\langle{}\mathrm{10},\mathrm{27},\mathrm{32}\rangle;\,\Varid{t}(\mathrm{1})} reduces to \ensuremath{\mathrm{27}} and \ensuremath{\Varid{t}(\mathrm{3})} reduces to \ensuremath{\textbf{fail}}.

What if we apply a tuple to a choice, thus \ensuremath{\langle{}\mathrm{10},\mathrm{27},\mathrm{32}\rangle(\mathrm{1}\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\mathrm{0}\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\mathrm{1})}?  First we must
desugar the application to the form \ensuremath{(\Varid{v}_{\mathrm{1}}\,\Varid{v}_{\mathrm{2}})}, because that is all \versecalc{} permits (\cref{fig:syntax}),
giving \ensuremath{\Varid{x}\hspace{-0.14em}:=\hspace{-0.14em}(\mathrm{1}\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\mathrm{0}\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\mathrm{1});\,\langle{}\mathrm{10},\mathrm{27},\mathrm{32}\rangle(\Varid{x})}, which readily reduces to \ensuremath{(\mathrm{27}\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\mathrm{10}\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\mathrm{27})}.

Tuples can be constructed by collecting all the results from a multi-valued expression, using the \ensuremath{\textbf{all}} construct:
if \ensuremath{\Varid{e}} reduces to \ensuremath{(\Varid{v}_{\mathrm{1}}\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\mydots\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\Varid{v}_{\Varid{n}})}, where $n \geq 2$, then \ensuremath{\textbf{all}\{\Varid{e}\}} reduces to
the tuple \ensuremath{\langle{}\Varid{v}_{\mathrm{1}},\mydots,\Varid{v}_{\Varid{n}}\rangle}; \ensuremath{\textbf{all}\{\Varid{v}\}} produces the singleton tuple \ensuremath{\langle{}\Varid{v}\rangle}; and \ensuremath{\textbf{all}\{\textbf{fail}\}} produces the empty tuple \ensuremath{\langle\rangle}.
Note that \ensuremath{\rule[-0.1em]{0.15em}{0.75em}} is associative, which means that we
can think of a sequence or tree of binary choices as really being a single
$n$-way choice.
\label{sec:all}

You might think that tuple indexing would be stuck until we know the index,
but in \versecalc{}, the application of a tuple to a value rewrites to a choice of all the possible
values of the index.
For example,
\ensuremath{\Varid{t}\hspace{-0.14em}:=\hspace{-0.14em}\langle{}\mathrm{10},\mathrm{27},\mathrm{32}\rangle;\,\exists\Varid{i}.\,\Varid{t}(\Varid{i})}
looks stuck because we have no value for \ensuremath{\Varid{i}}, but actually rewrites to:
$$
\ensuremath{\exists\Varid{i}.\,(\Varid{i}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{0};\,\mathrm{10})\;\rule[-0.1em]{0.15em}{0.75em}\;(\Varid{i}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{1};\,\mathrm{27})\;\rule[-0.1em]{0.15em}{0.75em}\;(\Varid{i}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{2};\,\mathrm{32})}
$$
which (as we will see in \cref{sec:rewrite-rules}) simplifies
to just \ensuremath{(\mathrm{10}\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\mathrm{27}\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\mathrm{32})}.
So \ensuremath{\textbf{all}} allows a choice to be reified into a tuple; and \ensuremath{(\exists\Varid{i}.\,\Varid{t}(\Varid{i}))} allows a
tuple to be turned back into a choice.
The idea of rewriting a call of a function with a finite domain into a finite choice
is called ``narrowing'' in the literature.

Do we even need \ensuremath{\textbf{one}} as a primitive construct, given that we have
\ensuremath{\textbf{all}}? Can we not use \ensuremath{(\textbf{all}\{\Varid{e}\})(\mathrm{0})} instead of \ensuremath{\textbf{one}\{\Varid{e}\}}? Indeed,
they behave the same if \ensuremath{\Varid{e}} fully reduces to finitely many choices of
values. But \ensuremath{\textbf{all}} really requires \emph{every} arm of the choice tree to resolve to a value
before proceeding, while \ensuremath{\textbf{one}} only needs the
\emph{first} choice to be a value. So, supposing that \ensuremath{\Varid{loop}} is a non-terminating function,
\ensuremath{\textbf{one}\{\mathrm{1}\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\Varid{loop}\langle\rangle\}} can reduce to 1, while
\ensuremath{(\textbf{all}\{\mathrm{1}\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\Varid{loop}\langle\rangle\})(\mathrm{0})} loops.

\subsection{For loops} \label{sec:for}

The expression \ensuremath{\textbf{for}(\Varid{e}_{\mathrm{1}})\,\textbf{do}\,\Varid{e}_{\mathrm{2}}} will evaluate \ensuremath{\Varid{e}_{\mathrm{2}}} for each of the choices in \ensuremath{\Varid{e}_{\mathrm{1}}},
rather like a list comprehension in languages like Haskell or Python.
% The |e_1| is called the \emph{domain} of the loop.
The scoping is peculiar\footnote{But similar to C++, Java, Fortress, and Swift, and explained in \versecalc{} by
  the subsequent desugaring into \ensuremath{\textbf{all}}.}
in that variables bound in \ensuremath{\Varid{e}_{\mathrm{1}}} also scope over \ensuremath{\Varid{e}_{\mathrm{2}}}.
So, for example, \ensuremath{\textbf{for}(\Varid{x}\hspace{-0.14em}:=\hspace{-0.14em}\mathrm{2}\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\mathrm{3}\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\mathrm{5})\,\textbf{do}\,(\Varid{x}\mathbin{+}\mathrm{1})} will reduce to the tuple \ensuremath{\langle{}\mathrm{3},\mathrm{4},\mathrm{6}\rangle}.
% This example can, with the |:==| shorthand, be written |for^(x :== 2 `choice` 3 `choice` 5) do (x+1)|.

Like list comprehension, \ensuremath{\textbf{for}} supports filtering; in \versecalc{}, this falls out naturally by just using
a possibly failing expression in \ensuremath{\Varid{e}_{\mathrm{1}}}.
So, \ensuremath{\textbf{for}(\Varid{x}\hspace{-0.14em}:=\hspace{-0.14em}\mathrm{2}\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\mathrm{3}\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\mathrm{5};\,\Varid{x}\mathbin{>}\mathrm{2})\,\textbf{do}\,(\Varid{x}\mathbin{+}\mathrm{1})} reduces to \ensuremath{\langle{}\mathrm{4},\mathrm{6}\rangle}.
Nested iteration in a \ensuremath{\textbf{for}} works as expected and requires nothing special.
So, \ensuremath{\textbf{for}(\exists\Varid{x}\hspace{0.1em}\Varid{y}.\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{10}\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\mathrm{20};\,\Varid{y}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{1}\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\mathrm{2}\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\mathrm{3})\,\textbf{do}\,(\Varid{x}\mathbin{+}\Varid{y})} reduces to \ensuremath{\langle{}\mathrm{11},\mathrm{12},\mathrm{13},\mathrm{21},\mathrm{22},\mathrm{23}\rangle}.

Just as \ensuremath{\textbf{if}} is defined in terms of the primitive \ensuremath{\textbf{one}} (\cref{sec:if}), we can define \ensuremath{\textbf{for}}
in terms of the primitive \ensuremath{\textbf{all}}.
Again, we have to be careful when \ensuremath{\Varid{e}_{\mathrm{2}}} itself fails or produces multiple results;
simply writing \ensuremath{\textbf{all}\{\Varid{e}_{\mathrm{1}};\,\Varid{e}_{\mathrm{2}}\}} would give the wrong semantics. So we put \ensuremath{\Varid{e}_{\mathrm{2}}} within a lambda expression, and
apply each element of the tuple to \ensuremath{\langle\rangle} afterwards, using a \ensuremath{\Varid{map}} function (as defined in \cref{fig:list-functions}):
\[
\ensuremath{\textbf{for}(\exists\Varid{x}_{\mathrm{1}}\hspace{0.1em}\mydots\hspace{0.1em}\Varid{x}_{\Varid{n}}.\,\Varid{e}_{\mathrm{1}})\,\textbf{do}\,\Varid{e}_{\mathrm{2}}} \quad \text{means} \quad
\ensuremath{\Varid{v}\hspace{-0.14em}:=\hspace{-0.14em}\textbf{all}\{\exists\Varid{x}_{\mathrm{1}}\hspace{0.1em}\mydots\hspace{0.1em}\Varid{x}_{\Varid{n}}.\,\Varid{e}_{\mathrm{1}};\,\lambda \langle\rangle.\,\Varid{e}_{\mathrm{2}}\};\,\Varid{map}\langle\lambda\Varid{z}.\,\Varid{z}\langle\rangle,\Varid{v}\rangle}
\]
for a fresh variable \ensuremath{\Varid{v}}.
Note how this achieves that peculiar scoping rule: the initial variables in \ensuremath{\exists\Varid{x}_{\mathrm{1}}\hspace{0.1em}\mydots\hspace{0.1em}\Varid{x}_{\Varid{n}}.\,\Varid{e}_{\mathrm{1}}} are in scope in \ensuremath{\Varid{e}_{\mathrm{2}}}.
Moreover, any effects (like being multi-valued) in \ensuremath{\Varid{e}_{\mathrm{2}}} will not affect the choices defined by \ensuremath{\Varid{e}_{\mathrm{1}}} since the effects are contained within that lambda.
So, for example, \ensuremath{\textbf{for}(\Varid{x}\hspace{-0.14em}:=\hspace{-0.14em}\mathrm{10}\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\mathrm{20})\,\textbf{do}\,(\Varid{x}\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\Varid{x}\mathbin{+}\mathrm{1})} will reduce to
\ensuremath{\langle{}\mathrm{10},\mathrm{20}\rangle\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\langle{}\mathrm{10},\mathrm{21}\rangle\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\langle{}\mathrm{11},\mathrm{20}\rangle\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\langle{}\mathrm{11},\mathrm{21}\rangle}.
At this point, it is crucial that the desugaring of \ensuremath{\textbf{for}} uses \ensuremath{\Varid{map}}, not \ensuremath{\Varid{flatMap}}.

Given that tuple indexing expands into choices, we can iterate over tuple indices and elements using \ensuremath{\textbf{for}}.
For example, \ensuremath{\textbf{for}(\exists\Varid{i}\hspace{0.1em}\Varid{x}.\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\Varid{t}(\Varid{i}))\,\textbf{do}\,(\Varid{x}\mathbin{+}\Varid{i})} produces a tuple with the elements of \ensuremath{\Varid{t}}, each increased by its index within \ensuremath{\Varid{t}}.  Notice the absence of the
fencepost-error-prone iteration of \ensuremath{\Varid{i}} over \ensuremath{(\mathrm{0}\mathinner{\ldotp\ldotp}\Varid{size}\;(\Varid{t})\mathbin{-}\mathrm{1})}, common in most languages.

\section{Rewrite rules} \label{sec:rewrite-rules}

How can we give a precise semantics to a programming language?
Here are some possibilities:
\begin{itemize}
\item \emph{A denotational semantics} is the classical approach,
  but it is tricky to give a (perspicuous) denotational semantics to a functional logic language
  because of the logical variables.  We have such a denotational semantics under development,
  which we offer for completeness in \cref{sec:den-sem}, but that is the subject of another paper.
\item \emph{A big-step operational semantics} typically involves explaining how a (heap, expression) starting point
  evaluates to a (heap, value) pair; Launchbury's natural semantics for lazy evaluation~\cite{launchbury:natural} is the classic paper.  The heap, threaded through the semantics, accounts for updating thunks as they are evaluated.
  Despite its ``operational semantics'' title, the big-step approach
  does not convey accurate operational intuition, because it goes all the way to a value
  in one step.
\item \emph{A small-step operational semantics} addresses this criticism:
  it typically describe how a (heap, expression, stack) configuration evolves, one small
  step at a time ({\eg}, ~\cite{pj:stg-machine}).
  The difficulty is that the description is now so low level that it is again hard to explain to programmers.
\item \emph{A rewrite semantics} steers between these two extremes.
  For example, Ariola {\etal}'s ``A call-by-need lambda calculus''~\cite{ariola:call-by-need} shows how
  to give the semantics of a call-by-need language as a set of rewrite rules. The great
  advantage of this approach is that it is easily explicable to programmers.
  In fact, teachers almost always explain the execution of Haskell or ML programs as a succession of rewrites
  of the program, such as: inline this call, simplify this \ensuremath{\textbf{case}} expression, \etc
\end{itemize}
Up to this point, there has been no satisfying rewrite semantics for functional logic languages
(see \cref{sec:related} for previous work).
Our main technical contribution is to fill this gap with a rewrite semantics for \versecalc{}, one that
has the following properties:
\begin{itemize}
\item The semantics is expressed as a set of rewrite rules (\cref{fig:rules}). These rules apply to the core language of \cref{fig:syntax}, after
  all desugaring.
\item Any rule can be applied, in either direction,
  anywhere in the program term (including under lambdas).
\item The rules are (mostly) oriented, with the intent that using them left to right makes progress.
\item Despite this orientation, the rules do not say which rule should be
  applied where; that is the task of a separate \emph{evaluation strategy} (\cref{sec:eval-strategy}).
\item The rules can be applied by programmers to reason about what their program does,
  and by compilers to transform (and hopefully optimise) the program.
\item There is no ``magical rewriting'' (\cref{sec:magical}):
  all the free variables on the
  right-hand side of a rule are bound on the left.
\end{itemize}

\subsection{Functions and function application rules} \label{sec:beta}

Looking at \cref{fig:rules}, the rule for a primitive operator like addition, \rulename{app-add},
should be familiar: it simply rewrites an application of \ensuremath{\textbf{add}} to integer
constants.  For example $\ensuremath{\textbf{add}\langle\mathrm{3},\mathrm{4}\rangle} \movesto \ensuremath{\mathrm{7}}$.  Rules \rulename{app-gt} and
\rulename{app-gt-fail} are more interesting: \ensuremath{\textbf{gt}\langle\Varid{k}_{\mathrm{1}},\Varid{k}_{\mathrm{2}}\rangle} fails if $k_1 \leq k_2$
(rather than returning \ensuremath{\Conid{False}} as is more conventional), and returns \ensuremath{\Varid{k}_{\mathrm{1}}} otherwise (rather than returning \ensuremath{\Conid{True}}).
An amusing consequence is that \ensuremath{(\mathrm{10}\mathbin{>}\Varid{x}\mathbin{>}\mathrm{0})} succeeds
iff \ensuremath{\Varid{x}} is between \ensuremath{\mathrm{10}} and \ensuremath{\mathrm{0}} (comparison is right-associative).

An application of a primitive operator can only rewrite when its
arguments are ground values; an application like \ensuremath{\textbf{gt}\langle\Varid{x},\mathrm{3}\rangle} or \ensuremath{\textbf{add}\langle\mathrm{7},\Varid{x}\rangle} is \emph{stuck} awaiting a value for \ensuremath{\Varid{x}},
which may arrive later, by substitution (\Cref{sec:unification-rules}).
However equality, written \ensuremath{\Varid{v}\mathrel{=}\Varid{e}}, is special: it is a language construct, not a primitive operator, and enjoys
a rich set of rules of built-in rules (\Cref{sec:unification-rules}).

% Rule \rulename{p-map} implements a map operation over tuples: It applies the function |f| to each value
% in turn, and collects the results into a tuple. If the function applications produce a choice
% of values, this will, as it should, produce a choice of tuples:
% \[
% |applyN map ((lam x (x choice  x+1)), (10,20))| \movesto^\ast |(10,20) choice (10,21) choice (11,20) choice (11,21)|
% \]
\textbeta-reduction is performed quite conventionally by \rulename{app-beta}; the
only unusual feature is that on the RHS of the rule, we use an \ensuremath{\exists{}} to bind \ensuremath{\Varid{x}},
together with \ensuremath{(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\Varid{v})} to equate \ensuremath{\Varid{x}} to the argument.
The rule may appear to use call-by-value, because the argument is a value \ensuremath{\Varid{v}},
but remember that values include variables, and a variable may be bound to an
as-yet-unevaluated expression.  For example:
$$
\ensuremath{\exists\Varid{y}.\,\Varid{y}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{3}\mathbin{+}\mathrm{4};\,(\lambda\Varid{x}.\,\Varid{x}\mathbin{+}\mathrm{1})(\Varid{y})}
\; \movesto \;
\ensuremath{\exists\Varid{y}.\,\Varid{y}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{3}\mathbin{+}\mathrm{4};\,\exists\Varid{x}.\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\Varid{y};\,\Varid{x}\mathbin{+}\mathrm{1}}
$$
Finally, the side condition $x \not\in \freevars{\ensuremath{\Varid{v}}}$ in \rulename{app-beta}
ensures that the $\ensuremath{\exists{}} \ensuremath{\Varid{x}}$
% hacky typesetting to ge the \exists right next to the x.
does not capture any variables free in \ensuremath{\Varid{v}}.  If \ensuremath{\Varid{x}} appears free in \ensuremath{\Varid{v}},
\textalpha-conversion may be used on \ensuremath{\lambda\Varid{x}.\,\Varid{e}} to rename \ensuremath{\Varid{x}} to $y \not\in \freevars{\ensuremath{\Varid{v}}}$.

In \versecalc{}, tuples behave like (finite) functions in which application is
indexing.  Rule \rulename{app-tup} describes how tuple application works on non-empty
tuples, while \rulename{app-tup-0} deals with empty tuples.
Notice that \rulename{app-tup} does not require the argument to be evaluated to an integer $k$; instead
the rule works by narrowing.  So the expression \ensuremath{\exists\Varid{x}.\,\langle{}\mathrm{2},\mathrm{3},\mathrm{2},\mathrm{7},\mathrm{9}\rangle(\Varid{x})\hspace{-0.14em}=\hspace{-0.14em}\mathrm{2};\,\Varid{x}}
does not suspend awaiting a value for \ensuremath{\Varid{x}}; instead it explores all the
alternatives (a form of narrowing), returning \ensuremath{(\mathrm{0}\;\rule[-0.1em]{0.15em}{0.75em}\;\mathrm{2})}.  This is a free design decision:
a suspending semantics would be equally easy to express.

\subsection{Unification rules} \label{sec:unification-rules}

Next we study unification, again in \cref{fig:rules}.
Rules \rulename{u-lit} and \rulename{u-tup}
are the standard rules for unification, going back nearly 60 years \cite{robinson:unification}.
% Note that
% when unification succeeds it yields the common value; hence |s == s| rewrites
% to |s|\footnote{An alternative choice would
% for unification to yield |emptytup| on success.  It does not make much difference
% either way.}.
Rule \rulename{u-fail} makes unification fail on two different head-normal forms (see
\cref{fig:syntax} for the syntax of \ensuremath{\mathit{hnf}}).
Note in particular that unification fails if you attempt to unify a
lambda with any other value, including itself (see \cref{sec:unifying-lambda}).

The standard ``occurs check'' is rule \rulename{u-occurs}, which makes use of a
\emph{context} \ensuremath{\Conid{V}}, whose syntax is given in \cref{fig:contexts}
\cite{felleisen:contexts1,felleisen:contexts2}.
In general, a context is a syntax tree containing a single hole, written \ensuremath{\hole}.
The notation \ensuremath{\Conid{V} [\mskip1.5mu \Varid{v}\mskip1.5mu]} is the term obtained by filling the hole
in \ensuremath{\Conid{V}} with \ensuremath{\Varid{v}}.  For example, \rulename{u-occurs} reduces \ensuremath{\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\langle{}\mathrm{1},\Varid{x},\mathrm{3}\rangle} to \ensuremath{\textbf{fail}}
using the context \ensuremath{\Conid{V}} = \ensuremath{\langle{}\mathrm{1},\hole,\mathrm{3}\rangle}.

The key innovation in \versecalc{} is the way bindings (that is, just ordinary equalities)
of logical variables are propagated.
The key rule is:
$$
\begin{array}{rrcll}
  \rulename{subst} & \ensuremath{X [\mskip1.5mu \Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\Varid{v};\,\Varid{e}\mskip1.5mu]} & \movesto & \ensuremath{(\subst{X}{\Varid{x}}{\Varid{v}}) [\mskip1.5mu \Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\Varid{v};\,\subst{\Varid{e}}{\Varid{x}}{\Varid{v}}\mskip1.5mu]} \hspace*{-0.3em}
  & \text{if $\ensuremath{\Varid{v}} \not= \ensuremath{\Conid{V} [\mskip1.5mu \Varid{x}\mskip1.5mu]}$} \\
\end{array}
$$
The rule says that if we have an equation \ensuremath{(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\Varid{v})},
we can replace the occurrences of \ensuremath{\Varid{x}} by \ensuremath{\Varid{v}} within the following expression and also within a surrounding context.
This rule uses context~\ensuremath{X} (\cref{fig:contexts}), and
uses the notation \ensuremath{\subst{\Varid{e}}{\Varid{x}}{\Varid{v}}} to mean ``capture-avoiding substitution of~\ensuremath{\Varid{v}} for~\ensuremath{\Varid{x}} in~\ensuremath{\Varid{e}}''
%(and similarly |subst X x v|, but |X| will have no bindings to be avoided).
(and similarly \ensuremath{\subst{X}{\Varid{x}}{\Varid{v}}} to mean ``capture-avoiding substitution of~\ensuremath{\Varid{v}} for~\ensuremath{\Varid{x}} in~\ensuremath{X}'').
There are several things to notice:
\begin{itemize}
\item \rulename{subst} fires only when the right-hand side of the equation
  is a \emph{value} \ensuremath{\Varid{v}}, so that the substitution does not risk
  duplicating either work or choices. This
restriction is precisely the same as the \rulename{let-v} rule of \citet{ariola:call-by-need}
and, by not duplicating choices, it neatly implements so-called \emph{call-time choice}~\cite{hanus:survey}.
We do not need a heap, or thunks, or updates; the equalities of the program elegantly
suffice to express the necessary sharing.
\item \rulename{subst} replaces all occurrences of \ensuremath{\Varid{x}} in \ensuremath{X} and \ensuremath{\Varid{e}}, but
  \emph{it leaves the original \ensuremath{(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\Varid{v})} undisturbed}, because \ensuremath{X} might not be
  big enough to encompass all occurrences of \ensuremath{\Varid{x}}.
  For example, we can rewrite \ensuremath{(\Varid{y}\hspace{-0.14em}=\hspace{-0.14em}\Varid{x}\mathbin{+}\mathrm{1};\,(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{3};\,\Varid{x}\mathbin{+}\mathrm{3}))} to \ensuremath{(\Varid{y}\hspace{-0.14em}=\hspace{-0.14em}\Varid{x}\mathbin{+}\mathrm{1};\,(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{3};\,\mathrm{3}\mathbin{+}\mathrm{3}))}, using
  \ensuremath{X\hspace{-0.14em}=\hspace{-0.14em}\hole}, where the \ensuremath{\hole} is \ensuremath{\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{3};\,\Varid{x}\mathbin{+}\mathrm{3}}.  But that leaves an occurrence of \ensuremath{\Varid{x}} in \ensuremath{(\Varid{y}\hspace{-0.14em}=\hspace{-0.14em}\Varid{x}\mathbin{+}\mathrm{1})}.
  When there are no remaining occurrences of \ensuremath{\Varid{x}} we may eliminate
  the binding: see \cref{sec:gc}.
\item The side condition $\ensuremath{\Varid{v}} \not= \ensuremath{\Conid{V} [\mskip1.5mu \Varid{x}\mskip1.5mu]}$ in \rulename{subst} avoids an overlap with \rulename{u-occurs}.
\end{itemize}

\subsection{Swapping and binding order} \label{sec:swapping}

Two rules allow the two sides of an equation to be swapped.
Rule \rulename{hnf-swap} helps \rulename{subst} to fire by putting the variable on the left.
Rule \rulename{var-swap}, which swaps two variables, is needed for a more subtle reason.  Consider the following example expression, where \ensuremath{\Varid{a}} and \ensuremath{\Varid{b}} are
bound within some containing context, perhaps by lambdas.  It can be rewritten in two different ways, where
each column is a reduction sequence starting from the same initial term:
\begin{center}                         % USING \begin{center} RATHER THAN $$ GIVES BETTER PAGE BREAK
$
\begin{array}{l@{\,}l@{\hspace{0.5cm}}|@{\hspace{0.5cm}}l@{\,}l}
  \multicolumn{4}{c}{\ensuremath{\exists\Varid{x}.\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\langle{}\Varid{a}\rangle;\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\langle{}\Varid{b}\rangle;\,\Varid{x}}} \\
  \movestoR{subst} & \ensuremath{\exists\Varid{x}.\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\langle{}\Varid{a}\rangle;\,\langle{}\Varid{a}\rangle\hspace{-0.14em}=\hspace{-0.14em}\langle{}\Varid{b}\rangle;\,\langle{}\Varid{a}\rangle} &
     \movestoR{subst}  & \ensuremath{\exists\Varid{x}.\,\langle{}\Varid{b}\rangle\hspace{-0.14em}=\hspace{-0.14em}\langle{}\Varid{a}\rangle;\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\langle{}\Varid{b}\rangle;\,\langle{}\Varid{b}\rangle} \\
  \movestoR{u-tup} & \ensuremath{\exists\Varid{x}.\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\langle{}\Varid{a}\rangle;\,\Varid{a}\hspace{-0.14em}=\hspace{-0.14em}\Varid{b};\,\langle{}\Varid{a}\rangle} &
     \movestoR{u-tup}  & \ensuremath{\exists\Varid{x}.\,\Varid{b}\hspace{-0.14em}=\hspace{-0.14em}\Varid{a};\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\langle{}\Varid{b}\rangle;\,\langle{}\Varid{b}\rangle} \\
  \movestoR{eqn-elim} & \ensuremath{\Varid{a}\hspace{-0.14em}=\hspace{-0.14em}\Varid{b};\,\langle{}\Varid{a}\rangle} &
     \movestoR{eqn-elim} & \ensuremath{\Varid{b}\hspace{-0.14em}=\hspace{-0.14em}\Varid{a};\,\langle{}\Varid{b}\rangle} \\
  \movestoR{subst} & \ensuremath{\Varid{a}\hspace{-0.14em}=\hspace{-0.14em}\Varid{b};\,\langle{}\Varid{b}\rangle} &
      \movestoR{subst} & \ensuremath{\Varid{b}\hspace{-0.14em}=\hspace{-0.14em}\Varid{a};\,\langle{}\Varid{a}\rangle} \\

\end{array}
$
\end{center}
The two sequences differ when it comes to which equation for \ensuremath{\Varid{x}} is chosen for \rulename{subst}
in the first step of each column. As you can see, they conclude with two terms that are ``obviously''
the same semantically, but which are syntactically different.  Rule \rulename{var-swap} allows
them to be brought together (for example, $\ensuremath{\Varid{a}\hspace{-0.14em}=\hspace{-0.14em}\Varid{b};\,\langle{}\Varid{b}\rangle} \; \movestoR{var-swap} \; \ensuremath{\Varid{b}\hspace{-0.14em}=\hspace{-0.14em}\Varid{a};\,\langle{}\Varid{b}\rangle} \; \movestoR{subst} \;  \ensuremath{\Varid{b}\hspace{-0.14em}=\hspace{-0.14em}\Varid{a};\,\langle{}\Varid{a}\rangle}$),
so that the unification rules can be syntactically confluent.

Rule \rulename{seq-swap}, which swaps adjacent equations within a sequence under certain conditions, is needed for a similar reason. Consider
this example:
% where |a,b,c| are bound in that order; \ie, $|c| \prec |b| \prec |a|$:
\begin{center}                         % USING \begin{center} RATHER THAN $$ GIVES BETTER PAGE BREAK
$
\begin{array}{l@{\,}l@{\hspace{0.5cm}}|@{\hspace{0.5cm}}l@{\,}l}
  \multicolumn{4}{c}{\quad\ensuremath{\Varid{c}\hspace{-0.14em}=\hspace{-0.14em}\Varid{a};\,\Varid{c}\hspace{-0.14em}=\hspace{-0.14em}\Varid{b};\,\Varid{c}}} \\
  \movestoR{subst} & \ensuremath{\Varid{c}\hspace{-0.14em}=\hspace{-0.14em}\Varid{a};\,\Varid{a}\hspace{-0.14em}=\hspace{-0.14em}\Varid{b};\,\Varid{a}} &
     \movestoR{subst} & \ensuremath{\Varid{b}\hspace{-0.14em}=\hspace{-0.14em}\Varid{a};\,\Varid{c}\hspace{-0.14em}=\hspace{-0.14em}\Varid{b};\,\Varid{b}} \\
  \movestoR{var-swap} & \ensuremath{\Varid{c}\hspace{-0.14em}=\hspace{-0.14em}\Varid{a};\,\Varid{b}\hspace{-0.14em}=\hspace{-0.14em}\Varid{a};\,\Varid{a}} &
     \movestoR{subst}   & \ensuremath{\Varid{b}\hspace{-0.14em}=\hspace{-0.14em}\Varid{a};\,\Varid{c}\hspace{-0.14em}=\hspace{-0.14em}\Varid{a};\,\Varid{a}} \\
\end{array}
$
\end{center}
Again, the concluding terms of the two columns are ``obviously'' the same
(they differ only in the order of the equations \ensuremath{\Varid{b}\hspace{-0.14em}=\hspace{-0.14em}\Varid{a}} and \ensuremath{\Varid{c}\hspace{-0.14em}=\hspace{-0.14em}\Varid{a}});
\rulename{seq-swap} allows one term to be rewritten to match the other, making explicit
our intuition that the order of equations of the form \ensuremath{\Varid{x}\mathrel{=}\Varid{v}} should not matter.

% not for \emph{progress}. \simon{I think this is true of both, isn't it?}
% \lennart{Without \rulename{var-swap} I think we need another \rulename{eqn-elim} rule for the |y==x| case.}

Observe the mysterious side condition $\ensuremath{\Varid{x}} \prec \ensuremath{\Varid{y}}$  in rule \rulename{var-swap},
and a similar one in % \rulename{subst} and
\rulename{seq-swap}.
In the overall proof of confluence, it turns out to
be very helpful if the unification rules
are \emph{terminating} (see \cref{sec:design-confluence}).
To achieve this, \rulename{var-swap} fires on \ensuremath{\Varid{y}\hspace{-0.14em}=\hspace{-0.14em}\Varid{x}} \emph{only if
\ensuremath{\Varid{x}}'s binding occurs in the scope of \ensuremath{\Varid{y}}'s binding}, written $\ensuremath{\Varid{x}} \prec \ensuremath{\Varid{y}}$, so that the innermost-bound variable
ends up on the left.
Similarly, the side condition on \rulename{seq-swap} prevents it firing infinitely.

Other rules, notably \rulename{exi-swap}, may change this binding order
and thereby re-enable \rulename{var-swap} or \rulename{seq-swap},
but the unification rules \emph{considered in isolation} are terminating and confluent,
and that is what we need for the proof (but see \Cref{sec:simpler}).

\subsection{Elimination and normalization rules} \label{sec:gc} \label{sec:normalization}

Four \emph{elimination} rules allow dead code to be dropped (\cref{fig:rules}):
\rulename{val-elim} discards a value to the left of a semicolon; \rulename{exi-elim} discards
a dead existential; \rulename{eqn-elim} discards an existential $\ensuremath{\exists{}} \ensuremath{\Varid{x}}$ that binds a variable whose only
occurrence is a single equation \ensuremath{\Varid{x}\mathrel{=}\Varid{v}}; and \rulename{fail-elim} discards the context surrounding a \ensuremath{\textbf{fail}}.
Note that none of these rules, except \rulename{fail-elim}, discard an unevaluated expression,
because that expression might fail and we don't want to ``lose'' that failure (see \cref{sec:lenient}).
The exception is \rulename{fail-elim}, which propagates failure.

Four \emph{normalization} rules help to put the expression in a form that
allows other rules to fire (\cref{fig:rules}):  \rulename{exi-float} allows an existential to float outwards;
\rulename{seq-assoc} makes semicolon right-associated; \rulename{eqn-float} moves work out
of the right-hand side of an equation \ensuremath{\Varid{v}\hspace{-0.14em}=\hspace{-0.14em}\Varid{e}}.
For example, we cannot use \rulename{subst} to substitute for \ensuremath{\Varid{x}} in \ensuremath{(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}(\Varid{e};\,\mathrm{3});\,\Varid{x}\mathbin{+}\mathrm{2})},
because the RHS of the \ensuremath{\Varid{x}}-equation is not a value; but we can instead apply \rulename{eqn-float}
to get \ensuremath{(\Varid{e};\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{3});\,\Varid{x}\mathbin{+}\mathrm{2}}, and then \rulename{seq-assoc} to get \ensuremath{\Varid{e};\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{3};\,\Varid{x}\mathbin{+}\mathrm{2}}; and now we \emph{can}
apply \rulename{subst}.

Rule \rulename{exi-swap} allows you to move an existential inward so that a dead equation can be eliminated
by \rulename{eqn-elim}.
Rule \rulename{exi-swap} is unusual because it can be infinitely applied;
avoiding that eventuality is easily achieved by tweaking the evaluation strategy (\cref{sec:eval-strategy}).

Note that all these swapping and normalization rules \emph{preserve the left-to-right sequencing of expressions},
which matters because choices are made left to right as we saw in \cref{sec:mixing}.
Moreover, the rules do not float equalities or existentials out of choices:
that restriction is the key to localizing unification (\cref{sec:mixing}) and
to the flexible/rigid distinction of
\cref{sec:rigid}. For example, consider the expression
\ensuremath{(\Varid{y}\hspace{-0.14em}=\hspace{-0.14em}((\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{3};\,\Varid{x}\mathbin{+}\mathrm{5})\;\rule[-0.1em]{0.15em}{0.75em}\;(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{4}));\,\langle{}\Varid{x}\mathbin{+}\mathrm{1},\Varid{y}\rangle)}.
We must not float the binding \ensuremath{(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{3})} up to a point where it might interact with the expression \ensuremath{(\Varid{x}\mathbin{+}\mathrm{1})},
because the latter is outside the choice, and a different branch of the choice binds \ensuremath{\Varid{x}} to \ensuremath{\mathrm{4}}.

\subsection{Rules for choice} \label{sec:choice-rules}

The rules for choice are given in \cref{fig:rules}:
\begin{itemize}
\item Rules \rulename{one-fail}, \rulename{one-value}, and \rulename{one-choice} describe
  the semantics of \ensuremath{\textbf{one}}, as in \cref{sec:one}.
\item Similarly, \rulename{all-fail}, \rulename{all-value}, and \rulename{all-choice} describe
  the semantics of \ensuremath{\textbf{all}} (\cref{sec:all}).
\item Rules \rulename{choose-r} and \rulename{choose-l} eliminate \ensuremath{\textbf{fail}}, which behaves as
  an identity for choice.
\item Rule \rulename{choose-assoc} associates choice to the right, so that \rulename{one-choice} or \rulename{all-choice}
  can fire.  (The dots on the left of \rulename{all-choice} should be read as
  a string of right-associated choices.)
\end{itemize}
The most interesting rule is \rulename{choose}, which, just as described in \cref{sec:choice},
``floats the choice outwards,'' duplicating the surrounding context.
But what ``surrounding context'' precisely? We use two new contexts, \ensuremath{\Conid{SX}} and \ensuremath{CX}, both
defined in \cref{fig:contexts}. A \emph{choice context} \ensuremath{CX} is like a
execution context \ensuremath{X}, \emph{but with no possible
choices to the left of the hole}:
\begin{center}$
CX ::= \hole \vb \ensuremath{\Varid{v}\hspace{-0.14em}=\hspace{-0.14em}CX} \vb \ensuremath{CX;\,\Varid{e}} \vb \ensuremath{\Varid{ce};\,CX} \vb \ensuremath{\exists\Varid{x}.\,CX}
$\end{center}
Here, \ensuremath{\Varid{ce}} is a guaranteed-choice-free expression (syntax in \cref{fig:choice-free-syntax}).
This syntactic condition is necessarily conservative; for example, a call \ensuremath{\Varid{f}(\Varid{x})}
is considered not guaranteed-choice-free because it depends on what function \ensuremath{\Varid{f}} does.
We must guarantee not to have choices to the left so that we preserve the order of choices---see \cref{sec:mixing}.

The context \ensuremath{\Conid{SX}} (\cref{fig:contexts}) in \rulename{choose} ensures that \ensuremath{CX} is as large as possible.
This is a very subtle point:
without this restriction we lose confluence.  To see this, consider\footnote{Remember, \ensuremath{\textbf{if}} is syntactic sugar for
  a use of \ensuremath{\textbf{one}} (see \cref{sec:one}), but using \ensuremath{\textbf{if}} makes the example easier to understand.}:
$$
\begin{array}{l@{\;\;}l}
                                 & \ensuremath{\exists\Varid{x}.\,(\textbf{if}\;(\Varid{x}\mathbin{>}\mathrm{0})\;\textbf{then}\;\mathrm{55}\;\textbf{else}\;(\mathrm{44}\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\mathrm{2}));\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{1};\,(\mathrm{77}\;\rule[-0.1em]{0.15em}{0.75em}\;\mathrm{99})}\\
 \movestoR{subst}                & \ensuremath{\exists\Varid{x}.\,(\textbf{if}\;(\mathrm{1}\mathbin{>}\mathrm{0})\;\textbf{then}\;\mathrm{55}\;\textbf{else}\;(\mathrm{44}\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\mathrm{2}));\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{1};\,(\mathrm{77}\;\rule[-0.1em]{0.15em}{0.75em}\;\mathrm{99})}\\
 \movestoC{\text{simplify \ensuremath{\textbf{if}}}} & \ensuremath{\exists\Varid{x}.\,\mathrm{55};\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{1};\,(\mathrm{77}\;\rule[-0.1em]{0.15em}{0.75em}\;\mathrm{99})} \;\;    % Tweaks to save a line
 \movestoR{val-elim, eqn-elim}   \;\;  \ensuremath{\mathrm{77}\;\rule[-0.1em]{0.15em}{0.75em}\;\mathrm{99}}\\
\end{array}
$$
But suppose instead we floated the choice out, \emph{partway}, like this\footnote{
  As in the previous example, here and elsewhere we freely rewrite terms that have not
  been fully desugared, but that is just an expository aid;
  formally, the rewrite rules of \cref{fig:rules} apply only to programs in the basic ``Syntax'' language of \cref{fig:syntax}.}:
$$
\begin{array}{ll}
\multicolumn{2}{l}{\ensuremath{\exists\Varid{x}.\,(\textbf{if}\;(\Varid{x}\mathbin{>}\mathrm{0})\;\textbf{then}\;\mathrm{55}\;\textbf{else}\;(\mathrm{44}\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\mathrm{2}));\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{1};\,(\mathrm{77}\;\rule[-0.1em]{0.15em}{0.75em}\;\mathrm{99})}}\\
\quad \movestoC{\text{Bogus \rulename{choose}}} & \ensuremath{\exists\Varid{x}.\,(\textbf{if}\;(\Varid{x}\mathbin{>}\mathrm{0})\;\textbf{then}\;\mathrm{55}\;\textbf{else}\;(\mathrm{44}\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\mathrm{2}));\,((\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{1};\,\mathrm{77})\;\rule[-0.1em]{0.15em}{0.75em}\;(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{1};\,\mathrm{99}))}\\
\end{array}
$$
Now the \ensuremath{(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{1})} is inside the choice branches, so we cannot use \rulename{subst} to
substitute for \ensuremath{\Varid{x}} in the condition of the \ensuremath{\textbf{if}}.
Nor can we use \rulename{choose} again to float the choice further out
because the \ensuremath{\textbf{if}} is not guaranteed choice-free (in this example, the \ensuremath{\textbf{else}} branch has a choice).
So, alas, we are stuck!
Our not-entirely-satisfying solution is to force \rulename{choose} to float the choice all the way
to the top.  The \ensuremath{\Conid{SX}} context (\cref{fig:contexts}) formalizes what we mean by ``the top'':
rule \rulename{choose} can float a choice outward only when it becomes part of
the choice tree (context \ensuremath{\Conid{SC}}) immediately under a \ensuremath{\textbf{one}} or \ensuremath{\textbf{all}} construct (context \ensuremath{\Conid{SX}}).

Rule \rulename{choose} moves choices around; only \rulename{one-choice} and \rulename{all-choice}
\emph{decompose} choices.
So choice behaves a bit like a data constructor, or normal form, of the language.
This contrasts with other semantic approaches that eliminate choice by non-deterministically
picking one branch or the other, which immediately gives up confluence.  However, in implementation
terms, treating choice a bit like a data constructor is the basis for \emph{pull-tabbing}
in logically-complete implementations of non-deterministic functional logic
languages like Curry \cite{antoy:pull-tabbing}.

\subsection{The Verse calculus is lenient} \label{sec:lenient}  \label{sec:laziness}

\versecalc{} is \emph{lenient} \cite{lenience},
not lazy (call-by-need), nor strict (call-by-value).
Under lenient evaluation, functions can run before their arguments have a ground value
(as in a lazy language), but (almost) everything is eventually evaluated (as in a strict language).

\versecalc{} cannot be lazy, because
of \versecalc{}'s commitment to determinism and, particularly, its first-class \ensuremath{\textbf{all}} operator.  Consider:
$$
\ensuremath{\textbf{all}\{\exists\Varid{x}.\,\Varid{x}\mathrel{=}\Varid{e};\,\mathrm{3}\}}
$$
In a lazy language like Curry, the expression \ensuremath{\exists\Varid{x}.\,\Varid{x}\mathrel{=}\Varid{e};\,\mathrm{3}} would return one result, \ensuremath{\mathrm{3}}, regardless of \ensuremath{\Varid{e}},
because \ensuremath{\Varid{x}} is unused, and so \ensuremath{\Varid{x}\mathrel{=}\Varid{e}} is simply dead code. But in \versecalc{} we must evaluate \ensuremath{\Varid{e}}, and the \ensuremath{\textbf{all}} expression will yield an array with one element
for each result yielded by \ensuremath{\Varid{e}}.  If \ensuremath{\Varid{e}} fails (returns zero results), then the result array is empty.  Since \ensuremath{\textbf{all}}
reifies these results into an array, the multiplicity of results is visible to the context of the \ensuremath{\textbf{all}} expression, via
the length of the returned array.
Moreover, consider
$$
\ensuremath{\textbf{all}\{\exists\Varid{x},\Varid{y}.\,\Varid{x}\mathrel{=}(\Varid{y}\mathrel{=}\mathrm{3};\,\mathrm{1}\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\Varid{y}\mathrel{=}\mathrm{4};\,\mathrm{2});\,\Varid{y}\}}
$$
In \versecalc{}, the expression to which \ensuremath{\Varid{x}} is equated must be evaluated, yielding two values, \ensuremath{\mathrm{1}} and \ensuremath{\mathrm{2}}, which are then discarded since
\ensuremath{\Varid{x}} is unused.  However, each of these
values is accompanied by a distinct binding for \ensuremath{\Varid{y}}, so the result of the whole expression is the array \ensuremath{\langle{}\mathrm{3},\mathrm{4}\rangle}.
In short, we must eventually evaluate the expression to which \ensuremath{\Varid{x}} is equated not only to get the \emph{size} of the array,
but also the \emph{values of its elements}, via the bindings of \ensuremath{\Varid{y}}.

% However, a function can proceed even before its argument is a ground value.
% Consider the call |apply1 f e|, where |e| is not a value.  In \versecalc{}, applications are in administrative normal form (ANF),
% so we must actually write |def x (x==e; apply1 f x)|. This expression will not return a value until
% |e| reduces to a value: that is, everything is eventually evaluated.  But even so, |apply1 f x| can proceed
% to \textbeta-reduce (\cref{sec:beta}), assuming we know the definition of |f|.

So it seems hard to reconcile laziness with a deterministic \ensuremath{\textbf{all}}.  But \versecalc{} is not strict, either;
in \versecalc{} a function can be called without its argument having a ground value.  For example:
$$
\ensuremath{\exists\Varid{f}.\,\Varid{f}\mathrel{=}(\lambda\Varid{x}.\,\Varid{x}\mathrel{=}\mathrm{3};\,\Varid{x});\,\exists\Varid{y}.\,\Varid{f}(\Varid{y})}
$$
Here we can call \ensuremath{\Varid{f}} on an uninstantiated existential variable \ensuremath{\Varid{y}};
we do not have to wait until \ensuremath{\Varid{y}} gets a ground value from the calling context.
Rather, in this example it is the body of \ensuremath{\Varid{f}} that then constrains \ensuremath{\Varid{y}} to be \ensuremath{\mathrm{3}}.
This is part of the essence of functional logic programming, and indeed in \versecalc{} a variable \emph{is} a value
(\cref{fig:syntax}).
When lenience was introduced in the data-flow language Id~\cite{lenience}, it was a way
to get \emph{parallelism}, and it certainly justifies parallelism in an implementation of \versecalc{}.
But for \versecalc{} lenience has \emph{semantic} significance, too, as we see in this example.
% Consider:
% $$
% |def (f ^^ x ^^ y) (f == (lam p (x==7;p)); y == (if (x>0) then 7 else 8); apply1 f y|
% $$
% Here, |y| does not get a value until |x| is known; but |x| does not get its
% value (in this case |7|) until |f| is called.  Without lenience this program would be stuck.

So in \versecalc{} functions can be called on as-yet-evaluated arguments,
but almost all redexes that are not under a lambda get evaluated eventually.   You can see this
enforced in the rule for \ensuremath{\textbf{one}} in \Cref{fig:rules}, which fires only when the first choice
in the body is a value $\ensuremath{\Varid{v}}$, and similarly for \ensuremath{\textbf{all}}.
Why ``almost'' all redexes? Because the \ensuremath{\textbf{one}} operator returns the value of its first choice and abandons all other choices.
For example \ensuremath{\textbf{one}\{\mathrm{1}\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\Varid{loop}\langle\rangle\}} returns \ensuremath{\mathrm{1}}, discarding the infinite \ensuremath{\Varid{loop}\langle\rangle}.

Note that lenience, like laziness, supports \emph{abstraction} in the presence of unification.
For example, we can replace an expression \ensuremath{(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\langle{}\Varid{y},\mathrm{3}\rangle;\,\Varid{y}\mathbin{>}\mathrm{7})} by
$$
\ensuremath{\exists\Varid{f}.\,\Varid{f}\hspace{-0.14em}=\hspace{-0.14em}(\lambda\langle{}\Varid{p},\Varid{q}\rangle.\,\Varid{p}\hspace{-0.14em}=\hspace{-0.14em}\langle{}\Varid{q},\mathrm{3}\rangle;\,\Varid{q}\mathbin{>}\mathrm{7});\,\Varid{f}\langle\Varid{x},\Varid{y}\rangle}
$$
Here, we abstract over the free variables of the expression and define a named function \ensuremath{\Varid{f}}.
Calling the function is just the same as writing the original expression.
This transformation would not be valid under call-by-value.

% However, moving to \emph{laziness} seems problematic.
% For example, consider: |def x (x == apply0 wombat; 3)|.  In a lazy language this expression would
% yield |3|, but in \versecalc{}, everything is evaluated, and
% the expression will not return a value until |apply0 wombat| converges.
% There is a good reason for this choice: |apply0 wombat| might fail, and we should not return |3| until
% we know there is no failure.  With laziness, we could easily lose confluence.

\subsection{Evaluation strategy} \label{sec:eval-strategy}

Any rewrite rule can apply anywhere in the term, at any time.  For example, in the term \ensuremath{(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{3}\mathbin{+}\mathrm{4};\,\Varid{y}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{4}\mathbin{+}\mathrm{2};\,\Varid{x}\mathbin{+}\Varid{y})}
the rewrite rules do not say whether to rewrite \ensuremath{\mathrm{3}\mathbin{+}\mathrm{4}} $\rightarrow$ \ensuremath{\mathrm{7}} and then \ensuremath{\mathrm{4}\mathbin{+}\mathrm{2}} $\rightarrow$ \ensuremath{\mathrm{6}},
or the other way around.
The rules do, however, require us to reduce \ensuremath{\mathrm{3}\mathbin{+}\mathrm{4}} $\rightarrow$ \ensuremath{\mathrm{7}} before substituting for \ensuremath{\Varid{x}} in \ensuremath{\Varid{x}\mathbin{+}\Varid{y}}, because
rule \rulename{subst} fires only when the RHS is a value.
The rewrite rules thereby express \emph{semantics}.

For example, in the lambda calculus, by changing the rewrite
rule \textbeta{} to \textbeta V, we change the language from call-by-name to call-by-value; by
adding {\bf let}, plus suitable rewrite rules, we can express call-by-need \cite{ariola:call-by-need}.
In \versecalc{}, the rewrite rules are carefully crafted in a similar way; for example, \rulename{subst} will
substitute \ensuremath{\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\Varid{v}} only when the equation binds a variable to a \emph{value}, rather like \textbeta V in
lambda calculus.  Similarly, the elimination rules never discard a term that could fail.

In any term there may of course be many redexes---that is good.
An \emph{evaluation strategy} answers the question: given a closed term, which unique redex,
out of the many possible redexes, should be rewritten next to make progress toward the result?
Let us call an evaluation strategy \emph{normalizing} if it guarantees to terminate if there
is \emph{any} terminating sequence of reductions; \ie, if any path terminates with a value,
then a normalizing evaluation strategy will terminate with that same value\footnote{
It would be even better if the strategy could guarantee to find the
result in the minimal number of rewrite steps---so-called ``optimal reduction'' \cite{levy1978reductions,lamping1989algorithm,asperti1998optimal}---but optimal reduction is typically very hard, even in theory, and
invariably involves reducing under lambdas, so for practical purposes it is well out of reach.}.
For example, in the pure lambda calculus, \emph{normal-order reduction},
sometimes called \emph{leftmost-outermost reduction}, is a normalizing strategy.

For \versecalc{}, a first step towards a normalizing strategy is to avoid no-ops.
For example, there is no point in applying rule \rulename{subst} if $\ensuremath{\Varid{x}} \not\in \freevars{e}$;
nor in applying \rulename{fail-elim} if \ensuremath{X\mathrel{=}\hole}.  Next, a normalizing strategy
must be careful with \rulename{exi-swap}, because it can easily apply infinitely;
its role is to make it possible to use \rulename{exi-float}.  Dealing with these
no-ops and flip-flops is not hard.

With that done, we believe that a fair outermost strategy is normalizing for
\versecalc{}: at each step, just select any redex that is not within a lambda or another redex.
That sounds easy, but is tricky in practice for two reasons.
First, a reduction may ``unlock'' a redex far to its left.
For example, consider \ensuremath{(\Varid{x}\mathbin{+}\Varid{y};\,\langle{}\Varid{x},\mathrm{3}\rangle\hspace{-0.14em}=\hspace{-0.14em}\langle{}\mathrm{2},\Varid{y}\rangle);\,\Varid{x}}.  The \ensuremath{(\Varid{x}\mathbin{+}\Varid{y})} is not a
redex, but the equation is; we can apply unification to get \ensuremath{(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{2};\,\Varid{y}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{3})}, and then
use substitution to rewrite the \ensuremath{(\Varid{x}\mathbin{+}\Varid{y})} to \ensuremath{(\mathrm{2}\mathbin{+}\mathrm{3})}; and
\emph{now} the \ensuremath{(\mathrm{2}\mathbin{+}\mathrm{3})} is a redex. Hence, a challenge for an implementation is to
find the next redex efficiently. Secondly, as with other functional-logic languages such as Curry, a normalizing
strategy must be able to pursue redexes within multiple subterms in parallel (or alternately), without starving any of them, hence ``fair''.  For example \ensuremath{(\Varid{loop}\langle\rangle;\,\textbf{fail})}
and \ensuremath{(\textbf{fail};\,\Varid{loop}\langle\rangle)} should both fail, so a normalizing strategy must, in both cases, avoid getting stuck
in the \ensuremath{\Varid{loop}}.

We have not yet, however, formalized such a strategy for \versecalc{} or proved it normalizing.
We have several prototype implementations of \versecalc{}, each involving an abstract
machine with a stack, a heap, a bunch of previously blocked computations, and so on. Exploring this
design space is, however, beyond the scope of this paper.

\subsection{Logical completeness} \label{sec:local-subst}

Our rewrite rules seek to implement the idea articulated in \Cref{sec:unify} that
a program executes by solving its equations, specifically by finding values for the logical variables
that satisfy the equations.  Ideally, we would like to promise that if any substitution for the
logical variables exists that satisfies the equations, we will find it: this is \emph{logical completeness}.
Alas, in practice logical completeness is well out of reach.
For example, consider \ensuremath{\exists\Varid{x}.\,(\Varid{x}\mathbin{*}\Varid{x}\mathbin{*}\Varid{x}\mathbin{+}\mathrm{3}\mathbin{*}\Varid{x}\mathbin{-}\mathrm{8})\mathrel{=}\mathrm{0}} (assuming a version of \versecalc{} extended to have
a multiplication operator $*$).
To solve this would require solving a cubic
equation, and it is equally easy to express intractable problems like Fermat's last theorem.
Moreover, remembering that \versecalc{} is deterministic, in what order would the (infinite sequence of)
results of, say, \ensuremath{\exists\Varid{x}.\,\Varid{x}\mathbin{+}\mathrm{1}} be returned?
So instead we satisfy ourselves with a computable and deterministic approximation
to this Platonic ideal, an approximation that is described precisely by our rewrite rules, using unification as the
mechanism.  Expressions that unification cannot solve, like the one above, are stuck; no
rewrite rule applies.  All functional logic languages share this challenge to logical completeness.\footnote{For countable
domains, like the integers, or even the algebraic numbers---which would be adequate to solve our cubic equation example---we could
in theory simply try all possible values of \ensuremath{\Varid{x}} one by one; but that is hardly practical.}

The boundary between ``stuck'' and ``soluble'' can be quite subtle.
Consider this tricky term:
\ensuremath{\exists\Varid{x}.\,\Varid{x}\mathrel{=}\textbf{if}\;(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{0};\,\Varid{x}\mathbin{>}\mathrm{1})\;\textbf{then}\;\mathrm{33}\;\textbf{else}\;\mathrm{55};\,\Varid{x}}.
At first we might think it was stuck---how can we simplify the \ensuremath{\textbf{if}}
when its condition mentions~\ensuremath{\Varid{x}}, which is not yet defined?  But in fact, rule \rulename{subst} allows
us to substitute \emph{locally} in any \ensuremath{X}-context surrounding the equation \ensuremath{(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{0})}
thus:
$$
\begin{array}{l@{\;}l}
\multicolumn{2}{l}{\ensuremath{\exists\Varid{x}.\,\Varid{x}\mathrel{=}\textbf{if}\;(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{0};\,\Varid{x}\mathbin{>}\mathrm{1})\;\textbf{then}\;\mathrm{33}\;\textbf{else}\;\mathrm{55};\,\Varid{x}}} \\
\movestoR{subst}                & \ensuremath{\exists\Varid{x}.\,\Varid{x}\mathrel{=}\textbf{if}\;(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{0};\,\mathrm{0}\mathbin{>}\mathrm{1})\;\textbf{then}\;\mathrm{33}\;\textbf{else}\;\mathrm{55};\,\Varid{x}} \\
\movestoR{app-gt-fail}     & \ensuremath{\exists\Varid{x}.\,\Varid{x}\mathrel{=}\textbf{if}\;(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{0};\,\textbf{fail})\;\textbf{then}\;\mathrm{33}\;\textbf{else}\;\mathrm{55};\,\Varid{x}} \\
\movestoR{fail-elim}     & \ensuremath{\exists\Varid{x}.\,\Varid{x}\mathrel{=}\textbf{if}\;\textbf{fail}\;\textbf{then}\;\mathrm{33}\;\textbf{else}\;\mathrm{55};\,\Varid{x}} \\
\multicolumn{2}{l}{\movestoC{\text{simplify \ensuremath{\textbf{if}}}} \; \ensuremath{\exists\Varid{x}.\,\Varid{x}\mathrel{=}\mathrm{55};\,\Varid{x}} \quad
\movestoR{subst} \; \ensuremath{\exists\Varid{x}.\,\Varid{x}\mathrel{=}\mathrm{55};\,\mathrm{55}} \quad
\movestoR{eqn-elim}  \; \ensuremath{\mathrm{55}}}
\end{array}
$$
Minor variants of the same example get stuck instead of reducing. For example, if we replace
the \ensuremath{\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{0}} with \ensuremath{\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{100}} then rewriting gets stuck, as the reader may verify
(we cannot eliminate the equation \ensuremath{\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{100}}); and yet there
is a substitution that will satisfy the equations, namely $\{55/x\}$.
And if we replace \ensuremath{\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{0}} with \ensuremath{\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\mathrm{55}}, then rewriting again
gets stuck; this time there is \emph{no} substitution that will satisfy the equations, and yet
the expression gets stuck rather than failing.

\section{Metatheory} \label{sec:metatheory}

The rules of our rewrite semantics can be applied anywhere,
in any order, and they give meaning to programs without
committing to a particular evaluation strategy.
%
But then it had better be the case that no matter how the
rules are applied, one always obtains the same result!
That is, our rules should be \emph{confluent}.  In this section, we describe
our proof of confluence.  Because the rule set is quite big (compared, say, to the
pure lambda calculus), this proof turns out to be a substantial undertaking.

\mypara{Reductions and confluence.}
%
A \emph{binary relation} is a set of pairs of related items.
%
A \emph{reduction relation} \rr is the \emph{compatible closure}\footnote{
``Compatible closure'' means that, for any context $E$ and any two terms $M$ and $N$,
if $(M,N)\in\rr$ then $(E[M],E[N])\in\rr$.}
of any binary relation on a set of tree-structured  \emph{terms,}
such as the terms generated by some BNF grammar.
%
We write $\rr^{*}$ for the reflexive transitive closure of $\rr$.
% We write $\rr^{k}$ for the $k$-step closure of $\rr$ and $\rr^{*}$
% for the reflexive and transitive closure of $\rr$, \ie $\rr^{*} \equiv \cup_{0 \leq k} \rr^k$.
%\lennart{We don't see $\rr^k$ being used anywhere below, so we dropped it here.}\guy{And now dropping Lennart's comment.}
%
We write $\rstep{\rr}{e}{e'}$ ($e$ \emph{steps to} $e'$) if $(e, e') \in \rr$ and
$\rsteps{\rr}{e}{e'}$ ($e$ \emph{reduces to} $e'$) if $(e, e') \in \rr^{*}$.
%
A reduction relation $\rr$ is \emph{confluent} if whenever $\rsteps{\rr}{e}{e_1}$ and $\rsteps{\rr}{e}{e_2}$,
there exists an $e'$ such that $\rsteps{\rr}{e_1}{e'}$ and $\rsteps{\rr}{e_2}{e'}$.
%
Confluence gives us the assurance that we will not get
different results when choosing different rules, or get
stuck with some rules and not with others.

\mypara{Normal forms and unicity.}
%
A term $e$ is an \emph{$\rr$-normal form} if there does
not exist any $e'$ such that $\rstep{\rr}{e}{e'}$.
Confluence implies uniqueness of normal forms (unicity): if $\rsteps{\rr}{e}{e_1}$ and $\rsteps{\rr}{e}{e_2}$,
and $e_1$ and $e_2$ are normal forms, then $e_1 = e_2$
\cite[Corollary 3.1.13(ii)]{Barendregt}.

\subsection{Recursion, and the notorious even/odd problem}  \label{sec:even-odd}
%
It is well known that adding {\bf letrec} to the lambda calculus
makes it non-confluent, in a very tiresome, but hard-to-avoid, way \cite{ariola:skew-confluence}.
In our context, consider the term:
%
$$
\begin{array}{lcl@{\qquad}l}
 \ensuremath{\exists\Varid{x}\hspace{0.1em}\Varid{y}.\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\langle{}\mathrm{1},\Varid{y}\rangle;\,\Varid{y}\hspace{-0.14em}=\hspace{-0.14em}(\lambda\Varid{z}.\,\Varid{x});\,\Varid{x}}
 & \rsteps{}{}{} & \ensuremath{\exists\Varid{y}.\,\Varid{y}\hspace{-0.14em}=\hspace{-0.14em}(\lambda\Varid{z}.\,\langle{}\mathrm{1},\Varid{y}\rangle);\,\langle{}\mathrm{1},\Varid{y}\rangle} &  \text {(1) substitute for \ensuremath{\Varid{x}} first} \\[0.5mm]

 \ensuremath{\exists\Varid{x}\hspace{0.1em}\Varid{y}.\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\langle{}\mathrm{1},\Varid{y}\rangle;\,\Varid{y}\hspace{-0.14em}=\hspace{-0.14em}(\lambda\Varid{z}.\,\Varid{x});\,\Varid{x}}
& \rsteps{}{}{} & \ensuremath{\exists\Varid{x}.\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\langle{}\mathrm{1},\lambda\Varid{z}.\,\Varid{x}\rangle;\,\Varid{x}} & \text {(2) substitute for \ensuremath{\Varid{y}} first}
\end{array}
$$
%
The results of (1) and (2) have the same meaning (are indistinguishable by a \versecalc{} context)
but cannot be joined by our rewrite rules. Nor is this easily fixed by adding new
rules, as we did when we added \rulename{var-swap} (\cref{sec:unification-rules})
and \rulename{seq-swap} (\cref{sec:normalization}).  Why not? Because the terms
are equivalent only under some kind of graph isomorphism.

We have tackled this problem in three different ways.  First, we can simply prohibit recursion, and prove
confluence under that restriction (\cref{sec:confluence-sketch}).
This is akin to proving confluence for the lambda calculus with {\bf let} but not {\bf letrec}.
In \versecalc{}, excluding recursion is not so simple because \versecalc{} has no {\bf letrec}; rather,
recursion emerges during execution.  For example, is this recursive: \ensuremath{\exists\Varid{x}\hspace{0.1em}\Varid{y}.\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\langle{}\mathrm{1},\Varid{y}\rangle;\,\Varid{f}\langle\Varid{x},\Varid{y}\rangle}?
It might be if \ensuremath{\Varid{f}\hspace{-0.14em}=\hspace{-0.14em}(\lambda\langle{}\Varid{v},\Varid{w}\rangle.\,\Varid{v}\hspace{-0.14em}=\hspace{-0.14em}\Varid{w})}!
For \emph{tuples}, we have a simple sol\-ution: rule \rulename{u-occurs} makes the entire
term fail if we get recursion through a tuple. But we cannot do this for lambdas because it
leads to non-confluence. Consider \ensuremath{\Varid{f}\hspace{-0.14em}=\hspace{-0.14em}(\lambda\Varid{x}.\,\Varid{const}\langle\Varid{x},\Varid{f}\rangle)}, where \ensuremath{\Varid{const}\hspace{-0.14em}=\hspace{-0.14em}(\lambda\langle{}\Varid{p},\Varid{q}\rangle.\,\Varid{p})}.
The equation for \ensuremath{\Varid{f}} \emph{looks} recursive because the RHS mentions \ensuremath{\Varid{f}}; but if we \textbeta-reduce the application
of \ensuremath{\Varid{const}}, the occurrence of \ensuremath{\Varid{f}} disappears.

Thus motivated, we restrict our attention to terms that have \emph{\norecursion}:
\begin{itemize}
\item A \emph{recursive equation} is an equation of the form
\ensuremath{\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\Conid{V} [\mskip1.5mu \lambda\Varid{y}.\,\Varid{e}\mskip1.5mu]}, where $x \in \freevars{e}$,
which equates a variable \ensuremath{\Varid{x}} with a value that contains a lambda in which \ensuremath{\Varid{x}} is free.
\item A term $e$ is \emph{recursive} if it contains a recursive equation.
\item A term $e$ is \emph{transitively recursive} if $\rsteps{}{e}{e'}$ where $e'$ is recursive.
\item A term $e$ has \emph{\norecursion} if it is not transitively recursive.
\end{itemize}
%

\noindent
The \noHYPHENrecursion condition is not as severe as it might first appear: it only prohibits
recursion through \emph{equations}.
But no expressiveness is lost thereby: in our untyped setting, one can still write recursive (and non-terminating) programs
using one's favorite fixpoint combinator, such as \textbf{Y} or \textbf{Z}.

This approach is not entirely satisfying: it is hard to prove that a term
has no recursion, and it is clumsy to write recursive programs using only \textbf{Y}-combinators.
Our second approach is to adopt the idea of \emph{skew confluence}
\cite{ariola:skew-confluence}, a clever technique developed
specifically to handle the even/odd problem; we give an overview of skew confluence
in \cref{sec:skew-overview-2}, and
provide details of our approach to a proof of skew confluence for \versecalc{} in \cref{sec:skew-confluence},
including several new lemmas, but we emphasize that the proof of skew confluence is not yet complete.

A third approach is simply to abandon confluence as a goal altogether.
Confluence is, after all, purely \emph{syntactic}, and hence
much stronger than what we really need, which is that each of our rules be
\emph{semantics}-preserving.  But, of course, that requires
an independent notion of semantics, a direction we sketch in \cref{sec:den-sem}.

\subsection{Proof of confluence} \label{sec:confluence-sketch}

Our main result is that \versecalc{}'s reduction rules are confluent for terms with \norecursion.
We sketch the proof here, with full details in \cref{sec:confluence} (and relevant preliminaries in \cref{sec:preliminaries}).


\begin{theorem}[Confluence] \label{thm:confluence-sketch}
%
The reduction relation in \cref{fig:rules}
is confluent for terms with \norecursion.
%
%  If $e$ has \norecursion and $\rsteps{\rr}{e}{e_1}$ and $\rsteps{\rr}{e}{e_2}$ then $\join{\rr}{e_1}{e_2}$.
%
\end{theorem}

\noindent
\emph{Proof sketch.}  Our proof strategy is to
%
(1)~divide the rules into groups for application, unification,
    \etc, approximately as in \cref{fig:rules},
(2)~prove confluence for each separately, and then
(3)~prove that their combination is confluent via commutativity.
%
Given two reduction relations $R$ and $S$, we say that $R$ \emph{commutes} with $S$ if for all terms
$e, e_1, e_2$ such that $\rsteps{R}{e}{e_1}$ and $\rsteps{S}{e}{e_2}$
there exists $e'$ such that $\rsteps{S}{e_1}{e'}$ and $\rsteps{R}{e_2}{e'}$.
%
We prove each individual sub-relation is confluent
and that they pairwise commute. Then confluence of
their union follows, using \citet{Huet80}:

\begin{lemma}[Commutativity] \label{lem:comm-summary}
If $R$ and $S$ are confluent and commute, then $R \cup S$ is confluent.
\end{lemma}
\noindent
%
Proving confluence for application, elimination and choice is easy:
%
they all satisfy the \emph{diamond property}---namely, that two
different reduction steps can be joined at a common term
\emph{by a single step}---which suffices to show the
relations are confluent \cite{Barendregt}.
%
The diamond property itself can be verified easily by considering
critical pairs of transitions.
%
The rules for unification and normalization, however, pose two problems.
%

\mypara{The unification problem.}
%
The first problem is that the unification relation does not satisfy the diamond property---it may need
multiple steps to join the results of two different one-step reductions.
%
For example, consider the term $\ensuremath{(\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\langle\mathrm{1},\Varid{y}\rangle;\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\langle\Varid{z},\mathrm{2}\rangle;\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\langle\mathrm{1},\mathrm{2}\rangle;\,\mathrm{3})}$.
%
The term can be reduced in one step by substituting \ensuremath{\Varid{x}}
in the third equation by either \ensuremath{\langle\mathrm{1},\Varid{y}\rangle} or \ensuremath{\langle\Varid{z},\mathrm{2}\rangle}.
After this, it will take multiple steps to join the two terms.

Following a well-trodden path in proofs of confluence
for the \textlambda-calculus (\eg, \cite{Barendregt}),
our proof of confluence for the unification rules works in two stages.
%
First, we prove that the reductions are \emph{locally confluent},
meaning if $e$ single-steps to each of $e_1$ and $e_2$, then $e_1$
and $e_2$ can be joined at some $e'$ by taking \emph{multiple}
unification rule steps.
%
Second, we prove that the unification reductions are \emph{terminating}, which relies
upon eliminating recursion in tuples via \rulename{u-occurs} and in lambdas
via the \noHYPHENrecursion condition.
%
Newman's Lemma \cite[Lemma 2.4]{Huet80} then implies that the
locally confluent, terminating unification relation is
also confluent.

\mypara{The normalization problem.}
%
The second problem is that the normalization rules do not commute with the unification rules.
%
Recall from \cref{sec:swapping} that the unification rules rely upon
variable ordering to orient equations between variables in a canonical fashion.
%
The normalization rule \rulename{exi-swap} can \emph{change}
the variable order and hence, its behavior is deeply intertwined
with unification and cannot be factored out via a commutativity argument.
%
Instead, we prove that the union of unification and normalization
is confluent by showing that unification \emph{postpones after}
normalization \cite{Hindley}; see \cref{sec:confluence} for the gory details.
%
% , \ie, informally speaking, that a
% mixed sequence of unification-or-normalization steps can be reordered
% into two blocks of normalization-only steps followed by unification-only
% steps. The confluence of the union can then be established by suitably
% composing the confluence properties for the individual relations.

\subsection{Overview of skew confluence} \label{sec:skew-overview-2}

We travel a path very similar to the one blazed by Ariola and her co-authors.
Ariola and Klop studied a form of the lambda calculus with an added $\mathsf{letrec}$ construct and
determined (like us) that their calculus was not confluent; then they added a specific
constraint on recursive substitution
and proved that the modified calculus is confluent \cite{ariola:cyclic-lambda,ariola:lambda-letrec}.
In a later paper, Ariola and Blom proved that their calculus without the constraint, while not confluent,
does obey a weaker related property, which they invented, called \emph{skew confluence} \cite{ariola:skew-confluence}.
We believe, and currently are trying to prove, that \versecalc{} without the
pesky \noHYPHENrecursion side condition of \cref{thm:confluence-sketch} is skew confluent.

You can understand skew confluence as follows: if two different reduction paths from \ensuremath{\Varid{e}} produce
terms \ensuremath{\Varid{e}_{\mathrm{1}}}, \ensuremath{\Varid{e}_{\mathrm{2}}}, then \ensuremath{\Varid{e}_{\mathrm{1}}} can be further reduced to some \ensuremath{\Varid{e'}} such that all of \ensuremath{\Varid{e}_{\mathrm{2}}}'s permanent structure

further reduction takes place.  For example, however far we reduce the term \ensuremath{\langle{}\mathrm{1},\lambda\Varid{z}.\,\Varid{e}\rangle}, the

replacing each occurrence of \ensuremath{\mathit{\Omega}} in $\omegarr(\ensuremath{\Varid{e}_{\mathrm{2}}})$ with an (individually-chosen) term.

    \rstep{}{}{} & \ensuremath{\exists\Varid{y}.\,\Varid{y}\hspace{-0.14em}=\hspace{-0.14em}\lambda\Varid{z}.\,\langle{}\mathrm{1},\Varid{y}\rangle;\,\langle{}\mathrm{1},\lambda\Varid{z}.\,\langle{}\mathrm{1},\Varid{y}\rangle\rangle}
  & \rstep{}{}{} & \ensuremath{\exists\Varid{x}.\,\Varid{x}\mathrel{=}\langle{}\mathrm{1},\lambda\Varid{z}.\,\Varid{x}\rangle;\,\langle{}\mathrm{1},\lambda\Varid{z}.\,\Varid{x}\rangle)} \\
    \rstep{}{}{} & \ensuremath{\exists\Varid{y}.\,\Varid{y}\hspace{-0.14em}=\hspace{-0.14em}\lambda\Varid{z}.\,\langle{}\mathrm{1},\Varid{y}\rangle;\,\langle{}\mathrm{1},\lambda\Varid{z}.\,\langle{}\mathrm{1},\lambda\Varid{z}.\,\langle{}\mathrm{1},\Varid{y}\rangle\rangle\rangle}
  & \rstep{}{}{} & \ensuremath{\exists\Varid{x}.\,\Varid{x}\mathrel{=}\langle{}\mathrm{1},\lambda\Varid{z}.\,\Varid{x}\rangle;\,\langle{}\mathrm{1},\lambda\Varid{z}.\,\langle{}\mathrm{1},\lambda\Varid{z}.\,\Varid{x}\rangle\rangle} \\
    \rstep{}{}{} & \cdots & \rstep{}{}{} & \cdots
\end{array}
$$
The two columns can never join up, but if you pick any term in either column, there is a term in the other
column that has a greater amount of permanent structure.  That in turn means that the terms in the left-hand
column are contextually equivalent to those in the right-hand column, because the context can
inspect only the permanent structure. This contextual equivalence is the real reason
for seeking confluence in the first place.

In \versecalc{}, the notion of permanent structure must be generalized from an outer shell of \hbox{tuples}, lambdas, and constants
to a \emph{sequence} (right-associative tree) of \emph{choices} of such outer shells.
For example, however far we reduce the term \ensuremath{\langle{}\mathrm{68},\lambda\Varid{x}.\,\Varid{e}_{\mathrm{1}}\rangle\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\langle{}\lambda\Varid{y}.\,\Varid{e}_{\mathrm{2}},\mathrm{57},\lambda\Varid{z}.\,\Varid{e}_{\mathrm{3}}\rangle\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\Varid{e}_{\mathrm{4}}},
the result will always look like \ensuremath{\langle{}\mathrm{68},\lambda\Varid{x}.\,\Varid{e}_{1}'\rangle\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\langle{}\lambda\Varid{y}.\,\Varid{e}_{2}',\mathrm{57},\lambda\Varid{z}.\,\Varid{e}_{3}'\rangle\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\Varid{e}_{4}'},
where $\rsteps{\rr}{e_k}{e_k'}$ for $1 \leq k \leq 4$.

In \cref{sec:skew-confluence} we show how to adapt the proof strategy
of \cref{sec:confluence-sketch} and \cref{sec:confluence} for skew confluence.
To do this we need a new result: if two relations are skew confluent
with respect to the same information content function and commute, then their union is also skew confluent.
(In fact, it is not required that the two relations fully commute: a slightly weaker precondition suffices.)
Using this result, our plan
is to (i) define an appropriate information content function for \versecalc{} expressions;
(ii) prove that all the rewrite rules for \versecalc{} are monotonic in this information content function;
(iii) prove that the Unification rules (modified to permit recursive substitution) together with the Normalization rules are skew confluent;
(iv) prove that this combined set of rules commutes in the necessary way with the rules for Application, Elimination, and Choice (which taken together are already known to be confluent);
and (v) then apply our new result to show that the entire set of rewrite rules is skew confluent.
At this time steps (iii) and (iv) are incomplete, so we emphasize that we do not yet have a complete proof of skew confluence for \versecalc{}.

\subsection{Simplifying the rules} \label{sec:simpler}

The rules of \Cref{fig:rules} are more complicated than we would like, because our
proof of confluence requires that the unification rules form a \emph{terminating}
rewrite system (\Cref{sec:design-confluence}).  Tantalizingly, we have
found simpler versions of four of the unification and elimination rules that
we believe are equally expressive and still support (modified
versions of) our proofs of termination and confluence:
$$
\begin{array}{@{\hskip0.5em}l@{\qquad}r @{\;\;}c@{\;\;}l@{\qquad}l@{}}
  \rulename{subst${}'$} & \ensuremath{\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\Varid{v};\,\Varid{e}} & \movesto & \ensuremath{\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\Varid{v};\,\subst{\Varid{e}}{\Varid{x}}{\Varid{v}}} \hspace*{-0.3em}
  & \text{if $\ensuremath{\Varid{v}} \not= \ensuremath{\Conid{V} [\mskip1.5mu \Varid{x}\mskip1.5mu]}$}, \\
  \rulename{var-swap${}'$} & \ensuremath{\Varid{y}\hspace{-0.14em}=\hspace{-0.14em}\Varid{x};\,\Varid{e}}    & \movesto & \ensuremath{\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\Varid{y};\,\Varid{e}} & \text{(unconditionally)} \\
  \rulename{seq-swap${}'$} & \ensuremath{eq;\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\Varid{v};\,\Varid{e}} & \movesto & \ensuremath{\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\Varid{v};\,eq;\,\Varid{e}} &
                      \text{(unconditionally)} \\
\rulename{eqn-elim${}'$} & \ensuremath{\exists\Varid{x}.\,\Varid{x}\hspace{-0.14em}=\hspace{-0.14em}\Varid{v};\,\Varid{e}} & \movesto & \ensuremath{\Varid{e}} & \text{if $\ensuremath{\Varid{x}} \notin \freevars{\ensuremath{\Varid{v}},\ensuremath{\Varid{e}}}$} \\
\end{array}
$$
Each of these rules is simpler than its counterpart in \Cref{fig:rules}. In particular, rules \rulename{subst${}'$} and \rulename{eqn-elim${}'$}
need no context~\ensuremath{X}: instead, they rely on other rules (\rulename{eqn-float}
and \rulename{seq-swap${}'$}) to float the equation of interest upward and to the left.  Our testing framework (\Cref{sec:testing})
has given us strong confidence that the entire set of rules, with these four simplifications, remains confluent; actually \emph{proving} it confluent
is work in progress. We speculate that these same simplifications will also support a proof of skew confluence.

\subsection{Ordering and choices} \label{sec:ord-choice}

As we discussed in \cref{sec:choice-rules}, rule \rulename{choose} is less than satisfying for two reasons.
First, the \ensuremath{CX} context uses a conservative, syntactic analysis for choice-free expressions; and second,
the \ensuremath{\Conid{SX}} context is needed to force \ensuremath{CX} to be maximal.  A rule like this would be more satisfying:
$$
\rulename{simpler-choose} \quad   \ensuremath{CX [\mskip1.5mu \Varid{e}_{\mathrm{1}}\;\rule[-0.1em]{0.15em}{0.75em}\;\Varid{e}_{\mathrm{2}}\mskip1.5mu]} \movesto \ensuremath{CX [\mskip1.5mu \Varid{e}_{\mathrm{1}}\mskip1.5mu]\;\rule[-0.1em]{0.15em}{0.75em}\;CX [\mskip1.5mu \Varid{e}_{\mathrm{2}}\mskip1.5mu]}
$$
The trouble with this is that it may change the order of the results (\cref{sec:mixing}).
Another possibility would be to accept that results may come out in the ``wrong'' order, but have
some kind of sorting mechanism to put them back into the ``right'' order.  Something like this:
$$
\rulename{labeled-choose} \quad   \ensuremath{CX [\mskip1.5mu \Varid{e}_{\mathrm{1}}\;\rule[-0.1em]{0.15em}{0.75em}\;\Varid{e}_{\mathrm{2}}\mskip1.5mu]} \movesto \ensuremath{CX [\mskip1.5mu L \triangleright\Varid{e}_{\mathrm{1}}\mskip1.5mu]\;\rule[-0.1em]{0.15em}{0.75em}\;CX [\mskip1.5mu R \triangleright\Varid{e}_{\mathrm{2}}\mskip1.5mu]}
$$
Here, the two branches are labeled with \ensuremath{\Conid{L}} and \ensuremath{\Conid{R}}.  We can add new rules to reorder such
labeled expressions, something in the spirit of:
$$
\rulename{sort} \quad   \ensuremath{(R \triangleright\Varid{e}_{\mathrm{1}})\;\rule[-0.1em]{0.15em}{0.75em}\;(L \triangleright\Varid{e}_{\mathrm{2}})} \movesto \ensuremath{(L \triangleright\Varid{e}_{\mathrm{2}})\;\rule[-0.1em]{0.15em}{0.75em}\;(R \triangleright\Varid{e}_{\mathrm{1}})}
$$
We believe this can be made to work, and it would allow more programs to evaluate, but it adds unwelcome clutter to program terms, and the cure may be worse than the disease. However, the idea inspired our denotational
semantics (\cref{sec:order-sensitive}), where it seems to work rather beautifully.

\subsection{Generalizing \texorpdfstring{\ensuremath{\textbf{one}}}{``one''} and \texorpdfstring{\ensuremath{\textbf{all}}}{``all''}} \label{sec:split}

In \versecalc{}, we introduced \ensuremath{\textbf{one}} and \ensuremath{\textbf{all}} as the primitive choice-consuming operators, and neither is more general than the other, as discussed in \cref{sec:all}.
We could have introduced a more general operator \ensuremath{\textbf{split}}\footnote{The name \ensuremath{\textbf{split}} was inspired by \citet{oleglogict}.} as $e \; ::= \; \cdots \vb \ensuremath{\textbf{split}(\Varid{e})\{\Varid{v}_{\mathrm{1}},\Varid{v}_{\mathrm{2}}\}}$ and rules:
\[\begin{array}{lrcll}
    \rulename{split-fail} & \ensuremath{\textbf{split}(\textbf{fail})\{\Varid{f},\Varid{g}\}} & \movesto & \ensuremath{\Varid{f}\langle\rangle} \\
    \rulename{split-value} & \ensuremath{\textbf{split}(\Varid{v})\{\Varid{f},\Varid{g}\}} & \movesto & \ensuremath{\Varid{g}\langle\Varid{v},\lambda \langle\rangle.\,\textbf{fail}\rangle}  \\
    \rulename{split-choice} & \ensuremath{\textbf{split}(\Varid{v}\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}\Varid{e})\{\Varid{f},\Varid{g}\}} & \movesto & \ensuremath{\Varid{g}\langle\Varid{v},\lambda \langle\rangle.\,\Varid{e}\rangle}  \\
\end{array}\]
The intuition behind \ensuremath{\textbf{split}} is that it distinguishes a failing computation from one that returns at least one value.
If \ensuremath{\Varid{e}} fails, it calls \ensuremath{\Varid{f}}; but if \ensuremath{\Varid{e}} returns at least one value, it passes that to \ensuremath{\Varid{g}} together with the remaining computation, safely tucked away within a lambda.
When adding more effects to \versecalc{} (see \cref{sec:store}), it is in fact crucial to use $\ensuremath{\textbf{split}}$ to exactly control the order of effects.

Indeed, this is more general, as we can implement \ensuremath{\textbf{one}} and \ensuremath{\textbf{all}} with \ensuremath{\textbf{split}}:
\[\begin{array}{r@{\;}l@{\;}l@{\;}l}
    \ensuremath{\textbf{one}\{\Varid{e}\}} & \equiv \ensuremath{\Varid{f}(\Varid{x})\hspace{-0.14em}:=\hspace{-0.14em}\textbf{fail};\,} & \ensuremath{\Varid{g}\langle\Varid{x},\Varid{y}\rangle\hspace{-0.14em}:=\hspace{-0.14em}\Varid{x};\,} & \ensuremath{\textbf{split}(\Varid{e})\{\Varid{f},\Varid{g}\}} \\
    \ensuremath{\textbf{all}\{\Varid{e}\}} & \equiv \ensuremath{\Varid{f}(\Varid{x})\hspace{-0.14em}:=\hspace{-0.14em}\langle\rangle;\,}    & \ensuremath{\Varid{g}\langle\Varid{x},\Varid{y}\rangle\hspace{-0.14em}:=\hspace{-0.14em}\Varid{cons}\langle\Varid{x},\textbf{split}(\Varid{y}\langle\rangle)\{\Varid{f},\Varid{g}\}\rangle;\,} & \ensuremath{\textbf{split}(\Varid{e})\{\Varid{f},\Varid{g}\}}
\end{array}\]
For this paper, we stuck to the arguably simpler \ensuremath{\textbf{one}} and \ensuremath{\textbf{all}},
to avoid confusing the presentation with these higher-order encodings, but there are no complications using \ensuremath{\textbf{split}} instead.

\section{\texorpdfstring{\versecalc{}}{The Verse calculus} in context: reflections and related work} \label{sec:related}

Functional logic programming has a rich literature; excellent
starting points are the CACM review article by \citet{antoy-hanus:survey} and
the longer survey by~\citet{hanus:survey}.  Now that we
know what \versecalc{} is, we can identify its distinctive features,
and compare them to other approaches.

\subsection{Choice and non-determinism} \label{sec:deterministic}

A significant difference between our presentation and earlier works is our
treatment of choice. Consider an expression like \ensuremath{(\mathrm{3}\mathbin{+}(\mathrm{20}\;\rule[-0.1em]{0.15em}{0.75em}\;\mathrm{30}))}.
Choice is typically handled by a pair of non-deterministic rewrite rules:
\\
\centerline{$\ensuremath{\Varid{e}_{\mathrm{1}}\;\rule[-0.1em]{0.15em}{0.75em}\;\Varid{e}_{\mathrm{2}}} \movesto \ensuremath{\Varid{e}_{\mathrm{1}}} \hspace{1cm} \ensuremath{\Varid{e}_{\mathrm{1}}\;\rule[-0.1em]{0.15em}{0.75em}\;\Varid{e}_{\mathrm{2}}} \movesto \ensuremath{\Varid{e}_{\mathrm{2}}}$}
\\[0.3em]
So our expression rewrites (non-deterministically) to either \ensuremath{(\mathrm{3}\mathbin{+}\mathrm{20})} or \ensuremath{(\mathrm{3}\mathbin{+}\mathrm{30})},
and that in turn allows the addition to make progress.
Of course, including non-deterministic choice means the rules are non-confluent by
construction.  Instead, one must generalize to say that a reduction does not change the
\emph{set} of results; in the context of lambda calculi, see
for example \citet{schmidt-schauss:non-det1,schmidt-schauss:non-det2}.

In contrast, our rules never pick one side or the other of a choice.
And yet, \ensuremath{(\mathrm{3}\mathbin{+}(\mathrm{20}\;\rule[-0.1em]{0.15em}{0.75em}\;\mathrm{30}))} can still make progress by floating out the
choice (rule \rulename{choose} in \cref{fig:rules}), thus
\ensuremath{(\mathrm{3}\mathbin{+}\mathrm{20})\;\rule[-0.1em]{0.15em}{0.75em}\;(\mathrm{3}\mathbin{+}\mathrm{30})}.
In effect, \emph{choices are laid out in space} (in
the syntax of the term), rather than being explored by
non-deterministic selection.  Rule \rulename{choose} is not a new idea:
it is common in calculi with choice,
%%see {\eg}, ~\citetext{\citealp[Section 6.1]{liguoro:non-det}, \citealp[Section 3]{lago:probabilistic}}
see {\eg}, ~\citet[Section 6.1]{liguoro:non-det} and \citet[Section 3]{lago:probabilistic},
and, more recently, has been used to describe functional logic languages, where it is variously
called \emph{bubbling} \cite{antoy:bubbling} or \emph{pull-tabbing} \cite{antoy:pull-tabbing}.
However, laziness imposes significant additional complications, including the need to attach
an identifier to each choice.

A recent paper by Barenbaum \emph{et al} describes a functional logic calculus that, like \versecalc{},
is just an extension of lambda calculus \cite{barenbaum:relational}.
They deal with lambda, existentials, choice, and equations
in a similar way to \versecalc{}, but lack the \ensuremath{\textbf{all}} and \ensuremath{\textbf{one}} operators which are central to \versecalc{}.
As a result they can always float choice up to the top level of the whole program, and
indeed they do so in the very structure of their terms.

\subsection{One and all}

Logical variables, choice, and equalities are present in many functional logic
languages.  However, \ensuremath{\textbf{one}} and \ensuremath{\textbf{all}} are distinctive features of \versecalc{},
with the notable exception of Fresh, a very interesting design introduced in a
technical report nearly 40 years ago~\cite{smolka:fresh} that also aims
to unify functional and logical constructs.
%
Fresh reifies choice into data via \emph{confinement} (corresponding to \ensuremath{\textbf{one}})
and \emph{collection} (corresponding to \ensuremath{\textbf{all}}).
%
However, Fresh differs from \versecalc{} in crucial ways.
%
First, it solves equations in a strictly left-to-right fashion, which means that
it is not lenient in the sense discussed in \cref{sec:lenient}.
%
Second, its semantics are presented in an operational fashion with explicit stacks and heaps,
in contrast to our focus on developing an equational account of functional logic programming.
%
Finally, Fresh appears not to have been implemented.
%
% It is a very interesting design, but one does not appear to have been implemented,

Several aspects of \ensuremath{\textbf{all}} and \ensuremath{\textbf{one}} are worth noting.
First, \ensuremath{\textbf{all}} \emph{reifies} choice (a control operator) into a tuple (a data structure);
for example, \ensuremath{\textbf{all}\{\mathrm{1}\;\rule[-0.1em]{0.15em}{0.75em}\;\mathrm{7}\;\rule[-0.1em]{0.15em}{0.75em}\;\mathrm{2}\}} returns the tuple \ensuremath{\langle{}\mathrm{1},\mathrm{7},\mathrm{2}\rangle}.
In the other direction, indexing turns a tuple into choice (for example, \ensuremath{\exists\Varid{i}.\,\langle{}\mathrm{1},\mathrm{7},\mathrm{2}\rangle(\Varid{i})} yields \ensuremath{(\mathrm{1}\;\rule[-0.1em]{0.15em}{0.75em}\;\mathrm{7}\;\rule[-0.1em]{0.15em}{0.75em}\;\mathrm{2})}).
Other languages can reify choices into a (non-deterministic)
list, via an operator called \text{\ttfamily bagof}, or a mechanism called
\emph{set-functions} in an extension of Curry \cite[Section 4.2.7]{curry:tutorial},
implemented in the Kiel Curry System interpreter \cite{brassel:encapsulated,kics,hanus:set-functions}.
But in Curry, this is regarded as a somewhat sophisticated feature,
whereas it is part of the foundational fabric of \versecalc{}.
Curry's set-functions need careful explanation about sharing across non-deterministic choices\footnote{
Again, this complexity in Curry
appears to be a consequence of laziness.},
or what is ``inside'' and ``outside'' the set function, something that appears as a straightforward
consequence of \versecalc{}'s single rule \rulename{choose}.

Second, even under the reification of \ensuremath{\textbf{all}},
\versecalc{} is \emph{deterministic}.
\versecalc{} takes pains to maintain order, so that
when reifying choice into a tuple, the order of elements in that
tuple is completely determined.
This determinism has a price: we have to take care to maintain the
left-to-right order of choices (see \cref{sec:mixing} and \cref{sec:choice-rules}, for example).
However, maintaining that order has other payoffs. For example, it is relatively easy to add
effects other than choice, including mutable variables and input/output, to \versecalc{}.
To substantiate this claim, \cref{sec:store} gives the additional syntax and rewrite rules
for mutable variables.

Thirdly, \ensuremath{\textbf{one}} allows us to reify failure; to try something and take different actions
depending on whether or not it succeeds. Prolog's ``cut'' operator has a similar flavor,
and Curry's set-functions allow one to do the same thing.

Finally, \ensuremath{\textbf{one}} and \ensuremath{\textbf{all}} neatly encapsulate the idea of ``flexible'' {\vs} ``rigid'' logical variables.
As we saw in \cref{sec:rigid}, logical variables bound outside \ensuremath{\textbf{one}}/\ensuremath{\textbf{all}} cannot be unified inside it;
they are ``rigid.''  This notion is nicely captured by the fact that equalities
cannot float outside \ensuremath{\textbf{one}} and \ensuremath{\textbf{all}} (\cref{sec:normalization}).

\subsection{The semantics of logical variables} \label{sec:magical}

Our logical variables, introduced by \ensuremath{\exists{}}, are often called
\emph{extra variables} in the literature, because they
are typically introduced as variables that appear on the right-hand side of a
function definition, but are not bound on the left.  For example, in Curry we can write:
\begin{tabbing}\ttfamily
~~~~~~first~x~\char124{}~x~\char61{}\char58{}\char61{}~\char40{}a\char44{}b\char41{}~\char61{}~a~where~a\char44{}b~free
\end{tabbing}
Here, \text{\ttfamily a} and \text{\ttfamily b} are logical variables, not bound on the left; they
get their values through unification (written ``\text{\ttfamily \char61{}\char58{}\char61{}}'').   In Curry, they are explicitly introduced
by the ``\text{\ttfamily where~a\char44{}b~free}'' clause, while in many other papers their
introduction is implicit in the top-level rules, simply by not being bound
on the left.  These extra variables (our logical variables) are
at the heart of the ``logic'' part of functional logic programming.

Constructor-based ReWrite Logic (CRWL) \cite{crwl} is the brand leader for
high-level semantics for non-strict, non-deterministic functional logic languages.
%
CRWL is a ``big-step'' rewrite semantics that rewrites a term to a value in a single
step.
%
\Citet{fraguas:rewriting} make a powerful case for instead giving the semantics
of a functional logic language using ``small-step'' rewrite rules, more like those
of the lambda calculus, that successively rewrite the term, one step at
a time, until it reaches a normal form.
Their paper does exactly this, and proves equivalence to the
CRWL framework.  Their key insight (like us, inspired by
\citet{ariola:call-by-need}'s formalization of the call-by-need lambda calculus)
is to use \ensuremath{\mathbf{let}} to make sharing explicit.

However, both CRWL and L\'opez-Fraguas {\etal} are in some ways \emph{too} high level:
they require something we call \emph{magical rewriting}. A key rewrite rule is this:
$$
\begin{array}{l}
f( \theta(e_1),\ldots, \theta(e_n) ) \movesto \theta( rhs ) \\
\text{if}
\begin{array}[t]{l}
  (e_1, \ldots, e_n) \movesto rhs
   \; \text{is a top-level function binding, and} \\
\text{$\theta$ is a substitution mapping variables to closed values, s.t.
$dom(\theta) = \mathit{fvs}(e_1, \ldots, e_n, rhs)$}
\end{array}
\end{array}
$$
The substitution for the free variables of the left-hand-side can
readily be chosen by matching the left-hand-side against the call.
But the substitution for the extra variables must be chosen ``magically''
\cite[Section 7]{fraguas:rewriting} or clairvoyantly, so as to make the future execution
work out. This is admirably high-level because it hides everything
about unification, but it is not much help to a programmer trying to
understand a program, nor is it directly executable.
In a subsequent journal paper, they refine CRWL to avoid magical rewriting using
``let-narrowing'' \cite[Section 6]{fraguas:let-narrowing}; this system
looks rather different to ours, especially in its treatment of choice,
but is rather close in spirit.

To explain actual execution, the state of the art is described by
\citet{hanus:semantics}.  They give both a big-step operational
semantics (in the style of \citet{launchbury:natural}), and
a small-step operational semantics.  These two approaches both thread a
\emph{heap} through the execution, which holds the unification variables and their unification
state; the small-step semantics also has a \emph{stack}, to specify
the focus of execution.
%
The trouble is that heaps and stacks are difficult to explain to a programmer,
and do not make it easy to reason about program equivalence.
%
In addition to this machinery, the model is further complicated with
concurrency to account for residuation.

In contrast, our rewrite rules give a complete, executable (\ie, no ``magic'')
account of logical variables and choice, directly as small-step
rewrites on the original program, rather than as the evolution of a
(heap, control, stack) configuration. Moreover, we have no problem with
residuation.

The Escher language \cite{Escher} extends a Haskell-like functional base with
logical (existentially quantified) variables and constraints to yield an integration
of functional and logical constructs, and informally describes a rewriting based
evaluation mechanism. However, the language does not have the equivalent
of \ensuremath{\textbf{one}} and \ensuremath{\textbf{all}} which reify choice as data, the rules are not precisely
formalized and no confluence result is established for them.

Goffin \cite{Goffin} also presents a way to extend a higher-order functional
language with existentially quantified logical variables. However, Goffin is
rather different from \versecalc{} in that it falls in the tradition of
Concurrent Constraint Programming \cite{Saraswat90} where the constraints
over logical variables are used to declare data dependencies as a means to
co-ordinate efficient concurrent execution of sub-computations, rather
than in logic programming (and \versecalc) where we seek to declare the properties
of solutions. Consequently, in Goffin, the logical variables are never ``unified''
but instead are ``updated'' as values are computed in parallel yielding a completely
different model of computation.

\subsection{Flat {\vs} higher order}

When giving the semantics
of functional logic languages, a first-order presentation is almost universal.
User-defined functions can be defined at top level only,
and function symbols (the names of such functions) are syntactically
distinguished from ordinary variables.
% As Hanus describes, it is possible
% to translate a higher-order program into a first-order
% form\footnote{
%   Hanus does not mention this, but for a language with arbitrarily nested lambdas
%   one would need to do lambda-lifting as well, but that is perhaps a minor point.}
% using defunctionalization~\cite[Section 3.3]{hanus:survey}, and a built-in
% |apply| function.
As Hanus describes, it is possible
to translate a higher-order program into a first-order
form using defunctionalization~\cite[Section 3.3]{hanus:survey} and a built-in
\ensuremath{\textbf{apply}} function.
(Hanus does not mention this, but for a language with arbitrarily nested lambdas,
one would need to do lambda-lifting \cite{johnsson:lambdalift} as well; this is perhaps a minor point.)
Sadly, this encoding is hardly a natural rendition of the lambda calculus, and
it obstructs the goal of using rewrite rules to explain to programmers how
their program might execute. In contrast, a strength of our \versecalc{}
presentation is that it deals natively with the full lambda calculus.

\subsection{Intermediate language}

Hanus's \emph{Flat Language}~\cite[Fig 1]{hanus:semantics}, FLC, plays the same role as \versecalc{}:
it is a small core language
into which a larger surface language can be desugared.
There are some common features: variables, literals, constructor applications,
and sequencing (written \text{\ttfamily hnf} in FLC).
However, it seems that \versecalc{} has a greater economy of concepts.
In particular, FLC
has two forms of equality \text{\ttfamily \char40{}\char61{}\char61{}\char41{}} and \text{\ttfamily \char40{}\char61{}\char58{}\char61{}\char41{}}, and two forms
of case-expression, \text{\ttfamily case} and \text{\ttfamily fcase}.  In each pair, the former suspends
if it encounters a logical variable; the latter unifies or narrows
respectively. In contrast, \versecalc{}
has a single equality \ensuremath{(\mathrel{=})}, and the orthogonal \ensuremath{\textbf{one}} construct, to deal with all four concepts.

FLC has let-expressions \text{\ttfamily \char40{}let~x\char61{}e~in~b\char41{}} where \versecalc{} uses
\ensuremath{\exists{}} and (again) unification.
FLC also uses the same construct for a different purpose, to
bring a logical variable into scope, using the strange binding \text{\ttfamily x\char61{}x}, thus \text{\ttfamily \char40{}let~x\char61{}x~in~e\char41{}}.
In contrast, \ensuremath{\exists\Varid{x}.\,\Varid{e}} seems more direct.

\subsection{Comparison with Icon} \label{sec:icon}

There are many obvious similarities between Verse and the Icon programming language
\cite{griswold:icon-users-manual,griswold:generators-in-icon,griswold:icon-book-1983,griswold:icon-overview,griswold:icon-book-2002}:
\begin{itemize}
\item An expression can (successively) produce any number of values.
\item An expression that produces zero values is said to \emph{fail} \cite[\S 3.1]{griswold:generators-in-icon};
  an expression that produces at least one value is said to \emph{succeed}.
\item The expression \ensuremath{\Varid{e}_{\mathrm{1}}\;\rule[-0.1em]{0.15em}{0.75em}\;\Varid{e}_{\mathrm{2}}} produces all the values of \ensuremath{\Varid{e}_{\mathrm{1}}} followed by all the values of \ensuremath{\Varid{e}_{\mathrm{2}}}.
\item There is a way to turn an array (or tuple) $a$ into a sequence of produced values.
  In Icon, this is written \texttt{!\ensuremath{\Varid{a}}} \cite[\S 3]{griswold:icon-overview}; in Verse, \texttt{\ensuremath{\Varid{a}}?}; in \versecalc{}, \ensuremath{\exists\Varid{i}.\,\Varid{a}(\Varid{i})}.
\item Most ``scalar'' operations (such as addition and comparisons) run through
  all possible combinations of values of their operand expressions,
  using a specific left-to-right evaluation order and automatic chronological backtracking.
\item Success and failure are used in place of boolean values for control-structure purposes.
  Some operations, especially comparisons, can fail as part of their defined semantics.
  The expression \ensuremath{\textbf{if}\;\Varid{e}_{\mathrm{1}}\;\textbf{then}\;\Varid{e}_{\mathrm{2}}\;\textbf{else}\;\Varid{e}_{\mathrm{3}}} checks to see whether \ensuremath{\Varid{e}_{\mathrm{1}}} succeeds;
  it then produces the values of \ensuremath{\Varid{e}_{\mathrm{2}}} (if \ensuremath{\Varid{e}_{\mathrm{1}}} succeeded) \emph{or} produces the values of \ensuremath{\Varid{e}_{\mathrm{3}}} (if \ensuremath{\Varid{e}_{\mathrm{1}}} failed).
  If \ensuremath{\Varid{e}_{\mathrm{1}}} succeeds and then \ensuremath{\Varid{e}_{\mathrm{2}}} fails, backtracking does \emph{not} attempt to examine further values from \ensuremath{\Varid{e}_{\mathrm{1}}}.
\item The ``\ensuremath{\hspace{0.2em}\mathop{\rule[-0.1em]{0.15em}{0.75em}}\hspace{0.2em}}'' construct is idiomatically used as a \emph{logical or} operation \cite[\S 3]{griswold:icon-overview}.
\item There is a control structure that executes a specified expression once for every value produced by another expression.
  In Icon, this is \texttt{every}~\ensuremath{\Varid{e}_{\mathrm{1}}} \texttt{do}~\ensuremath{\Varid{e}_{\mathrm{2}}} and in Verse, it is written \texttt{for(}\ensuremath{\Varid{e}_{\mathrm{1}}}\texttt{)} \texttt{do}~\ensuremath{\Varid{e}_{\mathrm{2}}};
%%  in each, the ``\texttt{do}~|e_2|'' may be omitted to simply evaluate |e_1| repeatedly until it has yielded all its values.
%%  (In Icon, \texttt{every} |e| may also be written as \texttt{repeat} |e|.)
\item It is impossible to name a generator (Icon) or choice (Verse); if \ensuremath{\Varid{e}} produces multiple values,
  \texttt{\ensuremath{\Varid{x}}~:=~\ensuremath{\Varid{e}}} will provide one value at a time from \ensuremath{\Varid{e}} to be named by variable \ensuremath{\Varid{x}}.
\end{itemize}

But there are also major differences between Verse and Icon.
Icon was designed primarily to use expressions as generators to automatically
explore a combinatorial space of possibilities (``goal-directed evaluation''), and secondarily to use
success/failure rather than booleans to drive control structure.
But in other respects, \emph{Icon is a fairly conventional imperative language}, relying
on side effects (assignments) to process the generated combinations.
The designers judged that the interactions of such side effects with completely
unrestrained control backtracking would be difficult for programmers to
understand \cite[\S 3.1]{griswold:generators-in-icon}; therefore, the design of Icon emphasizes limited scopes for control
backtracking and tools for controlling the backtracking process \cite[\S 3.3]{griswold:generators-in-icon}.

In contrast, \emph{Verse is a declarative language} and
avoids these difficulties by using a functional logic approach
rather than an imperative approach to processing generated combinations:
\begin{itemize}
\item While Icon typically processes multiple values from an expression by using \emph{assignment},
  Verse typically processes multiple values by using \emph{equations}
  (which are then \emph{solved}).
\item
  Verse also has a concise way to turn a finite sequence of multiple
  values into an array directly.
  For example, to make variable \texttt{a} refer to an array containing all values generated by expression \texttt{e},
  code such as the following (using a \texttt{repeat} loop containing an assignment) is idiomatic in Icon \cite[\S 8]{griswold:icon-overview}:\\
\centerline{
  \texttt{a := array 0 string; i := 0; repeat a[i+] := e; close(a)}
  }
%%  (In Icon, \emph{postfix} \texttt{+} is an autoincrement operator \cite[\S 3.1.2]{griswold:icon-users-manual}
%%  with the same behavior as \emph{prefix} \texttt{++} in~C \cite[\S 2.8]{kr-c-book}.)
  In Verse,
%\texttt{a~=~for(x :== e) do~x} or even just
  \texttt{a~=~for\{e\}} does the job; in \versecalc{}, \ensuremath{\Varid{a}\hspace{-0.14em}=\hspace{-0.14em}\textbf{all}\{\Varid{e}\}} is all it takes.
\item Backtracking in Icon is ``only control backtracking''; side effects, such as assignments, are not undone \cite[\S 3.1]{griswold:generators-in-icon}.
\item Both languages have an implicit ``cut'' (permanent acceptance of the first produced value) after the predicate part of an \ensuremath{\textbf{if}}-\ensuremath{\textbf{then}}-\ensuremath{\textbf{else}},
  but Icon furthermore has an implicit cut at each statement end (semicolon or end of line) \cite[\S 3.1]{griswold:generators-in-icon}, each
  closing brace ``\texttt{\}}'', and most keywords \cite{icon-programming-corner-4}.
\end{itemize}



\section{Looking back, looking forward}

We believe that this is the first presentation of a functional logic
language as a deterministic rewrite system.  A rewrite system has the
advantage (compared to more denotational, or more operational,
methods) that it is is sufficiently low-level to capture the
\emph{computational model} of the language; and yet sufficiently
high-level to be \emph{illuminating} to a programmer or compiler
writer.  Our focus on rewriting as a way to define the
semantics has forced us to focus on confluence, a rather syntactic
property that is stronger (and hence more delicate and harder to prove) than the
contextual equivalence that we really need.  That in turn led us to
study the elegant and ingenious notion of skew confluence, which has been
barely revisited during the last 20 years, but which we believe deserves a wider
audience.

% Defining a language by giving its low-level semantics is precise but not necessarily illuminating;
% for example, giving a reference compiler that compiles
% the program to x86 instructions is precise, but is not helpful
% to a human who is trying to understand exactly what the original program
%  meant. Likewise, a high-level (\eg, denotational) semantics simply provides the eventual answer
% produced by the program without insight into the computational steps
% that got us from program start to program completion.

%%\simon{dropped a short para here that interrupted the flow}
% The moral here is that \emph{formal specifications can be obfuscatory---or
% illuminating.} The latter kind shed light because they are defined \emph{in
% terms of the intended mechanisms of the language}. \versecalc{} does this; it
% respects the conceptual structures of the Verse language.

% Note that when we say ``illuminating'' we mean that in multiple ways.
% A semantics can be illuminating for humans who are trying to understand
% what a particular program does, or how a proposed change to the language
% will affect the language. It can drive analyses that help the compiler
% optimize programs. It underlies the use of formal methods and verification
% to provide machine-derived and -checkable proofs of correctness. All of
% these applications depend on the semantics being defined at the appropriate
% level: the level of the computational model that underlies the language.
% This has been our goal in this work.

We have much left to do.  The full Verse language has
statically checked types.  In the dynamic semantics, the types can be
represented by partial identity functions---identity for the values of
the type, and \ensuremath{\textbf{fail}} otherwise.  This gives a distinctive new
perspective on type systems, one that we intend to develop in future
work.  The full Verse language also has a statically checked effect system,
including both mutable references and input/output.  All these effects
must be \emph{transactional}, {\eg}, when the condition of an \ensuremath{\textbf{if}}
fails, any store effects in the condition must be rolled back.
We have preliminary reduction rules for updateable references,
see \cref{sec:store}.
