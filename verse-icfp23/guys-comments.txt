Big-picture notes I wrote on the front page:

There is an asymmetry between the "one" and "all" constructs, in that "all" returns a tuple of all the values, but "one" returns just one value rather than a tuple of the one value. I wonder whether this is somehow related to an asymmetry between the syntax for a choice (expressed using an infix operator "|") and the syntax for a tuple (expressed with angle brackets around a list of comma-separated items). The tuple syntax makes it easy to distinguish between a value and a singleton tuple of that value, whereas the choice syntax has no way to express "a choice of one item" as distinct from the item itself. All this is of course a perennial problem in programming language design. From Simon's talk I gather that in Verse one uses parentheses (around a list of comma-separated items) rather than angle brackets to express tuples, and therefore, given that parentheses are already committed to their usual use of being able to surround any expression without changing the meaning, in Verse there is no natural way to express a singleton tuple using the parenthesis syntax. Some languages use "(e,)", which I dislike. Simon's talk (slide 21) indicates that there is a more verbose alternate syntax ("for { v }" in verse, or "all { v }" in VC) for creating a singleton tuple, and that strikes me as a reasonable design decision. (I am reminded of all the contortions required in APL to distinguish scalar values from single-element arrays of various ranks when necessary, and to gloss over those distinctions when convenient. The rules are not obvious.) And all this in turn is related to the property that one can nest tuples but one cannot nest choices; choice construction is associative but tuple construction is not. These are all perfectly sensible design choices, but perhaps worth communicating to the reader even more clearly and explicitly in a longer document.

This idea of explaining logic programming using a term-rewriting system seems like an excellent research goal that would be a distinct improvement over prior literature on logic programming.

BIG-PICTURE COMMENTS

The text of Section 3 does not say explicitly whether the expectation is that a VC program should be fully desugared before applying the rewrite rules, and the indirect evidence is contradictory: on the one hand, some of the examples (line 548, line 662, line 689) apply rewrite rules to VC expressions that have not been desugared; on the other hand, if the intention is to desugar "on the fly" or not at all, then the discussion of confluence in Section 4 should include the rewriting rules for desugaring as part of the overall set of rules expected to be confluent, and the proof of confluence should take those rules into account.

The organization of the figures and the organization of the text are mismatched big-time. Avoid chiasmus! Please organize Figures 3, 4, and 5 so that the sets of rules (both rewrite rules and Gentzen rules) appear in the order that they are discussed in the text. (The current order in the Figures is Contexts, Unification, Application, Speculation, Normalization, Fail Propagation, Garbage Collection, Structural rules, and Well-formedness; the current order in the text is Application, Unification, the first two Contexts, Normalization, Garbage Collection, Fail Propagation, Speculation, and the other two contexts—and the Structural Rules are not mentioned at all in the text.)

The same issue occurs with the discussion of each set of rules. For example, the rules for Application appear in Figure 3 in the order APP-BETA, APP-TUP0, APP-TUP, APP-ADD, APP-GT, and APP-GT-FAIL, but they are discussed in the text (in Section 3.1) in the order APP-ADD, APP-GT, APP-GT-FAIL, APP-BETA, and APP-TUP, and rule APP-TUP0 is not mentioned in the text.

With so many sets of rules and so many rules in each set, it is really important to proceed in one chosen order, mentioning every set of rules and every rule, so that the reader can more easily follow the textual description and have confidence that every item has been addressed. (You will note that, when I write up my detailed comments, even though every comment is labeled with a line number, I also take care to present the comments in ascending order of line number.)

[Now that I have seen the "POPL version" of the paper, I can see that these figures were reorganized. They may yet be reorganized again. The basic principle remains: all else being equal, the order of presentation of figures and within each figure should match the order of discussion in the text.]

line 547: This sentence uses the word "equality" rather than "equation" (and see also "heap equalities" in line 570); this raises a big-picture question about terminology. It is generally a good rule in technical writing that when you invest a word with a specific technical meaning, then you should stick to using that word, and only that word, for that technical meaning, and only that meaning, unless there is a very good reason to do otherwise. So one should avoid unnecessary "synonyms" on the one hand and overloading of technical terms on the other.

In this paper, I find that not only are the terms "equality" and "equation" used to refer to the same thing, but also the word "unification" (see line 569)—indeed, the BNF nonterminal "eu" seems to hint at the word "unification"—and the word "binding" (see "heap bindings" in line 572).

I recommend that the term "unification" be reserved to refer to the actual process, or an instance of the process, of unifying two terms. This is an process or action that may be spurred by the processing of an equation. I recommend that the syntax form "x = e" always be referred to as an "equation", not an "equality" or "unification".
